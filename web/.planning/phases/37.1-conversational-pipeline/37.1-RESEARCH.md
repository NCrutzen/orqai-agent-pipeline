## Summary

The pipeline is currently a pure batch job: each stage calls `runPromptAdapter()` (non-streaming Orq.ai), stores results in Supabase, and broadcasts status updates via Supabase Broadcast. There is no user-facing conversational layer — it fires and the user watches status badges change.

The transformation requires four coordinated changes: (1) a new discussion stage prepended to the pipeline that loops with `step.waitForEvent()` for each conversation turn, (2) two streaming narrator "interjections" after architect and spec-generator that call the Orq.ai router with `stream: true` and fan tokens out via Supabase Broadcast, (3) a `pipeline_chat_messages` table to persist the conversation so it survives page reload, and (4) a reworked right panel that shows a chat thread with a text input instead of only status badges.

The critical constraint discovered in research: streaming cannot happen inside `step.run()` because Inngest serializes step return values and memoizes them — the step must complete atomically. The workaround is to run streaming Orq.ai calls **outside** `step.run()`, directly in the async function body between steps. This is the same pattern the existing `step.waitForEvent()` calls use. Tokens are pushed to the browser via Supabase Broadcast (the infrastructure already works and has 100 msg/sec free / 500 msg/sec pro headroom — well above typical token rates of 20-60 tokens/sec). Inngest Realtime (`@inngest/realtime`) exists but is a developer preview with at-most-once delivery and would add a new dependency; Supabase Broadcast is already proven in this codebase.

**Primary recommendation:** Streaming Orq.ai calls live outside `step.run()`, directly in the Inngest function body between steps. Fan tokens to the browser via existing Supabase Broadcast on `run:{runId}` with a new `chat-token` event. Store completed messages in a new `pipeline_chat_messages` table. The discussion loop uses numbered step IDs (`discussion-ask-0`, `discussion-wait-0`, `discussion-ask-1`, `discussion-wait-1`, ...) with a turn counter tracked in function scope.

### Pattern 1: Streaming Orq.ai Call Outside step.run()

**What:** Call the Orq.ai router with `stream: true` in the Inngest function body between two `step.run()` calls. Consume the ReadableStream line-by-line, parsing SSE chunks and broadcasting each token to the client.

**When to use:** Architect summary narration and spec-generator summary narration — after the stage completes (output available) but before the waitForEvent pause.

**Example:**
```typescript
// Source: adapter.ts pattern + Orq.ai streaming docs (https://docs.orq.ai/docs/ai-gateway-streaming)
// Called OUTSIDE step.run() — between steps, directly in the async function body
async function streamNarrator(
  runId: string,
  systemPrompt: string,
  userMessage: string,
  messageId: string
): Promise<string> {
  const apiKey = process.env.ORQ_API_KEY!;

  const response = await fetch("https://api.orq.ai/v2/router/chat/completions", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${apiKey}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: "anthropic/claude-sonnet-4-6",
      max_tokens: 2048,
      stream: true,
      messages: [
        { role: "system", content: systemPrompt },
        { role: "user", content: userMessage },
      ],
    }),
  });

  if (!response.ok || !response.body) {
    throw new Error(`Orq.ai stream error: ${response.status}`);
  }

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let fullText = "";
  let buffer = "";

  // Broadcast typing indicator first
  await broadcastChatToken(runId, { messageId, token: "", isStart: true });

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });

    // SSE lines are separated by \n\n; each line is "data: {...}" or "data: [DONE]"
    const lines = buffer.split("\n");
    buffer = lines.pop() ?? "";

    for (const line of lines) {
      if (!line.startsWith("data: ")) continue;
      const payload = line.slice(6).trim();
      if (payload === "[DONE]") break;
      try {
        const chunk = JSON.parse(payload);
        const token = chunk.choices?.[0]?.delta?.content ?? "";
        if (token) {
          fullText += token;
          await broadcastChatToken(runId, { messageId, token });
        }
      } catch {
        // skip malformed chunk
      }
    }
  }

  // Broadcast stream end
  await broadcastChatToken(runId, { messageId, token: "", isDone: true });
  return fullText;
}
```

### Pattern 2: Discussion Loop with Numbered waitForEvent

**What:** The discussion stage is a `while` loop (max N turns) inside the Inngest function. Each turn has a unique step ID using a turn counter. The loop breaks when the AI response includes a sentinel (`<discussion_complete>`) indicating it has enough context.

**When to use:** The new discussion stage, prepended before the architect stage.

**Example:**
```typescript
// Source: Inngest docs (https://www.inngest.com/docs/guides/multi-step-functions)
// Inngest memoizes by step ID. Numbered IDs make each turn independently replayable.
let conversationMessages: Array<{role: string; content: string}> = [];
let enrichedUseCase = useCase;

for (let turn = 0; turn < 5; turn++) {
  // Step: AI generates next question (or declares done)
  const aiTurn = await step.run(`discussion-ask-${turn}`, async () => {
    const response = await runDiscussionTurn(conversationMessages, useCase);
    return { text: response, done: response.includes("<discussion_complete>") };
  });

  // Store AI message + broadcast to chat UI
  conversationMessages.push({ role: "assistant", content: aiTurn.text });
  await saveChatMessage(runId, "assistant", aiTurn.text);
  await broadcastChatMessage(runId, { role: "assistant", content: aiTurn.text });

  if (aiTurn.done) break; // AI satisfied with context

  // Mark run as waiting for user input
  await step.run(`discussion-waiting-${turn}`, async () => {
    await broadcastRunUpdate(runId, { runId, status: "waiting", stepsCompleted: 0 });
  });

  // Wait for user response (reuses existing pipeline/review.responded event)
  const userEvent = await step.waitForEvent(`discussion-wait-${turn}`, {
    event: "pipeline/discussion.responded",
    timeout: "1h",
    if: `async.data.runId == "${runId}"`,
  });

  if (!userEvent) break; // Timeout — proceed with what we have

  conversationMessages.push({ role: "user", content: userEvent.data.message });
  enrichedUseCase = `${useCase}\n\nUser clarifications:\n${conversationMessages
    .filter(m => m.role === "user")
    .map(m => m.content)
    .join("\n")}`;
}
```

### Pattern 3: Client-Side Token Accumulation

**What:** The client subscribes to `chat-token` events on the existing `run:{runId}` channel. Tokens are appended to a `useRef` buffer and flushed into state via `requestAnimationFrame` to avoid per-token re-renders.

**When to use:** The new ChatPanel and ChatMessage components for rendering streaming AI messages.

**Example:**
```typescript
// Source: sitepoint.com streaming re-render pattern (2025)
// Accumulate tokens in ref, flush at animation frame rate
const tokenBufferRef = useRef<string>("");
const rafRef = useRef<number | null>(null);

useBroadcast<ChatTokenPayload>(`run:${runId}`, "chat-token", (payload) => {
  if (payload.isStart) {
    // New message starting — add placeholder entry
    setMessages(prev => [...prev, { id: payload.messageId, role: "assistant", content: "", streaming: true }]);
    tokenBufferRef.current = "";
    return;
  }
  if (payload.isDone) {
    // Flush final state, mark not streaming
    const finalContent = tokenBufferRef.current;
    setMessages(prev =>
      prev.map(m => m.id === payload.messageId
        ? { ...m, content: finalContent, streaming: false }
        : m
      )
    );
    if (rafRef.current) cancelAnimationFrame(rafRef.current);
    return;
  }

  tokenBufferRef.current += payload.token;

  // Schedule flush — collapses many tokens into one setState per frame
  if (!rafRef.current) {
    rafRef.current = requestAnimationFrame(() => {
      const accumulated = tokenBufferRef.current;
      setMessages(prev =>
        prev.map(m => m.id === payload.messageId
          ? { ...m, content: accumulated }
          : m
        )
      );
      rafRef.current = null;
    });
  }
});
```

### Anti-Patterns to Avoid

- **Streaming inside `step.run()`:** `step.run()` must return a serializable value atomically. Reading a ReadableStream inside it will either hang or return before streaming completes. All streaming Orq.ai calls must be in the function body, not inside a step.
- **Per-token `setState` calls:** Calling React `setState` for every token (~30-60/sec) causes excessive re-renders. Use the `requestAnimationFrame` buffer pattern above.
- **Per-token Supabase channel open/close:** The existing `broadcastStepUpdate` opens a channel, sends, then removes it. At 30-60 tokens/sec this would create thousands of channels. Build a dedicated `broadcastChatToken` function that keeps the channel open for the duration of a stream, then closes it when `isDone: true`.
- **Same waitForEvent ID across turns:** Each discussion turn must use a unique step ID (`discussion-wait-0`, `discussion-wait-1`). Reusing the same ID will cause Inngest memoization to return the stale first response for all subsequent turns.

## Common Pitfalls

### Pitfall 1: Streaming Call Memoization
**What goes wrong:** If a streaming call is accidentally placed inside `step.run()`, on Inngest function re-execution (common during retries), Inngest will skip it (memoized) — the client sees no tokens on resume.
**Why it happens:** Developers naturally reach for `step.run()` to wrap any async call.
**How to avoid:** Streaming calls go directly in the function body between steps, not wrapped in `step.run()`. Before streaming, call a `step.run()` to save the "about to stream" state in DB so resume can detect partial sends.
**Warning signs:** Streaming works first time; user sees no tokens after a retry.

### Pitfall 2: Broadcast Channel Per Token
**What goes wrong:** Reusing the existing `broadcastStepUpdate()` pattern (which opens a channel, sends, and removes) at token frequency creates ~3,000 channel operations per 60-second stream.
**Why it happens:** `broadcastStepUpdate` is the only broadcast pattern in the codebase.
**How to avoid:** Build a `ChatBroadcaster` class (or function with persistent channel) that opens once per stream, sends all tokens, then closes on `isDone`. The admin Supabase client supports this.
**Warning signs:** High Supabase connection counts; tokens arriving out of order.

### Pitfall 3: Discussion Event Race (waitForEvent timing)
**What goes wrong:** If the user submits their discussion response before `step.waitForEvent()` has registered (possible if the broadcast round-trip is very fast), the event is missed and the turn times out.
**Why it happens:** Inngest only catches events sent **after** waitForEvent is called. The broadcast "UI is ready for input" must happen before the user can type.
**How to avoid:** The `discussion-waiting-N` step (which sets run status to "waiting" and broadcasts the waiting state) must complete and return **before** `step.waitForEvent()` is called. This is the same dual-write pattern used in the existing approval gate. Never broadcast "waiting" and waitForEvent in the same step.run().
**Warning signs:** Discussion appears to hang; Inngest logs show waitForEvent timing out immediately.

### Pitfall 4: Discussion Turn ID Reuse
**What goes wrong:** Using a fixed step ID like `"discussion-ask"` for all turns causes Inngest to return the memoized result from turn 0 for every subsequent turn.
**Why it happens:** Inngest step memoization is keyed by step ID. In a loop, each iteration needs a distinct ID.
**How to avoid:** Include the turn index in every step ID: `discussion-ask-${turn}`, `discussion-wait-${turn}`, `discussion-waiting-${turn}`. Inngest increments an internal counter for same-ID steps in loops, but explicit numbering is clearer and more debuggable.
**Warning signs:** AI always asks the same question regardless of user input.

### Pitfall 5: Chat Messages Not Visible on Page Refresh
**What goes wrong:** Streaming tokens and chat messages only exist in React state; after a page reload the conversation is gone.
**Why it happens:** Broadcast events are ephemeral — they are not stored in the DB by default.
**How to avoid:** After each complete AI message (when `isDone: true`), write the full message to `pipeline_chat_messages`. On page load, the server fetches all messages for the run and hydrates the initial state. The `pipeline_runs` query in the run detail page must be extended to also fetch chat messages.
**Warning signs:** Conversation disappears on refresh; user cannot return to a paused discussion.

## Database Schema: pipeline_chat_messages

```sql
-- Source: Supabase Postgres Best Practices skill (schema-design category)
CREATE TABLE pipeline_chat_messages (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  run_id       UUID REFERENCES pipeline_runs(id) ON DELETE CASCADE NOT NULL,
  role         TEXT NOT NULL CHECK (role IN ('assistant', 'user')),
  content      TEXT NOT NULL,
  stage_name   TEXT,          -- which pipeline stage this message belongs to (e.g. 'discussion', 'architect-summary')
  turn_index   INTEGER,       -- conversation turn within a stage (NULL for narrator messages)
  created_at   TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_chat_messages_run_id ON pipeline_chat_messages(run_id);

-- RLS: same project-member pattern as pipeline_steps
ALTER TABLE pipeline_chat_messages ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Project members see chat messages" ON pipeline_chat_messages
  FOR SELECT USING (
    run_id IN (
      SELECT id FROM pipeline_runs
      WHERE project_id IN (
        SELECT project_id FROM project_members
        WHERE user_id = (SELECT auth.uid())
      )
    )
  );
-- INSERT is service-role only (Inngest admin client)
```

**Why new table (not extending pipeline_steps):** One pipeline step can produce many chat messages (N discussion turns). The 1:many relationship is clear in a dedicated table. Also, `pipeline_steps` is a fixed schema per stage; adding JSONB columns for chat would bloat non-conversation runs.

## Open Questions

1. **Discussion agent .md file location**
   - What we know: Stage prompt files are fetched from `https://raw.githubusercontent.com/NCrutzen/orqai-agent-pipeline/main/` at runtime
   - What's unclear: Does a `discussion.md` file already exist in the orqai-agent-pipeline repo? The CONTEXT.md references the discussion agent but doesn't confirm the file exists.

2. **Streaming persistence timing**
   - What we know: After stream completes, full message should be written to `pipeline_chat_messages`
   - What's unclear: If the Inngest function fails mid-stream (between streaming and the next `step.run()`), the message may be in Supabase Broadcast state but not yet persisted. On retry, the stream would re-run (since it's outside `step.run()`), creating a duplicate.
   - Recommendation: Before each streaming call, `step.run()` a "prepare-narrative-N" step that writes a `pending` chat message to DB with the messageId. After streaming, update it to `complete`. On retry, if `complete` message already exists, skip streaming and use stored content.

3. **Existing `needsReview` flag on architect stage**
   - What we know: `stages.ts` has `needsReview: true` on the architect stage, and `pipeline.ts` does NOT implement the review gate (the review event was wired separately or the flag is aspirational). The `TerminalReviewEntry` exists but the current pipeline.ts loop has no `needsReview` handler.
   - What's unclear: Is `needsReview` currently dead code, or is it used by another path?
   - Recommendation: Grep for `needsReview` usage in pipeline.ts before implementing — if it's dead code, the architect streaming narrator is the first real implementation of that gate.
