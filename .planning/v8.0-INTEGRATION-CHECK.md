## Cross-phase findings

### BLOCKERS

None.

### WARNINGS

1. **PipelineStageContext TS interface not codified (RFC-02 forward-ref unfulfilled).** The RFC (`docs/agentic-pipeline/context-shape-contract.md` line 56) forward-refs codification of `PipelineStageContext` in `web/lib/agentic-pipeline/types.ts` for "Phase 64 or 70 (whichever first needs the type at runtime)." The actual `types.ts` (19 lines) exports only `HandlerOutput` (Stage 4 output). The Stage 2→3 context fields (`customer_id`, `customer_name`, `language`, `entity_brand`, `recent_documents`, `context_version`) are passed between coordinator and orchestrator as **inline field plumbing** through `debtor-email-coordinator.ts` and `classifier-invoice-copy-handler.ts:383-389` (where `context_version: 1` is stamped). One handler (`classifier-invoice-copy-handler.ts`) imports neither a shared type nor matches the canonical field set verbatim — it composes the shape from scratch. Impact: a new swarm onboarding adds field-rename risk (no compile-time enforcement of the contract). Affected REQs: **RFC-02**.

2. **Phase 82.9 production migration + Zap edit deferred to operator.** Code wired; data flow degrades to `null`/`[]` fallbacks until the Zap projection adds `contact_person` + `recent_invoices` and migration `20260519_phase82_9_candidate_details_schema_extension.sql` applies to prod Supabase. The Zod schema absorbs the empty shape — no runtime error, just empty evidence in the audit panel for sender_match / identifier_match / llm_tiebreaker arms. Affected REQs: **TAG-03** (audit surface richness).

3. **Phase 83 V3/V4 verification PARTIAL pending live traffic.** Not a wiring break — the column writers and code paths exist (verified at `debtor-email-coordinator.ts:247,302`); the 24h window simply contained pre-hotfix rows or zero Stage 0 traffic. The legacy `email_pipeline.emails.body_text` column should still be dropped per the v8.2 backlog reminder in `83-07-VERIFICATION.md` §6.

### Anti-patterns observed

- **Default `?? "debtor-email"` fallbacks proliferate** in cross-swarm handlers (`classifier-invoice-copy-handler.ts` 6 sites, `classifier-label-resolver.ts:211`). They never gate dispatch (dispatch goes via `swarm_intents`), but they do silently bias telemetry toward the debtor swarm if a future caller forgets to thread `swarm_type` through the event payload. Defensible per Phase 65 input-validation rule (`stage-0-safety-worker.ts:121` and `classifier-screen-worker.ts:156` already hard-fail when `swarm_type` is missing on entry), but the inner-handler fallback is belt-and-braces in a way that masks bugs rather than surfacing them. Consider promoting to required-field assertions in v8.1.

- **`agent_runs.predictor` is forward-only** (D-09 decision in Phase 999.8). Historical rows stay NULL; `labeling-flip-cron` filters `predictor IS NOT NULL` to exclude them. Intentional. Cold-start window per `(mailbox, predictor)` until N≥50.

- **`classifier/screen.requires_review` and `classifier/calibration_drift.detected` declared with no subscriber** by design (Phase 999.8 D-10). The audit row in `classifier_rule_evaluations` is the source of truth; future learning-loop work (V9.0) picks them up. Verified in `lib/inngest/events.ts:336,387` and `:193`.
