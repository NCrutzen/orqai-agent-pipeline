Variant A — Column-board (literal Kanban)

Two columns within the Stage 3 tab: No handler and Low confidence. Each row becomes a card with subject + intent-pill + entity + age + confidence (when applicable). Drag-and-drop is technically implementable but conceptually wrong here — operators don't move rows between reasons; the system already classified them.

Counts visually obvious. Cards have room for richer per-row info (intent + confidence + age all visible without truncation). Consistent with V7 KanbanBoard mental model elsewhere in the app (familiar pattern for engineers).
Drag-and-drop is dead weight — operators reclassify intent via the detail-pane action, not by dragging columns. Two columns waste horizontal space when there are 4-7 rows. With 100+ rows, columns become tall scrolling lists no denser than a table. Mixing columns with the row-strip pattern from Bulk Review (Stage 1 tab) creates inconsistency across stage tabs.

Variant B — Filtered table + detail pane (★ candidate)

Reuses the Bulk Review pattern almost verbatim: dense table on the left, detail pane on the right. Chip strip filters by reason (All / No handler / Low confidence), with optional sort and bulk-select later. Each row carries a reason pill + intent pill + subject + (for low-conf) confidence bar + recipient + age. Selected row populates the detail pane with Stage 3 ranked output, body preview, and the three operator actions. Stage 4 tab inherits the same shape with reason='handler_error' and an "error_detail" expanding section in the pane.

Visual + interaction consistency with the Stage 1 (Bulk Review) tab — operators learn one row pattern. Density: 30+ rows visible without scroll on a typical laptop. Detail pane fits the Replay editor (intent dropdown), Reclassify dropdown, and Stage 3 ranked output without modal-juggling. Keyboard shortcuts (⏎ approve / N reclassify / Space close) match the existing Bulk Review keyboard cheat-sheet. Stage 4 tab can reuse the entire shape — only the reason values + detail-pane fields change.
"All" filter merges no_handler + low_confidence into one list — fine for triage but slightly less visually grouped than Variant A. Confidence bar adds visual noise on rows where it doesn't apply (no_handler rows have no meaningful confidence). Solution: only render the bar for low-conf rows (already implemented in this mock).

Variant C — Hybrid (chip filter + grouped sections)

Same chip strip as Variant B for global filtering. When "All" is active, rows render in collapsible sections grouped by reason. Section header shows count + a one-sentence explainer. When a chip filters to one reason, sections collapse to just that one. No detail pane in this mock (would slot in identically to B).

Visual grouping by reason is explicit — operator immediately sees how many rows per reason without reading individual pills. Section headers double as documentation ("intent picked, no Stage 4 worker registered yet" — explains the reason inline). Empty sections collapse cleanly when one reason is empty.
Chrome-heavy: every section adds ~40px of header height. When all reasons have rows, the page has 2-3 section headers eating viewport space. Doesn't share row-pattern parity with Stage 1 (Bulk Review) tab — Stage 1 is a flat list, Stage 3 would be grouped — small inconsistency. The "explainer text" inside section headers is nice for first-time operators but becomes visual noise for daily users (could collapse to a tooltip?).
