Q: Scope future stages + Approve semantics, with full surface stack visible (Stage 1 tab)

A — Keep Approve, relabel

Full surface stack preserved. Stage 0+1 expanded with audit panels and feedback panels visible. Stages 2-4 collapsed into a single pill ("Stages 2-4 haven't run yet · expand to override pre-emptively").

Per-stage Save/Confirm stays tight to the textarea inside each stage's SHOW DETAILS expander. The bottom Approve stays as a row-level shortcut but with clearer microcopy: "Approve verdicts that ran (Stages 0+1)". It batch-confirms what's been actioned, ignoring future stages.

  • Operator who wants nuance → uses per-stage chips inside expanders.
  • Operator clearing fast → bottom Approve one-shot.
  • Future stages → out of sight, out of mind unless explicitly expanded.
Trade-off

Two paths to the same write (per-stage Confirm chip AND bottom Approve). Could be redundant; could be a feature.

B — Drop bottom Approve

Same scope as A: full pipeline visible, future stages collapsed. The bottom Approve disappears entirely. Per-stage Confirm chips (inside each expander) are the only verdict writers. Bottom footer becomes pure queue-navigation: Reject row + Skip.

  • One way to write a verdict — no redundancy.
  • Bottom bar verbs are unambiguous (Reject, Skip).
  • To clear a row, operator clicks Confirm on each open expander, OR just skips to next row leaving stages as-is.
Trade-off

Loses the "approve everything in one click" affordance. Power-user case slows down if they're confirming many in a row. But the operator who's training the corpus (the actual job) loses zero functionality.

C — Approve = batch-confirm visible

Per-stage Confirm chips stay (same as A and B). The bottom Approve becomes an explicit batch: "Approve all 2 unconfirmed verdicts". The number changes dynamically with how many stages still need a verdict.

  • Bottom button is honest about what it does.
  • Once you confirm a stage individually, the count drops.
  • If nothing's left to confirm, the button disappears entirely — only Skip / Reject remain.
Trade-off

The dynamic count creates a moving target for muscle memory. Veterans who scan the same button position may feel disoriented when it disappears.
