# Phase 82.1: Stage shell polish — Context

**Gathered:** 2026-05-11
**Status:** Ready for planning
**Trigger:** Live smoke after Phase 82 revealed 4 polish issues on `/stage-1`. Decisions locked inline via AskUserQuestion.

<domain>
## Phase Boundary

Four targeted fixes to the unified `_shell/` library shipped in Phase 82. Three are CSS / layout polish (1-file changes); one is a real port (move Stage 1's override picker logic from `stage-1-override-pane.tsx` into the shared `Stage1Widget` inside `_shell/detail-pane.tsx`).

**In scope:**
1. **Stage label double-prefix bug** — `_shell/detail-pane.tsx:85-89` defines `STAGE_TITLES` as `"Stage 0 — Safety"`, etc. `stage-1/components/stage-step.tsx:62` then renders `Stage {stage.n} — {stage.title}`, producing `"Stage 0 — Stage 0 — Safety"`. Drop the `"Stage N — "` prefix from `STAGE_TITLES` (canonical label values: `"Safety"`, `"Category"`, `"Customer"`, `"Intent"`, `"Handler"`).
2. **MailboxFilter row placement** — each stage's `client-shell.tsx` renders `<MailboxFilter>` in its own flex row above the row-list/detail-pane grid. Per Phase 82 D-11, MailboxFilter belongs on the right edge of the primary stage-specific chip strip. Restructure: render MailboxFilter inline with the chip strip (Stage 1 = NoiseCategoryChipStrip; Stages 3/4 = FilterChips). Stages 0/2 with no primary chip strip can keep MailboxFilter on its own row above the empty row list.
3. **Row column widths** — `_shell/row-list.tsx` sender column is fixed at `minWidth: 160px, maxWidth: 220px`; the row-list pane is capped at `460px` by the client-shell grids. Result: subject column squeezed to ~30-50px, showing `"R.."` / `"A.."`. Widen the row-list pane to `minmax(540px, 720px)` and tighten the sender column to `minWidth: 120px, maxWidth: 200px` so subject has visible room.
4. **Stage 1 override picker port** — `Stage1OverridePane` is currently slotted into `UnifiedDetailPane.taggingFailuresSection`, which duplicates header/subject/body/pipeline-overrides chrome. Port the working override-picker logic (category dropdown, server-action wiring, optimistic-removal, `OverrideConfirmDialog` integration) from `stage-1-override-pane.tsx` into `Stage1Widget` (the `n === 1` cell inside `_shell/detail-pane.tsx`). Drop the slot. Preserve tagging-failures section + iController banner as separate slots if needed (or fold them into Stage 1 cell extras).

**Out of scope:**
- Stage 2/4 override picker logic (they're currently stubs by design — Stage 2 awaits Phase 77, Stage 4 has its own reclassify flow already wired in `Stage4ClientShell`).
- Backend pipeline changes.
- Phase 71 4-axis UI semantics — Stage1Widget inherits Phase 71's UX behavior (same picker, same server actions, same optimistic-removal flow), just relocated.
- Visual parity with Sketch 005 beyond the 4 fixes above.

</domain>

<decisions>
## Implementation Decisions

### Fix 1 — STAGE_TITLES prefix (locked)

- **D-01:** Replace `STAGE_TITLES` values in `_shell/detail-pane.tsx:85-89` with bare labels:
  ```
  0: "Safety",
  1: "Category",
  2: "Customer",
  3: "Intent",
  4: "Handler",
  ```
  `stage-step.tsx:62` already prefixes with `Stage {stage.n} —`, producing the correct `"Stage 0 — Safety"` shape.

### Fix 2 — MailboxFilter row placement (locked)

- **D-02:** Hoist MailboxFilter out of each `client-shell.tsx` and render it on the same flex row as the primary chip strip:
  - Stage 1: render at the page level next to `<NoiseCategoryChipStrip>` (stage-1/page.tsx wrap them in a flex row with `justifyContent: space-between` so the chip strip flows left, MailboxFilter sits right).
  - Stage 3: render next to FilterChips inside Stage3ClientShell (chip strip + MailboxFilter share a row).
  - Stage 4: same as Stage 3.
  - Stages 0/2: no primary chip strip exists — MailboxFilter renders alone, right-aligned, above the empty row list.
- **D-03:** MailboxFilter component itself stays unchanged — only its mount point moves.

### Fix 3 — Row column widths (locked)

- **D-04:** `_shell/row-list.tsx` sender column: `minWidth: 120px, maxWidth: 200px` (was 160/220 — 40px tighter).
- **D-05:** Each `client-shell.tsx` two-column grid: `minmax(540px, 720px) 1fr` (was `minmax(380px, 460px) 1fr` — adds 260px of headroom for the subject).
- **D-06:** Subject column keeps `flex: 1` + ellipsis truncation — the widening provides visible room without changing wrap behavior.

### Fix 4 — Stage 1 override picker port (locked)

- **D-07:** Port the following from `stage-1/stage-1-override-pane.tsx` into a new file `_shell/components/stage-1-widget.tsx` (which today exists as a stub inside `_shell/detail-pane.tsx`):
  - Category dropdown UI (the `<select>` / popover that lists `categories` and shows `Current: <key>`)
  - `OverrideConfirmDialog` integration
  - `confirmOverride` / `rejectPrediction` server-action wiring
  - Optimistic-removal hooks via `useSelection().markPendingRemoval`
  - Keyboard shortcut listeners (Enter to confirm, Space to reject, n to skip) — these already exist as window events; the widget subscribes via `useEffect`
- **D-08:** `Stage1Widget` prop contract (already declared in `_shell/detail-pane.tsx`):
  ```ts
  interface Stage1WidgetProps {
    categories: SwarmNoiseCategoryRow[];   // hard-sep: noise registry ONLY
    value: string | null;                  // current prediction (was: row.predicted_category)
    onChange: (next: string) => void;      // marks the stage cell dirty
    row: PredictedRow;                     // for server-action args
    swarmType: string;                     // for server-action args
  }
  ```
- **D-09:** `Stage1ClientShell` drops the `taggingFailuresSection={...}` prop on `<UnifiedDetailPane>`. The tagging-failures rendering + iControllerBanner that lived inside `Stage1OverridePane` move into:
  - **iControllerBanner** → `iControllerBanner` slot on `UnifiedDetailPane` (already exists per Phase 82 Plan 01).
  - **Tagging failures section** → a new optional `extrasBelowPipeline` slot on `UnifiedDetailPane` that renders below the 5-cell PipelineFlow. Stage 1 passes it; other stages don't.
- **D-10:** `stage-1-override-pane.tsx` becomes unused after the port. Delete the file. Verify no other consumers via grep.
- **D-11:** Hard-separation rule preserved: `Stage1Widget` reads `categories` (swarm_noise_categories) ONLY. Never `intents`. Grep gate in test: `! grep -q "swarm_intents\|SwarmIntentRow" _shell/components/stage-1-widget.tsx`.
- **D-12:** Realtime channel `${swarmType}-review` (Phase 81 D-19 lock) stays — `Stage1ClientShell` continues to wrap in `AutomationRealtimeProvider` at the same point.

### Claude's Discretion

- Whether `Stage1Widget` lives as `_shell/components/stage-1-widget.tsx` (separate file) or stays inline in `_shell/detail-pane.tsx`. Lean toward separate file given the ported logic is ~200 LOC.
- Exact prop name for the new pipeline-extras slot (`extrasBelowPipeline` vs `stage1Extras` vs `pipelineExtras`). Planner's call.
- Whether the port preserves the entire `Stage1OverridePane` server-action surface verbatim or trims unused fields. Default: verbatim. Optimizations are out of scope.
