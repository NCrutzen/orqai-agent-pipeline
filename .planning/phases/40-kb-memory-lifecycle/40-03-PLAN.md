3. **Constraints block (SKST-06):** NEVER / ALWAYS rules + "Why these constraints" paragraph. Include AT LEAST:

8. **Step-by-step flow** (the working body). Use Step N: headings. Include at minimum:

   **Step 1: Collect Context**
   - Accept from caller: swarm-dir, agent-slug (or agent list), existing agent spec paths.
   - If not provided, detect from ORCHESTRATION.md `## Memory Store Design` section.

   **Step 2: Ask User for Store Details (AskUserQuestion)**
   - Store name (default: `<agent-slug>-memory` per naming convention)
   - Purpose (free text, stored as description)
   - Expected schema — which descriptive keys will be written: `session_history`, `user_preferences`, `conversation_context`, or custom descriptive keys (reject generic keys like `data`, `x`, `state` — lint them inline)
   - Read/write pattern: per-invocation write, per-session read, periodic cleanup

   **Step 3: Create Memory Store**
   - MCP-first: call `create_memory_store` with `{name, description, keys}`.
   - REST fallback: `curl -X POST "$ORQ_BASE_URL/v2/memory-stores" -H "Authorization: Bearer $ORQ_API_KEY" -d '{"name": "...", "description": "...", "keys": [...]}'`.
   - Capture `memory_store_id` from response.
   - On failure, surface raw error + retry suggestion. NEVER fabricate an ID.

   **Step 4: Wire Agent**
   - Update `agents/{agent-slug}.md` spec:
     - Add to `settings.memory_stores`: `{memory_store_id, name, keys}`.
     - Inject instruction into system prompt (in the "Tools & Integrations" or equivalent section): "Use memory store `<name>` to recall user-specific context. Keys available: `<keys>`. Write updates via `set_memory(<key>, <value>)` and read via `get_memory(<key>)`."
   - Run AskUserQuestion confirm if the spec already has a `memory_stores` entry.

   **Step 5: Round-Trip Read/Write/Recall Test (KBM-05 core)**
   - Generate `test_write_<uuid>` value.
   - **Write phase:** MCP `set_memory(store_id, key=<first descriptive key>, value=<test_uuid_value>)`. Expect 200.
   - **Read phase:** MCP `get_memory(store_id, key=<same key>)`. Expect returned value == test value.
   - **Recall phase:** Spawn a minimal agent invocation via SDK `deployments.invoke({agent, prompt: "What value is stored under <key>?"})`. Expect the response to contain the test value (LLM-judge binary Pass/Fail is acceptable).
   - **Cleanup:** Delete the test key via `delete_memory(store_id, key)` OR overwrite with empty string.
   - Report each phase's status in the summary.

   **Step 6: Summary**
   - Output table:

         | Store | ID | Keys | Write | Read | Recall | Cleanup |
         |-------|----|------|-------|------|--------|---------|
         | {name} | {id} | {keys} | ✓ | ✓ | ✓ | ✓ |

   - On any phase failure: STOP, surface error, do NOT mark handoff complete.

9. **KB-vs-Memory Decision Rule (KBM-04 — embedded for lint anchor):**

   ```
   ## KB-vs-Memory Decision Rule (KBM-04)

   - **KB (static reference data):** Docs, FAQs, product catalogs, policies, structured knowledge. Chunked + embedded + queried by similarity.
   - **Memory Store (dynamic user context):** Session history, preferences, per-user facts. Keyed + written at runtime by agent decisions.
   - **Block:** Memory for docs/FAQs → use KB. KB for conversation context → use Memory Store.

   If the caller's requested purpose matches a KB pattern (docs / FAQ / policy catalog / structured knowledge), STOP and redirect to `/orq-agent:kb --mode kb`. Never silently coerce.
   ```
