# Phase 56.7: Swarm Registry — Research

**Researched:** 2026-04-29
**Domain:** Cross-swarm registry promotion (Supabase tables + module-cache loader + dynamic-segment Next route + Inngest verdict-worker switch)
**Confidence:** HIGH (all decisions locked in CONTEXT.md; this research catalogs concrete code shapes against existing precedent)

## Summary

Phase 56.7 promotes the queue-review pattern from "debtor-email-specific" to a swarm-agnostic, registry-driven system. Two `public.` tables (`swarms`, `swarm_categories`) drive a generic `/automations/[swarm]/review` surface and a switch-on-`action` verdict-worker. The architectural blueprint already exists in-repo: `zapier_tools` (registry-replaces-env-vars) gives the migration shape; `web/lib/classifier/cache.ts` gives the module-scope-Map TTL loader; `classifier_queue_counts(p_swarm_type)` already accepts swarm_type so no RPC change is needed; `/automations/classifier-rules` is already swarm-agnostic.

Phase 56.7 is mostly **assembly of established patterns** — minimal novel architecture. The riskiest bit is the verdict-worker generalization (single switch on `action`) because today's worker has Outlook side-effects coded inline; everything else is mirror-existing-pattern.

**Primary recommendation:** Land migration + loader + worker switch + dynamic route in one ship; redirect `/automations/debtor-email-review` via `next.config.ts` `async redirects()`. Amend 56-05 frontmatter with `depends_on: [56.7]` and rewrite its category-source line to read from `loadSwarmCategories('debtor-email')`.

### Pattern 2: 60s in-memory TTL cache, keyed by swarm_type (mirrors `web/lib/classifier/cache.ts`)
**What:** Module-scope `Map<string, {value, expires}>`. On miss → fetch → cache. On error → return last-known-good or empty.
**When to use:** Read-heavy, mutation-rare tables (registry, whitelist).
**Example:**
```typescript
// Source: web/lib/classifier/cache.ts (lines 12-55) — mirror this exact shape
// for web/lib/swarms/registry.ts.

import type { SupabaseClient } from "@supabase/supabase-js";

interface SwarmRow {
  swarm_type: string;
  display_name: string;
  description: string | null;
  review_route: string;
  source_table: string;
  enabled: boolean;
  ui_config: SwarmUiConfig;
  side_effects: Record<string, unknown> | null;
}

interface SwarmCategoryRow {
  swarm_type: string;
  category_key: string;
  display_label: string;
  outlook_label: string | null;
  action: "categorize_archive" | "reject" | "manual_review" | "swarm_dispatch";
  swarm_dispatch: string | null;
  display_order: number;
  enabled: boolean;
}

const SWARM_CACHE = new Map<string, { value: SwarmRow | null; expires: number }>();
const CATEGORIES_CACHE = new Map<string, { value: SwarmCategoryRow[]; expires: number }>();
const TTL_MS = 60_000;

export async function loadSwarm(
  admin: SupabaseClient,
  swarmType: string,
): Promise<SwarmRow | null> {
  const now = Date.now();
  const hit = SWARM_CACHE.get(swarmType);
  if (hit && hit.expires > now) return hit.value;

  const { data, error } = await admin
    .from("swarms")
    .select("*")
    .eq("swarm_type", swarmType)
    .eq("enabled", true)
    .maybeSingle();

  if (error) {
    if (hit) return hit.value; // last-known-good on error
    return null;
  }
  const value = (data as SwarmRow | null) ?? null;
  SWARM_CACHE.set(swarmType, { value, expires: now + TTL_MS });
  return value;
}

export async function loadSwarmCategories(
  admin: SupabaseClient,
  swarmType: string,
): Promise<SwarmCategoryRow[]> {
  const now = Date.now();
  const hit = CATEGORIES_CACHE.get(swarmType);
  if (hit && hit.expires > now) return hit.value;

  const { data, error } = await admin
    .from("swarm_categories")
    .select("*")
    .eq("swarm_type", swarmType)
    .eq("enabled", true)
    .order("display_order", { ascending: true });

  if (error) {
    if (hit) return hit.value;
    return [];
  }
  const value = (data as SwarmCategoryRow[] | null) ?? [];
  CATEGORIES_CACHE.set(swarmType, { value, expires: now + TTL_MS });
  return value;
}

// Test helper. Production callers MUST NOT use this.
export function __resetCacheForTests(): void {
  SWARM_CACHE.clear();
  CATEGORIES_CACHE.clear();
}
```

### Anti-Patterns to Avoid
- **Reading `swarms` directly from React server components.** Always go through `loadSwarm()` so the TTL cache can short-circuit.
- **Hardcoding `swarm_type='debtor-email'` in the [swarm] route.** The whole point of D-07 is generic. The only place `'debtor-email'` should remain hardcoded post-56.7 is the migration seed and (per D-12) the `queue-icontroller-delete` step inside the verdict worker — which stays Outlook+iController-coupled until a third swarm proves the abstraction.
- **FK from `swarm_categories.swarm_type` → `swarms.swarm_type`.** D-16 explicitly rejects FKs (lock-contention without enforcement value). Use a logical reference + composite PK on `(swarm_type, category_key)`.
- **Inline cache invalidation on every read.** TTL is the contract; mutations (rare) wait at most 60s.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Module-scope cache + TTL | Custom EventEmitter / WeakMap dance | Map + `Date.now()` (mirror `cache.ts`) | Already proven; trivial; serializable to test |

## Common Pitfalls

### Pitfall 1: Worker reads category synchronously inside step.run, but loader uses module cache
**What goes wrong:** First Inngest worker invocation after deploy/restart hits cold cache → 1 Supabase round-trip inside `step.run("load-category")`. Subsequent invocations within 60s reuse cache.
**Why it happens:** Module-scope state survives across function invocations within the same Vercel/Inngest container.
**How to avoid:** Treat the loader call as a normal step.run; it's idempotent. Don't try to pre-warm the cache.
**Warning signs:** Worker latency p99 spikes only after a deploy / cold start.

### Pitfall 2: `next.config.ts` redirects don't run for already-cached client navigations
**What goes wrong:** Operator clicks an in-app link to `/automations/debtor-email-review` cached in router. Redirect doesn't fire.
**Why it happens:** Next App Router client-side prefetcher cached the old URL.
**How to avoid:** ALSO update every internal link in the codebase that points to `/automations/debtor-email-review` (search-replace). Don't rely solely on the edge redirect for in-app traffic.
**Warning signs:** Internal `<Link>` clicks 404 instead of redirecting; only direct browser-bar navigation works.

### Pitfall 3: Worker switch falls through silently on unknown `action`
**What goes wrong:** Migration adds new action value at DB level, but the worker switch has no branch → silent no-op.
**Why it happens:** `switch` without exhaustiveness check.
**How to avoid:** Use TypeScript exhaustive-check pattern: `default: { const _exhaustive: never = category.action; throw new Error(...); }` — fails type-check on next migration without code update.
**Warning signs:** automation_runs row stays in 'pending' forever after approve.

### Pitfall 4: `outlook_label IS NULL` + `action='categorize_archive'` reaches archive step
**What goes wrong:** Reading D-11 strictly: when `outlook_label` is null, NEITHER categorize NOR archive should fire. But "archive" is currently a separate `step.run` decoupled from category lookup.
**Why it happens:** Today's worker calls `categorize` then `archive` unconditionally on approve.
**How to avoid:** Gate BOTH the `categorize` step and the `archive` step on `category.outlook_label !== null`. If null, skip both and go straight to `mark-completed`. (Open Q1 — confirm with planner whether archive without categorize ever makes sense.)
**Warning signs:** Sales-swarm rows get archived in the Sales mailbox after approve (wrong: they shouldn't touch a mailbox at all if no outlook_label).

### Pitfall 5: Phase 56-05 amendment forgets the type-only export from categories.ts
**What goes wrong:** `actions.ts` imports `OVERRIDE_CATEGORIES` and `type OverrideCategory` from `./categories`. Switch to registry → static `OVERRIDE_CATEGORIES` const becomes `await loadSwarmCategories('debtor-email').then(c => c.map(x => x.category_key))` — but Zod's `z.enum()` requires a literal tuple at parse-construction time.
**Why it happens:** `z.enum([...await loadSwarmCategories()])` is async — can't wrap a top-level Zod schema.
**How to avoid:** Either (a) build the schema lazily inside `recordVerdict` (`const schema = z.object({override_category: z.enum(allowedKeys), …})`), or (b) accept `z.string()` at the schema layer + post-validate against the loaded list (cheaper). Recommend (b).
**Warning signs:** TypeScript error "Argument of type 'string[]' is not assignable to parameter of type '[string, ...string[]]'".

### Pitfall 6: classifier_queue_counts RPC query already accepts swarm_type — but page.tsx hardcodes 'debtor-email'
**What goes wrong:** Even though the RPC is parameterized [VERIFIED: 20260428_classifier_queue_counts.sql], `loadPageData` at `web/app/(dashboard)/automations/debtor-email-review/page.tsx:94` calls it with hardcoded `'debtor-email'`. The [swarm] route must thread the dynamic-segment value through.
**Why it happens:** The current page was built before [swarm] existed.
**How to avoid:** `loadPageData(params, admin, swarmType)` adds a third argument; replace hardcoded `'debtor-email'` (5 occurrences in page.tsx + actions.ts).
**Warning signs:** Sales swarm queue page shows debtor-email rows.

### Migration shape (concrete DDL — copy/adapt to `supabase/migrations/20260429b_swarm_registry.sql`)

```sql
-- Phase 56.7. Swarm registry: drives the generic /automations/[swarm]/review
-- surface and the verdict-worker switch. Mirrors the zapier_tools registry
-- pattern (20260429_zapier_tools_registry.sql) and the classifier_rules RLS
-- posture (20260428_classifier_rules.sql).
--
-- Adding a new swarm = INSERT swarms + INSERT swarm_categories. No code
-- change for routing; one-line code change in the verdict-worker IF the new
-- swarm uses action='swarm_dispatch' (then a new Inngest worker listens on
-- the dispatch event).

create table if not exists public.swarms (
  swarm_type    text primary key,
  display_name  text not null,
  description   text,
  review_route  text not null default '/automations/[swarm]/review',
  source_table  text not null default 'automation_runs',
  enabled       boolean not null default true,
  ui_config     jsonb not null default jsonb_build_object(
    'tree_levels', jsonb_build_array('topic','entity','mailbox_id'),
    'row_columns', jsonb_build_array(),
    'drawer_fields', jsonb_build_array(),
    'default_sort', 'created_at desc'
  ),
  side_effects  jsonb,                   -- D-12: reserved for Phase 56.8+
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

create table if not exists public.swarm_categories (
  swarm_type      text not null,
  category_key    text not null,
  display_label   text not null,
  outlook_label   text,                  -- D-11: nullable
  action          text not null check (action in (
    'categorize_archive','reject','manual_review','swarm_dispatch'
  )),
  swarm_dispatch  text,                  -- D-02: Inngest event name when action='swarm_dispatch'
  display_order   int  not null default 0,
  enabled         boolean not null default true,
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),
  primary key (swarm_type, category_key)
```

### Generic verdict-worker switch (concrete shape)

```typescript
// web/lib/inngest/functions/classifier-verdict-worker.ts (rewrite)
//
// Phase 56.7 (D-10/D-11). Single dispatch function that loads the
// swarm_category row and switches on `action`. Outlook side-effects gated on
// outlook_label !== null (D-11). swarm_dispatch fires the Inngest event named
// in swarm_dispatch column.

import { inngest } from "@/lib/inngest/client";
import { createAdminClient } from "@/lib/supabase/admin";
import { emitAutomationRunStale } from "@/lib/automations/runs/emit";
import { categorizeEmail, archiveEmail } from "@/lib/outlook";
import { loadSwarmCategories } from "@/lib/swarms/registry";

export const classifierVerdictWorker = inngest.createFunction(
  { id: "classifier/verdict-worker", retries: 0 },
  { event: "classifier/verdict.recorded" },
  async ({ event, step }) => {
    const {
      automation_run_id,
      swarm_type,
      decision,
      message_id,
      source_mailbox,
      predicted_category,
      override_category,
    } = event.data;

    const admin = createAdminClient();

    // ---- Reject path (unchanged) -----------------------------------------
    if (decision === "reject") {
      await step.run("mark-complete-reject", async () => {
        await admin.from("automation_runs").update({
          status: "completed", completed_at: new Date().toISOString(),
        }).eq("id", automation_run_id);
        await emitAutomationRunStale(admin, `${swarm_type}-review`);
      });
      return { ok: true, decision };
    }

    // ---- Approve path: registry-driven dispatch --------------------------
    await step.run("flip-to-pending", async () => {
      await admin.from("automation_runs").update({ status: "pending" })
        .eq("id", automation_run_id);
      await emitAutomationRunStale(admin, `${swarm_type}-review`);
    });

    const finalCategoryKey = override_category ?? predicted_category;

    const categories = await step.run("load-categories", () =>
      loadSwarmCategories(admin, swarm_type),
    );
    const category = categories.find(c => c.category_key === finalCategoryKey);

    if (!category) {
      const errMsg = `no swarm_categories row for (${swarm_type}, ${finalCategoryKey})`;
      await step.run("mark-failed-no-category", async () => {
        await admin.from("automation_runs").update({
          status: "failed", error_message: errMsg,
          completed_at: new Date().toISOString(),
        }).eq("id", automation_run_id);
        await emitAutomationRunStale(admin, `${swarm_type}-review`);
      });
      throw new Error(errMsg);
    }

    try {
      switch (category.action) {
        case "reject": {
          // Worker-resolved 'reject' even though decision='approve' — the seed
          // category for 'unknown' is action='reject' (label-only-skip).
          break;
        }
        case "manual_review": {
          // No side-effects. Row left in feedback for human follow-up.
          break;
        }
        case "categorize_archive": {
          // D-11: outlook_label nullable. When null, skip BOTH categorize and
          // archive (graceful for non-Outlook-sourced swarms).
          if (category.outlook_label) {
            await step.run("categorize", async () => {
              const r = await categorizeEmail(source_mailbox, message_id, category.outlook_label!);
              if (!r.success) throw new Error(`categorize failed: ${r.error ?? "unknown"}`);
            });
            await step.run("archive", async () => {
              const r = await archiveEmail(source_mailbox, message_id);
              if (!r.success) throw new Error(`archive failed: ${r.error ?? "unknown"}`);
            });
          }
          // D-12: iController-delete remains code-coupled for the debtor-email
          // swarm. Until Phase 56.8 generalizes side_effects, keep this inline
          // and gated on swarm_type.
          if (swarm_type === "debtor-email") {
            await step.run("queue-icontroller-delete", async () => {
              await admin.from("automation_runs").insert({
                automation: "debtor-email-cleanup",
                status: "deferred",
                swarm_type,
                result: {
                  stage: "icontroller_delete",
                  source_automation_run_id: automation_run_id,
                  message_id, source_mailbox, icontroller: "pending",
                },
                triggered_by: "classifier-verdict-worker",
              });
              await emitAutomationRunStale(admin, "debtor-email-cleanup");
            });
          }
          break;
        }
        case "swarm_dispatch": {
          // D-02: fire the Inngest event named in swarm_dispatch column.
          if (!category.swarm_dispatch) {
            throw new Error(`swarm_dispatch action requires swarm_dispatch event name (${swarm_type}, ${finalCategoryKey})`);
          }
          await step.run("dispatch", async () => {
            await inngest.send({
              name: category.swarm_dispatch!,
              data: {
                automation_run_id, swarm_type,
                category_key: finalCategoryKey,
                message_id, source_mailbox,
              },
            });
          });
          break;
        }
        default: {
          // Pitfall 3: exhaustive-check guard. Unknown action = type error.
          const _exhaustive: never = category.action;
          throw new Error(`unhandled action: ${_exhaustive}`);
        }
      }

      await step.run("mark-completed", async () => {
        await admin.from("automation_runs").update({
          status: "completed", completed_at: new Date().toISOString(),
        }).eq("id", automation_run_id);
        await emitAutomationRunStale(admin, `${swarm_type}-review`);
      });
      return { ok: true, decision };
    } catch (err) {
      const msg = err instanceof Error ? err.message : String(err);
      await step.run("mark-failed", async () => {
        await admin.from("automation_runs").update({
          status: "failed", error_message: msg,
          completed_at: new Date().toISOString(),
        }).eq("id", automation_run_id);
        await emitAutomationRunStale(admin, `${swarm_type}-review`);
      });
      throw err;
    }
  },
);
```

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---|---|---|
| `[swarm]` dynamic segment SSRF / path traversal | Tampering | `loadSwarm` enforces `enabled=true` + exact match on `swarm_type` PK; `notFound()` on miss |
| Swarm-scoped data leak across reviewers | Information disclosure | RLS authenticated read on swarms is intentional (read-only); writes via service-role only — operator UI never bypasses RLS |
| Worker dispatches event with attacker-controlled name | Tampering | `swarm_categories.swarm_dispatch` writable only by service-role; `swarm_dispatch` only fires when `action='swarm_dispatch'` (CHECK enforces) |
| Override-dropdown injection of arbitrary category_key | Tampering | Post-validate `override_category` against `loadSwarmCategories(swarm_type)`; reject unknown |
