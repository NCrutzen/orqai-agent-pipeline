## Decisions Made

- **`inngest.send` cast for dynamic event names.** The Inngest client is typed against the `Events` EventSchemas record (`web/lib/inngest/client.ts`), so `name: category.swarm_dispatch!` (a generic `string`) is a type error. Cast through `unknown` to a `(payload: { name: string; data: Record<string, unknown> }) => Promise<unknown>` shape inside `step.run("dispatch", …)`. Cost: payload validation moves to the downstream worker. This is acceptable per D-02 — each swarm_dispatch row has a known consumer.
- **Single guard around both Outlook steps.** `if (category.outlook_label) { categorize; archive; }` rather than `if (cat.outlook_label) categorize; if (cat.outlook_label) archive;`. The single-guard form makes the "skip both" branch obvious and matches RESEARCH.md verbatim.
- **Manual-review marks completed.** Per RESEARCH.md lines 697-702 the post-switch `mark-completed` step runs for every action that didn't throw. Test 5 pins this behavior. If product later wants `manual_review` rows to remain in `feedback`, the change is a one-line `return` inside the `manual_review` case — well-pinned by the test.

## Deviations from Plan

- **[Rule 3 - Blocking issue] Inngest type error on `inngest.send({ name: category.swarm_dispatch! })`.** The client's `send` method is strictly typed against the `Events` map (`new EventSchemas().fromRecord<Events>()` in `client.ts`), so a runtime-string event name doesn't satisfy the discriminated union. Resolved with a narrow `unknown` cast to a `{ name: string; data: Record<string, unknown> }` payload shape (still inside `step.run("dispatch")`). Documented inline. Tracked here as a deviation because the plan didn't anticipate this — RESEARCH.md's worker-rewrite block had the same shape but wasn't compiled against the live `EventSchemas` until now.
- **[Rule 3 - Blocking issue] Test mock parameter typing.** `vi.fn(async () => ...)` plus a `(...args: unknown[]) => mock(...args)` re-spreader produced a TS2556 (spread arg without tuple type). Fixed by giving the mock itself a `(..._args: unknown[]) => ...` signature.

No Rule-1/Rule-2 fixes needed — old `CATEGORY_LABEL` + `payment_admittance → payment` normalization was already obsoleted by the seeded alias row from Wave 1.
