# Phase 37.1: Conversational Pipeline - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning
**Source:** UAT discussion — user wants CLI skill experience in web UI

<domain>
## Phase Boundary

Transform the pipeline from a silent batch job into a streaming conversation. The /orq-agent CLI skill (in orqai-agent-pipeline repo) works as a conversation where Claude narrates, spawns subagents behind the scenes, and asks the user for input at natural decision points. The web UI must replicate this experience.

The CLI skill flow: user provides use case → Claude (following /orq-agent instructions) asks discussion questions → spawns architect → shows swarm summary, asks for confirmation → spawns tool-resolver + researcher (silently) → spawns spec-generator → shows specs, asks for approval → continues through deployment/testing. Claude itself is the narrator — not any specific subagent.

</domain>

<decisions>
## Implementation Decisions

### Architecture: Keep Inngest, add conversation layer
- Keep Inngest pipeline for durability/retry/resume (a 15+ minute single API call is too fragile)
- Only 3 streaming Orq.ai calls: discussion + architect summary + spec summary
- Silent stages get template status messages (no extra API call)
- Feedback re-runs the relevant stage with user feedback appended to prompt context

### Conversation model
- One unified voice narrating the process (like Claude in terminal), not separate agent voices
- Subagents run behind the scenes — user doesn't see raw stage output
- The orchestrator narrates what's happening and surfaces key decisions
- Interaction happens at natural conversation points, not at every stage boundary

### Interaction points per stage
- **Discussion (NEW)**: Streaming multi-turn conversation asking clarifying questions. AI decides when it has enough context. Produces enriched use case.
- **Architect**: Stage runs behind the scenes → streaming narrator summarizes the designed swarm → pauses for confirmation/feedback
- **Tool Resolver**: Runs silently → brief template status message
- **Researcher**: Runs silently → brief template status message
- **Spec Generator**: Stage runs behind the scenes → streaming narrator highlights key specs → pauses for approval/feedback
- **Orchestration Generator**: Runs silently → brief template status message
- **Dataset Generator**: Runs silently → brief template status message
- **Readme Generator**: Runs silently → brief template status message

### UI layout
- Right panel transforms from status-only to chat-primary
- Top of right panel: compact status progress bar (stages with checkmarks/spinners)
- Middle: streaming chat messages from AI + user responses
- Bottom: text input field for user responses
- Left: swarm graph (still there, populates as stages complete)

### Streaming infrastructure
- Orq.ai chat completions API with `stream: true` for the 3 conversational stages
- Supabase Broadcast pushes streaming tokens to client (`agent-message` events on `run:{runId}`)
- Chat messages stored in DB for persistence across page refreshes
- User input → server action → Inngest event → pipeline resumes via waitForEvent

### Claude's Discretion
- Chat message DB schema design (new table vs extending pipeline_steps)
- Token chunking strategy for broadcast (per-token vs batched)
- Narrator prompt engineering (how to summarize architect/spec output conversationally)
- Discussion agent prompt (what questions to ask, when to stop)
- Mobile/responsive layout considerations
- Auto-scroll behavior in chat panel

</decisions>

<specifics>
## Specific Ideas

- The discussion phase should feel like a brief conversation (3-5 questions max), not an interrogation
- Architect summary should show agents as cards (name, role, tools) similar to how graph nodes show them
- Use the same Orq.ai API the pipeline already uses, just with streaming enabled
- The `pipeline/review.responded` Inngest event already exists for the feedback/confirmation flow
- Status progress bar at top of chat could reuse the existing step status badges in a horizontal layout
- Consider a typing indicator animation while AI is generating (before first token arrives)

</specifics>
