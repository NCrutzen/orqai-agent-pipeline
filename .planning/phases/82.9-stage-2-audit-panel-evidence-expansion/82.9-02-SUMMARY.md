# Phase 82.9 Plan 02: Resolver + writer changes for Stage 2 audit panel evidence

## One-liner

Widened `ResolveResult` with a discriminated `Stage2Inputs` union (5 arms, one per `method`) and extended the Inngest label-resolver writer to persist that payload at `pipeline_events.decision_details.inputs` — Plan 03's mapper now has a typed, exhaustive contract to read against.

## Final shape of `Stage2Inputs`

```typescript
export type Stage2Inputs =
  | { kind: "thread_inheritance"; prior_email_label_id: string; conversation_id: string }
  | { kind: "sender_match"; sender_email: string; candidates: Candidate[] }
  | { kind: "identifier_match"; matched_identifiers: string[]; candidates: Candidate[] }
  | { kind: "llm_tiebreaker"; sender_email: string | null; matched_identifiers: string[]; candidates: Candidate[]; llm_reason: string }
  | { kind: "unresolved"; sender_email: string | null; matched_identifiers: string[] };

export type Candidate = {
  id: string;
  name: string;
  contact_person: string | null;
  recent_invoices: string[]; // top 5, newest first (Tier 3 bound)
};
```

`kind` mirrors the outer `method` literal — the union is discriminated by both fields in lockstep so illegal combinations (e.g. `method:"sender_match"` with `kind:"identifier_match"`) cannot compile.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Plan 01 CandidateDetailMatchSchema extension absent on base commit**

- **Found during:** Task 1 (initial grep for `contact_person` in `nxt-zap-client.ts` returned empty)
- **Issue:** Plan 02 depends on Plan 01 having extended `CandidateDetailMatchSchema` with `contact_person` + `recent_invoices` so Zod parsing of Zap responses tolerates the new fields. Base commit `2402095` did not contain the change; Plan 01 may be landing in parallel worktree.
- **Fix:** Added the two optional fields directly in this commit (`contact_person: z.string().nullable().optional()`, `recent_invoices: z.array(z.string()).nullable().optional().transform((v) => v ?? [])`). Forward-compatible: legacy Zap responses without these fields still parse cleanly (optional + nullable + default).
- **Files modified:** `web/lib/automations/debtor-email/nxt-zap-client.ts`
- **Commit:** `4c388ba`
- **Coordination note:** If Plan 01 also lands the same Zod extension in a separate worktree, the merge will be a textual conflict (one wins, the other is a no-op since the Zod shape is identical). Both versions converge on the same runtime contract.

**2. [Rule 3 — Blocking] Legacy `/api/automations/debtor/label-email/route.ts` and Inngest worker `runThreadInheritanceOnly` constructed ResolveResult literals without the new required `inputs` field**

- **Found during:** Task 1 (first `tsc --noEmit` after widening `ResolveResult`)
- **Issue:** Two non-test files outside the plan's `files_modified` declared ResolveResult literals: the legacy webhook handler at `route.ts` (4 literals) and the no-NXT fallback `runThreadInheritanceOnly` in `classifier-label-resolver.ts` (3 literals).
- **Fix:** Populated discriminated `inputs` on every literal. Added `id` to the `email_labels` SELECT projection in both files so `prior_email_label_id` can be persisted. Extended both helpers' arg signature with `from_email`.
- **Files modified:** `web/app/api/automations/debtor/label-email/route.ts`, `web/lib/inngest/functions/classifier-label-resolver.ts`
- **Commit:** `4c388ba`

**3. [Rule 2 — Critical functionality] `llmTiebreak` candidate-details fetch + tiebreaker call lacked graceful failure paths in the new discriminated-inputs world**

- **Found during:** Task 1 (mapping behaviour ↔ types)
- **Issue:** Pre-Phase-82.9, `lookupCandidateDetails` and `callTiebreaker` failures inside `llmTiebreak` would bubble up and be caught by the outer `try/catch` in `classifier-label-resolver.ts`. With the new mandatory `inputs` field, even failure paths must populate a valid discriminated payload — otherwise the writer would emit `decision_details.inputs = undefined` and Plan 03's mapper would mis-classify the row as legacy.
- **Fix:** Wrapped both calls in try/catch. On fetch failure: return `llm_tiebreaker` with empty `candidates` + sentinel `llm_reason: "candidate-details fetch failed"`. On tiebreaker LLM failure: return `llm_tiebreaker` with `candidates: richCandidates` (we still got the details) + sentinel `llm_reason: "llm tiebreaker failed: <err>"`. In both cases `customer_account_id: null, confidence: "none"` — matching prior failure semantics, no throw.
- **Files modified:** `web/lib/automations/debtor-email/resolve-debtor.ts`
- **Commit:** `4c388ba`
