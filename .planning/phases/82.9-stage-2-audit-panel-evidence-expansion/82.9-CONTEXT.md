## Locked Decisions

### D-01 · Decision-details schema: per-path discriminated union

`decision_details.method` is the discriminant. TS discriminated union in the writer ensures only valid combinations exist. Mapper exhaustively branches on `method`.

```ts
type Stage2DecisionDetails =
  | { method: "thread_inheritance"; customer_account_id: string; customer_name: string | null;
      inputs: { prior_email_label_id: string; conversation_id: string } }
  | { method: "sender_match"; customer_account_id: string; customer_name: string | null;
      inputs: { sender_email: string; candidates: Candidate[] } }
  | { method: "identifier_match"; customer_account_id: string; customer_name: string | null;
      inputs: { matched_identifiers: string[]; candidates: Candidate[] } }
  | { method: "llm_tiebreaker"; customer_account_id: string; customer_name: string | null;
      inputs: { sender_email: string | null; matched_identifiers: string[]; candidates: Candidate[]; llm_reason: string } }
  | { method: "unresolved"; customer_account_id: null; customer_name: null;
      inputs: { sender_email: string | null; matched_identifiers: string[] } };
```

`customer_account_id` and `customer_name` stay at the top level (back-compat with the current panel + with rows written before Phase 82.9). The new richness lives under `inputs`.

### D-02 · LLM tiebreaker reproducibility: reason only

Persist `inputs.llm_reason: string` only. Do NOT store model, agent_key, temperature, or prompt version in `decision_details` — Orq.ai trace IDs and the existing `agent_runs` row already carry that for forensic replay. Smallest payload, no duplication.

When `method === "llm_tiebreaker"` the panel renders a `REASONING` section showing `inputs.llm_reason`.

### D-03 · Candidate richness: include contact_person + recent_invoices

Each candidate is persisted as:
```ts
type Candidate = {
  id: string;
  name: string;
  contact_person: string | null;
  recent_invoices: string[];  // top 5, newest first
};
```

**Derived requirement (downstream):** `lookupCandidateDetails` in `nxt-zap-client.ts` currently returns the narrower `CandidateDetail` (id + name + a few fields, NOT contact_person/recent_invoices — see the `toTiebreakerCandidate` comment in `resolve-debtor.ts:201-207`). The corresponding NXT/Zapier endpoint must be extended (or a new sibling endpoint added) to return contact_person + recent_invoices before this phase can write them. Planner: confirm whether this is a Zapier-side schema change or a new lookup and budget it as its own plan.

**Bound:** `recent_invoices.length <= 5`. Avoids unbounded growth on customers with thousands of invoices.

### D-04 · Historical-row treatment: forward-only

New rows after deploy: full evidence. Legacy rows: panel renders the existing chip-only render plus a muted "Legacy run — limited evidence captured" line in the INPUTS section. No backfill — rows age out of operator review within ~2 weeks.

Mapper detects legacy rows by `decision_details.inputs == null`.

### D-05 · Audit-types + mapper + panel updates (delegated to planner)

These follow mechanically from D-01..D-04 and were NOT discussed individually — the planner makes the local calls:

- Mapper bridge reads each `method` branch; legacy fallback preserved.
- `Stage2EvidencePanel.tsx` adds INPUTS / REASONING / CANDIDATES sections. Visual reference: `Stage3EvidencePanel.tsx` (commit `ff12322`).

## Deferred Ideas

- **Stage 2 backfill** — D-04 chose forward-only. If operators later report repeated friction reviewing legacy rows, revisit a bounded 7-day backfill in a follow-up phase. Capture cost: per-row Zapier `lookupCandidateDetails` round-trip × ~hundreds of rows = roughly an hour of Zapier task budget.
- **LLM tiebreaker full repro envelope** — model + agent_key + prompt_version + temperature + trace_id. Use case is debugging a regression in the tiebreaker agent. Deferred because Orq.ai trace IDs + the agent_runs row already cover this; revisit if those prove insufficient.
- **Clickable prior-label link in thread_inheritance inputs** — UI affordance to navigate to the originating email's audit panel. Deferred because route-shape for cross-email navigation isn't standardised yet.
- **Stage 4 audit panel** — no audit panel exists for Stage 4 today; tracked as a separate need.
