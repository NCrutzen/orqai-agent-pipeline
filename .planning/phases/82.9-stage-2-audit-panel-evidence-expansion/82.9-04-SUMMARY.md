# Phase 82.9 Plan 04: Stage 2 Audit Panel UI Summary

Rewrites `Stage2EvidencePanel.tsx` to mirror the Stage 3 expanded layout (commit `ff12322`): INPUTS at the top with per-method content (5-arm switch), conditional REASONING for `llm_tiebreaker`, existing EVIDENCE chips preserved verbatim, new CANDIDATES list with `contact_person` + `recent_invoices` (rich) and slim `top_candidates` fallback for legacy rows, untouched SCREENSHOTS + raw JSON sections. Extends the panel test file with 6 new fixtures covering each method arm plus the legacy fallback.

### 3. CANDIDATES rendering

`renderCandidates(payload)` picks one of:
- **Rich** (`payload.candidates` non-empty) — `<ol>` with each candidate showing `<code>{id}</code> {name}`, then `Contact: {contact_person}` or italic muted "No contact recorded", then `recent_invoices[]` as inline chips.
- **Slim fallback** (`top_candidates` non-empty when rich is absent) — `<ol>` with `{account_id} — {name} ({score})`. Sliced to 3 to keep the existing `truncates top_candidates to first 3` test contract.

Section skipped entirely on `isUnresolved` (matches the existing "No candidates returned" UX contract under EVIDENCE).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `isEmpty` early-return missed Phase-82.9 fields**
- **Found during:** Task 1 self-check.
- **Issue:** Original `isEmpty` checked only back-compat fields (`identifier_source`, `confidence`, `top_candidates`, `screenshot_paths`). A row with `inputs` set but all back-compat fields null would falsely hit the empty branch, hiding the new INPUTS section.
- **Fix:** Widened the predicate to also require `inputs == null` and `(!candidates || empty)`.
- **Files modified:** `Stage2EvidencePanel.tsx`
- **Commit:** `add9eee`

**2. [Rule 1 — Bug] Acceptance-criteria grep false-positives from comments**
- **Found during:** Task 1 verification.
- **Issue:** Initial draft had `"use client"` and `Legacy run — limited evidence captured.` in JSDoc comments, causing the acceptance `grep -c '"use client"'` to return 1 (should be 0) and the `Legacy run …` grep to return 2 (should be 1).
- **Fix:** Reworded comments to avoid the literal patterns — `client-boundary directive at the top` and `muted "limited evidence captured" line`.
- **Files modified:** `Stage2EvidencePanel.tsx`
- **Commit:** `add9eee` (same commit)

**3. [Rule 1 — Bug] `INV-2026-999` text not unique in llm_tiebreaker fixture**
- **Found during:** Task 2 vitest run.
- **Issue:** `INV-2026-999` appears both as a `matched_identifiers` chip (INPUTS) and as a `recent_invoices` chip (CANDIDATES). `screen.getByText("INV-2026-999")` threw "Found multiple elements".
- **Fix:** Replaced with `expect(screen.getAllByText("INV-2026-999").length).toBeGreaterThanOrEqual(2)` — explicitly asserts the chip appears in both sections.
- **Files modified:** `Stage2EvidencePanel.test.tsx`
- **Commit:** `398c525`

**4. [Rule 1 — Bug] `fixtureLlmTiebreaker` missing required `llm_reason` field**
- **Found during:** Task 2 `tsc --noEmit`.
- **Issue:** `Stage2InputsView` `llm_tiebreaker` arm requires `llm_reason: string`. Initial fixture omitted it (placed `reasoning` only at payload top level).
- **Fix:** Added `llm_reason: "Best match — closest brand."` to the discriminated union arm.
- **Files modified:** `Stage2EvidencePanel.test.tsx`
- **Commit:** `398c525` (same commit)

### Rule 3 — Blocking issue (transient, no commit)

**5. [Rule 3 — Fix blocking issue] Worktree missing `web/node_modules` for vitest/tsc**
- **Found during:** Task 1 verification.
- **Issue:** Worktree has no installed `node_modules` — `npx tsc --noEmit` and `npx vitest run` cannot resolve dependencies.
- **Fix:** Symlinked main repo `web/node_modules` into the worktree to run verification, then removed the symlink before commit (kept out of git). Same approach as Plan 82.9-03 Deviation 2.
- **Files modified:** None committed.
