**Phase 4: Read-Back Verification (DEPLOY-05)**
After all resources are deployed, read back EVERY deployed resource from Orq.ai and compare to the intended local spec:

- For each deployed agent: `agents-retrieve` (MCP) or `GET /v2/agents/{agent_key}` (REST). Compare key fields: `instructions`, `model`, `role`, `description`, `settings.tools`, `team_of_agents`, `knowledge_bases`, `memory_stores`.
- For each deployed tool: `tools-retrieve` (MCP) or `GET /v2/tools/{tool_id}` (REST). Compare key fields: `type`, `description`, configuration object (varies by tool type: `function`, `http`, etc.).
- Comparison must exclude server-added metadata fields: `_id`, `created`, `updated`, `workspace_id`, `project_id`, `status`, `version_hash`, `created_by_id`, `updated_by_id`.
- Build an allowlist of fields to compare (fields present in local spec). Only compare those fields.

Per user decision (LOCKED): Verification discrepancies — warn and continue. Log as 'warning' in deploy-log, surface to user at end, don't block deploy.

Collect all discrepancies into a warnings list with: resource name, field name, expected value summary, actual value summary.

**Phase 5: Annotate Local Spec Files with YAML Frontmatter (DEPLOY-07)**
After successful deployment and verification, update each local agent spec `.md` file with YAML frontmatter:

```yaml
---
orqai_id: "{resource_id_from_orqai}"
orqai_version: "{version_hash_from_response}"
deployed_at: "{ISO_8601_timestamp}"
deploy_channel: "mcp|rest"
---
```

Rules for frontmatter handling:
- If the spec file already has YAML frontmatter (`---` delimiters at top), MERGE new fields into existing frontmatter. Preserve all existing fields.
- If the spec file has no frontmatter, prepend the `---` block before the first heading.
- Use standard YAML frontmatter pattern: split on `---` delimiters, parse YAML between them. Do NOT hand-roll a regex parser.
- Also annotate tool entries: store tool_id in a frontmatter section or in-line annotation in TOOLS.md so that re-runs can look up tools by stored ID first (per user decision — LOCKED: use frontmatter metadata first, fall back to key-based API search).

Per user decision (LOCKED): Resource lookup on re-deploy uses frontmatter metadata (stored agent ID) first, falls back to key-based API search if no metadata.
