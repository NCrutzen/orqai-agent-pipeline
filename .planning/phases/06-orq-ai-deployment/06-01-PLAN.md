Create `orq-agent/agents/deployer.md` following the established subagent pattern (see researcher.md, spec-generator.md for structure: frontmatter -> files_to_read -> role -> decision framework -> output format -> anti-patterns).

The deployer agent receives a swarm directory path and deploys all resources to Orq.ai. It implements a 4-phase pipeline:

**Phase 0: Pre-flight Validation**
- Check MCP reachability by calling `models-list` MCP tool. If fails, set `mcp_available = false` (will use REST for all operations).
- Validate API key via `GET /v2/models` with `Authorization: Bearer $ORQ_API_KEY`. If 401, STOP with clear error: "Invalid Orq.ai API key. Set ORQ_API_KEY environment variable."
- Read swarm directory: parse ORCHESTRATION.md for dependency order, parse TOOLS.md for tool definitions, parse each agent spec `.md` file.
- Build deploy manifest: ordered list of resources — tools first, then sub-agents (non-orchestrator), then orchestrator last.

**Phase 1: Deploy Tools**
- For each tool defined in TOOLS.md:
  - Lookup: List all workspace tools via `tools-list` (MCP) or `GET /v2/tools?limit=200` (REST). Search response `data` array for matching `key` field. Cache the tool list to avoid repeated calls.
  - If NOT found: Create via `tools-create` (MCP) or `POST /v2/tools` (REST). Record tool_id from response.
  - If found: Compare local spec fields against existing tool (exclude server-added fields: `_id`, `created`, `updated`, `workspace_id`, `project_id`, `status`, `created_by_id`, `updated_by_id`). If different: PATCH via `tools-update` (MCP) or `PATCH /v2/tools/{tool_id}` (REST). If same: skip (mark as "unchanged").
- Report progress: "Deploying tools... (N/M)"

**Phase 2: Deploy Sub-Agents**
- For each sub-agent (non-orchestrator) from ORCHESTRATION.md, in listed order:
  - Lookup: Use frontmatter `orqai_id` if exists, otherwise `GET /v2/agents/{agent_key}` (REST) or `agents-retrieve` (MCP). 404 = new, 200 = existing.
  - If NOT found: Create via `agents-create` (MCP) or `POST /v2/agents` (REST) with fields from agent spec: `key`, `role`, `description`, `instructions`, `model`, `settings` (including `tools` array from TOOLS.md per-agent section), `path: "Default"`.
  - If found: Diff local spec against Orq.ai state (compare: `instructions`, `model`, `fallback_models`, `settings.tools`, `team_of_agents`, `knowledge_bases`, `memory_stores`, `role`, `description`). If different: PATCH. If same: skip.
- Report progress: "Deploying sub-agents... (N/M)"

**Phase 3: Deploy Orchestrator**
- Deploy orchestrator agent last (identified in ORCHESTRATION.md as the agent with `team_of_agents` assignments).
- Include `team_of_agents` field referencing sub-agent keys. Per research open question: try array of strings first (`["agent-key-1", "agent-key-2"]`); if 422 validation error, switch to array of objects (`[{"key": "agent-key-1"}, {"key": "agent-key-2"}]`). Document which format works.
- Wire `retrieve_agents` and `call_sub_agent` tools in orchestrator's `settings.tools`.
- Report progress: "Deploying orchestrator... (1/1)"

**MCP-first/REST-fallback pattern (per user decision — LOCKED):**
Every API operation tries MCP first. If MCP call fails (timeout, connection error, MCP unavailable), retry same operation via REST. Only log "via REST (fallback)" when fallback actually occurs. This is per-operation, not per-session.

**Retry with exponential backoff (per user decision — LOCKED):**
On transient errors (429, 500, timeouts): retry up to 3 times with exponential backoff (base 1s, multiplied by 2^attempt, capped at 30s, with random jitter 0-500ms). Fail permanently on 4xx client errors (except 429).

**On resource failure (per user decision — LOCKED):**
Stop deploying remaining resources. Do NOT roll back already-deployed resources. Report what succeeded and what failed.

**Abort entire deploy (per user decision — LOCKED):**
If a critical resource cannot be deployed through ANY available channel (MCP + REST both fail after retries), abort with clear error listing the failed resource and reason.

**files_to_read for the deployer agent:**
- `orq-agent/references/orqai-api-endpoints.md` (REST API details)
- `orq-agent/references/orqai-agent-fields.md` (agent create/update field reference)
- `orq-agent/references/naming-conventions.md` (key naming patterns)

**Anti-patterns to document in the agent:**
- Creating before checking (always GET/list first)
- Deploying agents before their tools
- Deploying orchestrator before sub-agents
- Using tool key for PATCH (must use tool_id)
- Installing @orq-ai/node@latest (must be ^3.14.45)
