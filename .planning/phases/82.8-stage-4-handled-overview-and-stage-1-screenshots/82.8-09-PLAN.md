# 82.8-09: Restrict Stage 4 auto-archived backfill to Stage-1-noise emails

## Problem

UAT review of Phase 82.8 surfaced a scope drift in the backfill migration
`20260518b_backfill_stage4_auto_archived_noise.sql` (Plan 04).

`debtor.email_labels` holds rows from **two** distinct iController paths:

1. **Stage 1 noise → archive** (the intended source of `auto_archived_noise`)
   — emitted live by Plan 82.8-02 in `classifier-verdict-worker.ts` only on the
   `categorize_archive` code path.
2. **Stage 3 intent → label/move-to-folder** — written by
   `classifier-label-resolver.ts` for any successful intent labeling.

Both paths set `icontroller_tag_status = 'tagged'`. The current backfill
filters only on `icontroller_tag_status='tagged'` and does a LEFT JOIN to
`pipeline_events stage=1` without enforcing `pe1.id IS NOT NULL`. Result: a
Stage-3-intent-labeled row would be backfilled as `auto_archived_noise` with
`noise_category = NULL`, violating the hard separation contract from
`docs/agentic-pipeline/README.md` (Stage 1 noise registry vs Stage 3 intent
registry never cross).

This drift is masked in the test env (zero `tagged` rows) but would manifest
the moment the migration runs against production.

## Root cause

Plan 04's WHERE clause assumed a 1:1 mapping between
`email_labels.icontroller_tag_status='tagged'` and "Stage 1 noise archive",
which is not the case. The matching live code path (Plan 02) does scope
correctly — only the backfill is over-broad.

## Fix

Edit `supabase/migrations/20260518b_backfill_stage4_auto_archived_noise.sql`
Pass A. The migration has **not been applied in any environment yet**
(confirmed against the active dashboard env on 2026-05-18), so amending the
existing file is preferred over a corrective `20260518c`. This avoids
shipping a no-op + corrective sequence.

Add one line to Pass A's WHERE clause:

```sql
-- existing:
WHERE el.icontroller_tag_status = 'tagged'
  AND el.labeled_at >= NOW() - INTERVAL '30 days'
  AND el.email_id IS NOT NULL
-- add:
  AND pe1.id IS NOT NULL          -- restrict to rows that had a Stage 1 event
```

This makes the LEFT JOIN behave as an inner-equivalent for the backfill
selection while preserving the LEFT JOIN syntax (so the `pe1.decision`
projection in `decision_details.noise_category` still works for genuine Stage
1 rows where `pe1.decision` could itself be NULL, e.g. unknown noise — that
NULL is correctly carried as `noise_category_unknown: true`).

Pass B (screenshot path extraction) is unaffected and stays as-is — it
operates on the URL columns directly and does not have a stage-routing concern.

- Not splitting the migration file — single in-place edit is cheaper than a
  corrective follow-up given nothing has been deployed yet.
