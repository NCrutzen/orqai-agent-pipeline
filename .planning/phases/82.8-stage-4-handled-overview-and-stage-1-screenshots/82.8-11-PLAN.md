# 82.8-11: Mount detail-pane (with screenshot strip) for Auto-archived rows

## Problem

UAT 2026-05-18: clicking an Auto-archived row on Stage 4 does not surface
a detail-pane with body / timeline / screenshot strip. Plan 05 explicitly
deferred this behaviour ("auto-archived rows do NOT mount the detail-pane
(plan §8 proposed it). They render as a plain row-list, which satisfies
the D-02 acceptance criteria; widening the selection model is left for a
follow-up plan.").

Plan 07 mounted `StageScreenshotStrip` in the **Stage 1** detail-pane and
fed `screenshotPathsByRowId` into the Stage 4 `UnifiedDetailPane`, but the
pane itself is rendered **inside the Handler-error CollapsibleContent** in
`stage-4/client-shell.tsx:298-330`. Concretely:

1. Click an Auto-archived row → `setSelected(r.id)` fires (RowList wires it).
2. `selectedUnified` resolves correctly (against `visibleUnified`).
3. The `UnifiedDetailPane` re-renders — but it lives inside Section 1
   (Handler error) which is scrolled away when the user is reading
   Auto-archived rows.
4. `body`/`timeline` are derived from `selectedEmailId`, which is
   computed from `selectedRow` (handler-error `KanbanRow` only). For
   Auto-archived rows `selectedRow=null` → `body=null` → empty pane.
5. `screenshotPathsByEmailId` only covers email_ids reachable from
   `rows` (handler-error kanban). Auto-archived rows' screenshot paths
   are not loaded server-side, so even when the pane mounts the strip
   has nothing to display.

## Fix

Four small tasks. All client-shell + page-level — no DB changes, no new
loaders.

### Task 1 — Hoist `UnifiedDetailPane` out of Section 1

Move the `UnifiedDetailPane` block from inside Handler error's
`<CollapsibleContent>` to a sibling element rendered at the page level
(below the three sections, or sticky on the right). The pane is
visibility-gated by `selectedUnified !== null` so it stays hidden until a
row is selected from any section.

Acceptance: clicking an Auto-archived row updates the pane visibly,
regardless of which section is expanded.

### Task 2 — Generalise `selectedEmailId` resolution

Currently `selectedEmailId = selectedRow?.result?.email_id ?? null`
(handler-error only). Widen to also pull from the matching
`AutoArchivedNoiseRow` (which carries `email_id` natively) or the
needs-review counterpart when those buckets get rows.

Pseudocode:
```ts
const selectedEmailId = useMemo(() => {
  if (!selectedId) return null;
  // Handler-error path (existing)
  const k = rows.find((r) => r.id === selectedId);
  if (k?.result?.email_id) return k.result.email_id as string;
  // Auto-archived path (new) — row.id IS pipeline_events.id; map back to email_id
  const a = autoArchivedRows.find((r) => r.id === selectedId);
  if (a?.email_id) return a.email_id;
  // Needs-review (future) — same pattern when wired
  return null;
}, [selectedId, rows, autoArchivedRows]);
```

Then `body`/`timeline` lookups continue to work unchanged — they're
already keyed by `selectedEmailId`.

### Task 3 — Page-level body & timeline coverage for Auto-archived email_ids

`bodyMap` and `timelineMap` in `stage-4/page.tsx` are currently built for
handler-error kanban rows + the `body_text` carried inline on
`AutoArchivedNoiseRow`. Two cases:

- **body_text** — already carried on the row by `loadAutoArchivedNoiseRows`
  Pass 2 (`select(... body_text)`). Page §157-171 already inlines this
  into `bodyMap`. Keep.
- **timeline** — emit pipeline_events query to also cover auto-archived
  email_ids. Either:
  - reuse the existing timeline loader and union `autoArchivedRows.map(a => a.email_id)` into its input set, OR
  - lazy-load timeline client-side on first auto-archived selection.
  Pick (a) for consistency with Stage 1 page-pattern.

### Task 4 — Wire `screenshotPathsByEmailId` to cover auto-archived rows

The page builds `screenshotPathsByEmailId` from a query against
`debtor.email_labels`. Currently filtered to handler-error email_ids.
Widen the WHERE-IN set to include all auto-archived `email_id`s as well.

Note for acceptance env: this lookup returns no paths for the 241
backfilled rows because that env never reached
`icontroller_tag_status='tagged'`. UI must gracefully render the strip's
empty-state ("No screenshots available") — Plan 06 already implemented
the empty-state in `StageScreenshotStrip`, so this is a render-time
no-op for acceptance.

For **prod**, only rows whose iController capture succeeded will have
screenshot paths; backfilled rows older than 30 days or where the
capture failed will gracefully empty-state.
