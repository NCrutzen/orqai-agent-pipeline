## Implementation Decisions

### D-01 — Stage 1 success → Stage 4 telemetry write

Today `classifier-verdict-worker.ts` voert `categorize_archive` succesvol uit (Outlook label + archive + iController cleanup queued) en sluit af met `automation_runs.status = 'completed'` (line ~286). **Geen `pipeline_events` row op `stage=4` wordt geschreven** — Stage 4-writes bestaan vandaag alleen in `classifier-invoice-copy-handler.ts` voor handler-failures.

**Decision:** Direct na succesvolle `categorize_archive` (en vóór `mark-completed`), één `step.run("emit-stage-4-handled", ...)` toevoegen die schrijft:

```ts
await emitPipelineEvent(admin, {
  swarm_type: swarm_type ?? "debtor-email",
  stage: 4,
  email_id: emailRow.id,
  decision: "auto_archived_noise",
  decision_details: {
    noise_category: <category_key>,
    source_stage: 1,
    archived_at: <ISO timestamp>,
  },
  automation_run_id,
  triggered_by: "pipeline",
});
```

**Replay-safety:** `emit-stage-4-handled` MOET in een eigen `step.run()` zodat Inngest replay 'm niet dubbel-schrijft. Dedupe op DB-laag via D-06 unique index.

**Out of scope voor D-01:** future-handlers die schrijven na hun eigen succespad (volgt vanzelf als handlers landen — D-02 schema staat dat al toe).

### D-02 — Stage 4 overview: drie collapsible secties, English labels (LOCKED)

Vandaag `stage-4/page.tsx` filtert `kanban_reason === 'handler_error'` (line ~85) — alleen handler-fouten zichtbaar. Operator wil drie buckets.

**Decision:** Stage 4 tab rendert **drie collapsible sections** (geen tabs, geen flat list) met **English badge labels** en counts. Volgorde top-down naar urgentie:

1. **Handler error** (`rood/oranje accent`, default OPEN) — `automation_runs` met `kanban_reason='handler_error'`. Mens moet ingrijpen. Huidige sectie, ongewijzigd qua loader.
2. **Needs review** (`amber/gele accent`, default COLLAPSED) — `automation_runs` met `kanban_reason='handler_needs_review'` (toekomstige value — vandaag leeg, lege-state copy "No handlers awaiting review"). Mens moet bevestigen vóór finale actie.
3. **Auto-archived** (`groen/neutraal accent`, default COLLAPSED) — `pipeline_events WHERE stage=4 AND decision='auto_archived_noise'`. Vangnet om terug te zoeken — geen actie nodig.

**Header pattern:** elke sectie krijgt een collapsible header met badge `[Label] ([count])` zodat alle drie tellingen direct zichtbaar zijn zonder uitklappen.

**Sortering:** **nieuwste eerst (chronologisch desc)** voor alle drie secties. Uniform, geen per-sectie sort-strategie.

**Loader:** combineer `loadKanbanRows` (sectie 1+2, client-side filter op kanban_reason) met een nieuwe `loadAutoArchivedNoiseRows(admin, swarmType, { before, limit: 100 })` die `pipeline_events` direct queryt — JOIN met `email_pipeline.emails` voor sender/subject. Pagination per sectie onafhankelijk.

**Hard-separation:** sectie 3 gebruikt `email_id` (niet `automation_run_id`) als key — kan dezelfde row-list / detail-pane infrastructure hergebruiken als Stages 0-3.

### D-04 — Screenshot read-path: path columns + sign-on-demand (REVISED 2026-05-18)

**Researcher onthulde:** `debtor.email_labels.screenshot_before_url / .screenshot_after_url` slaan **1-uur signed URLs** op (`web/lib/browser/screenshots.ts:25-52` — `createSignedUrl(path, 3600)`). Operator opent detail-pane vrijwel altijd > 1u na archive → 403 op image. D-04 zoals eerst gedraft (direct lezen van `_url` kolommen) werkt **niet** voor bestaande rows.

**Decision (LOCKED 2026-05-18 post-research) — Optie O1:**

1. **Additieve migratie:** voeg twee nieuwe kolommen toe aan `debtor.email_labels`:
   - `screenshot_before_path text NULL`
   - `screenshot_after_path text NULL`
   Bestaande `_url` kolommen blijven bestaan (geen breaking change).

2. **Capture-side wijziging:** `label-email-in-icontroller.ts` (huidig: discardt `path`, persisteert alleen `url`) wordt aangepast om ook `path` mee te geven. `debtor-email-icontroller-tagger.ts:165-170` schrijft beide kolommen.

3. **Read-side:** detail-pane loader joint `debtor.email_labels` op `email_id`, leest `screenshot_before_path` en `screenshot_after_path`. `StageScreenshotStrip` component vraagt verse signed URL via bestaand endpoint `/api/automations/audit/signed-url` (zelfde patroon als `ScreenshotThumb` component dat Stage 2 al gebruikt).

4. **Historische backfill van paths:** D-07 backfill-migratie regex'et de bucket-path uit bestaande `screenshot_before_url` / `screenshot_after_url` strings (`/object/sign/automation-screenshots/<path>?token=...`). Eenmalig, niet brittle voor toekomstige rows want die krijgen het path direct van de capture.

**Hergebruik:**
- `ScreenshotThumb` component bestaat al (`web/components/automations/bulk-review/audit/ScreenshotThumb.tsx` of vergelijkbare locatie — planner verifieert exacte path).
- `/api/automations/audit/signed-url` endpoint bestaat al (gebruikt door Stage 2 EvidencePanel).
- Geen nieuwe Supabase Storage configuratie nodig — bucket `automation-screenshots` blijft.

### D-05 — decision_details payload: minimaal (LOCKED)

**Decision:** `pipeline_events.decision_details` voor `decision='auto_archived_noise'` bevat **alleen** velden die Stage 4 zelf produceert:

```json
{
  "noise_category": "<key from swarm_noise_categories>",
  "source_stage": 1,
  "archived_at": "<ISO timestamp>"
}
```

**NIET in payload:**

### D-06 — Idempotency: hergebruik bestaande Phase 82.2 unique index (REVISED 2026-05-18)

**Researcher onthulde:** Phase 82.2 voegde reeds `pipeline_events_one_per_stage_email UNIQUE (email_id, swarm_type, stage) WHERE email_id IS NOT NULL` toe (`supabase/migrations/20260513a_pipeline_events_dedupe_and_unique_index.sql:34-36`). Deze blokkeert al elke 2e Stage 4 row per email per swarm.

**Decision (LOCKED 2026-05-18 post-research):** **Geen nieuwe migratie.** Vertrouw op de bestaande Phase 82.2 unique index.

Worker wrapt de `emit-stage-4-handled` insert in:
```ts
try {
  await emitPipelineEvent(admin, { ... });
} catch (err: any) {
  if (err?.code === '23505') {
    // Idempotent — Stage 4 row already exists for this email
    return;
  }
  throw err;
}
```

**Trade-off geaccepteerd:** Eén Stage 4 row per email per swarm. Een handler die later na error succesvol retried krijgt geen 2e Stage 4 row (eerste error-row blijft staan). Acceptabel voor v8.0; bredere Stage 4 multi-row support is V11.0 werk.

**Deferred:** Multi-Stage-4-row support per email (vereist drop+replace van Phase 82.2 index met smallere partial indexen) — out of scope voor 82.8 deadline.

### D-07 — Backfill: 30 dagen, éénmalige migratie (REVISED 2026-05-18)

**Researcher corrigeerde:** (a) `debtor.email_labels.automation_run_id` bestaat NIET — kolom uit pseudocode verwijderen. (b) `noise_category` zit niet op `email_pipeline.emails` maar op `pipeline_events.decision` waar `stage=1` (`classifier-screen-worker.ts:342` schrijft hier `finalCategoryKey` naartoe). (c) Pre-Phase-82.2 emails kunnen geen Stage 1 row hebben (Phase 82.2 was de coverage backfill).

**Decision (LOCKED 2026-05-18 post-research):** Eenmalige backfill-migratie:

```sql
-- Inserts één Stage 4 auto_archived_noise row per gearchiveerd email in 30d venster.
-- Idempotent via Phase 82.2 unique index (pipeline_events_one_per_stage_email) — re-run veilig
-- want tweede insert wordt door 23505 geweigerd. Gebruik DO NOTHING om migratie niet te laten falen.
INSERT INTO public.pipeline_events (
  swarm_type, stage, email_id, decision, decision_details, triggered_by, created_at
)
SELECT
  'debtor-email',
  4,
  el.email_id,
  'auto_archived_noise',
  jsonb_build_object(
    'noise_category', pe1.decision,                            -- NULL als geen Stage 1 row
    'noise_category_unknown', (pe1.decision IS NULL),          -- marker voor forensics
    'source_stage', 1,
    'archived_at', el.labeled_at,
    'backfilled', true
  ),
  'backfill',
  el.labeled_at
FROM debtor.email_labels el
LEFT JOIN public.pipeline_events pe1
  ON pe1.email_id = el.email_id
 AND pe1.stage = 1
 AND pe1.swarm_type = 'debtor-email'
WHERE el.icontroller_tag_status = 'tagged'
  AND el.labeled_at >= NOW() - INTERVAL '30 days'
  AND el.email_id IS NOT NULL
ON CONFLICT ON CONSTRAINT pipeline_events_one_per_stage_email DO NOTHING;
```

**Marker:** `decision_details.backfilled = true` (alle backfilled rows) + `decision_details.noise_category_unknown = true` voor pre-82.2 emails zonder Stage 1 trace. Operator kan deze emails nog steeds vinden via sender/subject in Auto-archived sectie.

**Path-kolommen backfill (in dezelfde migratie of als 2e statement):** voor elke `debtor.email_labels` row in 30d venster met `screenshot_before_url IS NOT NULL` en `screenshot_before_path IS NULL`: extract path uit URL via regex (`/object/sign/automation-screenshots/(.+?)\?token=`), vul `screenshot_before_path` / `screenshot_after_path` kolommen. Eenmalig — nieuwe rows krijgen path direct van capture (zie D-04).

**Volume estimate (researcher):** ~enkele honderden tot lage duizenden rows in 30d venster. Migratie runtime: < 5 sec.

**Niet in scope:** retroactief NIEUWE screenshots capturen voor rows waar `screenshot_*_url` NULL is — accepteer lege-state strip voor die rows.
