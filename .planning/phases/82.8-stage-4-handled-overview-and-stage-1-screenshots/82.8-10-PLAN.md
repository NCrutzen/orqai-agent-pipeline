# 82.8-10: Re-source Stage 4 auto-archived backfill from `pipeline_events stage=1`

## Problem

UAT against acceptance env (2026-05-18) surfaced a structural blind spot in
Plan 04's backfill source choice. The original migration
(`20260518b_backfill_stage4_auto_archived_noise.sql`) sources from:

```sql
FROM debtor.email_labels el
WHERE el.icontroller_tag_status = 'tagged'
```

But `email_labels` is only populated by the iController tagger
(`debtor-email-icontroller-tagger.ts`), which is **gated by dry-run env**.
In acceptance / any env where iController is unhealthy, this table never
reaches `'tagged'` even when the email was genuinely auto-archived.

The **real** "auto-archived" event is the Outlook Graph archive that runs
inside `classifier-verdict-worker.ts:118-123` (case `categorize_archive`):

```ts
await step.run("archive", async () => {
  const r = await archiveEmail(source_mailbox, message_id);
  ...
});
```

That archive runs in **every** env regardless of iController dry-run status,
because it uses Microsoft Graph directly. It is the side-effect that makes
the email "auto-archived" from the operator's perspective. iController
tagging is a separate cleanup step that doesn't affect whether the email
was archived.

**Measured impact:** acceptance env has 543 Stage 1 noise classifications
in the last 14d that map to `categorize_archive` (auto_reply 151,
payment_admittance 78, ooo_temporary 10, ooo_permanent 2 + the
`unknown` subset that the LLM 2nd-pass mapped to a categorize_archive
key). Zero of them reached `tagged`. The current backfill ignores all 543.

## Root cause

Plan 04 chose `email_labels` as the source because it had the screenshot
URL columns (needed for Pass B). But Pass A (event backfill) doesn't need
screenshots — it only needs the email_id + the noise category. Conflating
the two passes' data sources caused Pass A to inherit the iController gate
that only Pass B actually needs.

The correct architecture for Pass A: query `pipeline_events stage=1`
directly, joined to `swarm_noise_categories` to filter to rows whose
`action='categorize_archive'`. This matches the live emit's gating
condition in `classifier-verdict-worker.ts` (the emit fires inside the
`categorize_archive` switch case).

Pass B stays as-is — it correctly needs `email_labels` because that's
where the screenshot URLs live.

## Fix

New corrective migration `20260518c_resource_stage4_backfill_from_stage1_events.sql`
(NOT an in-place edit of `20260518b`, because that migration has already been
applied to acceptance and may be applied to prod by the time this lands).

### Pass A (new)

```sql
INSERT INTO public.pipeline_events (
  swarm_type, stage, email_id, decision, decision_details,
  automation_run_id, triggered_by, created_at
)
SELECT
  'debtor-email',
  4,
  pe1.email_id,
  'auto_archived_noise',
  jsonb_build_object(
    'noise_category',          pe1.decision,
    'noise_category_unknown',  (pe1.decision IS NULL),
    'source_stage',            1,
    'archived_at',             pe1.created_at,
    'backfilled',              true
  ),
  pe1.automation_run_id,
  'backfill',
  pe1.created_at
FROM public.pipeline_events pe1
JOIN public.swarm_noise_categories snc
  ON snc.swarm_type = pe1.swarm_type
 AND snc.category_key = pe1.decision
WHERE pe1.swarm_type = 'debtor-email'
  AND pe1.stage = 1
  AND pe1.email_id IS NOT NULL
  AND pe1.created_at >= NOW() - INTERVAL '30 days'
  AND snc.action = 'categorize_archive'
ON CONFLICT (email_id, swarm_type, stage)
  WHERE email_id IS NOT NULL
  DO NOTHING;
```

## Open question / follow-up

The `automation_run_id` is now sourced from the original Stage 1 row
(`pe1.automation_run_id`). `20260518b` set it to NULL because email_labels
has no such column. Carrying the upstream run id makes the audit trail
better. Confirm this isn't surprising for the Stage 4 UI loader (it
projects `decision_details` mostly, not `automation_run_id`).

  scoped via `email_labels`.
- Not touching live emit in `classifier-verdict-worker.ts`. That code
  path was independently verified during Plan 02 — it fires after the
  Outlook archive, not after iController, so the source semantics already
  match this re-sourced backfill.
- Not deleting rows backfilled by `20260518b`. In acceptance it inserted 0
  rows; in prod (if applied) the iController-tagged subset is a strict
  subset of what `20260518c` will surface, and the ON CONFLICT guard makes
  re-inserts safe.
