# Phase 82.8 Plan 07: Stage 1 / Stage 4 screenshot strip mount Summary

Mounts the Plan 06 `StageScreenshotStrip` inside the Stage 1 detail-pane audit expander and threads `debtor.email_labels.screenshot_*_path` through the Stage 1 + Stage 4 page loaders → client shells → `UnifiedDetailPane.screenshotPathsByEmailId`. Operator can now open any recent Stage 1 row (or Stage 4 "Auto-archived" row) and see the before/after iController screenshots inline next to the existing Stage 1 evidence panel.

## What shipped

- **`UnifiedDetailPane.screenshotPathsByEmailId` prop** (new, optional). Direct-prop channel from page loaders. Type: `Record<string, { before: string | null; after: string | null }>`.
- **Stage 1 audit-expander mount.** `effectiveStageAudit[1]` is wrapped in a Fragment so the existing `Stage1EvidencePanel` content stays first and `<StageScreenshotStrip beforePath={...} afterPath={...} />` renders immediately after — all inside the same `StageDetailExpander` rendered by `stage-step.tsx:155-179`. `build-stage-audit-map.ts` is NOT modified (Plan 07 constraint #7 honored).
- **Stage 1 page loader.** Adds a parallel `admin.schema("debtor").from("email_labels").select("email_id, screenshot_before_path, screenshot_after_path").in("email_id", emailIds)` alongside the existing bodies/timeline/coordinator/tagging `Promise.all`. Result reduces into `screenshotPathsByEmailId` and returns as the new `PageData.screenshotPathsByEmailId` field (also returned as `{}` on the `params.sub === "pending"` short-circuit).
- **Stage 4 page loader.** Same query, folded into the existing email-id Promise.all (bodies + timeline + labels). Map covers handler-error + needs-review + auto-archived email_ids in one shot.
- **Stage 1 client-shell pass-through.** Adds `screenshotPathsByEmailId` to `Stage1ClientShellProps` and passes verbatim — Stage 1's `PredictedRow.id === email_id` by construction (`page.tsx:608` writes `id: row.email_id`), so no remap is needed.
- **Stage 4 client-shell remap.** Builds a `screenshotPathsByRowId` `useMemo` that walks handler-error rows (kanban_id → `result.email_id`), needs-review rows (same shape), and auto-archived rows (`a.id` → `a.email_id`), looks each `email_id` up in the page's map, and emits a row.id-keyed map. That remapped map is passed to `UnifiedDetailPane.screenshotPathsByEmailId` so the detail-pane's `paths?.[row.id]` lookup resolves for every section.

## Deviations from Plan

### Auto-fixed / scope adjustments

**1. [Rule 3 — Adapted Stage 4 row.id ≠ email_id semantics]**
- **Found during:** Task 2 wiring of Stage 4 client-shell.
- **Issue:** The plan's literal code sample reads `paths?.[selected.email_id]` in detail-pane, but detail-pane only carries `Row.id`. In Stage 4 `Row.id` is a kanban_id (handler-error / needs-review) or `pipeline_events.id` (auto-archived) — never email_id. A naive page → detail-pane handoff would resolve `paths[row.id]` to `undefined` for every Stage 4 row → strip always empty.
- **Fix:** Stage 4 client-shell does the email_id-keyed → row.id-keyed remap before passing into the detail-pane. Stage 1 is unaffected (PredictedRow.id === email_id there). Page-side loader signatures stay uniform (both pages export an email_id-keyed map).
- **Files modified:** `stage-4/client-shell.tsx` (added `screenshotPathsByRowId` `useMemo`).
- **Commit:** `d44d158`

**2. [Plan-template adaptation] Mount location interpretation**
- **Found during:** Task 1 design.
- **Issue:** Plan's literal code snippet renders `<StageScreenshotStrip>` directly in the detail-pane body next to where `Stage1EvidencePanel` is `mounted`. But `Stage1EvidencePanel` is not mounted inline in detail-pane — it lives inside `stageAudit[1]` (built by `build-stage-audit-map.ts`) which is rendered by `stage-step.tsx` inside `StageDetailExpander`. The plan also constrains: "do NOT modify `build-stage-audit-map.ts`".
- **Fix:** Append the strip by wrapping `effectiveStageAudit[1]` in a `<Fragment>` inside the existing `effectiveStageAudit` useMemo. This places the strip directly under the existing evidence panel content, both rendered inside the same Stage 1 audit expander — semantically matching the plan's "right next to evidence panel" intent without editing the audit-map builder.
- **Files modified:** `_shell/detail-pane.tsx`
- **Commit:** `c96006b`
