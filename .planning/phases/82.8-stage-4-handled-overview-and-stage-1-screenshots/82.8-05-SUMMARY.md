# Phase 82.8 Plan 05: Stage 4 three-section overview — Summary

D-02 ships: Stage 4 tab is no longer a single "handler_error only" list. It now renders three Collapsible sections — Handler error (red, default OPEN), Needs review (amber, COLLAPSED, empty today), Auto-archived (lime, COLLAPSED, populated from new + 30d backfilled telemetry).

## What changed

### New: `loadAutoArchivedNoiseRows` loader

`web/app/(dashboard)/automations/[swarm]/stage-4/_lib/load-auto-archived-noise-rows.ts` — thin two-pass loader.

- **Pass 1:** `pipeline_events` where `swarm_type=$1` AND `stage=4` AND `decision='auto_archived_noise'`, `email_id IS NOT NULL`, sorted `created_at DESC`, default `limit=100`.
- **Pass 2:** `email_pipeline.emails` via `.in("id", emailIds)` for sender / subject / body / mailbox.
- Returns `AutoArchivedNoiseRow[]` with `noise_category`, `noise_category_unknown`, `backfilled` flags + body_text/body_html for inline detail-pane plumbing.

Hard separation locked into the file header: this loader only emits Stage 1 noise origins via `decision_details` (display only) — it never touches `swarm_intents`.

### `KanbanReason` union widened

`web/app/(dashboard)/automations/[swarm]/_lib/kanban-loader.ts`:

```ts
export type KanbanReason =
  | "no_handler"
  | "low_confidence"
  | "handler_error"
  | "handler_needs_review"; // Phase 82.8 D-02 — empty today; populated when first review-required handler ships.
```

Grep verified: NO writer in `web/` currently emits `'handler_needs_review'` → filter is empty-today by design.

### Stage 4 page (`page.tsx`)

- `Promise.all` widened to 3 loaders: `loadKanbanRows`, `loadSwarmNoiseCategories`, `loadAutoArchivedNoiseRows`.
- Single `stage4Rows` filter replaced by `handlerErrorRows` + `needsReviewRows`.
- Body+timeline pre-fetch covers handler-error + needs-review (kanban) emails AND auto-archived emails; for auto-archived rows the body is short-circuited into `bodyMap` from the loader's return so no second round-trip happens.
- `SelectionProvider.rowIds` extended to span all three sections so the URL `?selected=` key resolves regardless of which section a row lives in.
- `StageTabStrip` `counts[4]` deliberately stays `handlerErrorRows.length` — needs-review is empty and auto-archived is informational; the action lane stays the badge.

### Stage 4 client shell (`client-shell.tsx`)

Three `Collapsible` blocks replacing the single row-list/detail-pane grid:

1. **Handler error** — `<Collapsible defaultOpen>`, red label, badge count = `handlerErrorCount`. Contains the existing grid (`RowList` + `UnifiedDetailPane` with `intents=[]`, `categories=noiseCategories`, `activeStage=4`). Detail-pane + handler-error widget wiring unchanged from prior shell.
2. **Needs review** — `<Collapsible>` (COLLAPSED by default), amber label, badge count = `needsReviewCount`. Empty-state copy verbatim: **"No handlers awaiting review"**. When populated, renders a plain `RowList` of needs-review rows.
3. **Auto-archived** — `<Collapsible>` (COLLAPSED), lime label, badge count = `autoArchivedCount`. Renders a plain `RowList` of `autoArchivedUnified` rows (no detail-pane interaction in v1).

Mailbox filter moved to a single header row above the three sections; the `filterByMailbox` closure is applied uniformly to each section's unified row set.

### Test wiring

`stage-4/__tests__/page.test.tsx` updated to mock the new loader (default empty array) so the existing 9 test cases — which exercise handler-error rendering, hard-separation, and notFound() — continue to import the page module without runtime errors.

### Plan section 8 — auto-archived detail-pane (deferred)

Plan §8 proposed the auto-archived section feed the same detail-pane infrastructure (keying by `email_id`). I deferred this for v1 because:

1. Acceptance criteria do not require detail-pane interaction for auto-archived rows.
2. The detail-pane reads `selectedRow: KanbanRow | null` — auto-archived rows are a different shape; widening the selection model is more than a one-line change and would risk regressing the locked handler-error UX.
3. D-02 "where did my mail go" lookups are answered by the row list (sender + subject + timestamp visible).

If a future plan needs body inspection inside the auto-archived section, the body is already inlined into `bodyMap` keyed on `email_id` — only the selection-context bridge needs widening.
