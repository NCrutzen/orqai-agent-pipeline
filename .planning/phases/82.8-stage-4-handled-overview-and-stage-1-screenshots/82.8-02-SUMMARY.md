# Phase 82.8 Plan 02: Stage 4 auto-archive telemetry write Summary

Stage 1 `categorize_archive` success now writes a Stage 4 `pipeline_events`
row (`decision='auto_archived_noise'`) immediately before `mark-completed`,
making auto-archived noise visible in the Stage 4 Auto-archived overview
(Plan 04).

## What Shipped

A single `step.run("emit-stage-4-handled", ...)` inside the
`categorize_archive` branch of `classifier-verdict-worker.ts`. Sequence:

1. After `categorize` + `archive` Outlook steps and `evaluateSideEffects`
   dispatches complete (success path of the switch), the new step runs.
2. The step looks up `email_pipeline.emails.id` from the Outlook
   `message_id` (matching either `source_id` or `internet_message_id`).
3. If a Supabase email row exists, the step calls
   `emitPipelineEvent(admin, { stage: 4, decision: 'auto_archived_noise',
   ... })` with `decision_details = { noise_category: category.category_key,
   source_stage: 1, archived_at: <ISO timestamp generated inside this step> }`.
4. If `emitPipelineEvent` throws a 23505 unique-violation
   (Phase 82.2 index `pipeline_events_one_per_stage_email`), the step
   swallows it silently â†’ idempotent on replays / cross-mailbox forwards.
5. If no `email_pipeline.emails` row exists (rare), the step returns
   without writing â€” the unique index requires `email_id IS NOT NULL`.

Hard-separation preserved: the new row carries `decision='auto_archived_noise'`
only and references the Stage 1 noise key in JSON; the `swarm_intents`
registry is not touched, and the worker does NOT emit Stage 4 from any
other action branch (`reject`, `manual_review`, `swarm_dispatch`).

## Replay-Safety

- `archived_at` is generated via `new Date().toISOString()` INSIDE the
  `step.run` callback (Phase 65 lesson â€” outside-of-step non-deterministic
  values cause UPDATE-key drift on replay).
- The entire emit (lookup + insert) lives in one `step.run`. On replay
  Inngest skips the step entirely if it previously succeeded; on a
  partial-failure replay, the 23505 catch absorbs the second-pass insert.

## Idempotency Strategy

The Phase 82.2 unique partial index already prevents a second Stage 4 row
per `(email_id, swarm_type, stage)`. The new step relies on that index
rather than a pre-write `SELECT ... WHERE NOT EXISTS` race-prone check.

The `emitPipelineEvent` helper rethrows Postgrest errors wrapped as
generic `Error` (`"pipeline_events insert failed: ${error.message}"`).
The Postgrest error message includes the constraint name and SQLSTATE
literal, so the catch matches on:

- `err.code === "23505"` (defensive; preserved when PostgrestError leaks through)
- `err.message.includes("duplicate key value")` (canonical Postgres wording)
- `err.message.includes("23505")` (raw SQLSTATE in message)
- `err.message.includes("pipeline_events_one_per_stage_email")` (index name)

This matches the established pattern in `stage-0-safety-worker.ts:404-413`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] email_id lookup inside step (not reusing `emailRow`)**

- **Found during:** Task 1 implementation
- **Issue:** The plan specified `email_id: emailRow.id`, but the existing
  `emailRow` variable in scope is (a) conditionally loaded only when
  `stage1Dispatches.length > 0`, and (b) selects `sender_email, subject,
  received_at` â€” it does NOT select `id`. Using `emailRow.id` directly
  would either be `null` (no dispatches) or a TypeScript error (field
  not in select tuple).
- **Fix:** Added an independent `emails` lookup INSIDE the new
  `emit-stage-4-handled` step that selects `id` via the same
  `source_id` / `internet_message_id` disjunction the existing
  `load-email-for-cleanup` step uses. If the lookup returns no row,
  the step returns silently (cannot write a Stage 4 row without a
  non-null `email_id` per the Phase 82.2 unique partial index).
- **Files modified:** `web/lib/inngest/functions/classifier-verdict-worker.ts`
- **Commit:** `044bd9d`

**2. [Rule 2 - Critical] 23505 detection via error message AND code**

- **Found during:** Task 1 implementation
- **Issue:** Plan specified `if (code === "23505")`, but
  `emitPipelineEvent` wraps the original `PostgrestError` into a generic
  `Error` with a templated message. The original `code` property is
  lost in the wrapped error. Matching on `code` alone would NOT catch
  any real-world 23505 â€” replays would erroneously rethrow.
- **Fix:** Combined `code === "23505"` (defensive, in case helper changes)
  with message-substring matching on `"duplicate key value"`, `"23505"`,
  and the index name `"pipeline_events_one_per_stage_email"`. Mirrors
  the established pattern in `stage-0-safety-worker.ts:404-413`.
- **Files modified:** `web/lib/inngest/functions/classifier-verdict-worker.ts`
- **Commit:** `044bd9d`
