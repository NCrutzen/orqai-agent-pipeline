5. **H-08 URL-param hook:** `predictor-confidence-chip-strip.tsx` already uses `useSearchParams()` from `next/navigation` (lines 39-41, 71). The auto-expand logic reuses `search.get("predictor")` / `search.get("confidence")` on initial render — no new pattern needed.

### Pattern 1: Hard-delete JSX block (H-01)
**What:** Remove a `<div>...</div>` JSX subtree from a client component; preserve sibling state hooks and prop interfaces.
**When to use:** When CONTEXT locks "delete, not prop-thread" and no tests assert on the deleted subtree.
**Example:**
```tsx
// stage-step.tsx — DELETE L253-307 (entire <div>...</div>):
// <div className="flex items-center justify-end gap-2 mt-1">
//   <button data-testid={`stage-${stage.n}-cancel-override`}>...</button>
//   <button data-testid={`stage-${stage.n}-submit-override`}>...</button>
// </div>
//
// PRESERVE: the `onSubmitDirty`/`onCancelDirty` props on `StageStepProps` (CONTEXT H-01).
// PRESERVE: the surrounding "Note and override are saved together" copy at L240-245.
```

### Pattern 2: Filter array before sum (H-05)
**What:** Defensive filter on RPC-returned array before client-side aggregation.
**When to use:** When the RPC contract is correct for most consumers but one consumer needs a stricter predicate. Cheap when the array is small.
**Example:**
```tsx
// noise-category-chip-strip.tsx L82-89 — current:
const countByTopic = new Map<string, number>();
let totalCount = 0;
for (const c of counts) {
  totalCount += c.count;            // counts ALL rows
  if (c.topic !== null) {
    countByTopic.set(c.topic, (countByTopic.get(c.topic) ?? 0) + c.count);
  }
}

// H-05 + H-06 — replace:
const countByTopic = new Map<string, number>();
let needsReviewCount = 0;
for (const c of counts) {
  if (c.topic !== null) {
    countByTopic.set(c.topic, (countByTopic.get(c.topic) ?? 0) + c.count);
  }
  // H-05: exclude "skip" (label-only noise) from the review-needed subset.
  // `classifier_queue_counts` already filters status='predicted' so
  // auto-handled + overridden rows are excluded at the RPC layer.
  if (c.topic !== "skip") needsReviewCount += c.count;
}

// L103 — H-06 rename:
{ key: "all", label: "Needs review", count: needsReviewCount },
```

### Pattern 3: useSearchParams auto-expand (H-08)
**What:** Initialise `useState` from URL search params on first mount; subsequent URL changes do NOT re-trigger.
**When to use:** Per-page-load expanded state that responds to deep-links but stays operator-controlled after.
**Example:**
```tsx
// predictor-confidence-chip-strip.tsx (already imports useSearchParams at L41):
const search = useSearchParams();
const [expanded, setExpanded] = useState(() => {
  const p = search?.get("predictor");
  const c = search?.get("confidence");
  // H-08 edge case: ?predictor=all / ?confidence=all → treat as not-active.
  const pActive = p && p !== "all";
  const cActive = c && c !== "all";
  return Boolean(pActive || cActive);
});
// After mount, `setExpanded` is the only path to flip state — URL changes
// from chip clicks below do NOT re-fire the initialiser (`useState`
// initialiser runs once).
```

### Pattern 4: Optional prop with graceful degrade (H-04)
**What:** Add an OPTIONAL prop to a presentation primitive so existing call-sites compile; threaded value is used only when present.
**When to use:** When extending shared primitives without forcing every consumer to update at once.
**Example:**

### Anti-Patterns to Avoid

- **Hex literals or new CSS tokens:** SPEC.md §Constraints "all six fixes use existing components, helpers, and patterns. No new dependencies." Use `var(--v7-*)` tokens already in the file. The mailbox-chip example above reuses `--v7-panel-2`, `--v7-text-muted`, `--v7-radius-pill`.
- **Snapshot tests on `noise-category-chip-strip.tsx`:** CONTEXT H-06 explicitly warns "Snapshot tests on `noise-category-chip-strip.tsx` will regenerate" — but grep against `web/app/(dashboard)/automations/[swarm]/` shows NO snapshot tests exist (`__snapshots__/` folders absent). Confirmed: no snapshots to update.
- **Hand-roll a collapse animation:** H-07 says "no border by default, underline on hover or chevron prefix is acceptable." Use `aria-expanded` and either conditionally render or `hidden` — no `framer-motion` etc.
- **Re-filter `counts` server-side via a new RPC:** CONTEXT-H-05 fallback says "If predicate field is missing on Row: add to the mapper" — but the count source is the RPC, not Row. Filter the RPC output client-side; do NOT migrate the RPC.
- **Forget Stage 0 / Stage 2 / Stage 3 / Stage 4 in H-03:** The grid `"minmax(0, 1fr) 460px"` is in 5 files; G-03 acceptance is "verified on all four stage tabs (Stage 0, 1, 3, 4)". ## Common Pitfalls

### Pitfall 1: Counting `counts` array vs counting `data.rows.length` (H-05)
**What goes wrong:** Replacing `totalCount` with `data.rows.length` undercounts because rows are paginated (PAGE_SIZE).
**Why it happens:** Operator says "the row list shows 459, the chip says 459" — but the row list itself only shows the page; the 459 is the swarm-wide RPC sum.
**How to avoid:** Filter `counts` array (still RPC-aggregate), not the in-memory `rows` array.
**Warning signs:** New count drops to ≤ PAGE_SIZE on swarms with > PAGE_SIZE pending rows.

### Pitfall 2: Deleting the wrong button block in `stage-1-widget.tsx` (H-01)
**What goes wrong:** `stage-1-widget.tsx` has two button clusters — the "Discard / Submit" pair at ~L355-415 (target) AND override-form fields elsewhere. Deleting too aggressively breaks the form.
**Why it happens:** "Discard changes" is the only literal-string anchor; "Submit override" appears in many places.
**How to avoid:** Anchor the delete on the unique containing `<div className="flex items-center justify-end gap-2" style={{paddingTop: "var(--space-4)"}}>` at L355-358 — the WHOLE div is the delete target.
**Warning signs:** Override form no longer submits at all, or notes-area disappears.

### Pitfall 3: `stage_1_overridden` ≠ `status` flip (H-05 misinterpretation)
**What goes wrong:** Confusing the per-row `stage_1_overridden: boolean` column on `pipeline_events_email_summary` with `automation_runs.status` transitions. The RPC filters `status='predicted'` — after an override is submitted, the row's `status` flips and it disappears from the RPC payload. So "overridden" is implicitly excluded; explicit `stage_1_overridden` filtering is redundant AND wrong (an in-progress override doesn't flip until commit).
**Why it happens:** The summary view exposes both signals; one is mid-pipeline state, one is terminal state.
**How to avoid:** Trust the RPC's `status='predicted'` gate. Only add the `topic !== "skip"` filter.
**Warning signs:** Count goes too low (excludes rows still in review).

### Pitfall 4: Forgetting H-04 thread to other stage shells
**What goes wrong:** H-04 says "all stage tabs". Threading `mailboxLabels` into Stage 1 `<RowList>` only; Stage 0 / Stage 3 / Stage 4 still render no chip.
**Why it happens:** Stage 1 is the primary surface; easy to stop there.
**How to avoid:** Grep `<RowList` across `web/app/(dashboard)/automations/[swarm]/` — 5 mounts (stage-0/page.tsx, stage-1/client-shell.tsx, stage-2/page.tsx, stage-3/client-shell.tsx, stage-4/client-shell.tsx). All 5 need the `mailboxLabels={mailboxLabels}` prop. Each call-site's parent already loads `mailboxLabels` for its `UnifiedDetailPane` — re-use the same variable.
**Warning signs:** Operator UAT on Stage 3 / Stage 4 reports "still no mailbox label".

### Pitfall 5: Auto-expand fires on every URL change (H-08)
**What goes wrong:** Implementing H-08 with `useEffect(() => setExpanded(...), [search])` causes the section to re-expand every time a user clicks a chip inside it (because the chip click writes a URL param).
**Why it happens:** Reactive subscription to `search` re-runs the auto-expand logic.
**How to avoid:** Use `useState(() => initFromURL())` — the initialiser runs ONCE on mount. After mount, only the toggle button mutates state.
**Warning signs:** Operator clicks "Hide advanced filters", then clicks a chip → section snaps back open.
