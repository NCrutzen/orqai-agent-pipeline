### BLOCK-01 — `noise-category-chip-strip.test.tsx` asserts `"All"` label on at least 5 lines; Plan 02 renames it to `"Needs review"` without a test-update task

**Dimension:** Task completeness / regression guard
**Severity:** BLOCK

**Evidence:**
- `web/app/(dashboard)/automations/[swarm]/stage-1/__tests__/noise-category-chip-strip.test.tsx` contains multiple hardcoded assertions on the literal label `"All"`:
  - L85: `screen.getByRole("tab", { name: /^All —/ })`
  - L115: test description `"click on All chip → push with no topic param"`
  - L125: `fireEvent.click(screen.getByRole("tab", { name: /^All —/ }))`
  - L145: `const all = screen.getByRole("tab", { name: /^All —/ })`
  - L149/161: `expect(screen.getByRole("tab", { name: /^All —/ }))`

- Plan 02 Task 1 renames `label: "All"` to `label: "Needs review"` but contains **no test-update task** and no instruction to update `noise-category-chip-strip.test.tsx`.

- RESEARCH.md §Anti-Patterns states "CONTEXT H-06 explicitly warns 'Snapshot tests on `noise-category-chip-strip.tsx` will regenerate' — but grep shows NO snapshot tests exist." This is accurate — there are no snapshot files — but the researcher did NOT check (or misread) the existing RTL test file which contains explicit `getByRole` label assertions on `"All"`. These are NOT snapshot tests; they are explicit string assertions that will break on rename.

- RESEARCH.md §"Open assumptions" item 1 about H-01 correctly found zero inline testid references. This finding was then over-generalized in CONTEXT.md's assumption wording: "any test asserting the literal string `'All '` against this chip needs to be updated." That warning appears in CONTEXT H-06 but was **not acted on** by the planner.

**Impact:** Plan 02 as written will turn the test suite red on the `noise-category-chip-strip` test. CI will block merge.

**Fix:** Add a Task 0 (or a step inside Task 1) to Plan 02 that updates `noise-category-chip-strip.test.tsx`:
- Replace all `/^All —/` role-name matchers with `/^Needs review —/` (or equivalent matching the rendered chip format after the rename).
- Update test descriptions (lines 4, 6, 8, 72, 115) to reference "Needs review" so the test intent is clear.
- Verify: `npm --prefix web test -- --run noise-category-chip-strip` passes.

### BLOCK-02 — `stage-1-widget.test.tsx:235` asserts `"submit override"` by role name; Plan 01 deletes the button it finds via that role in stage-1-widget.tsx without a corresponding test update

**Dimension:** Task completeness / regression guard
**Severity:** BLOCK

**Evidence:**
- `stage-1-widget.test.tsx` line 235–239 (confirmed above):
  ```
  const submit = await screen.findByRole("button", { name: /submit override/i });
  fireEvent.click(submit);
  ```
  This `findByRole` query for a button named "submit override" will either:
  (a) find the footer button (if detail-pane.tsx is rendered in this test's component tree), or
  (b) fail with "Unable to find an accessible element" if the widget is rendered in isolation without the footer.

- RESEARCH.md (§"Open assumptions" item 1, HIGH confidence claim): "The only test files that reach into `stage-1-widget` either (a) mock the entire widget or (b) test the widget in isolation without asserting on the inline action-row buttons." This claim is **incorrect**: `stage-1-widget.test.tsx:235–239` asserts on a `findByRole("button", { name: /submit override/i })` and clicks it. Whether this hits the inline button or the footer depends on the test scaffold.

- PATTERNS.md §H-01 (under "Test pattern to update") states: "`stage-1/__tests__/stage-1-widget.test.tsx:235` ('Click Submit override') — switch assertions to footer testids." This confirms the PATTERNS author knew this test needed updating, but Plan 01 contains **no test-update task** for `stage-1-widget.test.tsx`.

**Impact:** If the test scaffold does not render the footer and the inline button is deleted, `findByRole("button", { name: /submit override/i })` will throw, breaking CI. Even if the test happens to still pass because the footer button is rendered, the test assertion references the wrong surface post-H-01, which is a test-debt issue that will fail on the next edit.

**Fix:** Add a step to Plan 01 Task 2 (or as a separate Task 2b) to update `stage-1-widget.test.tsx`:
- Verify whether the test scaffold renders `detail-pane.tsx` (and thus the footer) alongside the widget. If not, re-target the test's submit-click to the footer's `data-testid` per CONTEXT H-01 guidance, or confirm the test passes cleanly post-delete.
- Run: `npm --prefix web test -- --run stage-1-widget` as a verify gate inside the task.
