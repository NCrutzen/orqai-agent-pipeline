### H-03 — G-03 widen-pane mechanism: shrink list column, grow pane

**Decision:** Adjust the parent split-layout's grid template so the detail
pane gets more horizontal space at the expense of the row-list column. Do
NOT raise pane min/max-width independently — that risks squeezing the list
below readable width on narrower viewports.

**Investigation needed during planning:**
- Locate the split-layout grid template. Likely candidates:
  `_shell/page-shell.tsx`, `_shell/two-pane-layout.tsx`, or similar. Researcher
  should grep for `gridTemplateColumns` / `grid-template-columns` in
  `_shell/`.
- Current ratio (approx): list ~60%, pane ~40%. Target: enough pane width
  for `Submit override (Stage 1) · Cancel override · Reject` + comfortable
  margin (≈8-12% additional pane width).
- Verify list column at the new ratio still renders the 5-column row grid
  without horizontal scroll (recipient 200px + from/subj 280px + 1fr stages
  + cost 84px + tag 70px = ~634px minimum). A list column of ~640-720px is
  the floor; the detail pane gets the remainder.

**Applies to all stage tabs (Stage 0, 1, 3, 4)** — the split layout is
shared. Stage 2 detail-pane is per Phase 82.3, untouched here.

### H-04 — G-04 mailbox chip placement: short alias inside recipient column

**Decision:** Render the mailbox label as a small chip placed BEFORE the
recipient name, inside the existing 200px recipient column. Format:
`[short-alias] recipient@example.com`. No new column. Applies to all stage
tabs that use `row-list.tsx`.

**Format details:**
- Short alias resolved via
  `web/app/(dashboard)/automations/[swarm]/_shell/_lib/load-mailbox-labels.ts`

  (researcher confirms naming during planning).
- Chip style: small pill / tag style, neutral background, mono font, ~10px
  font-size. Reuse existing chip primitive if one exists in `_shell/`;
  otherwise inline style matching the existing row's typography.
- If `mailbox_id` resolves to null or an unrecognized id, show NO chip (not
  a placeholder dash) — the recipient column degrades gracefully to just
  the recipient text. No errors thrown.

**Files to edit:**
- `web/app/(dashboard)/automations/[swarm]/_shell/row-list.tsx` — extend the
  recipient column render to include the mailbox chip before the
  recipient text.
- `web/app/(dashboard)/automations/[swarm]/_shell/_lib/load-mailbox-labels.ts`
  — verify exports and ensure the lookup map is available to row-list. May

**Boundary:** row column widths stay the same (200/280/1fr/84/70). If the
chip plus recipient exceeds 200px on long recipient strings, the recipient
text truncates with ellipsis (existing behavior).

### H-05 — G-05 "needs review" predicate: exclude handled + overridden + skipped

**Decision:** "Needs review" count = rows in the current view that are NOT
yet auto-handled, NOT yet manually overridden, AND NOT in the skip category.

**Predicate semantics:**
- A row is in the "needs review" subset if all three are true:
  1. The row has not been auto-handled (no handler completion marker)
  2. The row has not been manually overridden by an operator
  3. The row is not categorized as `skip` (label-only) per the Stage 1
     noise-category registry

**Where to enforce:**
- `web/app/(dashboard)/automations/[swarm]/stage-1/noise-category-chip-strip.tsx`
  — change the `count: totalCount` (line ~103) to a derived count using the
  predicate above. The exact field names (e.g. `row.is_handled`,
  `row.overridden_at`, `row.category === "skip"`) must be confirmed against
  the Row type during planning. Researcher checks
  `web/app/(dashboard)/automations/[swarm]/stage-1/page.tsx` mapper for
  what's available on the row shape.

**If predicate field is missing on Row:** add to the mapper in
`stage-1/page.tsx` similar to how 82.7.2-02 added `entity` hydration —
JOIN-back or compute from `pipeline_events.decision_details` if needed.
Document the data path in SUMMARY.md.

### H-07 — G-06 widget design: text toggle button

**Decision:** Plain text toggle button with the label "Show advanced filters"
when collapsed, "Hide advanced filters" when expanded. Click to toggle.
Placement: where the `predictor-confidence-chip-strip.tsx` currently mounts —
the toggle button replaces the always-visible strips when collapsed; when
expanded, the strips render below the toggle.

**Files to edit:**
- `web/app/(dashboard)/automations/[swarm]/stage-1/predictor-confidence-chip-strip.tsx`
  — wrap the existing `PREDICTOR_CHIPS` + `CONFIDENCE_CHIPS` render in a
  collapse-controlled section. Add local state for expanded/collapsed (no
  global state needed — this is per-page-load only, per SPEC.md G-06
  acceptance).

**Style:** small text button (e.g. `var(--font-mono)` 12px, brand-secondary
color), no border by default, underline on hover or chevron prefix is
acceptable. Reuse existing button styles where possible; do not introduce
new color tokens.

**Accessibility:** the toggle must have a clear `aria-expanded` state and
be keyboard-activatable (Enter / Space).

### H-08 — G-06 URL-param auto-expand behavior

**Decision:** When the page loads with `?predictor=...` or `?confidence=...`
URL params present, the section opens expanded. Otherwise it starts
collapsed.

**Implementation:**
- Inside `predictor-confidence-chip-strip.tsx`, on first render, inspect the
  URL's search params (the existing query-param hooks in the file already
  do this for filter state). If either `predictor` or `confidence` param
  is present AND not equal to `"all"`, initialize the expanded state to
  `true`. Otherwise initialize to `false`.
- After initial render, the toggle button controls expanded state normally.
  Subsequent URL-param changes during the same page session do NOT re-expand
  the section (avoids surprising state changes when the operator clicks
  chips inside).

**Edge case:** if both URL params equal `"all"`, treat as not-active and
start collapsed. Operator hasn't filtered anything — no reason to leak the
filter UI by default.

  snapshot test depends on will turn CI red. Mitigated by the H-01
  test-update task ordering.

  list column below the 5-column minimum (~640px), recipient or
  from/subj will truncate harshly. Mitigated by the floor-check in H-03

  "skipped" semantics differ from what's actually on Row, the

  ground truth. Mitigated by the H-05 field-availability check above.
