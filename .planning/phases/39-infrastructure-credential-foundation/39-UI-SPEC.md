## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (gap-1), badge internal spacing, inline icon-to-text gaps in credential rows |
| sm | 8px | Compact element spacing (gap-2), form field internal gaps, table cell padding-y |
| md | 16px | Default element spacing (gap-4), credential form field gaps, health card internal padding |
| lg | 24px | Section padding (p-6), page-level padding, card body padding, credential list container padding |
| xl | 32px | Layout gaps between page sections (mt-8), gap between credential list and "Add Credential" button area |
| 2xl | 48px | Empty state vertical padding (py-12) for "No credentials stored" |
| 3xl | 64px | Not used in this phase |

Exceptions: Health status indicator dots use 8px diameter (size-2). Touch target for credential delete action is 44px minimum (p-2.5 on icon button). Auth profile type selector radio/card items have 12px internal padding (p-3) for comfortable click targets.

### Destructive Actions

| Action | Confirmation Approach |
|--------|----------------------|
| Delete credential | Confirmation dialog (Dialog component). Shows credential name, lists linked automations count. Two buttons: "Cancel" (variant outline) and "Delete Credential" (variant destructive). Required because deletion is irreversible and may break running automations. |
| Replace credential | No confirmation dialog. The "Replace Credential" flow opens a modal with a paste-only input. Submitting overwrites the old value. The write-once model means the old value is already inaccessible, so replacement is a forward-only operation. |

### Credential List (Table Layout)

Table layout chosen over cards. Rationale: Credentials are a flat list of named items with metadata columns (type, status, linked projects, last used, actions). Tables are the established pattern for admin-style data with uniform structure. Cards work better for heterogeneous content (like projects with descriptions). The existing dashboard uses card grids for projects but the credential list is closer to a data table.

```
[Table]
  [thead]
    [th: "Name" (text-left)]
    [th: "Type" (text-left)]
    [th: "Status" (text-left)]
    [th: "Linked Projects" (text-left)]
    [th: "Created" (text-left)]
    [th: "" (actions, w-10)]

  [tbody]
    [tr (hover:bg-muted/50 transition-colors)]
      [td: credential name (text-sm font-semibold)]
      [td: auth type badge (text-xs, variant outline)]
      [td: CredentialStatusBadge]
      [td: project count or "None" (text-sm text-muted-foreground)]
      [td: relative time (text-xs text-muted-foreground)]
      [td: DropdownMenu (MoreHorizontal icon)]
        [DropdownMenuItem: "Replace" (RefreshCw icon)]
        [DropdownMenuItem: "View Projects" (FolderOpen icon)]
        [Separator]
        [DropdownMenuItem: "Delete" (Trash2 icon, text-destructive)]
```

Note: Native HTML table with Tailwind classes, not a shadcn Table component (not installed and not needed for this simple use case). Uses `w-full`, `text-left`, `border-b border-border` for rows.

### Paste-Only Secret Input Behavior

The credential value input fields enforce a paste-only interaction:

1. Input field renders with `type="password"` and `readOnly` attribute.
2. On focus, a tooltip appears: "Use Ctrl+V or Cmd+V to paste".
3. On paste event (`onPaste`): capture pasted text, set field value, remove `readOnly`.
4. On keydown: if not a paste shortcut (Ctrl/Cmd+V), prevent default and show brief helper text "Paste only -- use Ctrl+V".
5. After paste: field shows masked dots. Value is held in React state but never displayed as cleartext.
6. No "show password" toggle -- the write-once model means the value is never visible again.

## Interaction Patterns

### Create Credential Flow

1. User navigates to Settings > Credentials tab.
2. User clicks "Store Credential" button.
3. CreateCredentialModal opens.
4. User enters a descriptive name for the credential.
5. User selects an authentication type from the radio card selector.
6. Form fields update dynamically based on selected auth type.
7. User pastes credential value(s) into paste-only input fields.
8. Optionally links the credential to one or more projects.
9. User clicks "Store Credential" submit button.
10. Button shows loading state: disabled, Loader2 animate-spin, "Storing...".
11. On success: modal closes, credential appears in list with "Not tested" status badge, toast: "Credential stored securely".
12. On error: inline error message below form, modal stays open.

### Replace Credential Flow

1. User clicks DropdownMenu on a credential row, selects "Replace".
2. ReplaceCredentialModal opens showing credential name (read-only) and empty paste-only input.
3. User pastes new credential value.
4. User clicks "Replace Credential" button.
5. Loading state on button.
6. On success: modal closes, credential status resets to "Not tested", toast: "Credential replaced".
7. On error: inline error, modal stays open.

### Credential Failure Detection (Automatic)

1. An automation runs on Browserless.io and fails with an authentication error.
2. Inngest function identifies the auth failure pattern and updates the credential status to "needs_rotation" in the database.
3. CredentialFailureBanner appears on the credential list page for that credential.
4. Email notification sent to credential owner: "Credential '{name}' failed authentication. Replace it to resume automations."
5. The credential row in the list shows "Needs Rotation" amber badge.
6. Failure state persists until user replaces the credential.

### Health Check Flow

1. Admin navigates to Settings > Health tab.
2. Three HealthStatusCards show current status (or "Not yet checked" if first visit).
3. Admin clicks "Run Health Check" button.
4. Button shows loading state: RefreshCw icon with animate-spin, text changes to "Checking...".
5. Server action triggers Inngest function that tests each service sequentially.
6. As each service check completes, HealthStatusCard updates via Supabase Broadcast:
   - Card border-left color transitions to green/amber/red.
   - Status dot updates.
   - Status text updates.
   - "Last checked" timestamp updates.
7. On all connected: toast "All services healthy".
8. On any failure: toast "Some services have issues -- review the health dashboard".

### Delete Credential Flow

1. User clicks DropdownMenu on a credential row, selects "Delete".
2. DeleteCredentialDialog opens.
3. Shows credential name and warning if linked to automations.
4. User clicks "Delete Credential" (destructive button).
5. Loading state on button.
6. On success: dialog closes, credential removed from list (optimistic), toast: "Credential deleted".
7. On error: if credential is in active use, show error toast: "This credential is linked to active automations. Unlink it first before deleting."

### Settings Tab Navigation

- Uses URL-based tab state via query parameter: `/settings?tab=credentials`, `/settings?tab=auth-profiles`, `/settings?tab=health`.
- Default tab when no query param: "Credentials".
- Tab changes update the URL via `router.replace()` without full page navigation.
- Server component fetches data based on active tab for initial render.
