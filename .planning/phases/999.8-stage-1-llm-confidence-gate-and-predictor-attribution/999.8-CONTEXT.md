# Phase 999.8: Stage 1 LLM 2nd-pass confidence gate + predictor attribution — Context

**Gathered:** 2026-05-11
**Status:** Ready for planning
**Depends on:** Phase 81 (fold Stage 1 into stage-keyed shell) — UI half (filter chips D-05, predictor chip D-08) MUST sequence after Phase 81. Backend half (gate D-01, attribution D-07, migration D-09) is independent and can ship first if Phase 81 slips.

<domain>
## Phase Boundary

Add a hard confidence gate to the Stage 1 LLM 2nd-pass `categorize_archive` decision (only `llm_confidence === "high"` may auto-archive), AND attribute every Stage 1 prediction to its predictor (`regex` vs `llm_2nd_pass`) on the verdict row so that downstream feedback math (Wilson-CI promotion/demotion in `labeling-flip-cron`, future learning loops) groups on `(mailbox, predictor)` instead of mailbox alone. Pure Stage 1 — does not touch regex thresholds (Phase 71 owns those), does not touch Stage 3 (RFC hard-separation), does not change the LLM's output enum contract.

**In scope:**
- Code-level confidence gate in `web/lib/inngest/functions/classifier-screen-worker.ts` between the LLM call and `emit-verdict`. `medium`/`low` predictions keep `status=predicted` (queued for review) instead of emitting `decision: "approve"` unconditionally.
- Re-calibration of the `stage-1-category-classifier` Orq agent prompt so the `"high"` bucket maps to ~95%+ certainty. Prompt-side; the code gate enforces the boundary, the prompt defines what "high" means.
- New `agent_runs.predictor text` column (values: `'regex' | 'llm_2nd_pass'`). `recordVerdict` populates it server-side from the prediction's `pipeline_events.decision_details` row.
- Per-predictor Wilson-CI stream in `labeling-flip-cron.ts` — groups on `(mailbox, predictor)` with `predictor IS NOT NULL` filter (forward-only cutover).
- Stage 1 row UI affordance: predictor chip on each row card (`LLM · medium`, `regex · rule_key`). Two filter chips on `/stage-1` chip-strip (Phase 81 shell): `predictor` and `confidence`. Default view: unfiltered.
- High-confidence LLM false-positive tracking as a separate calibration-drift signal (alerting threshold and emission mechanism finalized at planning time; principle locked here).

**Out of scope:**
- Regex-pass logic, regex-rule promotion thresholds (Phase 71 owns; per RFC `stage-1-regex.md:115`).
- Per-rule regex attribution (`regex:<rule_key>` granularity) in `agent_runs.predictor`. Two streams only for Phase 999.8.
- Backfilling `predictor` on historical `agent_runs` rows. Forward-only cutover.
- Changing the LLM output contract from `"low"|"medium"|"high"` enum to a numeric top-2 + margin signal (explicit non-goal).
- Stage 3 confidence gating, Stage 0 budget breach (Phase 999.7), Stage 0 timeout (Phase 999.4).
- New operator verdict subclasses (`"LLM was wrong"` etc.) — verdict UI stays plain approve/reject/override.

</domain>

<decisions>
## Implementation Decisions

### Confidence gate scope

- **D-01:** Uniform `high`-only gate at the code level. In `classifier-screen-worker.ts`, the `emit-verdict` step branches on `llmConfidence`: `"high"` → emit `decision: "approve"` as today; `"medium" | "low"` → emit a different shape that leaves `automation_runs.status = "predicted"` (or equivalent — exact mechanism for keeping the row queued without firing `classifier/verdict.recorded` is a planning concern). The closed-list `"unknown"` escape remains (action=`reject`, label-only-skip).
- **D-02:** Re-calibrate the `stage-1-category-classifier` Orq agent so `"high"` means ~95%+ certainty. Prompt instruction: any doubt → `"medium"` (or `"low"`). Update the agent's instructions + few-shots; verify with `get_agent` after `update_agent` per CLAUDE.md Orq.ai patterns. The code gate enforces the boundary; the prompt defines what "high" means.
- **D-03:** Track high-confidence LLM false-positive rate as a separate calibration-drift signal. When a human marks a `high`-conf LLM prediction wrong, that's a model-calibration alarm — not just feedback noise. Concrete threshold (e.g. ">2% high-conf FP over rolling 50 high-conf predictions per mailbox") and emission mechanism (Inngest event vs dashboard flag vs both) decided at planning time. Principle: the signal exists and is separable from the standard Wilson-CI stream.

### Low-confidence routing (within Phase 81 `/stage-1` shell)

- **D-04:** Same `/stage-1` row list is the single landing place for medium/low LLM predictions. No new sub-view (`?sub=low-confidence` was considered and rejected as adding maintenance burden for low marginal benefit). No Kanban escalation. Rejected because Phase 81 is already rebuilding Stage 1 around a chip-strip + row list + detail pane; mirroring `?sub=pending` would have been clean but the user prefers fewer surfaces.
- **D-05:** Two new filter chips in the Stage 1 chip-strip (sit alongside the existing `swarm_noise_categories` chips per Phase 81 D-06): `predictor: regex | llm` and `confidence: high | medium | low`. URL state via `?predictor=` and `?confidence=` query params. `loadPageData` in `web/app/(dashboard)/automations/[swarm]/stage-1/` reads them and filters the row query accordingly.
- **D-06:** Default view = no predictor/confidence filter applied. Operator landing on `/stage-1` sees all `status=predicted` rows (LLM medium + LLM low + any regex shadow-mode predictions). Filters are opt-in for slicing.

### Predictor attribution model

- **D-07:** Two streams. Migration adds `agent_runs.predictor text NULL CHECK (predictor IS NULL OR predictor IN ('regex','llm_2nd_pass'))`. `labeling-flip-cron.ts:94-122` aggregates Wilson-CI on `(mailbox, predictor)` instead of mailbox alone. Per-rule granularity (`regex:<rule_key>`) is explicitly NOT in this phase — Phase 71 (which owns regex-rule promotion thresholds) reads `pipeline_events.decision_details.regex.matchedRule` directly if it needs the per-rule stream.
- **D-08:** Stage 1 row UI shows a predictor chip on each row card. Format: `LLM · medium` (with confidence variant) or `regex · <rule_key>` (rule name appears for regex predictions to give the operator instant context). Verdict UI stays plain: approve / reject / override-category. Server-side `recordVerdict` (currently in `web/app/(dashboard)/automations/[swarm]/review/actions.ts`, post-Phase 81 rename: `.../stage-1/actions.ts`) reads predictor from the prediction's `pipeline_events` row and writes it to the new `agent_runs.predictor` column. Operator doesn't pick an attribution — it's automatic.

### Backfill strategy

- **D-09:** Forward-only cutover. The migration adds `predictor` as NULL. Historical `agent_runs` rows stay NULL. `labeling-flip-cron` filters `predictor IS NOT NULL` so the new per-predictor Wilson-CI streams accumulate cleanly from cutover date. Accept the cold-start window (multiple weeks before promotion/demotion gates have N=50 per `(mailbox, predictor)` bucket). During cold-start, the *old* per-mailbox math is dead (gone with the cron change) — phase plan must confirm whether the cron should remain in shadow mode for the cold-start window or just go cold. Decided at planning time; principle of forward-only is locked.

### Claude's Discretion

- Exact mechanism for "keep row queued without firing classifier/verdict.recorded" in D-01 (e.g. new event name `classifier/screen.requires_review` vs reusing `verdict.recorded` with `decision: "defer"`).
- The exact column shape for predictor-aware filtering in `loadPageData` (single JOIN vs denormalized column on automation_runs).
- Exact threshold values for D-03 alerting.
- Whether the predictor chip shows the rule_key for regex predictions, or just `regex`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `web/lib/inngest/functions/classifier-screen-worker.ts:287-330` — the existing `emit-pipeline-event` → `emit-verdict` seam is exactly where the confidence gate goes. The LLM result is already in scope (`llmConfidence`, `llmCategoryKey`, `finalCategoryKey`, `agentRunId`). No new step ordering needed — just a conditional branch before `emit-verdict`.
- `web/lib/inngest/functions/classifier-verdict-worker.ts:45-57` — `reject` path is a no-op on Outlook side-effects (just `status=completed`). If we use a separate event for "deferred for review" rather than `decision: "reject"`, the verdict-worker stays untouched.
- `web/lib/inngest/functions/labeling-flip-cron.ts:87-122` — `evaluateMailbox` already groups via `.filter("context->>icontroller_mailbox_id", "eq", ...)`. Adding `predictor` is one more `.eq("predictor", ...)` filter and a loop over the two predictor values per mailbox.
- `web/lib/pipeline-events/types.ts:107-112` — `numericConfidence()` mapping (`high→0.9 / medium→0.7 / low→0.4`) stays; today it's display-only, and that doesn't change. The new gate reads `llmConfidence` (the enum), not the numeric.
- `web/app/(dashboard)/automations/[swarm]/review/actions.ts:131-146` — `recordVerdict`'s `agent_runs` insert is where the new `predictor` field gets written. Need a small extra query to read `pipeline_events` for this email and extract the predictor.

### Established Patterns
- **Registry-driven everything.** Per Phase 68 / SWRM-04, no `swarm_type === 'X'` literal branches. The confidence gate is per-confidence-bucket, not per-swarm — so this constraint is naturally honored.
- **Replay-safe id generation.** Per CLAUDE.md Phase 65 learning, any new UUIDs in `step.run` must be generated INSIDE the step. If D-01 introduces a new event payload with an id, follow the pattern.
- **Never destructure `inngest.send`.** Per CLAUDE.md commit dae6276 + the existing `SendFn` cast at classifier-screen-worker.ts:75-78. Any new `inngest.send` call follows the inline-cast pattern.
- **Service role for writes.** `recordVerdict` already uses `createAdminClient()` — adding the pipeline_events read for predictor extraction uses the same client.

### Integration Points
- `pipeline_events.decision_details` is the source of predictor truth (regex.matched_rule + llm_invoked are already there per Phase 70). `recordVerdict` reads this on verdict-write.
- Phase 81 `/stage-1` shell — the new filter chips (D-05) plug into the chip-strip Phase 81 is building. Need Phase 81 to land first OR explicit sequencing.
- `swarms.side_effects[]` — not touched by this phase; the stage1 cleanup dispatches in `classifier-verdict-worker.ts:130-218` fire only on `categorize_archive`, which by D-01 only fires on `high`-conf LLM (and any regex match, unchanged).

### Pitfalls flagged
- The current `pipeline_events.decision_details.regex.matched_rule` is `"no_match"` when the regex abstained — distinct from a real rule_key. `recordVerdict` must read `llm_invoked: true` to decide between `predictor='regex'` and `predictor='llm_2nd_pass'`, not assume "no_match means regex predicted no_match."
- The Phase 81 shell rename moves `review/actions.ts` to `stage-1/actions.ts`. If Phase 999.8 plans land before Phase 81 ships, the plan must hedge the path or sequence after Phase 81.
- `labeling-flip-cron.ts` uses `swarm_type="debtor-email-labeling"` (not `"debtor-email"`). The Wilson-CI stream split must respect that — `agent_runs.predictor` rows from the verdict path (swarm_type="debtor-email") must be reconciled with the cron's filter, OR the cron's filter loosens. Worth a planning-time check.

- **Backfilling historical `agent_runs.predictor`** from `pipeline_events`. Considered in D-09 and rejected. Forward-only cutover is the locked answer; backfill is available as a follow-on if the cold-start gap proves unacceptable.
