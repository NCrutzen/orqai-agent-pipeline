## Deviations from Plan

### Architectural Deviation: post-Phase-82 row composition

**[Rule 3 — Plan written pre-Phase-82]** Plan 08 was written assuming `row-strip.tsx` is the per-row composition site at `/stage-1/` and that `BudgetBreachBadge` / `TaggingFailureBadge` / `CoordinatorBadge` are mounted there. Phase 82 D-04 collapsed all per-stage row strips into the canonical `_shell/RowList` primitive (140px badge + 200px sender + 1fr subject + 150px timestamp + optional `rightEdgeSlot`). The legacy `row-strip.tsx` no longer exists at the renamed path; the surface badges (BudgetBreachBadge etc.) are unmounted dead code on the current shell.

**Action taken:** PredictorChip mounts via `_shell/RowList.rightEdgeSlot` in `Stage1ClientShell`. This is the canonical Phase 82 extension point. The chip is the first new badge to flow through `rightEdgeSlot`; PATTERNS.md's "BudgetBreachBadge / TaggingFailureBadge composition slot" guidance still applies stylistically (matching pill idiom + v7 tokens) but the mount point is the post-82 seam, not a now-extinct row-strip composition.

**Why not revive row-strip.tsx:** Phase 82 D-04's collapse decision was deliberate (single row primitive across Stages 0-4); re-introducing per-stage row strips would un-do the consolidation. The `rightEdgeSlot` extension point exists exactly to allow per-stage right-edge widgets without forking the primitive.

### Server-side derivation instead of explicit row-mapper plumbing

The plan's Task 2 action proposed populating `predictor` + `llmConfidence` inside `mapSummaryToPredictedRow` from `decisionDetails`. The summary view (`pipeline_events_email_summary`) does NOT expose `decision_details`, so a direct read from the row mapper is impossible without a second DB round-trip. Instead, derivation happens in a post-enrichment pass over `timelineMap` (the Stage 1 events already pre-fetched for the detail-pane PipelineFlow). Same observable end-state; zero extra round-trips; same hard-separation guarantee (filter `ev.stage === 1`).
