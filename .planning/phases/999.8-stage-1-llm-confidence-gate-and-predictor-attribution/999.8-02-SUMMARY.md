# Phase 999.8 Plan 02: Wave 1 — Migration + swarm_type Reconciliation + Predictor Denormalization Summary

Wave 1 lands the schema column, the load-bearing pre-existing swarm_type bug fix, and the denormalized predictor on `pipeline_events.decision_details` — all three independent of the gate logic (Plan 03) but prerequisite to waves 2-5 writing/reading `predictor` values.

## What Shipped

### 1. Migration — `agent_runs.predictor` column

File: [supabase/migrations/20260512_phase9998_agent_runs_predictor.sql](../../../../supabase/migrations/20260512_phase9998_agent_runs_predictor.sql)

```sql
alter table public.agent_runs add column if not exists predictor text;
alter table public.agent_runs drop constraint if exists agent_runs_predictor_check;
alter table public.agent_runs add constraint agent_runs_predictor_check
  check (predictor is null or predictor in ('regex', 'llm_2nd_pass'));
```

- Nullable column; CHECK enforces `predictor IN ('regex','llm_2nd_pass')`
- Idempotent DDL idiom matches `20260506_phase74_agent_runs_entity_nullable.sql`
- No index (RESEARCH §0 explicitly recommends against — low selectivity)
- Column comment documents Phase 999.8 D-07/D-09 rationale
- **NOT applied to production by this executor.** File committed only; apply runs in a separate orchestrator/verifier step or manual `mcp__supabase__apply_migration`.

Commit: `b39798e` feat(999.8-02): add agent_runs.predictor column migration

### 2. swarm_type reconciliation — `labeling-flip-cron`

File: [web/lib/inngest/functions/labeling-flip-cron.ts](../../../../web/lib/inngest/functions/labeling-flip-cron.ts) line 104

```diff
-    .eq("swarm_type", "debtor-email-labeling")
+    .eq("swarm_type", "debtor-email")
```

**Pre-flight production snapshot (2026-05-11, per Plan 02 directive 1):**

| swarm_type            | rows with `human_verdict IS NOT NULL` |
| --------------------- | ------------------------------------- |
| `debtor-email`        | **404**                               |
| `debtor-email-labeling` | **0**                               |

The cron has been a silent no-op since it was written — it filtered for a value `recordVerdict` never writes. Realignment makes the cron functional against the real production data shape.

**Why option (a) "realign cron" over option (b) "rewrite verdict-write":**
- (a) = one-line, zero-risk; touches one file and reads existing production data
- (b) = many-times-shipped server action + 404 historical row backfill + cross-swarm ripple

The `'debtor-email-labeling'` string only existed in two places in the codebase, both in `labeling-flip-cron.ts` (one read filter, one audit-row write). The audit-row write at line 123 is intentionally left at `'debtor-email-labeling'` — it's the cron's self-identification (downstream of the read filter), and Plan 06's per-predictor cron split owns that decision.

JSDoc above the query records the reconciliation rationale inline for future readers.

Commit: `44b8984` fix(999.8-02): realign labeling-flip-cron swarm_type filter to 'debtor-email'

### 3. Predictor denormalization on `pipeline_events.decision_details`

File: [web/lib/inngest/functions/classifier-screen-worker.ts](../../../../web/lib/inngest/functions/classifier-screen-worker.ts) line 311

Added one field to the existing `emit-pipeline-event` step's `decision_details` JSON:

```typescript
predictor: llmInvoked ? "llm_2nd_pass" : "regex",
```

Every new Stage 1 `pipeline_events` row now carries `decision_details->>predictor`, which the UI chip-strip filter (Plan 07, D-05/D-11) reads without a JOIN to `agent_runs`. Derivation mirrors `recordVerdict` (Plan 05) — both paths consume the same `llm_invoked` flag, so the two writers cannot diverge.

This is **not** a gate-logic change. `emit-verdict` still fires unconditionally; Plan 03 owns the high-confidence gate. The change is purely additive on the jsonb payload.

Commit: `e3e2646` feat(999.8-02): denormalize predictor onto pipeline_events.decision_details
