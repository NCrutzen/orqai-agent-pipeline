## Exact Gate Predicate (D-01)

`web/lib/inngest/functions/classifier-screen-worker.ts` Step 5:

```typescript
const gateClearsForAutoArchive =
  !llmInvoked ||
  llmConfidence === "high" ||
  finalCategoryKey === "unknown";

if (gateClearsForAutoArchive) {
  await step.run("emit-verdict", async () =>
    (inngest.send as unknown as SendFn)({
      name: "classifier/verdict.recorded",
      data: { automation_run_id, swarm_type, decision: "approve", message_id,
              source_mailbox, predicted_category: finalCategoryKey,
              override_category: null },
    }),
  );
} else {
  await step.run("emit-requires-review", async () =>
    (inngest.send as unknown as SendFn)({
      name: "classifier/screen.requires_review",
      data: { automation_run_id, agent_run_id: agentRunId, email_id,
              message_id, source_mailbox, swarm_type, entity: entity ?? null,
              llm_category_key: llmCategoryKey ?? finalCategoryKey,
              llm_confidence: llmConfidence as "medium" | "low",
              final_category_key: finalCategoryKey },
    }),
  );
}
```

## Unknown → reject Label-Only-Skip Path Preserved

When the LLM returns `category_key='unknown'` (at any confidence), the gate clears via the third disjunct. The downstream verdict-worker dispatches on `swarm_noise_categories.action`, and the 'unknown' seed row's `action='manual_review'` is the existing label-only-skip path — no Outlook label is applied, no archive happens. This pre-Plan-03 escape valve is intentionally untouched.

## emit-pipeline-event Unconditional

The Step 4 `emit-pipeline-event` step runs on every branch (gate-cleared OR gate-blocked). This means:

## Deviations from Plan

**[Rule 1 - Bug] Removed in-LLM-step `low → unknown` coercion** (planner expected this to be done; framed as superseding, not deviation)

- **Found during:** Task 2 verification (one gate test still RED after the first emit-branch edit).
- **Issue:** The prior Phase 74 D-10 coercion at `classifier-screen-worker.ts` line 201-202 (`finalKey = parsed.confidence === "low" ? "unknown" : parsed.category_key`) interacted badly with the new gate: when the LLM returned `(category='auto_reply', confidence='low')`, `finalCategoryKey` would be coerced to `"unknown"`, the third disjunct of `gateClearsForAutoArchive` would fire, and the verdict path would be taken — exactly what the gate is supposed to prevent.
- **Fix:** Replaced the ternary with `const finalKey = parsed.category_key;` so the predicted category survives into Step 5 where the new gate handles the confidence routing. Added a comment explaining the supersession.
- **Files modified:** `web/lib/inngest/functions/classifier-screen-worker.ts` (LLM step inside `step.run("llm-call", ...)`).
- **Commit:** `9e6abe7` (rolled into the Task 2 commit).
- **Knock-on:** Two pre-existing REQ-4 tests asserted the old coercion (`predicted_category='unknown'` on low). I updated their assertions to the new D-01 contract (`llm_confidence='low'` on `classifier/screen.requires_review`). The tests still describe the same code path; the contract they enforce is the new one.

**No other deviations.** Plan executed as written; the gate predicate matches the planner's `<interfaces>` block verbatim, the JSDoc cites D-01/D-10 + Therese Hendriks reference per the action spec, branches wrapped in their own `step.run`, inline `SendFn` cast used (no destructuring).
