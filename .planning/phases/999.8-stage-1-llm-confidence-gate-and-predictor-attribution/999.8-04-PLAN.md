<objective>
Re-tune the `stage-1-category-classifier` Orq agent's instructions + few-shots so that `"high"` means ~95%+ certainty and any ambiguity downgrades to `"medium"` or `"low"`. The code gate in Plan 03 enforces the boundary; this plan defines what the boundary IS.

Purpose: Without re-calibration, the gate alone won't help — if the LLM still produces "high" liberally, the gate becomes a no-op for the most dangerous cases. The asymmetric cost framing in the prompt (auto-archive vs queued for free) is the load-bearing instruction.

Output: Updated Orq agent (PATCH via MCP) + a SQL migration that records the prompt diff for version-control / audit (Phase 74 analog).

Runs in parallel with Plan 03 — both are Wave 2 and touch disjoint files (Plan 03: events.ts + classifier-screen-worker.ts; Plan 04: Orq agent + new migration file).
</objective>

Prompt addendum (RESEARCH §2 proposed; planner refines):
```xml
<confidence_calibration>
Choose confidence using THIS calibration:

- "high": You are ~95%+ certain. Signals MUST be unambiguous. If a reasonable
  human reviewer would need to read more than the subject + first 200 chars
  of body to agree with you, the confidence is NOT high.

- "medium": Signals present but not overwhelming. THE DEFAULT FOR ANY
  AMBIGUITY IS MEDIUM, not high.

- "low": You are guessing.

ASYMMETRIC COST: a wrong "high" prediction auto-archives the email and may
cause downstream financial damage. A "medium" or "low" prediction is sent
to a human reviewer at zero cost. WHEN IN DOUBT, downgrade.
</confidence_calibration>
```
