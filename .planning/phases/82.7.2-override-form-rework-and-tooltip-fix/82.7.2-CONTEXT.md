# Phase 82.7.2: Override form rework + tooltip fix + brand-color data audit

## Why this phase exists

Operator UAT of Phase 82.7.1 on 2026-05-18 surfaced 4 new defects across 3 of
the 5 closed E-IDs. The 82.7.1 fixes worked structurally (per-stage Submit
exists, button colors changed, em-dash placeholder added) but the resulting UX
needs further iteration based on operator-observed reality:

- **E-03 partial fail** — brand-color tooltip doesn't render on hover, AND only
  3 out of ~25 rows show any colored dot at all.
- **E-04 needs rework** — the override form is "clunky with a lot of text and
  colors." 82.7.1's polish addressed button styling but didn't restructure the
  surrounding text/state messaging.
- **E-05 was misunderstood at spec time** — operator does not understand what
  E-05 was for. Looking at current UAT screenshots, all rows render with chips
  on both Stage 0 and Stage 1 tabs. Either the original observation was
  off-screen/transient, OR Plan 05's defensive em-dash placeholder is working
  invisibly. Need to confirm whether this was a real defect.

E-01 (per-stage Submit) and E-02 (row fade-out) — no operator pushback.
Assumed working.

## Defect inventory

### F-01 — Brand-color swatch tooltip doesn't render on hover

**Observed:** Hovering the colored dot on a Stage 1 noise-tab row produces no
tooltip. The Radix Tooltip added in 82.7.1-03 (`predicted-row.tsx:102-120`) is
silent.

**Expected per 82.7.1 D-06:** Hover reveals tooltip with brand display name
("Fire Control", "Smeba", "Berki", "Sicli-Noord", "Sicli-Sud").

**Investigation needed:**
1. Is `TooltipProvider` wrapping the row, or only the individual swatch? Radix
   Tooltip requires a Provider somewhere up the tree.
2. Is the swatch element receiving pointer events at all? The 6×6px hit target
   may be too small or covered by a sibling.
3. Did the brand-color file refactor (Plan 03 adding `brandDisplayName()`)
   accidentally bypass the swatch render path?
4. Is the tooltip rendering but invisible (z-index, portal mount issue)?

**Resolution path:** Trace the Radix Tooltip render → verify Provider →
verify trigger element receives hover → verify Content renders in portal.
Likely a missing `TooltipProvider` wrapping at the `PredictedRow` parent.

### F-02 — Only 3 of ~25 rows show a colored brand dot

**Observed (image 12):** On the Stage 1 noise tab with 25+ rows visible, only
3 rows display a colored dot (one green, one green/yellow, one orange).
Remaining rows show no swatch at all.

**Expected:** Every row should show a brand-color dot (one of the 5 Benelux
brands per `project_brand_scope.md`).

**Root cause hypotheses (data-side):**
1. (a) `entity_brand` column is null/missing for most rows. Only rows where
   Stage 1 successfully attributed an entity get a brand.
2. (b) Rows belong to brands NOT in the 5-brand registry — the renderer
   correctly skips unknown brands and returns nothing.
3. (c) `brandColorToken()` returns a transparent / no-op value for some valid
   brands, hiding the swatch.

**Note:** This is partly orthogonal to F-01 (tooltip fix) — even with tooltip
working, if 22 of 25 rows have no swatch, hover would still produce nothing
on those rows.

**Resolution path:**
1. Query a sample of recent Stage 1 rows from `automation_runs` /
   `pipeline_events`: how many have `entity_brand` populated?
2. For populated rows, what's the distribution of values? How many match the
   5-brand registry?
3. If most rows have `entity_brand = null`: this is a Stage 1 enrichment gap,
   not a UI gap. Fix the swatch to show a neutral muted dot (or em-dash chip
   like Plan 05) so the visual column always has SOMETHING.
4. If most rows have brands not in the registry: expand the registry OR add a
   fallback "Other" muted swatch.

### F-03 — Override form needs structural rework (E-04 reopened)

**Operator-observed (image 13):** The form is "clunky with a lot of text and
colors". Specific list:

1. **Consolidate Primary and Secondary Actions:** Move `cancel override` link
   immediately adjacent to the green `Submit override` button. Currently they
   are visually separated. Grouping makes the final decision easier.

2. **Remove Redundant Labels:** The orange text "Submit Stage 1 override" is
   redundant with the green button labeled "Submit override". Delete the text
   label.

3. **Fix Color Semantics:** The orange/amber color used for helper text and
   "Override in progress" status reads as a warning. Switch to muted gray /
   secondary text color so operators don't think something is wrong.

4. **Clarify State Messaging:** Remove "Override in progress" entirely — the
   presence of the active dropdown + Submit/Cancel buttons already makes the
   editing state obvious.

5. **Refine Helper Text:** "↓ Override + note save together" is awkwardly
   phrased and the arrow adds confusion. Rephrase to "Note and override are
   saved together" (or similar) as a subtle muted hint directly under the
   dropdown.

**Why 82.7.1 didn't solve this:** Plan 04 (E-04) was scoped narrowly to button
fills + spacing. It left all surrounding text/state messaging untouched. The
operator's frustration is with the OVERALL composition, not the buttons in
isolation.

**Resolution direction (locked):** Apply all 5 operator-listed changes
verbatim. Wireframe target:

```
┌──────────────────────────────────────┐
│ 1  Stage 1 — Category                │
│                                      │
│ [ payment_admittance        ▾ ]      │
│                                      │
│ Note and override are saved together │  ← muted helper text
│                                      │
│ [Cancel override]  [Submit override] │  ← grouped, lime + ghost
└──────────────────────────────────────┘
```

(No "↓" arrow, no "Submit Stage 1 override" label duplicate, no "Override in
progress" message, no orange/amber on text.)

### F-04 — E-05 retrospective: clarify intent or close as no-op

**Operator (2026-05-18):** "Item 5 I don't understand."

**Context:** E-05 was logged in 82.7.1 from the 2026-05-16 UAT round
("not all rows are labelled or one has on label"). Plan 05 added an em-dash
placeholder to StageBadge so any null/empty chip would render a visible `—`.

**Current state (image 11 + 12):** Every visible row on both Stage 0 Safety
and Stage 1 Noise tabs has a chip. No em-dashes visible. Two possible
interpretations:
1. The original observation was off-screen/transient. Plan 05 is defensive but
   the bug never existed in normal traffic.
2. Plan 05 IS firing on some rows but they happen to be off-screen in the
   current screenshots.

**Resolution path:** Quick log-trace — search for any production logs that
would have shown a null-chip row in the past 7 days. If zero, document Plan 05
as a defensive guard with no observed live trigger and close. If some, capture
which rows and confirm the placeholder is showing correctly.

**Do NOT revert Plan 05** — defensive placeholders are cheap and have no
downside even if never triggered.

## Implementation Decisions (locked 2026-05-18)

### F-01 — Brand-color tooltip not firing

- **G-01:** Investigate-then-fix. Trace the Radix Tooltip render path in
  `predicted-row.tsx:102-120`: verify `TooltipProvider` is mounted in the
  ancestor tree (likely missing — Radix requires Provider somewhere up the
  tree); verify swatch element receives pointer events; verify Content
  renders in the portal. Most likely fix: wrap the row (or the swatch span)
  in a local `<TooltipProvider>`, OR confirm the noise-view row layout has
  Provider coverage. Note: F-02 changes the swatch render logic, so F-01
  + F-02 should land together (or F-02 first, then F-01 on whatever swatch
  shape survives).

### F-02 — Only 3 of ~25 rows show a colored dot

- **G-02:** Investigate first, then decide. Plan with a single
  data-investigation task BEFORE any UI change. Query a 7-day sample of
  Stage 1 rows (`automation_runs` joined to whatever holds `entity_brand`,
  per the codebase):
  - What % of rows have `entity_brand` populated?
  - For populated rows, what's the distribution of values?
  - How many values match the 5-brand registry (Smeba, Smeba-Fire, Berki,
    Sicli-Noord, Sicli-Sud)?
- **G-03:** Decision branching from the query result, documented in CONTEXT
  but executed in a follow-on plan only after the data lands:
  - If most rows have `entity_brand = null` → escalate as a Stage 1
    enrichment gap (separate phase). Ship a temporary neutral-dot UI fallback
    so the column isn't empty while the enrichment is fixed.
  - If most rows have brands NOT in the 5-brand registry → ask Nick whether
    to expand the registry (`iccafe`, `iccafe-france`, others per
    `project_brand_scope` memory) or add an "Other" neutral fallback.
  - If renderer bug (rare) → fix the renderer.
- **G-04:** Do NOT pre-emptively ship the neutral-dot fallback. Wait for
  query result. (Could change to "always-fallback" if query shows the
  enrichment fix won't happen this milestone.)

### F-03 — Override form structural rework

Locked verbatim from operator UAT 2026-05-18 (5 items, all applied):

- **G-05:** **Consolidate primary + secondary actions.** Move the
  `cancel override` link so it sits immediately adjacent to the green
  `Submit override` button. Same horizontal row. Cancel on the left (ghost
  outline, no fill), Submit on the right (brand-secondary lime fill from
  82.7.1-04). The `⌘⏎` keychip stays on Submit's right edge.

- **G-06:** **Remove redundant "Submit Stage 1 override" text label.** The
  orange/amber line that reads "Submit Stage 1 override" disappears — the
  green button labeled "Submit override" carries the full meaning. Single
  source of truth for the action verb.

- **G-07:** **Fix color semantics — orange/amber → muted gray on
  non-warning text.** Helper text and status messages currently use
  `var(--v7-amber)` or similar, which reads as warning/error. Switch to
  `var(--v7-muted)` (or whichever muted-secondary token the design system
  exposes) for all non-warning text inside the override form. Lime stays
  on the Submit button. Cancel stays brand-secondary outline.

- **G-08:** **Remove "Override in progress" status messaging entirely.**
  The presence of the active dropdown + Submit/Cancel buttons already makes
  the editing state obvious. Delete the line. No replacement.

- **G-09:** **Refine helper text "↓ Override + note save together".**
  Rewrite as a muted single-line hint directly under the dropdown (or above
  the action row — placement is Claude's discretion). Suggested copy:
  `Note and override are saved together`. No leading arrow glyph. No
  orange. Use `var(--v7-muted)` text color.
