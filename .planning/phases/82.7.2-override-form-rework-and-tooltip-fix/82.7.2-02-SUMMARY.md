## What changed in `stage-1/page.tsx`

Single insertion (~48 lines) between the `summaryRows.map(mapSummaryToPredictedRow)` call (L755) and the existing `mailbox_id` hydration block (L761-770).

Added a guarded `if (rows.length > 0)` block that:

1. Collects `entityRowIds = rows.map(r => r.id)` (= current page's `email_id` list, bounded by `PAGE_SIZE`).
2. One-shot `pipeline_events` query: `select("email_id, decision_details")` `.eq("swarm_type", swarmType)` `.eq("stage", 1)` `.in("email_id", entityRowIds)`.
3. Builds `entityByEmail: Map<string, string>` from `decision_details.entity` (only first match per email_id to be deterministic when Stage 1 re-emits).
4. Re-maps `rows` to `{ ...r, entity: entityByEmail.get(r.id) ?? r.entity }`.

`mapSummaryToPredictedRow` itself is unchanged — `entity: null` still
ships out of the mapper, then gets hydrated by the new block. This was
chosen over changing the mapper signature so `SummaryRow` shape and
`mapSummaryToPredictedRow` purity remain stable, and the diff for
future readers mirrors the analogous `mailbox_id` hydration block
immediately below it.

`mapEventToPredictedRow` (safety tab), `predicted-row.tsx`,
`brand-color.ts`, and all test files are untouched per plan
acceptance criteria.

## Out-of-scope flag: entity_brand swatch wiring

Per CONTEXT G-03 / G-04 and `project_brand_scope.md` (5 Benelux brands
locked):

- `fire-control` (109 / 275 rows = 40 %) is **NOT** in `brand-color.ts`
  `BRAND_COLOR_TOKEN`. After this plan ships, `fire-control` rows still
  render the muted-fallback swatch and the tooltip still reads
  "Unknown brand".
- The plan deliberately does NOT wire `PredictedRow.entity_brand` from
  the same `decision_details.entity` value, because that would surface
  lime/pink/amber dots for 3 brands but leave 40 % of rows muted — a
  partial fix that would read as still-broken to operators.
- Decision required from Nick: either expand the registry to include
  `fire-control` (one row in `BRAND_COLOR_TOKEN` + one in
  `BRAND_DISPLAY_NAME`, plus a CSS token choice), or add a generic
  "Other" muted swatch as the intended UX. Captured in F-02 findings,
  not addressed here.

## Does Plan 03 (F-01 tooltip) need to wait?

**No.** Plan 03 (Radix Tooltip wiring on the brand-color swatch) can
proceed immediately. The tooltip fix is orthogonal — it works on any
swatch, including the muted-fallback one. Operators will see the
tooltip fire on every row after Plan 03 lands; on the 165 rows with
mapped brands the tooltip will read the brand name once Plan 03 also
wires `entity_brand` from `row.entity` (a one-liner the F-01 planner
can choose to bundle, since `row.entity` is now populated by this
plan).
