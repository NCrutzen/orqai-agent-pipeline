## F-01 — Brand-color tooltip render trace

### Root cause (Investigation result, not just hypothesis)

`TooltipProvider` is **NOT missing**. The root layout already mounts one
globally:

**`web/app/layout.tsx` L4, L56:**
```typescript
import { TooltipProvider } from "@/components/ui/tooltip";
// ...
<TooltipProvider>{children}</TooltipProvider>
```

So the inner `<TooltipProvider>` at `predicted-row.tsx:102` is redundant
(nested Providers are legal — Radix tolerates them). Provider coverage
is fine.

Sibling proof that Radix Tooltip works in this exact file: the ↻
overridden-glyph tooltip at `predicted-row.tsx:194-211` uses the same
pattern (`TooltipProvider` → `Tooltip` → `TooltipTrigger asChild` →
`TooltipContent`) and per Plan 03 verification has been functional. The
↻ tooltip wraps a `<span>` of `font-size: 14px` (L200). The brand-color
swatch tooltip wraps a `<span>` of `width: 6, height: 6` (L110-111).

**Most likely real cause** (planner should verify with one DOM probe
before fixing): the 6×6 swatch's hover hit-target is too small AND/OR
the parent `<button>` element at L74-94 (the entire row, with
`onClick={onSelect}`) swallows pointer events so Radix never sees a
`pointerenter` on the swatch span. The ↻ glyph at 14×14 is 5× the hit
area and lives in the same parent button — but Radix `asChild` may
detect ancestor button semantics differently for inline spans inside
buttons (focus + hover propagation rules differ).

**Secondary hypothesis** (only if (1) is disproved): `TooltipContent`
renders via `TooltipPrimitive.Portal` (see `web/components/ui/tooltip.tsx`
L40) — z-index/portal collision with the detail pane overlay could hide
it. Default `sideOffset = 0` (`tooltip.tsx` L36) means the content sits
flush against the swatch — operators might see a 1-pixel sliver they
miss.

## F-02 — entity_brand coverage audit (READ-ONLY data plan)

### Root-cause finding from upstream code inspection (do not skip this)

**The "only 3 of 25 rows show a dot" is almost certainly NOT a Stage 1
enrichment gap. It is a UI bug: `entity_brand` is never set on the
PredictedRow shape that the noise tab renders.**

Evidence:

1. `swarms.entity_brand` is a **swarm-level registry**, not a per-email
   column. Migration `20260504b_swarms_registry_generalisation.sql:15`:
   ```sql
   add column if not exists entity_brand               jsonb;
   ```
   And `20260505a_entity_brand_expansion.sql` expands it into an array of
   per-brand JSONB objects (code/display_name/register_language/...) for
   the swarm. It enumerates which brands the swarm handles — not which
   brand a given email belongs to.

2. The view `pipeline_events_email_summary` does NOT select
   `entity_brand`. Confirmed at `stage-1/page.tsx` L662-668:
   ```typescript
   const listQuery = (admin.from("pipeline_events_email_summary") ...)
     .select(
       "email_id, swarm_type, subject, sender_email, sender_name, recipient_mailbox, email_received_at, " +
         "stage_0_decision, stage_1_decision, stage_2_decision, stage_3_decision, stage_4_decision, " +
         "stage_1_overridden, stage_2_overridden, stage_3_overridden, stage_4_overridden, " +
         "total_cost_cents, tool_call_count, first_event_at, last_event_at",
     )
   ```
   No `entity_brand` projected.

3. `mapSummaryToPredictedRow` (page.tsx L606-648) returns a `PredictedRow`
   with `entity: null` (L614) and **no `entity_brand` key at all**. The
   `SummaryRow` interface in `predicted-row.tsx` L28 declares
   `entity_brand: string | null`, but nothing assigns to it on the
   noise-tab path.

4. So `row.entity_brand` is `undefined` for every noise-tab row →
   `brandColorToken(undefined)` returns `FALLBACK_TOKEN = "--v7-muted"`
   (brand-color.ts L17, L25) → swatch background is muted gray on every
   row.

5. The 3 colored dots the operator saw are probably (a) the safety-tab
   path via `mapEventToPredictedRow` which reads
   `details.entity` (L442) — only populated when Stage 1 extracted an
   entity into `decision_details`, or (b) rows where a per-stage entity
   resolver wrote into `decision_details.entity` and that string happens
   to match one of the 5 registry codes.
