Important: the swatch reads `row.entity_brand`, not `row.entity`. The
`SummaryRow` interface (L513-535) has NO `entity_brand` field, and
the `pipeline_events_email_summary` SELECT (L662-668) does not project
one. PATTERNS.md Q4 query confirms whether the view exposes that
column at all.

**Implementation note for Task 2 (conditional on findings):** the
cleanest mapper change is to populate `entity` from `decision_details`
the same way the safety mapper does, AND have the SummaryRow JOIN-back
to raw `pipeline_events` for Stage 1 emits to fetch
`decision_details.entity`. The Task 2 design depends on Q4 result:

- If `pipeline_events_email_summary` view DOES expose an `entity_brand`
  column (Q4 lists it) → extend the view-SELECT to include it +
  assign `entity_brand: row.entity_brand` on the mapper. Simplest fix.
- If the view does NOT expose `entity_brand` (Q4 omits it) → fall back
  to JOIN-back: for the email_ids in the current page, query
  `pipeline_events` where stage=1 for `decision_details->>entity`,
  build a Map<email_id, entity>, populate `entity` on the mapper, and
  let `entity_brand` stay nulled out until a downstream enrichment
  surfaces brand-per-email. **In this branch, defer the swatch wiring
  to a follow-up phase** — document in findings, do not ship a partial
  fix.
- If Q2 shows entity is null on >50% of rows → escalation (G-03 branch
  1). Document in findings + skip the mapper change entirely; the
  existing fallback-to-muted behavior is the correct interim state.
