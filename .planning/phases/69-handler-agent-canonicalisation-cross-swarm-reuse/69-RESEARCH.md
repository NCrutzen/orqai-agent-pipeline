### Pattern 1: Registry-driven loader with Map cache
**What:** Mirror the established `loadSwarmCategories` pattern.
**When to use:** Read-heavy, mutation-rare registry tables.
**Example:**
```typescript
// Source: web/lib/swarms/registry.ts (existing pattern, lines 17, 48-70)
const ENTITY_BRAND_CACHE = new Map<string, { value: BrandRegister[]; expires: number }>();
const TTL_MS = 60_000;

export async function loadAllBrandRegisters(
  admin: SupabaseClient,
  swarmType: string,
): Promise<BrandRegister[]> {
  const now = Date.now();
  const hit = ENTITY_BRAND_CACHE.get(swarmType);
  if (hit && hit.expires > now) return hit.value;

  const swarm = await loadSwarm(admin, swarmType); // already TTL-cached
  const raw = swarm?.entity_brand;
  if (!Array.isArray(raw)) {
    if (hit) return hit.value;
    return [];
  }
  // Phase 69 invariant: entries are objects (post D-01 migration). If we
  // see strings, the migration hasn't run — throw rather than degrade.
  if (raw.length > 0 && typeof raw[0] !== "object") {
    throw new Error(
      `swarms.entity_brand for swarm_type='${swarmType}' is still string-shape; run migration 20260505a`,
    );
  }
  const value = (raw as BrandRegister[]);
  ENTITY_BRAND_CACHE.set(swarmType, { value, expires: now + TTL_MS });
  return value;
}

export async function loadBrandRegister(
  admin: SupabaseClient,
  swarmType: string,
  brandCode: string,
): Promise<BrandRegister> {
  const all = await loadAllBrandRegisters(admin, swarmType);
  const found = all.find((b) => b.code === brandCode);
  if (!found) {
    throw new Error(
      `brand_register lookup failed: swarm_type='${swarmType}' brand_code='${brandCode}' not in registry`,
    );
  }
  return found;
}
```

### Pattern 2: Build-time codegen from Supabase jsonb
**What:** Read DB at build-time, emit TS literal-union.
**When to use:** Strict typing AND data-driven onboarding both required.
**Example:**
```typescript
// Source: scripts/gen-entity-types.ts (NEW — pattern adapted from
// supabase/codegen.ts conventions seen in similar projects)
import { createClient } from "@supabase/supabase-js";
import { writeFileSync } from "node:fs";
import { resolve } from "node:path";

const SUPABASE_URL = process.env.SUPABASE_URL!;
const SERVICE_KEY = process.env.SUPABASE_SERVICE_ROLE_KEY!;
const SWARM_TYPE = "debtor-email";
const OUT_PATH = resolve(
  __dirname,
  "../web/lib/automations/debtor-email/coordinator/entity.generated.ts",
);

const supa = createClient(SUPABASE_URL, SERVICE_KEY);
const { data, error } = await supa
  .from("swarms")
  .select("entity_brand")
  .eq("swarm_type", SWARM_TYPE)
  .single();
if (error) throw error;

const brands = (data.entity_brand as Array<{ code: string }>) ?? [];
const codes = brands.map((b) => b.code).sort(); // sort → stable diffs
const union = codes.map((c) => `"${c}"`).join(" | ");

const content = `// AUTO-GENERATED by scripts/gen-entity-types.ts. DO NOT EDIT.
// Source: public.swarms.entity_brand for swarm_type='${SWARM_TYPE}'.
// Run: npm run codegen
// Brands: ${codes.length} (${codes.join(", ")})
export const ENTITY_BRANDS = [${codes.map((c) => `"${c}"`).join(", ")}] as const;
export type Entity = ${union};
`;
writeFileSync(OUT_PATH, content);
console.log(`Wrote ${OUT_PATH} with ${codes.length} brands.`);
```

### Anti-Patterns to Avoid

- **Hand-rolling JSON-Schema string→union conversion at runtime.** The `Entity` type must come from a build-time codegen, not a runtime dynamic type. Runtime "types" don't survive `tsc`.
- **Inserting `loadBrandRegister(...)` outside `step.run` boundaries.** Inngest replays will re-run the lookup; idempotent today (registry is read-only) but adds latency. Wrap inside the same `step.run("generate-body")` that already exists at line 255.
- **Falling back from registry to a hardcoded brand list when the registry read fails.** Per Phase 68 D-12 precedent: throw a structured error so misconfiguration surfaces in Bulk Review. Defensive fallback creates slow drift.
- **Calling Orq's `/v2/agents/{id}` GET endpoint with the invocation API key.** Returns 403 (verified live 2026-05-04). Use `mcp__orqai-mcp__get_agent` from the orchestrator/execute-phase agent context, which has management-key access.
- **Using `email_entity` in any new code path.** D-04 removes it; the canonical field is `entity_brand` (string code) plus `brand_register` (object). Grep audit at Wave 6: `grep -rn "email_entity" web/lib/` should return zero hits in production code (test fixtures may reference for backwards-compat verification only).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| DB-row → TS literal union | Runtime type derivation, AST manipulation, JSON-Schema-to-TS converters at runtime | Build-time `tsx` script that writes a static `.generated.ts` file (Pattern 2 above) | TS strict types must exist at compile time. Runtime types are fiction. |
| Brand metadata caching | Custom Redis layer, Supabase Realtime listener | Module-level `Map` with 60s TTL (Pattern 1; Phase 68 D-15 precedent) | Brand list mutates ~once per quarter; cold-start invalidation is sufficient. |
| Orq prompt versioning | A new `orq_agent_prompt_versions` table, custom diff tooling | Bump `body_version` literal in 5 places (per `Pitfall §"Version-bump fragmentation"`) + git history of `Agents/.../debtor-copy-document-body-agent.md` | Existing CLAUDE.md discipline suffices. The agent.md file IS the audit trail. |
| Test fixtures for multi-brand assertion | Custom snapshot framework | vitest `describe.each` over the 10 fixture rows + per-row assertions on body_html structure | vitest's `each` already does this idiomatically; no new framework. |
| Live-vs-mock discrepancy detection | A custom replay harness | One vitest test per smoke fixture marked `it.runIf(process.env.LIVE_SMOKE === "1")` calling real Orq via existing `invokeOrqAgent` | Phase 65 dae6276 learning encoded; gating on env var keeps CI fast and local-dev unblocked. |

**Key insight:** Phase 69's value is *removing* hardcoding, not *adding* abstraction. Every new helper is the minimum needed for the registry read; do not introduce a "BrandService" class or a generic-template renderer. The Orq prompt itself is the renderer.

## Common Pitfalls

### Pitfall 1: Version-bump fragmentation
**What goes wrong:** `BODY_VERSION` lives in 5 places that must change atomically. Forgetting one breaks idempotency invariants and post-validators reject every output.
**Why it happens:** No single source-of-truth file; the literal string is duplicated across runtime + Orq + spec + few-shots.
**How to avoid:** Wave 4 must update ALL five in lockstep:
1. `web/lib/automations/debtor-email/coordinator/types.ts` — `BODY_VERSION` const
2. Orq agent prompt's `<body_version>2026-05-04.v2</body_version>` literal
3. Orq agent's `response_format.json_schema.schema.properties.body_version.const`
4. Orq agent's `variables.body_version.const`
5. All 6 few-shot examples' footer text (`body_version: 2026-05-04.v2`)
6. (Sixth and seventh, if counting): the `Agents/.../debtor-copy-document-body-agent.md` frontmatter `version:` and the file-level `body_version:` literal.
**Warning signs:** Post-validator §7 rejection on first live invocation = mismatch detected.

### Pitfall 2: Live Orq divergence from spec file
**What goes wrong:** The `Agents/.../debtor-copy-document-body-agent.md` is *supposed* to be the authoritative paste-source for Orq Studio's "Instructions" field, but Orq Studio is editable directly. Operators (or earlier Claude sessions) may have edited the live prompt without updating the spec.
**Why it happens:** Orq has no enforcement linking the two. CLAUDE.md learning `cba7352b-4feb-4d11-94f8-0ebd24f15cd0` documents that `create_agent` silently drops `response_format`; similar drift can happen on prompt content.
**How to avoid:** Wave 4 step 1 = `mcp__orqai-mcp__get_agent` BEFORE PATCH, capture verbatim to `orq-baseline-prompt.txt`, diff against spec. Material drift = pause and reconcile before applying the new prompt.
**Warning signs:** A `get_agent` response whose prompt body doesn't byte-match the spec's `<instructions>` block.

### Pitfall 3: `inferLanguageFromEntity` orphan callers
**What goes wrong:** D-13 says one call site (`classifier-invoice-copy-handler.ts:249`). Grep audit (verified 2026-05-04) confirms ONLY one production call site. But Phase 69 also touches `agent-runs.ts` (line 15: `entity: Entity`) and `classifier-spotcheck-sampler.ts` (FALLBACK_ENTITY = "smeba"). If those aren't migrated, the codegen-removed `Entity` type breaks tsc.
**Why it happens:** The original Phase 56 author hardcoded `Entity` across multiple files; CONTEXT.md D-13 only enumerates the body-agent invocation site.
**How to avoid:** Wave 0 audit task: `grep -rn "from .*coordinator/types.*Entity\b" web/lib/` and migrate ALL imports to the new `entity.generated.ts` path (or fix the remaining call sites to use `string`).
**Warning signs:** `tsc --noEmit` failure post-Wave 2 with "Cannot find name 'Entity'" or "Type 'string' is not assignable to type 'Entity'" errors.

### Pitfall 4: Stage 2 resolver does NOT yet populate `entity_brand` into a PipelineStageContext
**What goes wrong:** CONTEXT.md D-12 says "Stage 2 resolver populates `entity_brand` (string code) into `PipelineStageContext`". Verified via reading `web/lib/automations/debtor-email/resolve-debtor.ts`: the resolver's `ResolveResult` interface has only `customer_account_id`, `customer_name`, `confidence`, `method` — NO `entity_brand` field. The brand is read from `debtor.labeling_settings.entity` per-mailbox in `classifier-invoice-copy-handler.ts:163`, NOT from a PipelineStageContext.
**Why it happens:** Phase 68 D-05 declared the canonical_context_shape including `entity_brand` but Phase 68 did NOT wire the resolver to emit it (Phase 68 was registry-shape work; runtime emit was deferred).
**How to avoid:** Phase 69 must NOT fix this in scope. The handler reads `entity` from `labeling_settings` (existing path) and uses it as the `brand_code` argument to `loadBrandRegister`. The PipelineStageContext "shape" is conceptual at this stage; Phase 70 (TELE-01 `pipeline_events`) is when the runtime envelope materialises.
**FLAG FOR PLANNER:** This is a *legitimate scope clarification* — CONTEXT.md D-12 should be read as "the agent input shape conforms to PipelineStageContext field names", not "Stage 2 emits a PipelineStageContext object". The plumbing detail (settings → handler → loadBrandRegister) stays as today; only the agent input *naming* and *shape* changes.

### Pitfall 5: Replay-id risk in `loadBrandRegister`
**What goes wrong:** CLAUDE.md Phase 65 learning `dd2583a` warns that non-deterministic ID generation outside `step.run` causes replay corruption. `loadBrandRegister` is async and reads from Supabase — if called inside a `step.run`, replays re-run the lookup (acceptable, registry is idempotent). But if its result is captured into a closure variable USED LATER inside a *different* `step.run`, replay re-runs the outer scope and the second step.run sees a fresh fetch result.
**Why it happens:** It's tempting to hoist `const brandReg = await loadBrandRegister(...)` to function top-level for re-use across draft/audit steps.
**How to avoid:** Call `loadBrandRegister` INSIDE the `step.run("generate-body")` block at line 255 specifically. Pass through the necessary fields (display_name, code) into subsequent `step.run` blocks via the data flow that already exists (the `body` result object). Do NOT close over `brandReg` from the outer scope.
**Warning signs:** Inngest replay test produces different log output between first run and second run.

### Pitfall 6: Sales-stub fixture leaks into production DB
**What goes wrong:** D-Discretion-5 recommends teardown — but if a test crashes mid-run, the synthetic `sales-email-stub` row may persist.
**How to avoid:** Test setup uses a transaction-scoped admin client OR a `beforeEach`/`afterEach` pair with explicit DELETE. If using the same Supabase project as production (highly likely — there's no test-DB seam in this repo), prefer a UUID-suffixed swarm_type (`sales-email-stub-${randomUUID()}`) so accidental survivals don't collide with the real Phase 73 row.
**Warning signs:** `SELECT * FROM swarms WHERE swarm_type LIKE 'sales-email-stub%'` returns >1 row outside test runtime.

### Pitfall 7: `mistral/mistral-large-latest` in fallback chain
**What goes wrong:** The agent.md spec line 36 lists `mistral/mistral-large-latest` as fallback #4. CLAUDE.md explicitly forbids that ID — it does NOT exist in Orq's catalog. Verified at the CLAUDE.md "Anthropic model routing" section: use `mistral/mistral-large-2411` (dated pin).
**How to avoid:** Wave 4 step 1 = `list_models`. If the live agent has `mistral-large-latest` in its model fallback chain, replace with `mistral/mistral-large-2411` as part of the same PATCH. Audit finding: this is pre-existing tech debt, but Wave 4 is the right moment to fix it because a model-config PATCH is happening anyway.
**Warning signs:** `list_models` doesn't return `mistral/mistral-large-latest`. Studio's Model dropdown for the agent appears empty (per CLAUDE.md learning `f980a2a1`).
