<decisions>

## Implementation Decisions

### D-01 — Rename event names too, stage-prefixed

Function ids AND event names move together. Producers (e.g. coordinator dispatching to Stage 2) update their `inngest.send({ name: "..." })` calls. Operator dashboards then show consistent stage-prefixed vocabulary end-to-end.

New event name shape: `debtor-email/stage-{N}.{verb}.requested` (e.g. `debtor-email/stage-2.customer-resolve.requested`). This matches the existing `debtor-email/retro-classify.requested` shape from Phase 87.

### D-02 — Single atomic deploy, quiesce-window cutover

One PR. Operator picks a low-traffic merge window (target: 06:00 Amsterdam Monday — outside business hours, before the daily cron load picks up). Vercel redeploys with new function ids + event names atomically. Any events fired against the old names in the seconds between merge and deploy may fail (Inngest retries handle it; the four functions are not on hot paths during the chosen window).

No dual-emit transition — would double code temporarily and complicate revert. Atomic is cleaner and the blast radius during 06:00 Amsterdam is verified-empty by the pipeline_events historical rate.

### D-03 — Sibling cleanup-dispatcher renames for consistency

`automations/debtor-email-icontroller-dispatch` → `stage-1-icontroller-noise-cleanup-dispatcher`. Same PR, aligned vocabulary. Avoids "we renamed everything except this one" follow-up.

### D-04 — Telemetry alignment is verify-only

Audit step in Wave 0: confirm that the two Stage 2 functions (customer-resolver + label-applier) already emit `pipeline_events.stage=2` and `swarm_type='debtor-email'`. If they do (Phase 80 should have handled this), no code change — the verification is the work product.

If audit surfaces missing `stage=2` emit, the discovery is logged as a deviation and a Wave 2 task is added to patch it. Do NOT preemptively add `stage=2` emit; verify first, then patch only if needed.

### D-05 — File renames track function-id renames

When function `id:` changes, the source file is renamed to match. Test files mirror source filenames. Same PR. This means existing filename-mismatches (e.g. `cleanup-worker.ts` registering id `shard-worker`) get fixed in this pass.

### D-06 — Inngest historical run preservation

Inngest treats a function-id change as a new function. Historical runs under the OLD ids remain queryable in the Inngest dashboard under those old ids forever — they're just orphaned from the new function. No manual migration of historical data; accept the discontinuity as documented in the PR description + operator runbook.

### D-07 — No table renames in this phase

`classifier_rules`, `classifier_rule_evaluations`, `agent_runs`, `coordinator_runs`, `pipeline_events` stay as-is. Table renames are V9 scope. This phase touches function ids + event names + file paths + test paths + UI/script references only.

### D-08 — Plan boundary: one PR per "Wave"

- **Wave 0:** audit + grep-clean baseline (no code changes; produces a list of every file that needs updating).
- **Wave 1:** the four renames in one PR (function ids + event names + file renames + test renames + every call-site + UI references + script references). Single atomic commit per file group preferred but not required — what matters is that the PR merges as one unit.
- **Wave 2:** deploy + smoke (operator picks merge window; Vercel deploy; confirm new function ids appear in Inngest dashboard; fire one test event under each new event name to confirm receipt).

</decisions>

<verification>

## Success criteria

1. **Grep clean:** zero references to the four old function ids OR four old event names in any `.ts` / `.tsx` / `.md` file under `web/` (allowed exception: this CONTEXT.md, the PR description, and the operator runbook section that documents the rename — they may carry the old names as historical reference).
2. **Inngest dashboard:** after Vercel deploy, the four new function ids appear as registered functions; the four old ids show no new runs (only their historical orphan tail).
3. **Live smoke:** one test event per new event name fires and is received by the corresponding new function. Operator runbook captures the exact `inngest.send` commands.
4. **Telemetry verified:** Wave 0 audit confirms `pipeline_events.stage=2` + `swarm_type='debtor-email'` already emitted for the two Stage 2 functions (or, if not, Wave 2 task patches it).
5. **CI green:** `cd web && npm run lint && npm run typecheck && npm run build && npm run test` all exit 0.
6. **Production deploy:** Vercel deploy succeeds; no Inngest function-registration errors.

</verification>

<risks>

## Risks

- **R-01 — Producer site missed during rename.** A `inngest.send({ name: "<old-event>" })` call survives the cutover and silently fails after deploy. **Mitigation:** Wave 0 grep produces the exhaustive call-site list; Wave 1 PR checklist asserts every entry was updated; Wave 2 smoke fires one event per new name and verifies receipt.
- **R-02 — Deploy lands during traffic burst.** Cutover-window strategy assumes 06:00 Amsterdam is quiet. If a backlog from the prior night is still draining, events under old names may queue up and fail post-deploy. **Mitigation:** operator checks Inngest dashboard for in-flight runs in the 5 minutes before merge; aborts if any of the four functions show queued events.
- **R-03 — Telemetry audit surfaces missing `stage=2` emit.** If Phase 80 didn't cover it, the rename PR balloons into a behavior change. **Mitigation:** D-04 splits this — verify first in Wave 0, patch only in Wave 2 if needed. Keeps Wave 1 a pure rename.
- **R-04 — Inngest function-id history orphaned.** Operator looking up "the last cleanup-worker run before the rename" needs to know to query the OLD id. **Mitigation:** PR description + operator runbook flag the discontinuity with exact old-id → new-id mapping and the cutover timestamp.
- **R-05 — File path rename confuses git blame.** History for renamed files split. **Mitigation:** use `git mv` (not delete + add) so git rename detection preserves blame.

</risks>
