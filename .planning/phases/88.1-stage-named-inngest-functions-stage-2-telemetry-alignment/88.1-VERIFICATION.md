## Incident: registry oversight (2026-05-22 03:44 → 04:50 UTC, ~1h window)

**What:** Plan 06 Task 2 specified two registry UPDATE statements (against `swarm_noise_categories.swarm_dispatch` and `swarms.side_effects[].event`) to migrate the runtime dispatch strings off the legacy event names. The cutover operator (claude) shipped the code-side renames but forgot the registry data migration.

**Symptom:** During the ~1h window between deploy and detection, Stage 1 classified one real production email (Factuurportal at 04:46Z) as `category_key='unknown'`, read `swarm_dispatch` from the registry, fired the **legacy** event name `debtor-email/label-resolve.requested`. No listener exists under that name in the new code → Inngest queued the event but no function ran. `automation_run` row reads `status='completed'` because the upstream classifier-screen-worker finished its own job; no Stage 2 telemetry emit.

**Detection:** Plan 07 1h verification SQL surfaced zero `pipeline_events.stage=2` post-deploy despite the cron firing correctly. Cross-referencing the email's stage chain showed Stage 0+1 emitted but Stage 2 missing.

**Fix:** Two registry UPDATEs at ~04:50Z (applied via Supabase MCP `execute_sql`):
1. `swarm_noise_categories.swarm_dispatch` for `category_key='unknown'` → `debtor-email/stage-2.customer-resolve.requested`
2. `swarms.side_effects[0].event` → `debtor-email/stage-2.icontroller-label.requested`

Verification sweep returned zero remaining legacy event-name references across all three registry tables (`swarm_noise_categories`, `swarm_intents`, `swarms.side_effects`).

**Stranded email recovered:** Operator re-fired the Factuurportal email under the new event name with the existing `automation_run_id`; Stage 2 + Stage 3 emitted normally (see Real-traffic table above).

**Lesson captured:**
- Runbook `docs/runbooks/inngest-function-rename-phase-88.1.md` now flags the registry UPDATEs as MANDATORY post-deploy, with the exact SQL + verification sweep + recovery procedure
- Memory `feedback-inngest-rename-registry-updates.md` (loaded into future sessions automatically)
- Key principle: **Inngest function/event renames are TWO-step (code + registry data)**. Code-side grep guards do not see registry strings.
