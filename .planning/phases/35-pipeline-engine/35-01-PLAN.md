  <action>
Create `supabase/schema-pipeline.sql` with the pipeline state machine tables. This is a SEPARATE file from the Phase 34 schema.sql -- users execute it after schema.sql.

Tables to create:

1. **pipeline_runs** - One row per pipeline execution:
   - `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
   - `project_id UUID REFERENCES projects(id) ON DELETE CASCADE NOT NULL`
   - `name TEXT` (optional -- auto-generated from use_case if blank)
   - `use_case TEXT NOT NULL` (the user's freeform description)
   - `status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','running','complete','failed'))`
   - `inngest_run_id TEXT` (links to Inngest for debugging)
   - `step_count INTEGER DEFAULT 0`
   - `steps_completed INTEGER DEFAULT 0`
   - `agent_count INTEGER DEFAULT 0`
   - `started_at TIMESTAMPTZ`
   - `completed_at TIMESTAMPTZ`
   - `created_at TIMESTAMPTZ DEFAULT now()`
   - `created_by UUID REFERENCES auth.users(id) NOT NULL`

2. **pipeline_steps** - One row per pipeline stage within a run:
   - `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
   - `run_id UUID REFERENCES pipeline_runs(id) ON DELETE CASCADE NOT NULL`
   - `name TEXT NOT NULL` (machine name: 'architect', 'researcher', etc.)
   - `display_name TEXT NOT NULL` (human name: 'Architect', 'Research', etc.)
   - `status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','running','complete','failed','skipped'))`
   - `step_order INTEGER NOT NULL`
   - `result JSONB` (step output stored for retry/resume -- large content stored here, NOT in Inngest state)
   - `log TEXT` (expandable log content for UI)
   - `error_message TEXT` (plain-English error for UI display)
   - `started_at TIMESTAMPTZ`
   - `completed_at TIMESTAMPTZ`
   - `duration_ms INTEGER`

3. **pipeline_files** - Reference files uploaded for a pipeline run:
   - `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
   - `run_id UUID REFERENCES pipeline_runs(id) ON DELETE CASCADE NOT NULL`
   - `file_name TEXT NOT NULL`
   - `file_path TEXT NOT NULL` (Supabase Storage path)
   - `file_size INTEGER`
   - `mime_type TEXT`
   - `uploaded_at TIMESTAMPTZ DEFAULT now()`

RLS Policies:
- **pipeline_runs SELECT**: project members can see runs (join through project_members on project_id + auth.uid())
- **pipeline_runs INSERT**: project members can create runs (same membership check + created_by = auth.uid())
- **pipeline_steps SELECT**: via run's project membership (subquery through pipeline_runs -> project_members)
- **pipeline_steps INSERT/UPDATE**: service role only (Inngest functions use admin client) -- no user RLS policy for mutations
- **pipeline_files SELECT**: via run's project membership
- **pipeline_files INSERT**: project members can upload files for their runs

Use `(SELECT auth.uid())` wrapper pattern (established in Phase 34) for query planner caching.

Add index: `CREATE INDEX idx_pipeline_runs_project_id ON pipeline_runs(project_id);`
Add index: `CREATE INDEX idx_pipeline_steps_run_id ON pipeline_steps(run_id);`

**web/lib/pipeline/adapter.ts:**
The prompt adapter:
1. Fetches the .md file content from GitHub raw URL (using `fetch()`)
2. Strips YAML frontmatter using `gray-matter`
3. Builds a user message from the stage context (use case description, previous stage outputs as XML-tagged context)
4. Calls Claude `messages.create()` with:
   - `model: "claude-sonnet-4-6"` (fast, capable, cost-effective for pipeline stages)
   - `max_tokens: 8192`
   - `system`: the stripped .md file content
   - `messages`: single user message with context
5. Returns the text response content

Export `runPromptAdapter(stage: string, context: Record<string, string>): Promise<string>`.

The `buildUserMessage(stage, context)` helper should format context as XML tags:
```
<use_case>{context.useCase}</use_case>
<blueprint>{context.blueprint}</blueprint>
...
```

Handle fetch errors (GitHub rate limit, 404) by throwing with classifiable error codes.
