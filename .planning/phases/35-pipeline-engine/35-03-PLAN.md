**web/components/step-status-badge.tsx:**
Client component that renders a status badge for pipeline steps:
- `pending`: gray outline badge, "Pending"
- `running`: blue badge with pulse animation (CSS animate-pulse), "Running..."
- `complete`: green badge with CheckCircle icon, "Complete"
- `failed`: red/destructive badge with XCircle icon, "Failed"
- `skipped`: gray badge, "Skipped"

Use the existing Badge component from shadcn/ui. Add appropriate Lucide icons (CheckCircle2, XCircle, Clock, Loader2).

**web/components/step-log-panel.tsx:**
Client component for expandable step details:
- Props: `step` (pipeline step data), `onRetry?: () => void`
- Collapsed state: Shows step display_name, status badge, duration (if complete)
- Expanded state (click to toggle): Shows:
  - Log content (pre-formatted text, monospace, scrollable max-height)
  - If status === "failed": error_message in a red/destructive alert box + Retry button (calls onRetry)
  - If status === "complete": duration in human-readable format
- Use Collapsible from shadcn/ui or a simple div with state toggle. ChevronDown/ChevronRight for expand indicator.

**web/app/(dashboard)/projects/[id]/runs/[runId]/page.tsx:**
Server component that fetches run data and renders the detail view.

**Data fetching:**
```typescript
const supabase = await createClient();
const { data: run } = await supabase
  .from("pipeline_runs")
  .select("*, pipeline_steps(*)")
  .eq("id", runId)
  .single();
```
Steps should be ordered by step_order.

**Page structure:**
1. Breadcrumb: Dashboard > {Project Name} > Runs > {Run Name}
2. Run header:
   - Run name (h1)
   - Status badge (pending/running/complete/failed) -- larger, prominent
   - Metadata row: Started {relative time}, Duration {if complete}, Steps {completed}/{total}
3. Use case description in a muted card (collapsed if long, expandable)
4. **Step timeline** (vertical list, the primary view per CONTEXT.md):
   - Each step rendered as a StepLogPanel
   - Visual connector line between steps (thin vertical line, Tailwind border-l)
   - Steps in step_order
5. If run status is "failed": Show a prominent retry section at the top with the retryPipeline action

**Polling for updates:**
Per RESEARCH.md: Use simple polling (5-second interval) for Phase 35. Phase 36 will replace with Supabase Realtime.

Create a client wrapper component that:
- Uses `useEffect` + `setInterval` to refetch run data every 5 seconds
- Only polls while run status is "pending" or "running"
- Stops polling when "complete" or "failed"
- Uses `router.refresh()` (Next.js App Router) to trigger server component re-render

**Style:** Build pipeline feel (per CONTEXT.md: "should feel like a build pipeline -- think GitHub Actions or Vercel deploy logs"). Each step clearly named, status visible at a glance. Vertical timeline with connector lines.
