**web/lib/inngest/functions/pipeline.ts:**

Create the main pipeline durable function using `inngest.createFunction()`.

Function configuration:
- `id: "pipeline/execute"`
- `retries: 3` (per step, not per function)
- `onFailure` handler that writes plain-English error to Supabase

Trigger: `{ event: "pipeline/run.started" }`

The function handler:
1. Extracts `runId`, `projectId`, `useCase`, `userId`, `resumeFromStep` from event data
2. Updates pipeline_runs status to "running" and sets started_at (inside a step.run)
3. Iterates through PIPELINE_STAGES in order. For each stage, wraps in `step.run(stage.name, async () => { ... })`:

   Inside each step.run:
   a. **Check for resume skip**: Query Supabase `pipeline_steps` for this run_id + stage name. If status is "complete" AND result is not null, return the stored result (this enables retry-from-failed-step -- per RESEARCH.md Pattern 3).
   b. **Update status to running**: Set pipeline_steps status = 'running', started_at = now()
   c. **Build context**: Assemble the context object from previous stage results. The context keys depend on the stage:
      - `architect`: `{ useCase }`
      - `tool-resolver`: `{ useCase, blueprint }` (blueprint from architect)
      - `researcher`: `{ useCase, blueprint }` (conditional -- only runs if architect indicates research needed)
      - `spec-generator`: `{ useCase, blueprint, tools, researchBrief }` (runs per agent in blueprint, but for Phase 35 run once with all agents)
      - `orchestration-generator`: `{ useCase, blueprint, agentSpecs }`
      - `dataset-generator`: `{ useCase, blueprint, agentSpecs }`
      - `readme-generator`: `{ useCase, blueprint, agentSpecs }`
   d. **Call prompt adapter**: `const result = await runPromptAdapter(stage.name, context)`
   e. **Store result in Supabase**: Update pipeline_steps with status = 'complete', result = { output: result }, completed_at = now(), duration_ms, log (summary of what happened)
   f. **Update run progress**: Increment pipeline_runs.steps_completed
   g. **Return a REFERENCE** (the step ID or a short summary), NOT the full output. Per RESEARCH.md Pitfall 6: store large outputs in Supabase, return only references from step.run() to avoid Inngest state size limits.

4. After all stages complete: Update pipeline_runs status to "complete", completed_at = now()

**CRITICAL anti-patterns to avoid (per RESEARCH.md):**
- Do NOT put side effects (DB writes, API calls) outside step.run() -- code outside steps runs on EVERY re-invocation
- Do NOT have multiple side effects in one step -- if it retries, all side effects re-execute
- Do NOT return full stage output from step.run() -- store in DB, return reference
- Do NOT use streaming for Claude API calls inside Inngest steps -- use non-streaming messages.create()
- Use admin client (createAdminClient()) for all Supabase operations inside Inngest functions -- no request context available

**onFailure handler:**
```typescript
onFailure: async ({ event, error }) => {
  const admin = createAdminClient();
  const { runId } = event.data;
  // Find the last running step
  const { data: failedStep } = await admin
    .from("pipeline_steps")
    .select("id, name")
    .eq("run_id", runId)
    .eq("status", "running")
    .single();
  if (failedStep) {
    await admin.from("pipeline_steps").update({
      status: "failed",
      error_message: toPlainEnglish(error),
      completed_at: new Date().toISOString(),
    }).eq("id", failedStep.id);
  }
  await admin.from("pipeline_runs").update({
    status: "failed",
  }).eq("id", runId);
}
```

Create a "use server" module with the `startPipeline` server action.

**startPipeline(formData: FormData):**
1. Get authenticated user via `createClient()` + `supabase.auth.getUser()`
2. Extract from formData: `projectId`, `useCase`, `runName` (optional)
3. Validate: useCase must be non-empty (use zod: `z.string().min(10, "Use case description must be at least 10 characters")`)
4. Create pipeline_runs record using admin client:
   ```typescript
   const admin = createAdminClient();
   const { data: run, error } = await admin
     .from("pipeline_runs")
     .insert({
       project_id: projectId,
       name: runName || useCase.slice(0, 60),
       use_case: useCase,
       status: "pending",
       step_count: PIPELINE_STAGES.length,
       created_by: user.id,
     })
     .select()
     .single();
   ```
5. Create initial pipeline_steps records for all stages (all status: "pending"):
   ```typescript
   await admin.from("pipeline_steps").insert(
     PIPELINE_STAGES.map((s) => ({
       run_id: run.id,
       name: s.name,
       display_name: s.displayName,
       status: "pending",
       step_order: s.stepOrder,
     }))
   );
   ```
6. Trigger Inngest:
   ```typescript
   await inngest.send({
     name: "pipeline/run.started",
     data: {
       runId: run.id,
       projectId,
       useCase,
       userId: user.id,
     },
   });
   ```
7. Redirect to run detail page: `redirect(\`/projects/${projectId}/runs/${run.id}\`)`

Also create a `retryPipeline(runId: string, projectId: string)` server action:
1. Authenticate user
2. Fetch the failed step from pipeline_steps (status = 'failed', ordered by step_order, take first)
3. Reset the failed step and all subsequent steps to status = 'pending', clear error_message
4. Update pipeline_runs status to 'running'
5. Re-trigger Inngest with `resumeFromStep` set to the failed step name
6. Revalidate the run detail page path
