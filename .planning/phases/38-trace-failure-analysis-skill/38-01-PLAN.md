### 2. Title + preamble

```markdown
# Trace Failure Analysis

You are running the `/orq-agent:trace-failure-analysis` command. This command walks the user through turning a pile of production traces into a 4-8 mode failure taxonomy using grounded-theory methodology — mixed sampling, open/axial coding, first-upstream-failure labeling, transition matrix, mode classification, and a handoff-ready error-analysis report.

**Tier:** deploy+ (requires authenticated workspace with ≥50 traces). Core-tier users see: "Trace failure analysis requires traces — run `/orq-agent:observability` first and ensure your workspace has ≥50 traces."
```

### 3. `## Constraints` (NEVER/ALWAYS rules + "Why these constraints:" paragraph)

Include these exact bullets (verbatim wording — grep anchors):

- **NEVER** label downstream cascading effects — label ONLY the first upstream failure per trace (TFAIL-03).
- **NEVER** force more than 8 or fewer than 4 failure modes — grounded theory yields 4-8 non-overlapping categories (TFAIL-02).
- **NEVER** fabricate trace IDs or span sequences when MCP/REST fails — STOP and surface the raw error (matches Phase 36 discipline).
- **NEVER** skip the sampling plan — the 50% random / 30% failure-driven / 20% outlier split is recorded in the report so results are reproducible (TFAIL-01).
- **NEVER** classify a mode as more than one of the 4 categories — specification / generalization-code-checkable / generalization-subjective / trivial-bug are mutually exclusive (TFAIL-05).
- **ALWAYS** emit the final report to `error-analysis-YYYYMMDD-HHMM.md` with taxonomy + rates + example trace IDs + recommended handoff (TFAIL-06).
- **ALWAYS** call MCP `list_traces` first; fall back to REST `GET /v2/traces` with `$ORQ_API_KEY` on MCP failure.
- **ALWAYS** ask the user to confirm / merge / split axial-coding clusters via `AskUserQuestion` — the taxonomy is their taxonomy, the skill only proposes.

Then a "Why these constraints:" paragraph explaining: first-upstream-failure prevents cascade inflation; 4-8 modes is the grounded-theory sweet spot (fewer = too coarse, more = overlapping); 50/30/20 sampling balances representativeness with failure coverage; MCP-first preserves Phase 36 never-fabricate invariant.

**## Step 2: Open Coding (TFAIL-02)**

Explain grounded-theory open coding (one-sentence freeform annotation per trace describing WHAT went wrong, not yet WHY). Reference `orq-agent/commands/trace-failure-analysis/resources/grounded-theory-methodology.md`.

Batch traces 10 at a time. For each batch, display trace excerpt (input + output + status + any error message) then use `AskUserQuestion` with free-text "Annotation for trace `<id>`?" — OR skill-LLM drafts an annotation from the span content and user confirms/edits.

Saturation heuristic (stop condition): "stop open coding when two consecutive batches produce no new annotation themes." User can override.

Emit annotation table: `| Trace ID | Bucket | Status | Annotation |`.

**## Step 3: Axial Coding (TFAIL-02)**

Explain axial coding (cluster open-coding annotations into categories). Reference `resources/grounded-theory-methodology.md`.

Skill-LLM proposes 4-8 candidate modes by clustering the annotation table. Render cluster table: `| Mode | Traces | Representative annotation |`.

Use `AskUserQuestion` to confirm / merge / split clusters. ENFORCE: final mode count MUST be between 4 and 8 inclusive. If the user converges on >8, ask them to merge; if <4, ask them to split or broaden sampling.

NEVER-OVERLAP rule: every trace belongs to exactly one mode (no multi-labeling).

**## Step 4: First-Upstream-Failure Labeling (TFAIL-03)**

For every sampled trace, fetch the span hierarchy via `mcp__orqai-mcp__list_spans(trace_id)`. Walk the DAG in topological order. Label ONLY the first span whose status is `error` OR whose output fails the mode's criterion.

Downstream spans that errored because the upstream already failed get `cascade-of: <parent_mode>` — NOT a new mode assignment.

Anti-pattern callout: "If span B errored because it received bad input from span A, B is a cascade — label A, mark B `cascade-of`."

Single-span traces: the whole trace is the first upstream failure.

**## Step 5: Transition Failure Matrix (TFAIL-04)**

If the pipeline is multi-step (max_depth > 1 across the sampled traces), build a matrix:

- Rows = last successful span name (or `<origin>` if the very first span failed).
- Columns = first failure mode.
- Cell value = count of traces with that (last_success → first_failure) transition.

Render as Markdown table. Include a "Transition hotspots" paragraph listing the top 3 cells by count.

If all sampled traces are single-step, emit: "Skipping transition matrix — all sampled traces are single-step (max depth = 1)."

**## Step 6: Mode Classification (TFAIL-05)**

For each of the 4-8 modes, classify as exactly ONE of:

- **specification** — prompt / instruction issue (ambiguous, conflicting, missing constraint). Fix: prompt engineering.
- **generalization-code-checkable** — output is wrong in a way verifiable by code (schema invalid, wrong tool called, typed field mismatch). Fix: build evaluator + guardrail.
- **generalization-subjective** — output is wrong in a way requiring human judgement (tone, reasoning quality, misinterpreted intent). Fix: LLM-judge evaluator + human labels.
- **trivial-bug** — plumbing / infra failure (auth error, env var missing, network timeout). Fix: developer patch, no AI-platform skill.

Reference `orq-agent/commands/trace-failure-analysis/resources/failure-mode-classification.md` for decision rules.

Use `AskUserQuestion` per mode to confirm the proposed classification (one AskUserQuestion call with N questions, one per mode). Store result as `| Mode | Count | Rate % | Classification |`.

**## Step 7: Generate Error-Analysis Report (TFAIL-06)**

Compose `error-analysis-YYYYMMDD-HHMM.md` with sections:

1. Sampling plan (from Step 1).
2. Taxonomy table (from Step 3) — mode, count, rate %, classification.
3. Per-mode detail: 3 example trace IDs + representative annotation.
4. Transition matrix (from Step 5, or "N/A — single-step pipeline").
5. Recommended handoffs — one row per mode, read from `resources/handoff-matrix.md`:

   | Classification | Recommended next skill | Rationale |
   |----------------|------------------------|-----------|
   | specification | `/orq-agent:prompt` | Prompt engineering fix |
   | generalization-code-checkable | `/orq-agent:harden` | Build evaluator + guardrail |
   | generalization-subjective | `/orq-agent:harden` + human annotation queue | LLM-judge + human labels |
   | trivial-bug | (none — developer fix) | Out of AI-platform scope |

6. Footer with generated-at timestamp + MCP tools used.

Resolve output path: if cwd contains `./Agents/<name>/` subdir, write into that subdir; otherwise write to cwd. Before overwriting an existing file of the same name, `AskUserQuestion` for confirmation.

### 10. `## Anti-Patterns` (Markdown table, ≥5 rows)

| Pattern | Do Instead |
|---------|-----------|
| Labeling every errored span in a cascade as its own mode | Label ONLY the first upstream failure (TFAIL-03); downstream spans get `cascade-of: <parent_mode>` |
| Forcing 3 or 12 modes "because the data says so" | Re-sample or merge — grounded theory's signal band is 4-8 non-overlapping modes |
| Sampling only errored traces ("failure-driven is enough") | Random 50% captures silent-failure patterns and "looks-ok-but-isn't" traces; failure-driven alone overfits to loud failures |
| Classifying the same mode as both specification AND generalization | Mutually exclusive categories — if the prompt is ambiguous AND the output is code-checkable, the SPEC fix eliminates the need for the evaluator; classify as specification |
| Skipping the transition matrix on multi-step pipelines | Transition hotspots reveal which handoffs are brittle — the matrix IS the diagnostic for pipeline design bugs |
| Writing a report without example trace IDs | Without IDs the report is un-verifiable — downstream handoffs need to pull the exact traces to build evaluators |
