<task type="auto">
  <name>Task 1: Create prompt-editor.md subagent with 6 internal phases</name>
  <files>orq-agent/agents/prompt-editor.md</files>
  <action>
Create `orq-agent/agents/prompt-editor.md` (~200 lines) extracting iterator.md Phases 5-7 into a focused subagent. Follow the same structural conventions as failure-diagnoser.md (YAML frontmatter, files_to_read, numbered phases with steps).

**Frontmatter:**
- name: orq-prompt-editor
- description: Applies approved prompt changes to agent spec files, delegates re-deploy and holdout re-test, computes before/after score comparison with regression flagging
- tools: Read, Write, Bash, Glob, Grep (needs Write for spec file editing)
- model: inherit

**files_to_read:** None needed — prompt-editor reads iteration-proposals.json, dataset-prep.json, and test-results.json from swarm_dir at runtime. Delegates load their own references.

**Inputs:** `swarm_dir` (path) and `iteration_number` (integer) — same pattern as failure-diagnoser.

**Phase 1: Read Inputs and Filter Approved Agents**
- Read `{swarm_dir}/iteration-proposals.json`
- Filter `per_agent` array to entries with `approval: "approved"`
- If NO approved agents: display "No approved changes. Skipping prompt editing." and STOP
- For each approved agent, collect `agent_key` and `changes[]` array
- Read `{swarm_dir}/dataset-prep.json` for `holdout_dataset_id` per agent
- Read `{swarm_dir}/test-results.json` for original scores (snapshot "before" scores in memory for comparison)
- Check `holdout_dataset_id` exists per approved agent. If missing, warn: "Holdout dataset IDs not found. Re-run /orq-agent:test to generate updated results." and STOP

**Phase 2: Apply Section-Level Changes to Spec Files (ITPIPE-04)**
For each approved agent:
1. Resolve spec file path: `{swarm_dir}/agents/{agent_key}.md` with Glob fallback `{swarm_dir}/**/agents/{agent_key}.md` for non-standard layouts (per Phase 30 decision)
2. Read the spec file
3. Parse file in 3 layers:
   - Layer 1: YAML frontmatter between first `---` and second `---`
   - Layer 2: Markdown sections (`## Configuration`, `## Model`, `## Tools`, `## Instructions`, `## Context`, `## Evaluators`, `## Guardrails`, `## Runtime Constraints`, `## Input/Output Templates`)
   - Layer 3: XML content within `## Instructions` section (between ```xml and ``` delimiters)
4. For each change in `changes[]`:
   - Find `<{change.section}>` opening tag and `</{change.section}>` closing tag in the XML content
   - Replace everything between the tags with `change.after`
   - If section tag NOT found: log warning "Section {change.section} not found in {agent_key}. Skipping change." and skip (do NOT create malformed XML)
5. Reassemble: Layer 1 (frontmatter) + Layer 2 (all markdown sections with modified Instructions) = full file
6. Write back to same file path

**Safety invariant (CRITICAL):** The file write MUST preserve:
- YAML frontmatter (orqai_id, orqai_version, deployed_at, deploy_channel, all custom fields)
- ALL markdown sections except the modified XML content within ## Instructions
- All XML sections within Instructions that were NOT in the changes array

Display progress:
```
Applying changes to {agent_key}...
  Modified <{section}>: {change.reason} (truncated to 80 chars)
Changes applied to {N} agent(s).
```

**Phase 3: Re-deploy Changed Agents (ITPIPE-05 part 1)**
Invoke `deployer.md` with swarm directory path. The deployer's idempotent create-or-update logic automatically detects which agents have changed instructions and PATCHes only those.

Do NOT modify the deployer subagent. Do NOT pass agent-key filters — the deployer processes the full swarm but only updates changed resources.

After deployer completes:
- Collect which agents were updated (status `updated`)
- Record new `orqai_version` and `deployed_at` from each agent's frontmatter
- If any agent fails to re-deploy: log failure but continue (test successfully deployed agents)
- If ALL agents fail: skip re-test, continue to Phase 6 (logging)

Display:
```
Re-deploying {N} changed agent(s)...
  {agent-key}: updated (version: {new_version})
Re-deploy complete.
```

**Phase 4: Holdout Re-test via Experiment-Runner and Results-Analyzer (ITPIPE-05 part 2)**
CRITICAL anti-pattern: Do NOT invoke dataset-preparer. Datasets already exist on Orq.ai.

Build the re-test agent list: only agents that were approved AND successfully re-deployed.

Invoke experiment-runner.md in holdout mode, once per agent:
- `dataset_id` = `holdout_dataset_id` from dataset-prep.json per agent
- `agent_key` = single agent key
- `evaluator_ids` = from test-results.json `results.per_agent[].scores` keys mapped to `orqai_evaluator_id` (pass through to skip re-resolution)
- `run_count` = 3
- Output: `experiment-raw-holdout.json`

Then invoke results-analyzer.md:
- `holdout = true` (reads experiment-raw-holdout.json instead of experiment-raw.json)

Error handling:
- If re-test fails for an agent: log error, keep old scores as "after" (no improvement), continue
- If ALL re-tests fail: continue to Phase 6 with no improvement data

**Phase 5: Before/After Score Comparison (ITPIPE-06)**
After holdout re-test, compute comparison for each changed agent using the "before" snapshot from Phase 1 and holdout results as "after".

For each evaluator per agent:
- `delta = ((new_median - old_median) / old_median) * 100`
- Status: `improved` if delta > 0, `unchanged` if delta == 0, `regressed` if delta < 0

Display comparison table:
```markdown
### Re-Test Results: {agent-key}

| Evaluator | Before | After | Delta | Status |
|-----------|--------|-------|-------|--------|
| {name} | {old_median} | {new_median} | {+/-delta}% | improved/unchanged/regressed |

**Bottleneck:** {old_bottleneck} -> {new_bottleneck} ({improvement}%)
```

Regression flagging: If ANY evaluator median decreased after changes (even if still passing):
```
Warning: {evaluator} regressed from {old} to {new} on {agent-key}.
```

After comparison display, update test-results.json with holdout re-test scores so iterate.md can evaluate stop conditions (min_improvement, all_pass). The before/after comparison uses the in-memory snapshot so original scores are preserved in the display.

**Phase 6: Logging (iteration-log.md and audit-trail.md)**
Append to both log files in the swarm directory. Write logs BEFORE returning results to ensure changes are recorded even if downstream operations fail.

iteration-log.md entry format (from iterator.md Phase 9):
```markdown
## Iteration {N} -- {TIMESTAMP}

### Changes Applied

**Agent: {agent-key}**
- Modified `<{section}>`: {reason}

### Re-Test Results (Holdout Split)

| Evaluator | Before | After | Delta |
|-----------|--------|-------|-------|
| {name} | {old} | {new} | {delta}% |

**Improvement:** {bottleneck_delta}% on bottleneck evaluator.
```

audit-trail.md entry format:
```markdown
## [{TIMESTAMP}] Iteration {N}

- **Agent:** {agent-key}
- **Changes proposed:** {count} (modified {sections})
- **Approval:** Approved
- **Scores before:** {evaluator}={score}, ...
- **Scores after:** {evaluator}={score}, ...
- **Bottleneck improvement:** {old} -> {new} ({delta}%)
- **Stop condition:** None (continuing)
```

If log write fails: display content in terminal as warning but do not block.

**Anti-patterns to explicitly document in the subagent:**
- Do NOT invoke dataset-preparer (ARCHITECTURE.md Anti-Pattern 4)
- Do NOT replace entire instructions content — only modify specific XML-tagged sections
- Do NOT embed stop conditions — those belong in iterate.md (Phase 32)
- Do NOT modify deployer.md, experiment-runner.md, or results-analyzer.md
- Do NOT create new XML tags that don't exist in the original spec
