**File structure — write Steps 1-3 now, leave Steps 4-7 as section headers that Plan 02 will fill:**

**Step 1: Capture Input**
- If `$ARGUMENTS` is provided (non-empty), use it as the use case description
- If `$ARGUMENTS` is empty, prompt the user: ask them to describe the use case they want to build agents for, noting it can be brief (1-3 sentences) or detailed (multi-paragraph)
- Store the input for use in subsequent steps

**Step 2: Classify Input Depth (LLM-based, per-stage)**
Write the embedded classification logic. The orchestrator itself performs this analysis (NOT a separate subagent). Classification evaluates the user's input against each pipeline stage's requirements and outputs a skip/run decision per stage.

Classification dimensions (from RESEARCH.md):
- **Architect**: Always RUN — topology is the architect's job regardless of input detail
- **Researcher**: RUN unless user explicitly provides ALL of these per agent: model selection with rationale, tool requirements (specific Orq.ai tool types), guardrail requirements (domain-specific), prompt strategy (role, constraints, edge cases), context needs (knowledge bases, variables, memory). If ANY dimension is missing for ANY agent, RUN researcher. Only SKIP when user explicitly provides agent configuration details — long business descriptions that don't mention models/tools/guardrails are NOT sufficient to skip research.
- **Spec Generator**: Always RUN
- **Orchestration Generator**: RUN if multi-agent pattern (determined after architect runs), N/A if single-agent
- **Dataset Generator**: Always RUN
- **README Generator**: Always RUN

IMPORTANT: Only the researcher stage is ever skippable. The classification should make this clear and not over-engineer by trying to skip other stages.

Output format — a table showing each stage, decision (RUN/SKIP), and brief reasoning:
```
| Stage | Decision | Reasoning |
|-------|----------|-----------|
```

**Step 3: Confirm with User**
Display the classification table using GSD-style banner formatting:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► CLASSIFYING INPUT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

After showing the table, present a checkpoint asking the user to confirm or override:
- "proceed" to accept and continue
- "change: researcher=skip" (or =run) to override a specific stage
