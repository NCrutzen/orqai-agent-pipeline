<decisions>
## Implementation Decisions (from discussion)

### D-1 — Read-back fetch wiring: server-side pre-fetch, pass via shell props

**Decision:** Each stage page (`stage-{0,1,2,3}/page.tsx`) pre-fetches a `feedbackMap: Record<string, FeedbackReadBack>` for the visible row set server-side, then passes it down to `Stage1ClientShell` / `Stage3ClientShell` / `Stage4ClientShell` / `OptionZDetailPane` exactly like the existing `bodyMap` and `timelineMap` props.

**Why this pattern wins:**
- Matches the established pattern (no new architectural concept to learn).
- No client-side fetch on every panel mount — fewer network calls, no spinner flash.
- Re-renders on URL navigation pick up fresh data because the page is `dynamic = "force-dynamic"`.
- No new client deps (SWR / react-query not installed; not introducing them just for this).

**Shape:** `FeedbackReadBack = { own_latest: { prose_notes, verdict, created_at } | null; others: Array<{ display_name, verdict, prose_notes, created_at }> }`.

**Endpoint coexists, doesn't replace:** the `GET /api/automations/debtor-email/feedback?email_id=X&stage=N` endpoint specified in SPEC.md R2 still ships — for refresh-after-write inside the client component when the operator commits a new verdict and we need to reflect the new "Saved at HH:MM" plus updated own_latest without a full page reload. Implementation: after a successful Save/Confirm, `StageFeedbackPanel` re-fetches its own row's `(email_id, stage)` via the GET, and merges into local state.

### D-2 — Row-strip dot mount: absolute-positioned overlay

**Decision:** Render the 8px verdict dot as an `absolute`-positioned span inside the existing row container. No changes to `RowList`'s grid template.

**Why this pattern wins:**
- RowList's grid is currently `grid-template-columns: 140px 160px 1fr 100px` (stage-badge / sender / subject / timestamp). Adding a 14px column requires touching every existing column width AND every Row consumer.
- Absolute overlay lives in `.row { position: relative }` (already a wrapper) and pins at `left: 4px; top: 50%; transform: translateY(-50%)`. Zero grid disruption.
- Selected-row left-border treatment (`--brand-primary`) stays in column 0; the dot sits on top of the border at a lower z-index so the orange selection line still reads.

**Implementation hint for the planner:**
- New `<RowVerdictDot verdict="confirm|override|unclear|null" />` client component (~20 lines, inline `<span style={…}>`).
- Mounted inside `RowList`'s row map, after the `<li>` open tag, before grid children.
- Color tokens: `--v7-lime` / `--v7-amber` / `--v7-brand-secondary` / dashed `--v7-text-dim`.

### D-3 — Operator identity in "others said": join auth.users for email local-part

**Decision:** The read-back endpoint joins `public.email_feedback.operator_id` → `auth.users.email` and exposes the email local-part (everything before `@`) as `display_name`. Example: `n.crutzen@curagroep.com` → `n.crutzen`.

**Why this pattern wins:**
- Humane in the UI — operators recognize colleagues by name, not pseudonym.
- Stable identifier — even if a user rotates auth tokens, the email local-part stays.
- Service-role server-side fetch — the operator email never reaches the browser as raw `auth.users.id`.

**Constraints the researcher / planner must respect:**
- The join MUST happen server-side via `createAdminClient()`. Never expose raw `auth.users` rows to the client.
- Cache the operator_id → display_name map per request (small map: <20 ops in real use) to avoid N+1 queries. A single `IN` against `auth.users.id` per page render is sufficient.
- If `auth.users.email` is null (shouldn't happen, defensive), fall back to first 8 chars of `operator_id` rendered mono.
- The map is computed inside the read-back endpoint and inside the server-side feedbackMap pre-fetch (D-1) so both paths surface the same `display_name`.

**Privacy posture:** email local-parts are internal-to-MR-Automations and already visible in iController / NXT logs operators access. No additional disclosure concern.

### D-4 — Future-pill expansion: inline-expand all hidden stages

**Decision:** Clicking the collapsed-stages pill ("Stages N-M haven't run yet · expand to override pre-emptively") replaces the pill with the today-current "didn't run — override to set" rows for all hidden stages, inline in place.

**Why this pattern wins:**
- Mirrors current behavior — operators who already know the old layout aren't surprised.
- No new component (sub-popup), no progressive disclosure click cascade.
- Operator's mental model: pill = compact, expanded = full. One toggle, two states.

**Implementation hint:**
- Single boolean `futureExpanded` state in the parent component (Stage1ClientShell etc.). Default `false`.
- When expanded: render the existing skipped-state stage steps for indices > active stage.
- The pill itself becomes the toggle — clicking it again collapses.
- ⌘+s and ⏎ shortcuts only act on the active stage, not on inline-expanded future ones (avoid surprise writes on stages the operator just clicked open to peek).

### Claude's Discretion (planner picks, not user-blocking)

- **Read-back endpoint file location:** Add `export async function GET(...)` to the existing `web/app/api/automations/debtor-email/feedback/route.ts` next to the POST handler. Same auth guard, same admin client, same zod-style guard on query params. Don't fork a new file.
- **Soft confirm-without-note dialog implementation:** Native `window.confirm("Confirm this stage without writing a note?")` for 82.5. No shadcn `<AlertDialog>` precedent in `_shell/` surfaces; the V7 dialog story is unbuilt. Native is one line, accessible, and not the long-term answer (we can swap to a custom modal in a later phase).
- **Bottom-button morph state source:** Derive from the existing `stages` array passed to `<PipelineFlow>` — `const anyDirty = stages.some(s => s.state === 'dirty')`. No new context, no new hook.
- **Override + note coupling helper line placement:** One copy under the picker (in `state='dirty'` widget), one copy under the textarea label. Both `--v7-amber`, both `font-mono`, `var(--fs-xs)`. Don't add a third (already piloted in Sketch 010-C).
- **Save / Confirm button two-line layout fallback at 1280px:** If width is the constraint, drop the `<sub>` microcopy lines on viewports < 1366px (CSS-only via `@media` — no JS). Icons + primary label stay.
- **Keyboard shortcuts:** ⏎ fires the bottom primary button only when no `textarea` is focused. ⌘+s saves the note only when a `textarea` is focused. `n` and `Esc` keep their existing semantics (skip / close-expander). All conditional via `useEffect` listener on `document.keydown`.

</decisions>

## Assumptions to Verify in Planning

- `auth.users.email` is non-null for every operator_id present in `email_feedback`. (Verify via SQL count check — planner can run via MCP.)
- The number of distinct operator_ids per (email_id, stage) is small (< 5 in real use). If it grows large, the "What others said (N)" collapsible needs a max-height + scroll affordance not currently in Sketch 008.
- The existing detail-pane footer (Approve / Reject / Skip) keyboard shortcuts (`KEYBOARD_EVENTS.approve / reject / skip`) are reusable — the morph just swaps the click handler bound to the primary button, not the keybinding itself.
- `feedbackMap` server-side fetch latency is acceptable when combined with `bodyMap` + `timelineMap` (already parallel). Should add < 50ms in practice; budget < 100ms.

</assumptions>
