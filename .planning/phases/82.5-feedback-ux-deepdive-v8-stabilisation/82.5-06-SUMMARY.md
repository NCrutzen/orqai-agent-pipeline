# Phase 82.5 Plan 06: Page-Level + Shell-Level feedbackMap Plumbing Summary

Plumb the server-side `loadFeedbackMap` prefetch from Plan 01 through every stage page (0/1/2/3) into the row strip and detail pane prop entry points, with per-shell client-side useMemo reductions for `rowVerdictMap` (row strip dot, R3) and `paneFeedbackMap` (per-stage detail pane readback, R1). Forward-compatible optional `feedbackMap` props on RowList, UnifiedDetailPane, and OptionZDetailPane so Plans 04/05/07 can land their renders without re-touching shells or pages.

## What changed

### Stage pages (server-side prefetch + reduction)

- **stage-0/page.tsx** and **stage-2/page.tsx** (no Stage{N}ClientShell wrapper — they mount `<RowList>` directly):
  - Added imports: `loadFeedbackMap`, `FeedbackMap` type.
  - Hardcoded `const ACTIVE_STAGE = 0|2 as const;` per page.
  - Reused existing `user` binding (line ~101/99) for `viewerId = user?.id ?? null`.
  - Extended existing `Promise.all([bodiesRes, timelineRes])` into a 3-tuple with `loadFeedbackMap(admin, emailIds, ACTIVE_STAGE, viewerId)`.
  - `let feedbackMap: FeedbackMap = {}` default outside the `if (rows.length > 0)` guard preserves type-safety on empty pages.
  - Server-side reduction → `rowVerdictMap` (Pattern E: own latest wins, else first "other" operator's verdict).
  - `<RowList feedbackMap={rowVerdictMap} />` + `<OptionZDetailPane feedbackMap={feedbackMap} />`.

- **stage-1/page.tsx** (delegates to Stage1ClientShell):
  - Added imports: `createClient`, `loadFeedbackMap`, `FeedbackMap`.
  - `const ACTIVE_STAGE = 1 as const;` at module scope.
  - Inside `SwarmReviewPage`: `viewerId` resolved via `createClient().auth.getUser()`.
  - After the existing 4-tuple `Promise.all([loadPageData, loadSwarmNoiseCategories, loadSwarmIntents, loadStage2WeeklyCount])`, a sequential `loadFeedbackMap(admin, data.rows.map(r => r.id), ACTIVE_STAGE, viewerId)` is awaited (row ids are only known after loadPageData resolves due to cursor pagination).
  - `feedbackMap={feedbackMap}` threaded into `<Stage1ClientShell>`.

- **stage-3/page.tsx** (delegates to Stage3ClientShell):
  - Added imports: `createClient`, `loadFeedbackMap`, `FeedbackMap`.
  - `const ACTIVE_STAGE = 3 as const;` at module scope.
  - Inside `Stage3Page`: `viewerId` resolved server-side.
  - `loadFeedbackMap` joined into the existing body+timeline `Promise.all` as a 3-tuple — single parallel block under the 100ms budget.
  - `feedbackMap={feedbackMap}` threaded into `<Stage3ClientShell>`.

### Client shells (per-row useMemo derivations)

All three shells (`stage-1`, `stage-3`, `stage-4`) gained:

1. `feedbackMap?: FeedbackMap` prop (optional during cross-wave landings).
2. `const ACTIVE_STAGE = N as const;` literal at module scope.
3. `rowVerdictMap` via `useMemo(() => ...)` — Pattern E reduction.
4. `paneFeedbackMap` via `useMemo(() => ...)` — selected-row-only, ACTIVE_STAGE-keyed; Stage 4 always undefined (guarded by `ACTIVE_STAGE > 3`).
5. `feedbackMap={rowVerdictMap}` passed to `<RowList>`.
6. `feedbackMap={paneFeedbackMap}` passed to `<UnifiedDetailPane>`.

**Stage 3 + 4 bridge:** Their row strip is keyed by `KanbanRow.id`, but `feedbackMap` is keyed by `email_id`. The shells iterate `rows` and bridge via `k.result.email_id` to produce a `kanban_id → verdict` map for the row strip.

### Primitive shell components (forward-compat optional props)

- **`_shell/row-list.tsx`** — added `feedbackMap?: Record<string, verdict|null>` to `RowListProps`. Marked `void feedbackMap` until Plan 04 wires the dot.
- **`_shell/detail-pane.tsx`** — added `feedbackMap?: Partial<Record<0|1|2|3, FeedbackReadBack>>` to `UnifiedDetailPaneProps` (Plan 05 will consume in StageFeedbackPanel).
- **`_shell/option-z-detail-pane.tsx`** — added `feedbackMap?: FeedbackMap` to `Props` and destructured it as accepted-but-ignored (`void feedbackMap`) — Plan 07 derives per-stage paneFeedbackMap from `useSelection()`.

## Hard-Separation Contract Preserved

Per `docs/agentic-pipeline/README.md`: a row exists in EXACTLY ONE of `swarm_noise_categories` (Stage 1) or `swarm_intents` (Stage 3). This plan respects that contract:

- Each page/shell hardcodes `ACTIVE_STAGE` to its single stage and threads it as the `stage` argument to `loadFeedbackMap` (which filters `email_feedback.stage = ACTIVE_STAGE`).
- Stage 0 (safety) and Stage 2 (entity mapping) bucket their own feedback under stage=0 / stage=2 — orthogonal to both registries.
- No code path reads Stage 1's feedback while rendering Stage 3 (or vice versa).

**1. [Rule 3 - Plumbing] Added optional `feedbackMap` prop to RowList, UnifiedDetailPane, OptionZDetailPane.**

- **Commit:** `4b447fd`.

**2. [Rule 3 - Path correction] Plan listed shell paths as `_shell/stage-{1,3,4}-client-shell.tsx`; actual paths are `stage-{1,3,4}/client-shell.tsx`.**
- **Found during:** Task 2 setup.
- **Issue:** Plan files_modified frontmatter referenced non-existent paths.
- **Fix:** Modified the actual files at `stage-{1,3,4}/client-shell.tsx`. Frontmatter path-pattern drift is cosmetic — content of the wiring identical.

**3. [Rule 3 - Latency tradeoff] Plan 1.4 told stage-1 to put loadFeedbackMap inside the existing Promise.all of `[bodyMap, timelineMap, mailboxLabels]`.**
