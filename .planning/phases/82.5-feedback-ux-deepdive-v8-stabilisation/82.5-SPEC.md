# Phase 82.5: Feedback UX Deepdive ‚ÄĒ Specification

**Created:** 2026-05-13
**Ambiguity score:** 0.13 (gate: ‚Č§ 0.20)
**Requirements:** 8 locked

## Goal

Operators reviewing rows in `automations/[swarm]/stage-{0,1,2,3}` can see prior feedback (own and others') on every revisit, scan verdict status at the row strip, understand the override-note coupling, and disambiguate Save vs Confirm ‚ÄĒ without breaking the durable 82.4 infrastructure or the locked agentic-pipeline hard-separation contract.

## Background

Phase 82.4 shipped the durable feedback layer:
- `public.email_feedback` table (`(email_id, stage, operator_id, created_at)` natural key, RLS service-role only)
- `POST /api/automations/debtor-email/feedback` writes a row per call (server-stamps `operator_id`)
- `StageFeedbackPanel` mounted inside `StageDetailExpander` for Stage 0‚Äď3 sidebar entries (textarea + Save + ‚úď Confirm)
- Override surfaces dual-write a `verdict='override'` row via `fire-feedback.ts` (additive to the existing Phase 71/82 Inngest dispatch)
- Nightly Inngest snapshot cron to `email-feedback-snapshots` storage bucket

**What's broken today (operator-confirmed during 82.4 UAT 2026-05-13):**
- After Save, the textarea cleared (now patched in commit 9fb5777 to keep prose visible + show "‚úď Saved at HH:MM"), but re-opening the row STILL shows an empty textarea ‚ÄĒ no fetch-back of prior notes.
- No row-strip indicator showing which rows already have feedback. Operators triage blind on 400+ row queues.
- The override picker and the textarea look independent ‚ÄĒ operators don't realize their note is being attached to the override on submit.
- Save vs Confirm both render as pill buttons in the same color family; semantics aren't intuitive ("which one closes the row?").
- Detail pane renders all 5 stages even when only Stage 0+1 ran. "Stages 2-4 didn't run ‚ÄĒ override to set" placeholders pad the pane.
- Bottom footer "Approve" is ambiguous ‚ÄĒ it cannot coherently approve stages that haven't run.

**Locked design decisions (from sketches 008‚Äď011):**
- **008-A + context-morph:** Full pipeline visible, future-not-yet-run stages collapse into a single pill. Bottom button morphs between `‚úď Approve verdicts that ran` (default) and `‚úď Submit override (Stage N)` (when any stage is dirty). Per-stage Save/Confirm chips stay tight to the textarea inside each stage's expander.
- **009-A:** 8px colored dot before the stage badge encodes verdict (lime=confirm, amber=override, blue=unclear, empty=none).
- **010-C:** Inline microcopy under the override picker ‚ÄĒ "‚§ď Override + note save together".
- **011-C:** Two-line buttons with icons + microcopy ("ūüíĺ Save note ¬∑ come back later" vs "‚úď Confirm ¬∑ looks right").

## Requirements

1. **Feedback read-back on mount**: When the operator opens a row that already has feedback for the active stage, the StageFeedbackPanel pre-populates with the operator's own latest prose, and a collapsible "What others said (N)" block surfaces cross-operator notes.
   - Current: textarea always empty on mount; cross-operator notes invisible.
   - Target: textarea pre-filled with latest own prose (verdict chip in stage label); "What others said (N)" collapsed by default below the Save/Confirm buttons, listing each prior cross-operator note (author handle, ISO timestamp, verdict pill, note text).
   - Acceptance: opening a row with ‚Č•1 own prior `email_feedback` row shows the latest `prose_notes` in the textarea on first render (no extra click). Opening a row with ‚Č•1 cross-operator row shows a "What others said (N)" toggle with N matching the count of distinct other-operator rows for that `(email_id, stage)`.

2. **Read-back fetch endpoint**: The panel fetches its read-back via a per-row GET when it mounts on a row with feedback.
   - Current: no read-back endpoint exists; the panel only writes.
   - Target: new endpoint `GET /api/automations/debtor-email/feedback?email_id=<uuid>&stage=<0-3>` returns `{ own_latest: {prose_notes, verdict, created_at} | null, others: [{operator_id_hash, verdict, prose_notes, created_at}] }`. Server-side service-role read; the response strips raw `operator_id` and substitutes a display-safe hash or handle.
   - Acceptance: GET request returns 200 with the shape above for a row that has feedback; returns `{own_latest: null, others: []}` for a row with no feedback; p95 round-trip latency < 300ms on the operator's network.

3. **Row-strip verdict indicator**: Each row in Stage 0‚Äď3 lists shows an 8px colored dot before the stage badge, encoding the latest verdict for that `(email_id, stage)`.
   - Current: no row-strip signal that a row has feedback.
   - Target: an 8px dot inside the leftmost row column. Color tokens: `--v7-lime` (confirm), `--v7-amber` (override), `--v7-brand-secondary` (unclear), transparent dashed circle (no feedback yet). The dot is decoupled from the stage badge ‚ÄĒ no badge restyling.
   - Acceptance: every row with ‚Č•1 `email_feedback` row for the active stage renders a filled dot in the correct color; rows with no feedback render the empty dashed dot. The selected-row left border treatment (`--brand-primary`) is preserved.

4. **Override-note coupling cue**: The override picker shows persistent inline microcopy stating that the textarea note will be attached on submit.
   - Current: picker and textarea are visually independent; operators don't realize their note is attached on override submit.
   - Target: a small helper line under the override picker (and one under the textarea label) reads `‚§ď Override + note save together`, colored `--v7-amber`. No animation, no tooltip ‚ÄĒ persistent text.
   - Acceptance: in `state='dirty'` mode the helper line is present in the rendered DOM; the existing Phase 82.4 Plan 04 dual-write (override + `fireFeedback`) fires unchanged.

5. **Save vs Confirm clarity**: The two write actions inside `StageFeedbackPanel` have differentiated icons + microcopy so the operator's first read identifies which one closes the row.
   - Current: two pill buttons of equal weight differentiated only by color.
   - Target: Save button ‚ÄĒ outline treatment, icon `ūüíĺ`, two-line label `Save note ¬∑ come back later`. Confirm button ‚ÄĒ filled lime, icon `‚úď`, two-line label `Confirm ¬∑ looks right`. Save fires `verdict='unclear'`; Confirm fires `verdict='confirm'` and auto-collapses the expander.
   - Acceptance: a snapshot test of the panel confirms both buttons render with the specified icon, label text, and visual treatment. Empty Confirm shows a soft confirm dialog ("Confirm without note?") ‚ÄĒ single click to proceed.

6. **Detail-pane stage scope**: The detail pane renders only stages that actually ran for the current row, plus a single collapsed pill for any future-not-yet-run stages.
   - Current: all 5 stages render with "Stage didn't run ‚ÄĒ override to set" placeholders for unrun ones.
   - Target: stages with `state === 'ok'` or `'dirty'` render as today; consecutive trailing `state === 'skipped'` stages collapse into a single `<div class="future-pill">` reading "Stages N-M haven't run yet ¬∑ expand to override pre-emptively". Click expands the original placeholders. Same behavior on all 4 stage tabs.
   - Acceptance: on a Stage 1 row that only ran Stage 0+1, the pane renders 2 expanded stage steps + 1 future-pill (covering Stages 2-4). Clicking the pill expands to the 4-stage current behavior. No regression on Stage 3 / Stage 4 rows where all stages ran.

7. **Contextual bottom-button morph**: The detail-pane footer's primary button morphs between Approve and Submit-override based on whether any visible stage is in `state='dirty'`.
   - Current: footer shows a constant `Approve` button alongside `Reject` and `Skip`.
   - Target: when `stages.some(s => s.state === 'dirty')` is false, the primary button reads `‚úď Approve verdicts that ran (Stages 0+N)` ‚ÄĒ batch-confirms every visible and unconfirmed stage. When one or more stages are dirty, it reads `‚úď Submit override (Stage N)` (composite "Submit overrides (Stages N, M)" when multiple). Visual treatment (primary lime) stays constant ‚ÄĒ only label + click handler swap.
   - Acceptance: with all stages clean the button label matches the "Approve verdicts that ran" pattern; clicking it issues one POST per visible stage with `verdict='confirm'`. After clicking "override stage" on one stage, the button label changes to "Submit override (Stage N)"; clicking it dispatches the existing 71/82 override path AND a `verdict='override'` `email_feedback` row.

8. **Narrow-viewport pass at 1280px**: The detail pane and feedback panel remain usable without overflow at 1280px viewport width.
   - Current: panel tested at ‚Č•1440px only; at 1280px buttons or textarea may overflow.
   - Target: detail pane sidebar fits 460px column; feedback panel buttons stack vertically if needed; textarea remains the full panel width. No horizontal scroll on the row strip.
   - Acceptance: a Playwright screenshot at 1280√ó800 shows the full feedback panel with both buttons visible and no horizontal scrollbar; a manual operator pass on a 1280px monitor confirms no overflow.

## Boundaries

**In scope:**
- Read-back of latest own + collapsed cross-operator notes on `StageFeedbackPanel` mount (per Requirement 1)
- New `GET /api/automations/debtor-email/feedback` read endpoint (Requirement 2)
- Row-strip verdict indicator dot (Requirement 3) across Stage 0‚Äď3 lists
- Override coupling microcopy (Requirement 4)
- Save vs Confirm visual + microcopy refresh (Requirement 5)
- Detail-pane stage scope rewrite ‚ÄĒ future-pill collapse (Requirement 6)
- Bottom-button context morph (Requirement 7)
- 1280px viewport fitness (Requirement 8)
- Tests for each new behavior (vitest for client components, route handler test for the GET endpoint)

**Out of scope:**
- Multi-stage flag-on-override (auto-cascade or chip strip prompt) ‚ÄĒ deferred to V9.0 synthesis. The 82.5 model is **manual only**: operator switches tabs to flag upstream.
- Bulk operations on feedback (multi-row confirm, batch override) ‚ÄĒ not a 82.5 problem.
- V9.0 synthesis layer reading from `email_feedback` ‚ÄĒ separate phase.
- Feedback analytics dashboard ‚ÄĒ separate phase.
- Cross-operator permission model ‚ÄĒ Phase 82.5 assumes all operators see all notes (no scoping).
- Hard guard on Confirm-without-note ‚ÄĒ 82.5 uses a soft dialog (Requirement 5).
- Mobile / <1280px viewports ‚ÄĒ explicitly deferred; documented as known limitation.
- Stage 4 (Handler) feedback surface ‚ÄĒ 82.4 already excluded Stage 4 from the panel; 82.5 inherits that exclusion.
- Changes to the snapshot cron (`emailFeedbackSnapshot`) ‚ÄĒ 82.4 ships it; 82.5 doesn't touch it.

## Constraints

- **Hard-separation lock (RFC `docs/agentic-pipeline/README.md`)**: a row exists in EXACTLY ONE of `swarm_noise_categories` (Stage 1) or `swarm_intents` (Stage 3) ‚ÄĒ never both. No change in 82.5; the read-back endpoint MUST query `email_feedback.stage` directly and never blur the two registries.
- **No DB schema changes.** Reuse `public.email_feedback` as shipped by 82.4. No new tables, no new columns.
- **Service-role only.** All reads/writes go through `createAdminClient()` server-side. RLS denies anon and authenticated entirely.
- **Server-stamped operator_id.** The read endpoint MUST NOT accept `operator_id` from the request body or expose raw operator IDs in the response (`others[].operator_id_hash` is a display-safe substitute).
- **V7 design tokens only.** No new color values introduced; all surfaces use existing tokens from `web/app/globals.css` `[data-theme="dark"]` block.
- **p95 read-back latency < 300ms** at the operator's network (per Requirement 2 acceptance).
- **No new MCP tools, no new env vars.** 82.5 is purely a UI + read-endpoint phase.
