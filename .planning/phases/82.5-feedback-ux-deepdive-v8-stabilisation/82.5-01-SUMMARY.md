# Phase 82.5 Plan 01: Feedback Data Layer & Read Endpoint Summary

Server-side data layer for Phase 82.5: shared types, two server loaders (operator display map + feedback map), and a new `GET` handler on the existing feedback route — enabling R2 read-back and the D-1 server-side prefetch infrastructure that downstream UI plans consume.

## What Shipped

### 1. `web/lib/automations/debtor-email/feedback/types.ts` (new)

Plain TS contract module. Three named exports:

- `FeedbackOtherNote` — `{ display_name, verdict, prose_notes, created_at }`
- `FeedbackReadBack` — `{ own_latest: { prose_notes, verdict, created_at } | null, others: FeedbackOtherNote[] }`
- `FeedbackMap = Record<string /* email_id */, FeedbackReadBack>`

Closed `verdict` union (`"confirm" | "override" | "unclear"`) matches the POST handler's `Verdict`. No imports, no runtime code.

### 2. `web/lib/automations/debtor-email/feedback/load-operator-display.ts` (new)

`loadOperatorDisplayMap(admin, operatorIds) → Promise<Record<string, string>>`.

- Empty input → `{}` (no DB roundtrip).
- Dedups operatorIds, single `.schema("auth").from("users").select("id, email").in("id", unique)` IN query — no N+1.
- Resolves `email.split("@")[0]` per id; falls back to `id.slice(0, 8)` on null email OR DB error.
- Service-role required (caller contract; not verified inside). Raw `auth.users` rows never leave the function — CONTEXT D-3 privacy posture.

### 3. `web/lib/automations/debtor-email/feedback/load-feedback-map.ts` (new)

`loadFeedbackMap(admin, emailIds, stage, viewerId) → Promise<FeedbackMap>`.

- Seeds every input emailId with `{ own_latest: null, others: [] }` BEFORE the query — callers receive a total map even for ids with no feedback.
- Empty input → seeded empty map (no DB roundtrip).
- One `.from("email_feedback").select(...).in("email_id", ids).eq("stage", stage).order("created_at", { ascending: false })`.
- Calls `loadOperatorDisplayMap` with the distinct operator_id set after the main SELECT settles (minimal IN list).
- First-write-wins (Pattern E): tracks `ownClaimed: Set<email_id>` and `otherClaimed: Set<email_id|operator_id>`; first viewer row per email_id becomes `own_latest`; first row per other operator becomes that operator's entry in `others`.
- Response never includes raw `operator_id` (only resolved `display_name`).

### 4. `web/app/api/automations/debtor-email/feedback/route.ts` (extended — POST untouched)

New named export `GET(req: NextRequest): Promise<NextResponse>` alongside existing POST.

- `FeedbackReadQuery = z.object({ email_id: z.string().uuid(), stage: z.union([z.literal(0..3)]) })`.
- Same auth guard as POST (`createClient` + `auth.getUser` → 401 on null user).
- Reads `email_id` and `stage` from `req.nextUrl.searchParams` (stage parsed via `Number()`).
- 400 on `safeParse` failure.
- Delegates to `loadFeedbackMap(admin, [email_id], stage, user.id)` and returns `map[email_id] ?? { own_latest: null, others: [] }` with status 200.
- Response body: `{ own_latest, others }` — NOT keyed by email_id (single-row API).
