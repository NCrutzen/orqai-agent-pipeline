# Phase 7.1: Test Pipeline Tech Debt - Research

**Researched:** 2026-03-01
**Domain:** Integration gap resolution and template hygiene across deploy/test pipeline
**Confidence:** HIGH

## Summary

Phase 7.1 addresses three discrete tech debt items identified in the V2.0 milestone audit. These are not feature work -- they are integration gaps and orphaned artifacts that, if left unresolved, create confusion or silent failures when the Phase 8 iteration loop depends on deploy and test infrastructure.

The three items are: (1) the `agents.responses.create()` SDK method has no documented REST mapping in `orqai-api-endpoints.md`, meaning the tester's REST fallback path for agent invocation is underspecified, (2) `@orq-ai/evaluators` is referenced in `tester.md` but never declared with a version pin in any install surface, and (3) `deploy-log.json` is an orphaned JSON template that does not match the live `deploy-log.md` markdown format and is missing `unchanged` status support.

**Primary recommendation:** Address all three items as straightforward file edits -- no new features, no architectural changes, no code to write. This is purely documentation and template cleanup.

## Architecture Patterns

### Pattern 1: SDK-to-REST Mapping Documentation

**What:** Each SDK method that subagents use must have a documented REST equivalent in `orqai-api-endpoints.md` so the MCP-first/REST-fallback pattern works.

**Current gap:** `tester.md` calls `agents.responses.create()` but the REST mapping `POST /v2/agents/{id}/execute` is already documented in `orqai-api-endpoints.md` -- the issue is the lack of an explicit SDK-to-REST cross-reference.

**What needs to happen:**
1. Add an "SDK Method Mapping" section to `orqai-api-endpoints.md` that explicitly maps `agents.responses.create()` to `POST /v2/agents/{agent_id}/execute`
2. Add a REST fallback note in `tester.md` Step 7.1 that shows the equivalent `curl` / fetch call for agent invocation when SDK is unavailable

**Files affected:**
- `orq-agent/references/orqai-api-endpoints.md` -- add SDK mapping section
- `orq-agent/agents/tester.md` -- add REST fallback path for agent invocation in Step 7.1

### Pattern 2: Package Declaration in Install Surface

**What:** Every npm package referenced by subagents must be declared with a version pin in at least one install-surface file.

**Current state:**
- `@orq-ai/node@^3.14.45` -- declared in ROADMAP stack constraints (serves as source of truth)
- `@orq-ai/evaluatorq@^1.1.0` -- declared in ROADMAP stack constraints
- `@orq-ai/evaluators` -- NOT declared anywhere

**What needs to happen:**
1. Add `@orq-ai/evaluators@^1.1.0` to the ROADMAP stack constraints alongside the other two pins
2. Add it to `orq-agent/SKILL.md` in a "V2.0 Dependencies" section or alongside existing references
3. The install script does NOT install npm packages (it only copies files) -- so the declaration is informational, not executable. The tester subagent is responsible for ensuring packages are available at runtime.

**Files affected:**
- `orq-agent/SKILL.md` -- add package declaration
- Optionally: `orq-agent/commands/test.md` -- mention required packages in a prerequisites note

### Pattern 3: Template Lifecycle (Remove Orphaned Template)

**What:** Templates in `orq-agent/templates/` must match the live format used by their consumers. Orphaned templates cause confusion.

**Current state of `deploy-log.json`:**
- JSON format with `"status": "created|updated|failed"` -- missing `unchanged`
- Live format is `deploy-log.md` (markdown) with append-only sections and `unchanged` status
- `deploy-log.json` is listed in SKILL.md template table but NOT referenced by deployer.md or deploy.md
- deploy.md Step 7.1 generates markdown directly -- never reads the JSON template

**Decision: Remove `deploy-log.json`.**

Rationale:
- The deployer generates `deploy-log.md` directly from its Phase 5 instructions -- never reads a template
- The JSON template was created during Phase 5 (V2.0 Foundation) as a placeholder before the deployer was built
- Updating it to match the markdown format would create a JSON representation of a markdown document -- structurally nonsensical
- Other templates (`test-results.json`, `iteration-log.json`) are actually consumed by their subagents (tester reads `test-results.json` schema); `deploy-log.json` is not

**Files affected:**
- `orq-agent/templates/deploy-log.json` -- DELETE (orphaned)
- `orq-agent/SKILL.md` -- remove `deploy-log.json` from template table, add note that deploy-log format is defined inline in `deployer.md` Phase 5

### Anti-Patterns to Avoid

- **Adding the REST fallback as a separate code path in tester.md:** The tester already uses the `@orq-ai/node` SDK for agent invocation. The SDK handles the REST call internally. The "REST fallback" documentation is for when the SDK itself is unavailable or for manual debugging -- it should be a documentation note, not a parallel implementation.
- **Creating a new deploy-log.md template file:** The deployer generates deploy-log.md from inline instructions. Adding a template would create two sources of truth. The inline format in deployer.md Phase 5 IS the template.
- **Pinning @orq-ai/evaluators to a different version than @orq-ai/evaluatorq:** They are peer dependencies and share the same version. Pin both to `^1.1.0`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Agent invocation during tests | Custom HTTP client for agent execution | `@orq-ai/node` SDK `agents.responses.create()` | SDK handles auth, retries, response parsing |
| Local evaluator scoring | Custom scorer implementations | `@orq-ai/evaluators` package | Provides tested cosine similarity and threshold evaluators |

**Key insight:** This phase involves zero custom code. Every item is a documentation or file-system edit.

## Common Pitfalls

### Pitfall 1: Over-Engineering the SDK-to-REST Mapping

**What goes wrong:** Researcher/planner interprets "tester.md REST fallback path covers agent invocation" as requiring a full REST-based agent invocation code path alongside the SDK path.
**Why it happens:** The tester already has a MCP-first/REST-fallback pattern for dataset operations and agent retrieval. It seems natural to extend this to agent invocation.
**How to avoid:** The SDK `agents.responses.create()` IS the REST call -- the SDK wraps `POST /v2/agents/{id}/execute`. The documentation gap is about making this mapping explicit, not about building a parallel path. Add a note in tester.md that says "SDK method maps to REST endpoint X" so that if the SDK is unavailable, a developer knows the raw endpoint.
**Warning signs:** Plan includes "implement REST fallback for agent invocation" as a task.

### Pitfall 2: Breaking SKILL.md Template Table Format

**What goes wrong:** Removing `deploy-log.json` from the SKILL.md template table but not cleaning up the reference or leaving a broken table row.
**Why it happens:** SKILL.md is a carefully formatted index file. Editing table rows can break alignment.
**How to avoid:** After removing the row, verify the table renders correctly. The remaining templates are: `agent-spec.md`, `orchestration.md`, `dataset.md`, `readme.md`, `tools.md`, `test-results.json`, `iteration-log.json`.

### Pitfall 3: Confusing @orq-ai/evaluators with @orq-ai/evaluatorq

**What goes wrong:** Treating them as interchangeable or declaring only one.
**Why it happens:** Similar names, same version, peer dependency relationship is non-obvious.
**How to avoid:** Be explicit: `evaluatorq` is the framework (runs experiments, orchestrates evaluations). `evaluators` is the library (provides scorer implementations like cosine similarity). Both are needed. Both are `^1.1.0`.
