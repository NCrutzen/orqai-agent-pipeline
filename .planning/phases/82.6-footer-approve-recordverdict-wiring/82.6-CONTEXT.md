# Phase 82.6 — Wire footer "Approve" into the recordVerdict pipeline

## Goal

Make the Phase 82.5 footer "✓ Approve (Stages X+Y)" actually advance rows out of the Stage 1 queue. Currently the click only writes feedback rows to `email_feedback`; `automation_runs.status` stays `predicted` so the operator sees the row stay visible (UAT surfaced 2026-05-15).

## Background

Phase 82.5 introduced the morphing footer button (`detail-pane.tsx:432-436` — "Approve" / "Submit override"). The approve branch posts `verdict='confirm'` per ok stage to `/api/automations/debtor-email/feedback` — a Phase 82.4 telemetry/learning capture, **not** a pipeline progression action.

The action that progresses rows is `recordVerdict` (`app/(dashboard)/automations/[swarm]/stage-1/actions.ts`, Phase 56.7-03):
1. Flips `automation_runs.status: predicted → feedback`
2. Writes `agent_runs` telemetry
3. Fires the `verdict-recorded` Inngest event → registry-driven `classifier-verdict-worker` performs the swarm dispatch (per `swarm_noise_categories.action`)
4. Emits broadcast → queue UI invalidates

Stage 1 widget's override flow already calls `recordVerdict` (decision='approve') alongside the override POST (`_shell/components/stage-1-widget.tsx:174`). The approve flow needs equivalent wiring.

**Operator-visible symptom:** Click "✓ Approve (Stages 0+1)" → 2 POSTs to `/api/automations/debtor-email/feedback` succeed (200 OK, rows written) → row stays visible in Stage 1 list. Confirmed via Browserless probe on commit 7112c78.

## Required behavior

After operator clicks footer approve on Stage 1:
1. (current) POST `verdict='confirm'` to `/api/automations/debtor-email/feedback` per ok stage — keep, this is the feedback substrate.
2. (NEW) Call `recordVerdict({ swarm_type, automation_run_id, email_id, rule_key, decision: 'approve', message_id, source_mailbox, entity, predicted_category })` — flips status, fires dispatch.
3. Row leaves Stage 1 queue via the existing realtime broadcast path.

Failure modes (per `stage-1-widget.tsx:188-191`): if `recordVerdict` fails after the feedback write, treat as non-fatal — the queue UI will reconcile on the next server roundtrip.

## Open design questions for /gsd-discuss-phase 82.6

- **D-01: Coupling.** `recordVerdict` requires `message_id` + `source_mailbox` (currently fetched via `/api/automations/debtor-email/email-body?id={row.id}` inside the widget). Should detail-pane fetch the same, or should the page-level shell pre-resolve via a prop, or should the action signature change to accept just `(row_id, decision)` and resolve internally?
- **D-02: Other stages.** Stage 3 has its own dispatch mechanism (Phase 80 stage-3-dispatcher per CLAUDE.md). Stage 0, 2, 4 may not need a footer approve at all. Design: how does the footer behave per-stage? Per-stage `onApprove` callback prop? Per-stage server action registry?
- **D-03: Pre-existing T8 test.** The hard-separation test (`detail-pane.test.tsx:167`) has been failing since Phase 82-01 (commit `f2de682`). Worth fixing as part of this phase or out-of-scope?

## Acceptance

- Operator clicks footer approve → row leaves Stage 1 queue within 1-2s
- `automation_runs.status` transitions `predicted` → `feedback` → eventually `completed`
- Classifier-verdict-worker fires the swarm dispatch (handler action per `swarm_noise_categories.action`)
- Browserless probe equivalent of `probe-approve-mode.ts` confirms the full round-trip including row disappearance

## Non-goals

- Refactoring `recordVerdict` itself — the action is stable, only its callers change.
- Touching Stage 3's dispatch path (handled by Phase 80).
- The pre-existing T8 hard-separation test failure (carry forward unless trivially fixable).
