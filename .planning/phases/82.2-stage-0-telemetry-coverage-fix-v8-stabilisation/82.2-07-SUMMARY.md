# Phase 82.2 Plan 07: Thin debtor-email ingest + align sales-email payload — Summary

D-A thin-ingest refactor of `web/app/api/automations/debtor-email/ingest/route.ts`. The route's responsibilities are now exactly: validate auth → load `labeling_settings` (with `ingest_enabled` gate) → fetch Outlook meta+body (via Zapier-SDK NXT-S3 flow) → resolve canonical `email_pipeline.emails.id` → INSERT pending placeholder `automation_runs` row → fire `stage-0/email.received` UNCONDITIONALLY → return 200. The regex / whitelist / auto-action / Stage 1 emit chain that used to live here is now wholly owned by the Plan-06 thick Stage 1 worker (`classifier-screen-worker.ts`), reached only after Stage 0 returns `verdict='safe'`. Plus a field-set alignment in `sales-email/ingest` and a Stage 0 → Stage 1 payload extension (`mailbox_id`, `from`, `fromName`, `receivedAt`) so the Plan 06 worker can write its audit rows without re-fetching.

## What Changed

### Task 1 — Stage 0 worker payload extension
- Added `mailbox_id`, `from`, `fromName`, `receivedAt` as optional/nullable fields on **both** `stage-0/email.received` and `classifier/screen.requested` events (`events.ts`).
- Stage 0 worker (`stage-0-safety-worker.ts`) now destructures + threads these fields through both emit sites: the happy-path `forward-to-classifier` and the operator-driven `emit-classifier-override` (safety_overridden short-circuit). The budget-breach emit does NOT need them (halts before dispatch).
- Verified by `stage-0-safety-worker.test.ts` (12/12 green) — existing assertions still pass because the new fields are optional/nullable; no fixture lock-in needed.

**Stage 0 worker UPDATE-by-id semantics confirmed compatible with universal placeholder**: the worker at L217–252 does `UPDATE automation_runs … WHERE id=automation_run_id AND status='pending'` with an INSERT fallback only when `automation_run_id` is absent (legacy/non-ingest entry points). Post-D-A, the UPDATE path always fires for ingest-originated emails; the INSERT fallback only runs for non-ingest entry points. → No INSERT-duplication risk.

**Replay-safety**: API route is NOT an Inngest function. Stage 0 placeholder INSERT happens once per HTTP request; Zapier's at-most-once delivery + `email_pipeline.emails.source_id` dedupe handle Zapier-side retries. Worst case = orphan pending row (swept by `automation-runs-sweeper`).
