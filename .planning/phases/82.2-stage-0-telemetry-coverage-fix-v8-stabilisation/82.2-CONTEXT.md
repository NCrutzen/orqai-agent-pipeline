<domain>
## Phase Boundary

**Re-scoped 2026-05-12 after research.** The original framing ("telemetry-only fix to the worker") was wrong. The dominant Stage 0 coverage gap (26–45% on debtor mailboxes) is not in `stage-0-safety-worker.ts` — it is in `web/app/api/automations/debtor-email/ingest/route.ts`, which only fires `stage-0/email.received` when the pre-classifier returns `category==='unknown'`. Whitelisted and predicted-category debtor emails currently bypass Stage 0 entirely and fire Stage 1 directly. This is a **safety-ordering bug** (Stage 0 was added after Stage 1 already existed; the wiring was never inverted), not a telemetry bug.

**True boundary:** make Stage 0 the unconditional first step of the canonical pipeline for every inbound email on every active swarm. Stage 1 (and any auto-action) only fires on a Stage 0 `safe` verdict. Plus the original telemetry hygiene work (worker emit refactor, ≤30d backfill, daily coverage probe).

The worker's safety verdict logic itself is still untouched — this phase changes **when** Stage 0 runs and **what listens for its safe verdict**, not how the safety decision is computed.

</domain>

<hypothesis>
## Root cause (CONFIRMED by research 2026-05-12 — see `82.2-RESEARCH.md`)

The original telemetry hypothesis was **mostly wrong**. Verified findings:

1. **Worker is mostly fine.** `stage-0-safety-worker.ts` main-path `emitPipelineEvent` (L255) lives inside a single `step.run("persist-verdict")` that handles BOTH UPDATE-existing and INSERT-fresh — there is no bifurcation gap. Test at L249–254 confirms both `safe` and `injection_suspected` emit.
2. **Real dominant gap is in `debtor-email/ingest/route.ts`.** `fireStage0Event()` is gated on `r.category === "unknown"` (L374–376). Whitelisted (auto-action L468+) and predicted-category-non-unknown (L401–423) emails fire Stage 1 directly and **never trigger Stage 0**. Sales-email ingest fires Stage 0 unconditionally — that's why sales = 91% and debtor = 26–45%.
3. **Small worker gaps remain** — the `safety_overridden` short-circuit (L86–105) and budget-breach branch (L167–182) DO skip `emitPipelineEvent`. Low volume but real; D-01 still applies.
4. **`suspected` branch already emits today.** Zero rows in 7d = zero injection attempts in 7d (plausible mailbox mix), not a missing emit. D-06 test is still valuable as regression cover.
5. **The wiring is backwards.** Stage 0 was added after Stage 1 already existed; the ingest route was never inverted to put Stage 0 first. This is a **safety-ordering bug**, not a telemetry bug.

</hypothesis>

<must_haves>
## Acceptance criteria

**Ordering (new — added on re-scope):**
- Every inbound email on every active swarm fires `stage-0/email.received` BEFORE any Stage 1 work. The ingest route returns 200 immediately after firing Stage 0; it does not run regex classification, auto-actions, or emit Stage 1 events itself.
- Stage 1 (`classifier/screen.requested`) is emitted only by the Stage 0 worker after a `safe` verdict. `injection_suspected` and `over_budget` verdicts halt the pipeline (no Stage 1 fired).
- Whitelist auto-actions and predicted-category bulk-review handling move OUT of the ingest route. Stage 1 worker (or its downstream) becomes the single owner of category-based action dispatch.

**Telemetry (original — preserved):**
- Every email that triggers `classifier/screen.requested` (Stage 1) has exactly one corresponding `pipeline_events` row with `stage=0` for the same `email_id` and `swarm_type`.
- Stage 0 coverage ≥ 99% on all 5 active mailboxes for the 7 days following the fix.
- Backfill: historical emails from the last 30 days that have Stage 1+ rows but no Stage 0 row get a Stage 0 row reconstructed from `automation_runs` where possible, or a `decision='unknown_legacy'` marker where source data is unavailable.
- `suspected` branch retains its emit (regression cover via D-06 test).
- No regression in Stage 0 safety **verdict** logic — the regex screen, LLM injection check, and budget counter are not modified. Only their **call site** and **what listens for the safe verdict** change.

</must_haves>

<decisions>
## Implementation Decisions

### Safety ordering (added on re-scope 2026-05-12)

- **D-A: Thin ingest, Stage 0 first, always.** `web/app/api/automations/debtor-email/ingest/route.ts` reduces to: persist `email_pipeline.emails` row → fire `stage-0/email.received` (unconditional, for every inbound) → return 200. **All other logic moves out of the ingest route:** regex classification, auto-action dispatch, predicted-category bulk-review INSERT, Stage 1 `pipeline_events` emit. This matches the canonical architecture in `docs/agentic-pipeline/README.md` and the existing sales-email ingest pattern.
- **D-B: Stage 1 is fired only by the Stage 0 worker on `safe` verdict.** The Stage 0 worker's existing `classifier/screen.requested` emit (today fires inside the worker for `safe` verdicts only) becomes the sole entry point into Stage 1. The Stage 1 worker (or its downstream chain) becomes the new owner of regex classification + auto-action dispatch + bulk-review queueing — whatever the old ingest route did post-classification. `injection_suspected` and `over_budget` verdicts continue to halt the pipeline (no Stage 1 fired); they keep their existing audit/operator surfaces.
- **D-C: Hard cutover at deploy.** No feature flag, no shadow mode, no per-swarm rollout. After deploy, every new inbound email runs the new ordering. In-flight emails finish on the old path. The backfill (D-03) covers the 30d historical gap. The change is a safety improvement — keeping the old unsafe path behind a flag is wrong by default.

### Emit-point architecture

- **D-01: Single emit at function exit.** Refactor `stage-0-safety-worker.ts` so every code path falls through to ONE `emitPipelineEvent` call at the end (or via try/finally). New branches added later cannot forget to emit. This is the structural fix for the class of bug, not a per-branch patch. The override (L87) and budget-breach (L169) branches no longer emit inline — they set a verdict variable and fall through to the single emit at the bottom. **Post-research scope note:** the main-path emit (L255) already fires correctly on both `safe` and `injection_suspected` and on both UPDATE-existing and INSERT-fresh — D-01 only needs to add `emitPipelineEvent` coverage to the override and budget-breach branches.
- **D-02: Preserve all three event sub-types.** The same payloads as today — `operator-override`, `budget-breach`, and `main-path` stay distinguishable in `pipeline_events`. Refactor only changes WHERE the emit happens, not WHAT it emits. Downstream consumers (Bulk Review, V9 trace) read the existing shape unchanged.

### Backfill strategy + sources

- **D-03: Inngest one-shot function as the backfill runtime.** Lives at `web/lib/inngest/functions/stage-0-backfill.ts`, triggered by event `pipeline.stage0.backfill`. Replay-safe via `step.run` boundaries (per CLAUDE.md Phase 65 learning: replay-id generation must be inside `step.run`). Idempotent via UPSERT on `(email_id, stage)`. Run once manually after the worker fix lands; re-runnable safely.
- **D-04: Source priority `automation_runs` → `unknown_legacy`.** For each email_id missing `pipeline_events.stage=0`: look up `automation_runs` row with `stage='stage_0_safety_pending'` or later; reconstruct `decision='safe'` from the fact that Stage 1 ran successfully (Stage 1 only fires post-safety verdict). If `automation_runs` is missing, write `decision='unknown_legacy'`. **Do NOT re-run safety logic** — risks divergence if rules changed since the original run, plus LLM cost.
- **D-05: Window = 30 days back from worker-fix deploy.** Captures the 219+ Stage-1-no-Stage-0 rows currently visible plus the prior 3 weeks. Aligns with the V9 corpus window. Older history not backfilled.

### Suspected-branch verification

- **D-06: Code-read + synthetic injection test.** Researcher reads `stage-0-safety-worker.ts` to identify the suspected-branch trigger conditions; planner writes an integration test that submits a known prompt-injection payload (e.g. `"IGNORE PREVIOUS INSTRUCTIONS"`) through the relevant ingest path and asserts a `pipeline_events` row with `decision='suspected'` is written. Test lives in `web/lib/inngest/functions/__tests__/` alongside existing worker tests for regression coverage.

### Coverage guard / regression prevention

- **D-07: Daily Inngest cron coverage probe.** New scheduled function `web/lib/inngest/functions/stage-0-coverage-probe.ts`. Runs at `TZ=Europe/Amsterdam 0 9 * * 1-5` (per CLAUDE.md cron default + TZ requirement). Window = trailing 24h. For every `email_pipeline.emails` row with a Stage-1 `pipeline_events`, asserts a Stage-0 `pipeline_events` row exists.
- **D-08: Breach surface = STATE.md session-note.** If coverage <99% on any of the 5 active mailboxes, append a structured session-note line to `.planning/STATE.md` so it surfaces in `/gsd-progress`. No Slack/email/Twilio. Operator/owner reads `STATE.md`. (If a future incident warrants paging, that's its own phase — telemetry hygiene doesn't.)

### Claude's Discretion

- Exact UPSERT key for the backfill (likely `(email_id, stage, swarm_type)`) — planner determines from current `pipeline_events` unique constraints.
- Whether the daily probe writes to a new `pipeline_health` table OR only to STATE.md — left to planner. Decision D-08 only requires STATE.md as the breach surface; whether to additionally persist coverage history is the planner's call.
- Test framework choice for D-06 (vitest vs whatever the existing `__tests__/` directory uses) — match existing pattern.

</decisions>

<specifics>
## Specific Ideas

- The class of bug here is "per-branch emit responsibility" — D-01 (single emit at function exit) is chosen because it removes the class, not just the instance. Future branches added to the worker cannot forget to emit.
- Backfill writes `decision='unknown_legacy'` as an honest marker rather than `decision='safe'` when the source data isn't there to prove it was safe. V9 trace surface should treat `unknown_legacy` as "we don't have the original verdict" rather than hide it.
