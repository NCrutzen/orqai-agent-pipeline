<hypothesis>
## Root cause (to verify in discuss-phase)

`web/lib/inngest/functions/stage-0-safety-worker.ts` has three emit branches:
- L87 — operator override re-emit (`emit-classifier-override`)
- L169 — budget breach (`emit-budget-breach`)
- L255 — main path (`emitPipelineEvent`)

The debtor-email `/ingest/route.ts` pre-creates an `automation_runs` row with `stage='stage_0_safety_pending'` (Phase 64 SAFE-01..03 pattern). The worker likely takes an UPDATE-existing branch on these and skips the `emitPipelineEvent` call that the INSERT-fresh branch makes. Sales-email's `/ingest/route.ts` does its own double-emit per the inline warning ("double-emit stage-0/email.received"), which is why sales-email coverage is 91% and debtor coverage is 26–45%.

Alternative: a fast-path return in the safety logic when the regex screen says safe AND no LLM call is needed.

</hypothesis>
