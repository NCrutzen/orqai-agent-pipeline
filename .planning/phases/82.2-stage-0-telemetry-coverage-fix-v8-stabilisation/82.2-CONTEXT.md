<hypothesis>
## Root cause (researcher to verify)

`web/lib/inngest/functions/stage-0-safety-worker.ts` has three emit branches:
- L87 ‚ÄĒ operator override re-emit (`emit-classifier-override`)
- L169 ‚ÄĒ budget breach (`emit-budget-breach`)
- L255 ‚ÄĒ main path (`emitPipelineEvent`)

The debtor-email `/ingest/route.ts` pre-creates an `automation_runs` row with `stage='stage_0_safety_pending'` (Phase 64 SAFE-01..03 pattern). The worker likely takes an UPDATE-existing branch on these and skips the `emitPipelineEvent` call that the INSERT-fresh branch makes. Sales-email's `/ingest/route.ts` does its own double-emit per the inline warning ("double-emit stage-0/email.received"), which is why sales-email coverage is 91% and debtor coverage is 26‚Äď45%.

Alternative: a fast-path return in the safety logic when the regex screen says safe AND no LLM call is needed.

</hypothesis>

<decisions>
## Implementation Decisions

### Emit-point architecture

- **D-01: Single emit at function exit.** Refactor `stage-0-safety-worker.ts` so every code path falls through to ONE `emitPipelineEvent` call at the end (or via try/finally). New branches added later cannot forget to emit. This is the structural fix for the class of bug, not a per-branch patch. The override (L87) and budget-breach (L169) branches no longer emit inline ‚ÄĒ they set a verdict variable and fall through to the single emit at the bottom.
- **D-02: Preserve all three event sub-types.** The same payloads as today ‚ÄĒ `operator-override`, `budget-breach`, and `main-path` stay distinguishable in `pipeline_events`. Refactor only changes WHERE the emit happens, not WHAT it emits. Downstream consumers (Bulk Review, V9 trace) read the existing shape unchanged.

### Backfill strategy + sources

- **D-03: Inngest one-shot function as the backfill runtime.** Lives at `web/lib/inngest/functions/stage-0-backfill.ts`, triggered by event `pipeline.stage0.backfill`. Replay-safe via `step.run` boundaries (per CLAUDE.md Phase 65 learning: replay-id generation must be inside `step.run`). Idempotent via UPSERT on `(email_id, stage)`. Run once manually after the worker fix lands; re-runnable safely.
- **D-04: Source priority `automation_runs` ‚Üí `unknown_legacy`.** For each email_id missing `pipeline_events.stage=0`: look up `automation_runs` row with `stage='stage_0_safety_pending'` or later; reconstruct `decision='safe'` from the fact that Stage 1 ran successfully (Stage 1 only fires post-safety verdict). If `automation_runs` is missing, write `decision='unknown_legacy'`. **Do NOT re-run safety logic** ‚ÄĒ risks divergence if rules changed since the original run, plus LLM cost.
- **D-05: Window = 30 days back from worker-fix deploy.** Captures the 219+ Stage-1-no-Stage-0 rows currently visible plus the prior 3 weeks. Aligns with the V9 corpus window. Older history not backfilled.

### Suspected-branch verification

- **D-06: Code-read + synthetic injection test.** Researcher reads `stage-0-safety-worker.ts` to identify the suspected-branch trigger conditions; planner writes an integration test that submits a known prompt-injection payload (e.g. `"IGNORE PREVIOUS INSTRUCTIONS"`) through the relevant ingest path and asserts a `pipeline_events` row with `decision='suspected'` is written. Test lives in `web/lib/inngest/functions/__tests__/` alongside existing worker tests for regression coverage.

### Coverage guard / regression prevention

- **D-07: Daily Inngest cron coverage probe.** New scheduled function `web/lib/inngest/functions/stage-0-coverage-probe.ts`. Runs at `TZ=Europe/Amsterdam 0 9 * * 1-5` (per CLAUDE.md cron default + TZ requirement). Window = trailing 24h. For every `email_pipeline.emails` row with a Stage-1 `pipeline_events`, asserts a Stage-0 `pipeline_events` row exists.
- **D-08: Breach surface = STATE.md session-note.** If coverage <99% on any of the 5 active mailboxes, append a structured session-note line to `.planning/STATE.md` so it surfaces in `/gsd-progress`. No Slack/email/Twilio. Operator/owner reads `STATE.md`. (If a future incident warrants paging, that's its own phase ‚ÄĒ telemetry hygiene doesn't.)

### Claude's Discretion

- Exact UPSERT key for the backfill (likely `(email_id, stage, swarm_type)`) ‚ÄĒ planner determines from current `pipeline_events` unique constraints.
- Whether the daily probe writes to a new `pipeline_health` table OR only to STATE.md ‚ÄĒ left to planner. Decision D-08 only requires STATE.md as the breach surface; whether to additionally persist coverage history is the planner's call.
- Test framework choice for D-06 (vitest vs whatever the existing `__tests__/` directory uses) ‚ÄĒ match existing pattern.

</decisions>

<specifics>
## Specific Ideas

- The class of bug here is "per-branch emit responsibility" ‚ÄĒ D-01 (single emit at function exit) is chosen because it removes the class, not just the instance. Future branches added to the worker cannot forget to emit.
- Backfill writes `decision='unknown_legacy'` as an honest marker rather than `decision='safe'` when the source data isn't there to prove it was safe. V9 trace surface should treat `unknown_legacy` as "we don't have the original verdict" rather than hide it.

</specifics>
