## Conclusion — Does the data support D-A re-wire as primary fix?

**YES, decisively.**

| Fix axis | Branch coverage | Gap share (7d) | Steady-state impact |
|---|---|---:|---|
| **D-A** (thin ingest + Stage-0-first re-wire, Plans 06+07) | `icontroller_delete` + `skipped_not_whitelisted` + `skipped_disabled` ingest branches | **184 / 232 (79.3%)** | Closes the dominant gap |
| **D-01** (worker single-emit, Plan 04) | L86 override + L169 budget-breach branches | **2 / 232 (0.9%)** | Closes the worker-emit class of bug, but the gap it visibly closes today is tiny |
| Worker `stage_0_safety_pending` orphans (NOT covered by either fix) | L154 rethrow path | **46 / 232 (19.8%)** | Out of scope — D-01 preserves the throw deliberately; these surface in Inngest failure logs, not pipeline_events |

**Implications for the locked plan execution order:**
1. **Plans 06 + 07 (D-A re-wire) carry the bulk of the coverage gain.** Without them, the ≥99% acceptance criterion is unreachable — confirmed by RESEARCH Risk #1 and now empirically anchored.
2. **Plan 04 (D-01 worker single-emit) is correct-by-design but small-by-volume.** It closes ~1% of the visible gap right now. The reason to ship it is **structural** (no more per-branch emit responsibility) and **forward-looking** (future operator-override traffic will emit correctly), not telemetry-coverage in week-1.
3. **Plan 09 (30d backfill) sizing is conservative.** 266 candidates fits comfortably in one batch; expect <60s wall-time for the manual one-shot trigger.
4. **The 46 `stage_0_safety_pending`-orphan worker failures (Q3, 81 `failed` + 34 `completed`-stuck) deserve a separate notice** — they are NOT closed by D-A or D-01. If the planner wants Stage 0 coverage to reach 100% (not just 99%), either:
   - (a) the L154 LLM-error rethrow needs a `decision='worker_error'` emit before throwing, OR
   - (b) the backfill's `unknown_legacy` marker absorbs them.
   The locked plan chooses (b) implicitly via D-04. That is acceptable, but the planner should know the steady-state floor is ~80% live coverage + ~20% backfill-derived `unknown_legacy` rows for the debtor swarm until the rethrow gets a separate emit.

**Operator has empirical confidence to approve D-C (hard cutover at deploy).** The downside risk (mis-firing Stage 0 for emails that should not run through the LLM check) is bounded by the existing budget counter and regex screen — both untouched by D-A/D-B/D-C — and the upside (closing 79% of the visible gap) is large enough to justify shipping without a feature flag.

## Methodology footnote

The plan specifies `supabase mcp execute_sql` for query execution. Because the MCP tool surface is not available in this worktree's agent, the queries were translated into PostgREST GETs + in-memory aggregation. The translations preserve query semantics:

- **Q1, Q5** — CTE + LEFT JOIN replaced by two paginated fetches into Sets keyed on `(email_id, swarm_type)`; the LEFT JOIN exclusion is `!s0Key.has(key)`.
- **Q2** — the LATERAL join into `automation_runs` is replaced by a Map keyed on `result.message_id`, populated from a single 8d fetch of all debtor-email `automation_runs`. The "most recent per message_id" tie-break is preserved.
- **Q3** — the OR predicate on `safety_overridden` / `error_message ILIKE '%budget%'` / `stage='stage_0_safety_pending'` is applied in JS over a 7d debtor-email fetch.
- **Q4** — single count via PostgREST `Range: 0-0` + `Prefer: count=exact` (returns the total via `Content-Range`).

No data was written. No PII (email content, sender names, subjects) was fetched — only `id`, `mailbox`, `source_id`, `swarm_type`, `stage`, `decision`, `result`, `status`, `error_message`. Q3 raw counts are aggregated; no individual `automation_runs.error_message` strings are reproduced.
