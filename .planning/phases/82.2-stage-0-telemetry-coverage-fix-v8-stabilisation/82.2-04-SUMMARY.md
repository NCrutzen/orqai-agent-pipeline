# Phase 82.2 Plan 04: Single-emit refactor of Stage 0 safety worker — Summary

Refactored `stage-0-safety-worker.ts` so the operator-override, budget-breach, and main-path branches all fall through to ONE `emitPipelineEvent` call inside a single `persist-and-emit` `step.run`. Closes the worker-side telemetry gaps in the override (pre-refactor L86–105) and budget-breach (pre-refactor L167–182) branches without changing what the worker decides; new branches added later cannot forget to emit because emission is mechanically guaranteed by the function exit shape.

## Behavior Changes

### Pipeline events (telemetry)

| Branch | Pre-refactor | Post-refactor |
|--------|---------------|----------------|
| operator-override (safety_overridden=true) | NO pipeline_events emit | One row, decision='safe', decision_details.emit_source='operator-override', safety_overridden=true |
| budget-breach (budgetCheck.breached=true) | NO pipeline_events emit | One row, decision='over_budget', decision_details.emit_source='budget-breach', reason+budget captured |
| main-path safe | One row, decision='safe' | One row, decision='safe', decision_details.emit_source='main-path' (+ existing keys) |
| main-path injection_suspected | One row, decision='injection_suspected' | One row, decision='injection_suspected', decision_details.emit_source='main-path' (+ existing keys) |

### Downstream sends (unchanged semantics)

- operator-override → `classifier/screen.requested` (preserved)
- budget-breach → `pipeline/budget_breached` (preserved)
- main-path safe → `classifier/screen.requested` (preserved)
- main-path injection_suspected → no downstream send (preserved)

### automation_runs writes (unchanged semantics, restructured)

- operator-override → NO write (preserved)
- budget-breach → NO write in worker (preserved; `budget-breach-handler` owns the failed-flip)
- main-path → UPDATE-by-id+status=pending / INSERT fallback (preserved, same payload shape)

### Verdict logic — completely untouched

`regexScreen`, `llmInjectionVerdict`, `budgetCheck`, `stripQuotedHistory`, `OrqClientTimeoutError` coercion, strip telemetry, original-body forwarding to Stage 1 — all identical to pre-refactor.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Removed worker-side automation_runs write on budget-breach**
- **Found during:** Task 3 grep-audit (Step 5 — automation_runs path audit + downstream consumer check).
- **Issue:** The initial Task 2 GREEN refactor (commit `2dd39e3`) added an `automation_runs` UPDATE in the worker's budget-breach branch (step "persist-and-emit"). The downstream `web/lib/inngest/functions/budget-breach-handler.ts` Inngest function (listens on `pipeline/budget_breached`) already owns this state transition — it UPDATEs the originating row to `status='failed'` (L36–50) and INSERTs a fresh Kanban-review row (L52–70). The worker's added write would race with the handler's UPDATE on the same row, double-writing the result jsonb + error_message.
- **Fix:** Removed the worker's automation_runs write on budget-breach; restored the pre-82.2-04 contract that the handler owns budget-breach state on automation_runs. The worker still emits the `pipeline_events` row + the `pipeline/budget_breached` event so telemetry coverage (the goal of D-01) is met.
- **Files modified:** `web/lib/inngest/functions/stage-0-safety-worker.ts`
- **Commit:** `1b06110`

### Note on plan Task 2 step 5 wording

The plan said: "budget-breach → write `status='failed'`, `topic=null`, `error_message=verdictSource.reason`, `result={ stage: 'stage_0_safety', budget: verdictSource.budget }` (preserve today's budget-breach-handler contract; verify against `budget-breach-handler.ts` so it doesn't double-write)." The grep verification surfaced exactly the double-write case the parenthetical warned against; the resolution was to NOT do the worker write — only the handler does it. The plan's verification clause was the load-bearing instruction.
