# Phase 82.2: Stage 0 telemetry coverage fix — Research

**Researched:** 2026-05-12
**Domain:** Inngest event-driven telemetry hygiene + one-shot backfill + daily probe
**Confidence:** HIGH (all critical claims verified against codebase + RFC docs)

## Summary

Phase 82.2 is a **telemetry-only** fix to close the Stage 0 `pipeline_events` coverage gap (26–45% on debtor mailboxes, 91% on sales). All eight implementation decisions (D-01..D-08) are locked in CONTEXT.md; this research fills in the codebase-specific details the planner needs to write executable tasks.

Three concrete findings flip the original hypothesis:

1. **The hypothesis in CONTEXT.md is mostly wrong about *where* the gap lives.** The worker (`stage-0-safety-worker.ts`) already emits `pipeline_events` on both the UPDATE-existing and INSERT-fresh paths — they share a single `step.run("persist-verdict", ...)` that emits unconditionally. The two early-return branches (L86–105 `safety_overridden` and L167–182 budget breach) are the ones that skip the emit. The "main path" L255 emit fires on **both** safe and `injection_suspected`. So the 26% debtor coverage is **not** caused by the UPDATE vs INSERT bifurcation — it's caused by something else upstream that prevents `stage-0/email.received` from being fired at all (or that prevents the worker from completing). Planner needs to budget a small diagnostic task to determine root cause before refactoring.
2. **The `suspected` branch *does* emit today** — line 255 fires on every `llmResult.verdict` including `injection_suspected` (and the test file at L249–254 asserts this). The CONTEXT.md claim that "suspected branch has zero rows in 7 days — either it never fires or it doesn't emit" is resolvable by code-read: it currently does emit; therefore zero rows means zero injection attempts in 7d (which is plausible for this mailbox mix). The synthetic-injection test (D-06) is still needed as regression coverage.
3. **The `automation_runs.completed_at` column is the canonical backfill timestamp source** — `persist-verdict` writes it on both UPDATE and INSERT paths (L227, L246).

**Primary recommendation:** Plan tasks in this order:
1. Diagnostic SQL task (find the 219 Stage-1-no-Stage-0 rows; trace which path they took)
2. Single-emit refactor + suspected-test (D-01, D-02, D-06) in parallel
3. Backfill + probe (D-03, D-04, D-05, D-07, D-08) in parallel after refactor
4. Manual one-shot backfill trigger as final task

## Worker emit-branch analysis

### What's there now (verified from file read)

`web/lib/inngest/functions/stage-0-safety-worker.ts` has exactly **three** `inngest.send` / `emitPipelineEvent` call sites and **two** early returns:

| Branch | Lines | Inngest emit | pipeline_events emit | Returns |
|--------|-------|--------------|----------------------|---------|
| **`safety_overridden=true` short-circuit** | L86–105 | `classifier/screen.requested` (L87 `emit-classifier-override`) | **NONE** | `{ skipped: "safety_overridden" }` |
| **Budget breach** | L167–182 | `pipeline/budget_breached` (L169 `emit-budget-breach`) | **NONE** | `{ halted: true }` |
| **Main path (safe OR injection_suspected)** | L184–298 | `classifier/screen.requested` on safe only (L279 inside conditional); never on injection | `emitPipelineEvent` at L255 inside `step.run("persist-verdict")` — fires on **both** verdicts | normal return |

**Critical correction to CONTEXT.md hypothesis:** The main-path `emitPipelineEvent` is INSIDE the same `step.run("persist-verdict")` that does the UPDATE-existing OR INSERT-fresh branch (L217–275). Both branches fall through to the emit at L255 — there is no UPDATE-vs-INSERT gap. [VERIFIED via direct file read of L217–275]

**Where the gap really is:**

| Branch | Misses pipeline_events? | Today's volume signal |
|--------|------------------------|----------------------|
| `safety_overridden` (L86–105) | YES — no emit at all | Rare; only operator-driven re-emit (Plan 05) sets this |
| Budget breach (L167–182) | YES — no emit at all | `pipeline/budget_breached` consumers exist; volume unknown |
| Main path safe | NO — emits | ~292 rows / 7d (159 sales + 133 debtor) [CITED: CONTEXT.md evidence table] |
| Main path injection_suspected | NO — emits (test asserts this at L249–254 of test file) | 0 / 7d [CITED] |
| **Throw before persist-verdict** (e.g. parse error rethrown at L154) | YES — exception bubbles, no emit | Unknown — diagnostic task needed |
| **`stage-0/email.received` never fired** (ingest-side gate) | YES — worker never runs | Plausible primary cause — see below |

**Suspected root cause of the 26% debtor gap (research-grade hypothesis):**

Reading `debtor-email/ingest/route.ts`:
- Line 374–376: `isLlmBound = r.category === "unknown" && settings.triage_shadow_mode && settings.entity`. Only `unknown`-category emails go through Stage 0.
- Lines 282–344: BEFORE the `isLlmBound` gate, the route emits Stage 1 `pipeline_events` (L347) and creates an automation_runs row.
- **Whitelist-match emails (auto_label_enabled=true) take the AUTO-ACTION branch at L468+** and **never fire `stage-0/email.received`**. They get Stage 1 emit but no Stage 0 emit.
- **Predicted-category non-unknown emails (not in whitelist) take the L401–423 INSERT branch** and **never fire `stage-0/email.received`** either. Stage 1 emit happens, Stage 0 does not.

In other words: **the debtor-email ingest route only calls `fireStage0Event()` for emails where the regex classifier returned `category='unknown'`**. All categorised (auto-label or bulk-review-pending) emails skip Stage 0 entirely. Sales-email ingest fires Stage 0 unconditionally (L233–247), which is why sales is at 91%.

This is the real shape of the gap. The "single-emit refactor" in D-01 fixes the worker's three-emit-site problem (which is real but small — `safety_overridden` and budget-breach branches), but the **dominant volume gap is upstream in the ingest route**, not in the worker.

**This needs surfacing to the planner — the D-01 refactor alone will NOT raise debtor coverage to 99%.** The planner needs to either:
- (a) extend D-01 scope to also fire `stage-0/email.received` from the auto-action / predicted-category branches in `debtor-email/ingest/route.ts`, OR
- (b) accept that for non-`unknown` debtor emails, the "backfill from automation_runs" path (D-04) is the steady-state mechanism (NOT just historical) — i.e. the daily probe writes synthetic Stage 0 rows for any `unknown_legacy` cases. This contradicts the spirit of D-04 ("safety logic ran"), so (a) is more honest.

The planner must decide. The locked CONTEXT.md doesn't anticipate this — re-discussion may be warranted before execution.

### Single-emit refactor (concrete sketch for D-01, D-02)

Pseudocode for the refactored worker — preserves all three event sub-types via a discriminated `verdictSource` variable:

```typescript
type VerdictSource =
  | { kind: "operator-override"; }
  | { kind: "budget-breach"; budget: BudgetState; reason: string }
  | { kind: "main-path"; verdict: "safe" | "injection_suspected"; ... };

export const stage0SafetyWorker = inngest.createFunction(
  { id: "stage-0/safety-worker", retries: 0 },
  { event: "stage-0/email.received" },
  async ({ event, step }) => {
    const data = ...; // unchanged
    let verdictSource: VerdictSource;
    let downstreamEmit: { name: string; data: any } | null = null;

    if (safety_overridden) {
      verdictSource = { kind: "operator-override" };
      downstreamEmit = {
        name: "classifier/screen.requested",
        data: { ...passthrough, safety_overridden: true },
      };
    } else {
      // strip → regex → llm-verdict → budget unchanged
      const stripResult = await step.run("strip-quoted-history", ...);
      const regexResult = await step.run("regex-screen", ...);
      const llmResult   = await step.run("llm-verdict", ...);
      const budgetVerdict = await step.run("check-budget", ...);

      if (budgetVerdict.breached) {
        verdictSource = { kind: "budget-breach", budget: budgetState, reason: budgetVerdict.reason };
        downstreamEmit = {
          name: "pipeline/budget_breached",
          data: { automation_run_id, email_id, budget: budgetState, reason: ... },
        };
      } else {
        verdictSource = { kind: "main-path", verdict: llmResult.verdict, ... };
        if (llmResult.verdict === "safe") {
          downstreamEmit = {
            name: "classifier/screen.requested",
            data: { ...original-body passthrough... },
          };
        }
        // injection_suspected → no downstream emit, persist topic='safety_review'
      }
    }

    // SINGLE step.run that:
    //  1. UPDATEs or INSERTs automation_runs (existing logic, gated on verdictSource.kind)
    //  2. ALWAYS emits pipeline_events with decision_details.emit_source = verdictSource.kind
    //  3. Optionally sends the downstreamEmit if non-null
    await step.run("persist-and-emit", async () => {
      // automation_runs write — same as today for main-path; for budget-breach
      // write status='failed'+error_message; for operator-override, write
      // status='completed' with topic=null and result.verdict='safe' (or
      // skip automation_runs write entirely and document why).

      await emitPipelineEvent(admin, {
        swarm_type,
        stage: 0,
        email_id,
        decision: decisionFor(verdictSource),  // 'safe' | 'injection_suspected' | 'over_budget' | 'safe' (override)
        confidence: null,
        decision_details: {
          emit_source: verdictSource.kind,  // <-- D-02 distinguisher
          // ... existing fields per branch
        },
        cost_cents: ..., automation_run_id, triggered_by: "pipeline",
      });

      if (downstreamEmit) {
        await inngest.send(downstreamEmit);
      }
    });

    await emitAutomationRunStale(admin, staleChannel);
    return verdictSource;  // or shape that satisfies existing test assertions
  },
);
```

**D-02 preservation:** the three sub-types remain distinguishable via `pipeline_events.decision_details.emit_source ∈ {operator-override, budget-breach, main-path}`. Downstream consumers (Bulk Review, V9 trace) already read `decision_details` as a free-form jsonb — adding one field is non-breaking.

**Early-return paths that block fall-through (must be eliminated):**

- L181 `return { halted: true }` → eliminate, fall through to single emit
- L154 `throw err` (non-timeout LLM error) → KEEP as throw; intentional retry-disabled exception per existing comment. Document that exceptions still won't emit pipeline_events — they're observable via Inngest's failure surface instead.

## Backfill query + source mapping

### Schema facts (verified from `20260506a_pipeline_events.sql`)

- **No UNIQUE constraint** on `(email_id, stage, swarm_type)` — only PK is `id uuid DEFAULT gen_random_uuid()`. **Backfill cannot rely on UPSERT to dedupe** — must explicitly check existence before INSERT to be idempotent.
- Indexes: `pipeline_events_email_id_idx`, `pipeline_events_swarm_stage_created_idx`. Both are non-unique. Both make the "find gaps" query cheap.
- `email_id` is nullable (per L29 `email_id uuid NULL`) — so a "stage=0 row exists" check should be `WHERE email_id = $1 AND swarm_type = $2 AND stage = 0`.

**Claude's Discretion item resolved:** there is NO unique constraint to UPSERT against. Two viable paths:
1. **Application-level idempotency** — `SELECT 1 FROM pipeline_events WHERE email_id=$1 AND swarm_type=$2 AND stage=0 LIMIT 1`, skip if exists. Race-safe enough for a one-shot backfill (no concurrent writers expected during the backfill window).
2. **Add a partial UNIQUE index** in a new migration: `CREATE UNIQUE INDEX pipeline_events_one_per_stage_email ON pipeline_events (email_id, swarm_type, stage) WHERE email_id IS NOT NULL` — gives DB-level guarantee for both the live worker and backfill going forward. **Recommended** — closes the duplicate-row class of bug permanently. Doesn't break existing rows because `email_id IS NULL` rows are excluded.

### Gap-finding SELECT

```sql
-- Backfill candidates: emails that have a Stage-1 pipeline_events row in the
-- last 30 days but no Stage-0 row for the same (email_id, swarm_type).
WITH stage1_in_window AS (
  SELECT DISTINCT email_id, swarm_type
  FROM pipeline_events
  WHERE stage = 1
    AND email_id IS NOT NULL
    AND created_at >= now() - interval '30 days'
),
stage0_present AS (
  SELECT email_id, swarm_type
  FROM pipeline_events
  WHERE stage = 0
    AND email_id IS NOT NULL
    AND created_at >= now() - interval '31 days'  -- 1d buffer for stage0-before-stage1 races
)
SELECT s1.email_id, s1.swarm_type
FROM stage1_in_window s1
LEFT JOIN stage0_present s0 USING (email_id, swarm_type)
WHERE s0.email_id IS NULL;
```

Expected row count: ≥219 (CONTEXT.md evidence for 7d; 30d should be roughly 3–4×). Manageable in a single Inngest function with batching — see "Inngest function shapes" below.

### Per-row reconstruction (D-04 source mapping)

For each `(email_id, swarm_type)` from the gap query:

```sql
-- Try to find the original automation_runs row that ran Stage 0 for this email.
SELECT id, completed_at, result, status
FROM automation_runs
WHERE result->>'message_id' IS NOT NULL  -- ingest-side rows carry this
  AND swarm_type = $2
  AND (
    result->>'stage' IN ('stage_0_safety', 'stage_0_safety_pending')
    OR result->>'verdict' IS NOT NULL  -- post-Phase-64 INSERT-fresh rows
  )
  -- Join via email_pipeline.emails.source_id = automation_runs.result->>'message_id'
ORDER BY completed_at DESC NULLS LAST
LIMIT 1;
```

The join is awkward because `automation_runs` doesn't carry `email_id` directly — it has `result.message_id` (Outlook id) which matches `email_pipeline.emails.source_id`. Two-step lookup per row:

```sql
-- Step A: resolve email_id → Outlook message_id
SELECT source_id FROM email_pipeline.emails WHERE id = $1;

-- Step B: find automation_runs row by message_id + swarm_type
SELECT result, completed_at FROM automation_runs
WHERE result->>'message_id' = $source_id
  AND swarm_type = $swarm_type
  AND result->>'stage' IN ('stage_0_safety', 'stage_0_safety_pending')
ORDER BY completed_at DESC LIMIT 1;
```

Or — far simpler — fold both lookups into the main backfill SELECT:

```sql
SELECT
  s1.email_id,
  s1.swarm_type,
  ar.completed_at,
  ar.result
FROM stage1_in_window s1
LEFT JOIN stage0_present s0 USING (email_id, swarm_type)
LEFT JOIN email_pipeline.emails e ON e.id = s1.email_id
LEFT JOIN LATERAL (
  SELECT result, completed_at
  FROM automation_runs ar2
  WHERE ar2.result->>'message_id' = e.source_id
    AND ar2.swarm_type = s1.swarm_type
    AND ar2.result->>'stage' IN ('stage_0_safety', 'stage_0_safety_pending')
  ORDER BY ar2.completed_at DESC NULLS LAST
  LIMIT 1
) ar ON true
WHERE s0.email_id IS NULL;
```

### Row construction logic per source

For each gap row:

| `automation_runs` found? | `result.verdict` value | Write Stage 0 row with |
|--------------------------|------------------------|------------------------|
| Yes, `stage='stage_0_safety'` | `'safe'` | `decision='safe'`, `decision_details={ source: 'backfill', original_automation_run_id, regex_matched, llm_reason, ... }`, `cost_cents=result.cost_cents`, `triggered_by='backfill'`, `created_at=ar.completed_at` |
| Yes, `stage='stage_0_safety'` | `'injection_suspected'` | `decision='injection_suspected'`, same shape as above |
| Yes, `stage='stage_0_safety_pending'` (worker never completed) | n/a | `decision='unknown_legacy'`, `decision_details={ source: 'backfill', reason: 'pending_orphan', original_automation_run_id }`, `triggered_by='backfill'`, `created_at=ar.created_at` |
| No `automation_runs` row at all | n/a | `decision='unknown_legacy'`, `decision_details={ source: 'backfill', reason: 'no_source_record' }`, `triggered_by='backfill'`, `created_at=` Stage-1 row's `created_at` minus 1 second (proxy) |

**Critical replay-safety:** the `created_at` timestamp must be set explicitly (not let it default to `now()`) so the row reflects the original safety verdict moment. The DB DEFAULT is `now()` so the INSERT must specify the column. **Phase 65 replay-id rule applies:** generate the explicit timestamp inside `step.run` (whether you use `ar.completed_at` from DB or `new Date(...)` for the synthetic). DB-derived timestamps are deterministic per replay; synthetic `Date.now()` is NOT and must be `step.run`-wrapped.

**emitPipelineEvent reuse:** the helper at `web/lib/pipeline-events/emit.ts` is a thin INSERT wrapper. It can be reused IF the `PipelineEventInput` type supports overriding `created_at`. Looking at types.ts: it does NOT — there's no `created_at` field on `PipelineEventInput`. **The backfill needs to either (a) extend the helper to accept an optional `created_at`, or (b) do the INSERT directly via `admin.from("pipeline_events").insert(...)` bypassing the helper.** Recommend (b) — keeps the helper's "one row per stage decision" semantics clean and signals via code that this is a non-live-path write. Triggered-by-`'backfill'` is also a strong signal.

## Inngest function shapes

### Backfill function (D-03, D-04, D-05)

```typescript
// web/lib/inngest/functions/stage-0-backfill.ts
//
// Phase 82.2 D-03/D-04/D-05 — one-shot Inngest backfill that writes a
// pipeline_events.stage=0 row for every email in the last 30 days that
// has a Stage-1 row but no Stage-0 row. Triggered manually via
//   inngest.send({ name: 'pipeline.stage0.backfill', data: { window_days?: 30 } })
//
// Source priority (D-04): automation_runs (with result.verdict reconstructed)
// → 'unknown_legacy' marker. Does NOT re-run safety logic.

import { inngest } from "@/lib/inngest/client";
import { createAdminClient } from "@/lib/supabase/admin";

const BATCH_SIZE = 500;  // see batching rationale below

type SendFn = (p: { name: string; data: Record<string, unknown> }) => Promise<unknown>;

export const stage0Backfill = inngest.createFunction(
  { id: "stage-0/backfill", retries: 1 },
  { event: "pipeline.stage0.backfill" },
  async ({ event, step }) => {
    const windowDays = (event.data as { window_days?: number })?.window_days ?? 30;

    // Phase 65 — uuid/timestamp generation inside step.run
    const runId = await step.run("resolve-run-id", () => crypto.randomUUID());

    // Stage 1: discover gaps
    const candidates = await step.run("find-gaps", async () => {
      const admin = createAdminClient();
      const { data, error } = await admin.rpc("stage0_backfill_candidates", {
        window_days: windowDays,
      });
      if (error) throw new Error(`gap query failed: ${error.message}`);
      return data as Array<{
        email_id: string;
        swarm_type: string;
        completed_at: string | null;
        result: any | null;
      }>;
    });

    // Stage 2: batch-insert backfill rows
    let written = 0;
    let skipped = 0;
    for (let i = 0; i < candidates.length; i += BATCH_SIZE) {
      const batch = candidates.slice(i, i + BATCH_SIZE);
      const batchIdx = Math.floor(i / BATCH_SIZE);
      const result = await step.run(`backfill-batch-${batchIdx}`, async () => {
        const admin = createAdminClient();
        const rows = batch.map((c) => buildBackfillRow(c, runId));
        // Defense: re-check no Stage-0 row appeared since gap query
        // (e.g. worker ran between gap detection and insert)
        const emailIds = batch.map((b) => b.email_id);
        const { data: existing } = await admin
          .from("pipeline_events")
          .select("email_id, swarm_type")
          .in("email_id", emailIds)
          .eq("stage", 0);
        const existingKeys = new Set(
          (existing ?? []).map((r) => `${r.email_id}|${r.swarm_type}`),
        );
        const toInsert = rows.filter(
          (r) => !existingKeys.has(`${r.email_id}|${r.swarm_type}`),
        );
        if (toInsert.length === 0) return { inserted: 0, skipped: rows.length };
        const { error } = await admin.from("pipeline_events").insert(toInsert);
        if (error) throw new Error(`backfill insert failed: ${error.message}`);
        return { inserted: toInsert.length, skipped: rows.length - toInsert.length };
      });
      written += result.inserted;
      skipped += result.skipped;
    }

    return { run_id: runId, candidates: candidates.length, written, skipped };
  },
);

function buildBackfillRow(c: { email_id: string; swarm_type: string; completed_at: string | null; result: any | null }, runId: string) {
  if (c.result?.verdict === "safe" || c.result?.verdict === "injection_suspected") {
    return {
      swarm_type: c.swarm_type,
      stage: 0,
      email_id: c.email_id,
      decision: c.result.verdict,
      confidence: null,
      decision_details: {
        source: "backfill",
        backfill_run_id: runId,
        original_automation_run_id: c.result.id ?? null,
        regex_matched: c.result.regex_matched ?? null,
        llm_reason: c.result.llm_reason ?? null,
        matched_span: c.result.matched_span ?? null,
      },
      cost_cents: c.result.cost_cents ?? null,
      created_at: c.completed_at ?? new Date().toISOString(),
      triggered_by: "backfill",
    };
  }
  // Fallback — no source record, mark unknown_legacy
  return {
    swarm_type: c.swarm_type,
    stage: 0,
    email_id: c.email_id,
    decision: "unknown_legacy",
    confidence: null,
    decision_details: {
      source: "backfill",
      backfill_run_id: runId,
      reason: c.result ? "stage_0_pending_orphan" : "no_source_record",
    },
    cost_cents: null,
    created_at: c.completed_at ?? new Date().toISOString(),
    triggered_by: "backfill",
  };
}
```

**Batching reasoning:** With ~219 rows/7d × ~4.3 weeks = ~940 candidate rows expected. Worst case (if upstream gap is larger than measured): ~12k as the original brief estimated. At 500/batch:
- 940 rows → 2 batches → comfortably under Vercel Pro 60s per step.run (each batch is a single bulk INSERT, ~1–2s).
- 12k rows → 24 batches → still safe, total run ~30–60s.

Vercel Pro maxDuration is set to 300s in the inngest route (route.ts L8). Inngest's `step.run` boundary is the right unit of replay safety — each batch is one step. If a batch fails mid-flight, Inngest replays it from the start with the same candidate slice. The pre-insert dedupe SELECT inside the batch handles partial-success replays.

**Idempotency:** Re-running the function on a window that already has Stage-0 rows is a no-op (every row hits the `existingKeys` skip path).

  async ({ step }) => {
    const stamp = await step.run("resolve-stamp", () => new Date().toISOString());

    const coverage = await step.run("measure-coverage", async () => {
      const admin = createAdminClient();
      // Per-mailbox: count emails with Stage-1 row in last 24h, count those
      // with a matching Stage-0 row, compute coverage.
      const results = [];
      for (const m of ACTIVE_MAILBOXES) {

          stage0_count: data?.stage0_count ?? 0,
          coverage: data?.stage1_count
            ? data.stage0_count / data.stage1_count
            : 1,
        });
      }
      return results;
    });

    const breaches = coverage.filter((c) => c.coverage < COVERAGE_FLOOR && c.stage1_count > 0);

    if (breaches.length > 0) {
      await step.run("append-state-md", async () => {
        const statePath = path.resolve(process.cwd(), "../.planning/STATE.md");
        const lines = [
          ``,
          `### Stage 0 coverage breach — ${stamp}`,
          ``,
          `| Mailbox | Stage 1 | Stage 0 | Coverage |`,
          `|---|---|---|---|`,

            (b) => `| ${b.mailbox} | ${b.stage1_count} | ${b.stage0_count} | ${(b.coverage * 100).toFixed(1)}% |`,
          ),
          ``,
        ].join("\n");
        // Append-only — never overwrite STATE.md

      });

    return { stamp, breaches: breaches.length, mailboxes: coverage.length, details: coverage };
  },

**STATE.md path resolution caveat:** Inngest functions run on Vercel serverless. `process.cwd()` is the function's runtime root. The `.planning/` directory is in the repo root but **likely NOT shipped to Vercel's serverless bundle** by default (Next.js doesn't bundle files outside the `app/` / public routes unless explicitly included). **This is a real blocker for the literal-STATE.md-write interpretation of D-08.** Three options for the planner:

1. **Vercel bundle inclusion** via `next.config.js` `outputFileTracingIncludes` — ship `.planning/STATE.md` into the function bundle. Allows direct file append. Caveat: the function's writes don't auto-commit to git; the change exists only on the Vercel filesystem until manually pulled.
2. **Write breaches to a `pipeline_health` Supabase table** (allowed under Claude's Discretion in D-08) and have a separate operator process (cron-on-laptop or human-run script) mirror them into STATE.md on commit. Cleanest separation of concerns.
3. **Write breaches into `automation_runs`** with `automation='stage-0-coverage-probe'`, `status='predicted'`, `result={breaches, ...}`. `/gsd-progress` could be taught to surface these. No new table, no file-write fiction.

**Recommendation:** Option 2. Create a `pipeline_health` table (a single migration) — D-08 explicitly permits this. The "STATE.md surface" requirement is satisfied operationally by a documented `/gsd-progress` task that reads the table. This is more honest about the "Inngest can't write to your git repo" reality.

**If the planner sticks with the literal-STATE.md interpretation,** they MUST validate Option 1 by running the deployed function once and confirming the file is writable.

## Coverage query

The `stage0_coverage_24h` Postgres function (referenced by the probe above):

```sql
CREATE OR REPLACE FUNCTION public.stage0_coverage_24h(mailbox_arg text, swarm_arg text)
RETURNS TABLE(stage1_count bigint, stage0_count bigint)
LANGUAGE sql STABLE AS $$
  WITH stage1 AS (
    SELECT DISTINCT pe.email_id
    FROM public.pipeline_events pe
    JOIN email_pipeline.emails e ON e.id = pe.email_id
    WHERE pe.stage = 1
      AND pe.swarm_type = swarm_arg
      AND e.mailbox = mailbox_arg
      AND pe.created_at >= now() - interval '24 hours'
      AND pe.email_id IS NOT NULL
  ),
  stage0 AS (
    SELECT DISTINCT email_id
    FROM public.pipeline_events
    WHERE stage = 0
      AND swarm_type = swarm_arg
      AND created_at >= now() - interval '25 hours'  -- 1h buffer
      AND email_id IS NOT NULL
  )
  SELECT
    (SELECT count(*) FROM stage1)::bigint AS stage1_count,
    (SELECT count(*) FROM stage1 JOIN stage0 USING (email_id))::bigint AS stage0_count;
$$;
```
