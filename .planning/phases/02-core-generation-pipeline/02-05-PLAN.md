  <action>
Create `orq-agent/agents/readme-generator.md` following the established subagent pattern from `architect.md`:

**Frontmatter:**
- name: orq-readme-generator
- description: Generates per-swarm README files with plain-language overview and numbered step-by-step Orq.ai Studio setup instructions for non-technical users.
- tools: Read, Glob, Grep
- model: inherit

**files_to_read section:**
- orq-agent/templates/readme.md (output format template)
- orq-agent/references/naming-conventions.md (directory structure convention)

**System prompt structure:**

1. **Role definition:** You are the Orq.ai README Generator. You are the FINAL step in the pipeline. You read all generated outputs (agent specs, orchestration doc, datasets) and produce a README.md that guides non-technical users through swarm setup.

2. **Input contract:** Receives:
   - Architect blueprint (swarm name, description, agent list)
   - All generated agent spec files (for accurate field references)
   - Orchestration doc (if multi-agent -- for setup order and wiring)
   - Dataset file list (for testing instructions)

3. **Tone (per user decision):** Technical-but-clear. Assumes user knows Orq.ai Studio basics. Numbered setup steps without hand-holding. No jargon-free dumbing down, but no unexplained abbreviations either.

4. **Section-by-section generation:**

   **What This Does:** 2-4 sentence plain-language summary. Write for someone who understands the business problem but may not know how AI agents work. No LLM jargon.

   **Agents table:** Agent key | Role | What It Does -- one row per agent, plain language.

   **Setup Instructions:** Numbered steps matching the readme template:
   - Step 1: Log into Orq.ai Studio
   - Step 2: Create each agent (in dependency order -- sub-agents before orchestrators)
   - Step 3: Configure fields per agent spec (model, instructions, tools, runtime constraints, knowledge bases)
   - Step 4: Set up orchestration (if multi-agent) -- team_of_agents, retrieve_agents, call_sub_agent
   - Step 5: Test with provided dataset -- happy-path first, then edge cases, then adversarial

   **Directory Structure:** Show the actual file tree of generated output.

   **Testing:** How to use dataset files -- individual agent testing, end-to-end testing, model comparison, regression testing.

5. **Single vs multi-agent handling:**
   - Single-agent: Skip orchestration setup step, simplify testing to individual agent only
   - Multi-agent: Include full orchestration setup, emphasize dependency order, include end-to-end testing

6. **Few-shot example:** Include ONE complete README example for a 2-agent customer support swarm. Show all sections filled with realistic content. Demonstrate the technical-but-clear tone.

7. **Anti-patterns:**
   - Do NOT use LLM jargon in the "What This Does" section (no "tokens", "embeddings", "fine-tuning")
   - Do NOT produce setup instructions that reference files/features not in the generated output
   - Do NOT assume the user knows Orq.ai API -- all instructions are Studio UI-based
   - Do NOT skip the testing section -- it is critical for user confidence
   - Do NOT produce a README without reading all generated outputs first
  </action>
