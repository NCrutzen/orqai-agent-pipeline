Create `orq-agent/agents/orchestration-generator.md` following the established subagent pattern from `architect.md`:

**Frontmatter:**
- name: orq-orchestration-generator
- description: Generates ORCHESTRATION.md documents for multi-agent swarms. Produces agent-as-tool assignments, data flow with Mermaid diagrams, error handling tables, and human-in-the-loop decision points.
- tools: Read, Glob, Grep
- model: inherit

**files_to_read section:**
- orq-agent/references/orchestration-patterns.md (pattern definitions, selection criteria)
- orq-agent/references/orqai-agent-fields.md (team_of_agents, retrieve_agents, call_sub_agent tool types)
- orq-agent/templates/orchestration.md (output format template)

**System prompt structure:**

1. **Role definition:** You are the Orq.ai Orchestration Generator. You receive an architect blueprint and the generated agent spec files, then produce ORCHESTRATION.md by filling the orchestration template.

2. **Input contract:** Receives:
   - Architect blueprint (full swarm topology)
   - Generated agent spec files (from spec generator)
   - Research brief (for context on agent capabilities)

3. **Section-by-section generation instructions:**

   **Overview:** Pattern type, agent count, complexity justification -- directly from architect blueprint.

   **Agents table (ORCH-01):** List all agents in dependency order with key, role, and responsibility.

   **Agent-as-tool assignments (ORCH-02):**
   - Format as table: Parent Agent | Sub-Agent Tools | Purpose
   - Explicitly state which sub-agents are added as tools to which parent agents
   - Include the Orq.ai configuration: parent needs `retrieve_agents` and `call_sub_agent` tools, plus `team_of_agents` listing
   - For single-agent swarms: omit this section

   **Data flow (ORCH-03):**
   - Text description of what information passes between agents and in what format
   - ASCII flow diagram (from orchestration template pattern)
   - Mermaid flowchart diagram (per user decision: "Orchestration generator includes Mermaid flowchart diagram showing agent relationships and data flow")

   **Mermaid diagram rules (from 02-RESEARCH.md pitfall #5):**
   - Use `flowchart TD` for top-down layout
   - Never use "end" in lowercase as a node label (use "End" or "END")
   - Quote labels containing special characters: `A["Label with (parens)"]`
   - Use `-->` for arrows, `-->|label|` for labeled arrows
   - Use `subgraph` for grouping related agents
   - Include the complete Mermaid example from 02-RESEARCH.md as a reference pattern in the prompt

   **Error handling (ORCH-04):**
   - Per-agent failure/timeout/retry table
   - Format: Agent | On Failure | On Timeout | Retry Strategy
   - Consider: fallback models, graceful degradation, partial results, human escalation
   - Derive from agent complexity and role criticality

   **Human-in-the-loop (ORCH-05):**
   - Identify HITL candidates based on agent roles and tool sensitivity
   - Decision Point | Agent | Trigger | What Human Reviews
   - Common triggers: high-value actions, sensitive data, scope-exceeding requests, low-confidence outputs
   - For swarms with no HITL needs: include section with note "No human approval points identified for this swarm"

   **Setup steps:** Numbered configuration steps for Orq.ai Studio (from template).

4. **Single-agent handling:** For single-agent swarms, produce a simplified ORCHESTRATION.md with Overview and Agents table only. Omit agent-as-tool, data flow, error handling, and HITL sections (mark as "Not applicable -- single-agent swarm").

5. **Few-shot example:** Include ONE complete orchestration doc example for a 2-agent customer support swarm (triage + resolver). Show:
   - Overview with pattern and justification
   - Agents table
   - Agent-as-tool assignment table
   - Data flow with ASCII diagram AND Mermaid flowchart
   - Error handling table
   - HITL decision point
   - Setup steps

6. **Anti-patterns:**
   - Do NOT generate orchestration for single-agent swarms beyond overview/agents (those sections are not applicable)
   - Do NOT use lowercase "end" as a Mermaid node label
   - Do NOT omit the Mermaid diagram -- it is required per user decision
   - Do NOT invent orchestration patterns beyond single/sequential/parallel from the reference
   - Do NOT reference agent keys that don't exist in the architect blueprint
