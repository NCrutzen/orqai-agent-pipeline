<objective>
Implement CONTEXT.md decision #3: add a per-`(channel, event-key)` 500ms debounce inside `web/lib/supabase/broadcast.ts`. Internal-only change — public API unchanged so all 22 callers in `pipeline.ts` and 4 in `pipeline/conversation-action.ts` / `pipeline/discussion-action.ts` keep working unmodified.

Purpose: pipeline runs cause rapid status flips (`waiting → running → complete` within ~100ms). Each emits a broadcast. With pipeline.ts having 22 call sites and a typical run firing 8-12 of them in rapid succession, debouncing collapses bursts into a single emit-the-latest message. Visually identical to the user (≤500ms staleness on a dashboard is imperceptible).

**Backward-compat is the load-bearing requirement.** The reason we debounce inside broadcast.ts (not at call sites) is to ship without touching any pipeline.ts line.

**Critical exclusion:** chat messages MUST NOT be debounced. Each chat token / chat message is a distinct semantic event (user/assistant turn or streaming token); collapsing would lose data. The chat code paths bypass the debounce entirely.

Output: 1 file modified + 1 new unit test. 1 atomic commit.
</objective>

<debounce_design>
**Key derivation:**
- `broadcastStepUpdate(runId, payload)` → key = `run:${runId}:step-update:${payload.stepName}` (per CONTEXT.md: runId+stepName)
- `broadcastRunUpdate(runId, payload)` → key = `runs:live:run-update:${runId}` (per CONTEXT.md: runId)
- `broadcastChatMessage` → BYPASS (direct emit)
- `createChatBroadcaster.send` → BYPASS (direct emit on the persistent channel)
- `broadcastHealthUpdate` → BYPASS (low frequency, distinct events)

**Coalescing semantics:** "emit-the-latest". When a call arrives within the 500ms window, replace `lastPayload` and reset the timer. When the timer fires, emit `lastPayload`. The intermediate states are dropped (acceptable — dashboard only needs to see the latest status).

**Storage:** module-local `Map<string, { timer: NodeJS.Timeout; lastPayload: { channel: string; event: string; payload: unknown } }>`. Single map for all debounced events.

**Per-instance state caveat:** module state is per-Vercel-function-instance. If two pipeline runs straddle two instances both emitting for the same key (rare; each run is single-threaded), the messages duplicate rather than coalesce. CONTEXT.md accepts this.
</debounce_design>

1. Add a private debounce machinery at the top of the module (after imports, before exports):
   ```typescript
   // ---------------------------------------------------------------------------
   // Internal debounce (Phase 59 D-03)
   // 500ms emit-the-latest coalescing for high-frequency status broadcasts.
   // Chat events bypass — each token / message is a distinct semantic event.
   // ---------------------------------------------------------------------------
   const DEBOUNCE_MS = 500;

   interface DebounceEntry {
     timer: ReturnType<typeof setTimeout>;
     channelName: string;
     event: string;
     payload: unknown;
   }
   const debounceMap = new Map<string, DebounceEntry>();

   function debouncedSend(
     key: string,
     channelName: string,
     event: string,
     payload: unknown,
   ): void {
     const existing = debounceMap.get(key);
     if (existing) {
       clearTimeout(existing.timer);
       existing.payload = payload;
       existing.timer = setTimeout(() => flushDebounced(key), DEBOUNCE_MS);
       return;
     }
     const timer = setTimeout(() => flushDebounced(key), DEBOUNCE_MS);
     debounceMap.set(key, { timer, channelName, event, payload });
   }

   function flushDebounced(key: string): void {
     const entry = debounceMap.get(key);
     if (!entry) return;
     debounceMap.delete(key);
     const admin = createAdminClient();
     const channel = admin.channel(entry.channelName);
     channel
       .send({ type: "broadcast", event: entry.event, payload: entry.payload })
       .catch((err) => console.warn("[broadcast.debounce] send failed", { key, err }))
       .finally(() => admin.removeChannel(channel));
   }
   ```
