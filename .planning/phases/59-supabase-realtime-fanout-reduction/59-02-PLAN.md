Prefix derivation rule (CONTEXT.md): `prefix = first segment of automation name before '-'`. Example: `debtor-email-classify` → prefix `debtor-email` (multi-segment); `uren-controle-process` → prefix `uren-controle`. Use the SAME rule callers already use when constructing the AutomationRealtimeProvider — the prefix is the string the consuming page passes in. For the EMIT side, we cannot guess: pages pass arbitrary prefixes. Solution: emit on BOTH the full-segment prefix AND broadcast wildcard-friendly key. SIMPLEST CORRECT APPROACH: emit on `automations:${automation}:stale` per row, AND have the provider listen on the prefix it owns by subscribing to `automations:${prefix}:stale` AS A WILDCARD via a different channel naming. **Clarification:** Supabase broadcast does not support wildcard channels. So we standardize the prefix derivation: the helper extracts everything BEFORE the last `-{verb}` segment is impossible without registry. Use the existing convention used by callers: the consuming pages pass `prefix="debtor-email"`, `prefix="uren-controle"`, etc. The emit helper accepts the full `automation` string and broadcasts on EVERY ancestor prefix — for `debtor-email-classify` it emits on `automations:debtor-email-classify:stale`, `automations:debtor-email:stale`, and `automations:debtor:stale`. Subscribers pick the level they need. This costs 2-3 broadcasts per write (still <<< full-table fan-out to all tabs) and keeps the provider's prefix contract honest without a registry.

1. **Create `web/lib/automations/runs/emit.ts`** (new file):
   ```typescript
   // Investigation 2026-04-26: all automation_runs writers are in-app (Vercel + Inngest).
   // External writers would require a Postgres NOTIFY trigger; not needed today.
   import type { SupabaseClient } from "@supabase/supabase-js";

   /**
    * Emit a "stale" broadcast for every ancestor prefix of an automation name,
    * so AutomationRealtimeProvider subscribers (which own a specific prefix)
    * get notified to refetch.
    *
    * Example: automation="debtor-email-classify" emits on:
    *   - automations:debtor-email-classify:stale
    *   - automations:debtor-email:stale
    *   - automations:debtor:stale
    *
    * Caller passes the admin client so we don't open a fresh connection per write.
    * Failures are logged, never thrown — the write succeeded; a missed broadcast
    * is recovered by manual refresh and is not data-loss.
    */
   export async function emitAutomationRunStale(
     admin: SupabaseClient,
     automation: string,
   ): Promise<void> {
     if (!automation) return;
     const segments = automation.split("-");
     const prefixes = new Set<string>();
     for (let i = 1; i <= segments.length; i++) {
       prefixes.add(segments.slice(0, i).join("-"));
     }
     try {
       await Promise.all(
         Array.from(prefixes).map(async (p) => {
           const ch = admin.channel(`automations:${p}:stale`);
           await ch.send({
             type: "broadcast",
             event: "stale",
             payload: { automation, at: new Date().toISOString() },
           });
           admin.removeChannel(ch);
         }),
       );
     } catch (err) {
       console.warn("[emitAutomationRunStale] broadcast failed", { automation, err });
     }
   }
   ```
