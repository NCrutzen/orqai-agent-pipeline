<objective>
Implement CONTEXT.md decision #1: replace the `agent_events` postgres_changes subscription on `swarm:${swarmId}` with a server-emitted broadcast + client refetch. Cuts ~99% of message volume on the hottest realtime path (50–200 row-level msgs/tick → 1 broadcast/tick).

Purpose: agent_events is the single largest fan-out source. Each bridge tick deletes+reinserts hundreds of span rows, every one of which fans out to every connected dashboard tab via postgres_changes. The fix is to keep the snapshot SELECT (already exists in the provider's fetchSnapshot) and trigger it from a single broadcast emitted at the end of the bridge tick.

Output: 3 files modified, 1 atomic commit. Other 3 postgres_changes blocks (swarm_jobs, swarm_agents, swarm_briefings) stay unchanged — they're low-churn and not driving the budget overrun.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Inngest worker → Supabase Realtime | server-side admin client emits broadcast (no new auth surface) |
| Browser tab → Supabase Realtime | client subscribes to `swarm:${swarmId}` via anon key + RLS (existing surface) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59.01-01 | I (Information disclosure) | broadcast `events-stale` payload | mitigate | Payload contains only `{reason, at}` — no row data. Even if cross-tenant subscriber attaches to wrong channel, they receive only a refetch trigger; data still gated by RLS on the SELECT. |
| T-59.01-02 | D (Denial of Service) | bridge tick fails to emit broadcast | mitigate | try/catch around `.send()`; log only, no throw. Client falls back to 15s poll (already in code). Bridge tick never blocked by broadcast failure. |
| T-59.01-03 | T (Tampering) | malicious client sends fake events-stale | accept | Worst case: dashboard does an extra fetchSnapshot. Cost is one SELECT; no data modification. Same risk profile as today's postgres_changes (Realtime channels are not authenticated emitters). |
| T-59.01-04 | R (Repudiation) | broadcast lost in transit | accept | 15s poll guarantees eventual consistency. Documented behavior; no audit log requirement on UI refresh. |
</threat_model>
