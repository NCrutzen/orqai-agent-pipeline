# Phase 82.3: Per-stage audit surface (v8.0 stabilisation)

**Gathered:** 2026-05-12
**Status:** Stub — needs `/gsd-discuss-phase 82.3` before planning
**Type:** INSERTED stabilisation phase — debtor-person prerequisite, blocks Phase 82.4 form

<domain>
## Phase Boundary

Give the operator full insight into what the pipeline did for any given email so they can audit each stage's decision before commenting on it. Click any cell of the 5-cell `PipelineFlow` in the Bulk Review detail pane → popup renders that stage's evidence (verdict + reasoning summary + screenshots).

This is the **read surface**. Writes (feedback forms) ride on top in Phase 82.4 — the form mounts inside the same popup, so 82.3 is a hard prerequisite for 82.4.

Scope is plumbing: the data already exists in `pipeline_events.decision_details`, `agent_runs.context`, `agent_runs.tool_outputs`, and `automation_runs.result` (verified 2026-05-12). Screenshots are in Supabase Storage bucket `automation-screenshots`. Phase 82.3 surfaces them.

</domain>

<rationale>
## Why this lands in v8.0 closure, not V9.0

Originally I scoped per-stage detail rendering into V9.0/V10.0 as a "future feature." Wrong call. V9.0's whole premise is *operator gives feedback on what the pipeline did* — that requires seeing what it did. Today's Bulk Review (Phase 71/82) shows verdicts but not evidence. Without the audit surface, the operator's prose feedback in 82.4 will be "I think Stage 2 was wrong" without being able to point at *which step in the resolver chain* failed, and V9.0's clusterer drowns in low-specificity signal.

The 2026-05-11 smoke-test note explicitly asked for this:
> *"For emails that got auto-handled, user wants to be able to drill into the details of what each stage did — including screenshots taken by the browser automation. Click any cell of the 5-cell PipelineFlow → popup with that stage's full reasoning + any screenshots captured."*

</rationale>

<scope>
## In scope (80/20 cut)

Render the verdict + key evidence per stage, not exhaustive agent transcripts. Default to summary view; offer "show full reasoning" expander for the ~20% of cases where deeper context is needed.

### Per-stage payload to render

| Stage | Data sources (already exists in DB) | Renders |
|---|---|---|
| **0 Safety** | `pipeline_events.decision_details` | Regex patterns triggered, LLM injection verdict + reasoning, budget headroom remaining |
| **1 Noise** | `pipeline_events.decision_details` + `agent_runs.tool_outputs` (Phase 999.8 predictor attribution) | Which rule fired (regex Pass-1 OR LLM Pass-2), confidence label (high/medium/low), predictor source chip |
| **2 Customer** | `agent_runs.context` (resolver layer that matched: thread / sender / identifier / unresolved) + `automation_runs.result` (iController screenshots from Phase 67 DOM tagging) | Identifier source, confidence, candidate list (top 3), iController before/after screenshots via signed URLs |
| **3 Intent** | `agent_runs.tool_outputs` (ranked-intent JSON from `debtor-intent-agent` Phase 65) | Ranked intent list with confidences, coordinator one-line reasoning, why-selected summary |
| **4 Handler** | `automation_runs.result` (tool calls, side-effects), screenshots from drafter / label-resolver / invoice-copy | Tool calls executed, browserless screenshots, side-effect log (one-liner per effect) |

### Screenshots

- Supabase Storage bucket: `automation-screenshots`, path `{automation}/{label}-{timestamp}.png`
- 1h signed URL expiry — **refresh on every popup open** (1 Storage API call per click; cheap; avoids 401 retries)
- Reachable via JSONB introspection on `automation_runs.result` and `agent_runs.tool_outputs` — no new column needed

### Data-quality fix (display layer)

The 2026-05-11 smoke flagged `sender_name = "Planning"` / `subject = "(no subject)"` rows on Stages 1 and 3. **Confirmed-by-user as a row-mapper display bug, NOT an ingest persistence bug** — raw `email_pipeline.emails` is correct. Fix lives in the per-row component that feeds the audit surface (likely in `web/lib/automations/swarm-bridge/configs.ts` or the Bulk Review row mapper). No data backfill needed.

</scope>

<must_haves>
## Acceptance criteria

- Clicking any of the 5 cells in `PipelineFlow` (detail pane) opens a stage-specific popup within 200ms
- For an email with complete pipeline_events coverage, each stage popup renders:
  - Verdict + 1–3 line reasoning summary
  - Evidence chips/badges where structured (regex match, confidence, identifier source)
  - Screenshots (when present) via fresh signed URL — no broken images, no 401s on first open
- "Show full reasoning" expander on each popup reveals the raw `decision_details` / `tool_outputs` JSON
- Sender / subject metadata renders correctly on every row of every Bulk Review page (Stage 0/1/2/3/4) — no "(unknown sender)" / "Planning" placeholders for rows where `email_pipeline.emails` actually has the data
- Read-only — no form fields on this surface (those land in 82.4)
- Works for both `debtor-email` and `sales-email` swarms (sales-email Stage 0 + Stage 1 will render; Stage 2/3 popups show "not yet implemented for this swarm" until V10.0)

</must_haves>
