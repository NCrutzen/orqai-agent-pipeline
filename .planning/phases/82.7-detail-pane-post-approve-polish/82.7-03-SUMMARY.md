# Phase 82.7 Plan 03: Per-Stage Cancel-Override Link Summary

Formally declared `onCancelDirty` on `PipelineFlowProps`, retired the Plan 02 `@ts-expect-error` bridge on the `<PipelineFlow>` call site in `detail-pane.tsx`, and added a `cancel override` text-button to each `<StageStep>` that is in `dirty` state — completing the per-stage half of D-03 (operator can undo a single accidental override without affecting other dirty stages or prose draft notes).

## What Was Built

- **Task 1 — Prop thread (commit `cda63f9`):**
  - `PipelineFlowProps`: added `onCancelDirty: (stageN: number) => void` directly after `onMarkDirty`. Component destructures it.
  - `<StageStep>` render: passes `onCancelDirty={() => onCancelDirty(s.n)}` as a sibling to the existing `onMarkDirty` callback. Future-pill rendering paths untouched.
  - `detail-pane.tsx`: removed the `// wired in 82.7-03 — PipelineFlow prop signature extended in Plan 03` comment + the `// @ts-expect-error` directive on the `<PipelineFlow onCancelDirty={...}>` call site. With `PipelineFlowProps.onCancelDirty` now declared, the directive would have become a TS2578 unused-directive error on first type-check — bundled into the same commit per worktree_branch_check.

- **Task 2 — Cancel-override link render (commit `40e2ced`):**
  - `StageStepProps`: added `onCancelDirty?: () => void` (optional).
  - Component destructure extended.
  - Dirty branch (`stage.state === "dirty"`) now renders a new sub-row AFTER the `⤓ Override + note save together` coupling helper:
    - Left: `Override in progress` label (`var(--v7-amber)`, `--font-mono`, 12px).
    - Right: `cancel override` text-button (`var(--v7-brand-secondary)`, `--font-mono`, underline-on-hover, focus-visible outline mirroring the existing `override stage` link styling).
    - Click invokes `onCancelDirty?.()` (optional-chained for safe omission).
    - `aria-label={`Cancel override on Stage ${stage.n}`}`, `data-testid={`stage-${stage.n}-cancel-override`}`.

## Deviations from Plan

**[Rule 3 — Cross-commit coupling]** The plan split Task 1 into "extend pipeline-flow" and Task 2 included "remove the @ts-expect-error directive in detail-pane." Splitting these across commits would briefly fail tsc (TS2578) at the Task 1 boundary. The worktree_branch_check explicitly directed combining them into a single commit. I followed the worktree_branch_check: Task 1's commit contains both the `PipelineFlowProps.onCancelDirty` declaration AND the `detail-pane.tsx` directive removal. Task 2's commit is now strictly the `stage-step.tsx` render. Behavioural outcome is identical; commit-atomicity is improved.
