## Phase Boundary

Four operator-observed UX defects on the Stage 1 Bulk Review surface after the Phase 82.6 footer-approve-recordVerdict wiring shipped. All four are confined to:
- `web/app/(dashboard)/automations/[swarm]/_shell/detail-pane.tsx`
- `web/app/(dashboard)/automations/[swarm]/_shell/selection-context.tsx` (for auto-advance)
- `web/components/automations/bulk-review/audit/StageFeedbackPanel.tsx` (for override widget buttons)

**Scope: visual + selection UX only.** No new server actions, no pipeline changes, no Inngest events, no DB writes beyond what 82.6 already wired. Hard-separation lock (Stage 1 ↔ swarm_noise_categories only) is preserved automatically — none of these fixes touch the dispatch path.

**Operator-reported symptoms (verbatim, 2026-05-16):**
1. "Side panel stays on the email you see it go on the left" — row vanishes from list but detail pane still renders the just-approved email.
2. "2 vinkjes on the approve" — Approve button shows `✓ ✓ Approve (Stages 0+1)` (icon glyph + label-prefix glyph render side-by-side).
3. "Override stage looks a bit off" + "buttons on the override could use some love" — `StageFeedbackPanel.tsx` Save note + Confirm buttons feel visually disconnected from the stage card, alignment/padding/contrast off.
4. "If you click override stage by accident you can't approve anymore — you need to override" — clicking `override stage` flips a stage to `dirty`, which morphs the footer into "✓ Submit override" mode (`detail-pane.tsx:450-453` `anyDirty` branch) with no exit. Operator wants a "cancel override / back to Approve" affordance.

</domain>

<decisions>
## Implementation Decisions

### D-01 — Auto-advance detail pane on approve

After footer Approve fires (Stage 1 only, `!anyDirty` branch in `handlePrimary` at `detail-pane.tsx:456-511`), the optimistic removal via `markPendingRemoval(row.id)` makes the row vanish from the visible list, but `selectedId` in `selection-context.tsx` still points at the removed row → detail pane keeps rendering it.

**Decision:** After `markPendingRemoval(row.id)`, the detail pane should advance to the **next visible row** in the current stage-list (the row immediately after the removed one, or the new first row if the removed one was last). If the list becomes empty, clear `selectedId` to `null` so the pane shows an empty/idle state.

**Implementation hint (planner decides):**
- Option A: extend `selection-context.tsx` with a `markPendingRemovalAndAdvance(id, nextId)` setter that atomically updates both — caller (`detail-pane.tsx`) computes `nextId` from its known `rows` prop / list ordering. **Recommended** — keeps selection ownership in the context.
- Option B: emit a window event (`bulk-review:row-removed`) and let `row-list.tsx` pick the next row. More moving parts; rejected.

The "next row" computation must use the **filtered visible list** (active stage tab + mailbox filter + needs-action chip), not the full `rows` prop. The detail pane already receives the visible list via props.

### D-02 — Drop the redundant checkmark from the Approve label

Current: `detail-pane.tsx:454` produces `` `✓ Approve (Stages ${visibleOkStages.join("+")})` `` — and the Button JSX (line ~651) likely renders an icon prefix on top, producing `✓ ✓ Approve`.

**Decision:** Single checkmark. Drop the `✓ ` prefix from the **label string** (`detail-pane.tsx:452-454`); keep the icon component the Button renders. Same fix applies to the override-mode labels (`✓ Submit override (Stage X)` / `✓ Submit overrides (Stages X, Y)`).

Verify in browser after the fix: only one checkmark glyph visible per primary button state (Approve / Submit override).

### D-03 — Cancel-override escape hatch

Operator's frustration: clicking `override stage` (the per-stage "override stage" link/button in `detail-pane.tsx` near each stage card, around line 1313 area — exact location to be confirmed during planning) flips the stage's `feedback` state to `dirty`. The footer `anyDirty` branch then locks into "Submit override" mode with no UI to undo the click.

**Decision:** Add a **"Cancel override · back to Approve"** affordance. Two surfaces:
1. **Per-stage card level** — next to the existing `override stage` link, when that stage is in `dirty` state, swap the link to `cancel override` (or render both). Click resets that stage's feedback to its initial `ok` state.
2. **Footer level** — when `anyDirty` is true, add a secondary subtle button next to "Submit override(s)" labeled `Cancel override` (or `↩ Cancel`). Click resets **all** dirty stages to `ok`, returning the footer to Approve mode.

Both must clear the stage's `feedbackMap` entry / dirty state without sending any server traffic — purely client-side undo of the local override intent.

**Implementation hint:** the `feedbackMap` + dirty tracking lives in `detail-pane.tsx` parent state (introduced by Phase 82.5 Plans 05/06). The planner needs to identify the reducer/setter and add a `resetStageFeedback(stageN)` + `resetAllStageFeedback()` helper.

### D-04 — Polish StageFeedbackPanel buttons + container

`web/components/automations/bulk-review/audit/StageFeedbackPanel.tsx:294,325` currently renders two buttons:
- `Save note · come back later` (secondary)
- `Confirm · looks right` (primary, green)

Operator says they feel off. Audit-and-polish pass:
- **Visual hierarchy:** Confirm should clearly be primary, Save note clearly secondary. Currently the labels' middle-dot separator + two-line wrapping make them feel equal-weight.
- **Container alignment:** the override stage block (the wrapper that contains the textarea + these buttons) "looks a bit off" — likely padding/border/spacing mismatch with the rest of the stage card. Audit and align with the stage card's existing visual rhythm (border radius, padding, border color from `--v7-*` tokens).
- **Compact labels:** shorten to fit the pane without wrapping. Suggested: `Save note` + `Confirm` (drop the trailing descriptor) **OR** keep descriptors but ensure no wrap at the 360-380px right-pane width.

**Decision:** Planner has discretion on the exact final labels + spacing values, but must:
- Confirm primary button is visually dominant (green or brand primary).
- Save note button is clearly secondary (border-only or muted fill).
- Container padding/border matches the stage card pattern (look at `detail-pane.tsx` stage card JSX for the reference rhythm).
- No label wraps at the operator's default pane width.

### Claude's Discretion

- Exact `nextId` selection algorithm when the approved row is the last one (advance to previous? clear?). Recommendation: clear (`null`) — operator's queue is empty for the current filter, showing an empty state is honest.
- Whether the "Cancel override" affordance is a button, a text link, or an icon-only button. Use whatever matches the existing per-stage `override stage` link style for consistency.
- StageFeedbackPanel exact button labels — within the constraints above.
- Whether to show a toast on auto-advance ("Approved · next row"). Recommendation: no toast — the visual change (row vanishes, next row's content fills the pane) is self-evident. Match the silent precedent from 82.6 D-03.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`markPendingRemoval`** (`selection-context.tsx:90`) — already wired. Extend its caller signature to also set the next selected id, or add a sibling `markPendingRemovalAndAdvance` setter.
- **`feedbackMap` state in `detail-pane.tsx`** — parent owns the per-stage dirty/feedback map (Phase 82.5 Plan 05/06). The reducer/setter is the anchor for D-03's `resetStageFeedback(n)` helper.
- **Button component** — same Button that renders `✓ Approve (Stages 0+1)` will be reused for the cancel-override affordance. Look up its prop API (likely supports `variant="secondary"` or similar).

### Established Patterns
- **Phase 82.5 controlled-component contract** — `StageFeedbackPanel` is fully controlled by parent state. Any D-03 "reset" must clear the parent state; do NOT add local state in the panel.
- **Silent UX precedent** — 82.6 D-03 + override-flow precedent both fail silently; auto-advance (D-01) should follow the same "let the visual change speak" rule. No toasts.
- **Footer hide pattern** (82.6 D-02b) — `activeStage === 1` ternary around the primary Button. The cancel-override footer button must also respect this (only render when on Stage 1, which is already implicit since override mode is Stage 1 only on this surface).

### Integration Points
- **Single primary integration site:** `detail-pane.tsx` `handlePrimary` (line 510 — right after `markPendingRemoval`).
- **Secondary integration site:** `detail-pane.tsx` footer JSX (line ~640-660) — add Cancel button next to primary in `anyDirty` mode.
- **Stage card area:** `detail-pane.tsx` per-stage block — find `override stage` link, add `cancel override` sibling that renders when stage state is `dirty`.

</code_context>

<specifics>
## Specific Ideas

- After D-01 ships, the operator should be able to approve 10 rows in a row without ever clicking on a row in the list — each Approve auto-advances to the next.
- After D-03 ships, an accidental `override stage` click is fully recoverable without leaving the keyboard: either click `cancel override` or hit a keyboard shortcut (consider `Esc` — but that may collide with existing shortcuts; planner verifies via `keyboard-shortcuts.tsx`).
- After D-04 ships, the override pane should read as "one card, two clear actions" — not "two buttons in a void".
- Visual verification: after the phase ships, the operator should re-screenshot Stage 1 with a row selected (no override) and a row selected (override mode active) — both should look polished and intentional.

</specifics>
