<!-- CI runtime measurement protocol (from RESEARCH §"CI Runtime Measurement Protocol"): -->

Cold-cache:
  1. Push to a brand-new branch name (cache key is per-branch via actions/setup-node + actions/cache@v4's restore-keys: nextjs-${{ runner.os }}-${{ hashFiles('web/package-lock.json') }}-).
  2. Open PR against main.
  3. Wait for all 5 pr-checks /* to complete.
  4. GitHub Actions UI → workflow run → "Total duration" badge. Record.

Warm-cache:
  1. Same branch, no-op commit (whitespace edit to README or similar).
  2. Wait for re-run.
  3. Record "Total duration".

Recording format (PR comment):
  ## CI runtime budget verification (Req 5)
  | Run | Commit | Workflow URL | Wall-clock |
  |-----|--------|--------------|-----------|
  | Cold cache (fresh branch) | <sha> | <url> | <mm:ss> |
  | Warm cache (re-run)       | <sha> | <url> | <mm:ss> |
  Budget: cold ≤ 8 min, warm ≤ 5 min. Status: ✅ / ❌

Per RESEARCH §"Risks & Landmines" Pitfall 6: the build cache invalidates on every
.ts/.tsx edit. Warm-cache budget should be measured on the doc-only follow-up
PR, NOT on the burn-down PR itself.
</interfaces>
