## User Constraints (from CONTEXT.md)

### Locked Decisions

**next/headers mock shape:**
- **D-01:** Mock lives globally in `web/test-setup.ts` via a single `vi.mock("next/headers", ...)`.
- **D-02:** Defaults are empty/safe — `cookies()` returns `{ get: () => undefined, getAll: () => [], set: () => {}, delete: () => {} }`, `headers()` returns `new Headers()`, `draftMode()` returns `{ isEnabled: false }`. Tests opt into auth state explicitly.
- **D-03:** Mock is overridable per-test via exported helpers from `test-setup.ts` (e.g. `__setMockCookies(map)`); cleared in a global `afterEach` so tests don't have to remember.

**Supabase admin mock design:**
- **D-04:** Chainable Proxy — every method access returns the same proxy; terminal awaits (`.then`, `.single`, `.maybeSingle`) resolve to per-test `{ data, error }`. Future chain methods (`.not`, `.schema`, `.order`, `.range`, `.ilike` …) work without code changes to the mock.
- **D-05:** Mock helper lives at **`web/test-utils/supabase-mock.ts`** (new directory). Imported via `import { createSupabaseAdminMock } from "@/test-utils/supabase-mock"` and applied with `vi.mock("@/lib/supabase/admin", ...)`.
- **D-06:** Per-call response API is queue-based: `mock.queueResponse({ data, error })` pushes in order; terminal awaits dequeue. Empty queue at terminal → throw with a clear message.

**Fixture-vs-skip cutoff:**
- **D-07:** Per-test cutoff time-boxed at 15 minutes. Inside window → fix; outside → `.skip()` + todo.
- **D-08:** Skipped-test todos are minimal — title = test name; body = failing line + one-line root-cause + which fixture needs work. One page max.
- **D-09:** Hard cap of **10 `.skip()`s** total across the phase. If exceeded, pause and reassess.

**any-narrowing style for lib/inngest/:**
- **D-10:** Default narrowing strategy = inline minimal interfaces, co-located with usage. No full domain model.
- **D-11:** For Inngest `event.data`: type per function via the registered event schema (hook into the `client.createFunction({...})` signature). If no registered schema, inline interface for `event.data` only + `// TODO(phase-88.2-followup): register event schema`.
- **D-12:** `unknown` + boundary assertion acceptable for genuine boundary code (HTTP parsing, JSON.parse of external payloads); NOT a shortcut for internal calls.

**Lint burn-down commit cadence:**
- **D-13:** Per-directory commits — one commit per top-level area (e.g. `fix(lint): narrow any in lib/inngest/functions/`). Not one big commit, not per-rule sprawl.

**React Compiler errors:**

## `next/headers` Mock — Recommended Vitest Pattern

D-01 + D-02 + D-03 land entirely in `web/test-setup.ts`. Recommended shape (consumed by the planner):

```typescript
// web/test-setup.ts
import "@testing-library/jest-dom/vitest";
import { vi, afterEach } from "vitest";

// Mutable per-test state — reset by the global afterEach below.
let mockCookies = new Map<string, string>();
let mockHeaders: Record<string, string> = {};
let mockDraftMode = false;

export function __setMockCookies(entries: Record<string, string> | Map<string, string>): void {
  mockCookies = entries instanceof Map ? entries : new Map(Object.entries(entries));
}
export function __setMockHeaders(entries: Record<string, string>): void {
  mockHeaders = entries;
}
export function __setMockDraftMode(enabled: boolean): void {
  mockDraftMode = enabled;
}

vi.mock("next/headers", () => ({
  cookies: () => ({
    get: (name: string) => {
      const value = mockCookies.get(name);
      return value === undefined ? undefined : { name, value };
    },
    getAll: () =>
      Array.from(mockCookies.entries()).map(([name, value]) => ({ name, value })),
    set: (_name: string, _value: string) => {},
    delete: (_name: string) => {},
    has: (name: string) => mockCookies.has(name),
  }),
  headers: () => new Headers(mockHeaders),
  draftMode: () => ({ isEnabled: mockDraftMode, enable: () => {}, disable: () => {} }),
}));

afterEach(() => {
  mockCookies = new Map();
  mockHeaders = {};
  mockDraftMode = false;
});
```

**Note on Next.js 16 async API:** In Next.js 15+, `cookies()`, `headers()`, and `draftMode()` are async (return `Promise<>`) when consumed in server components. `await cookies()` is the canonical call shape. The mock above returns synchronously — **planner must verify** whether the consuming code in stage-0..3 server components uses `await cookies()` vs sync. If async, wrap return in `Promise.resolve(...)`. [ASSUMED — Next 16 docs not re-verified this session; see Open Question OQ-1.]

**Hoisting hazard:** `vi.mock` is hoisted to the top of the file by Vitest's transformer. The `mockCookies`/`mockHeaders` state variables must be declared **before** the mock factory but the closure reference inside the factory is resolved at call-time, not hoist-time, so this works. Pattern validated in Vitest 1.x+ docs [CITED: vitest.dev/api/vi.html#vi-mock — pattern unchanged in v4].

## Supabase Admin Mock — Recommended Shape (D-04..D-06)

```typescript
// web/test-utils/supabase-mock.ts
import { vi } from "vitest";

type Response = { data: unknown; error: unknown };

export interface MockAdmin {
  from: (table: string) => ChainProxy;
  schema: (s: string) => MockAdmin;
  rpc: (fn: string, args?: unknown) => ChainProxy;
  /** Push a response that the next terminal-await will resolve to. */
  queueResponse: (r: Response) => void;
  /** Call log: every method invocation with its args, in order. */
  calls: Array<{ method: string; args: unknown[] }>;
}

// Chain proxy: every property access returns the same proxy except for the
// terminal awaitables (`then`, `single`, `maybeSingle`) which dequeue a
// queued response.
type ChainProxy = {
  // Indexed signature: any method name returns self.
  [k: string]: unknown;
} & Promise<Response>;

export function createSupabaseAdminMock(): MockAdmin {
  const responseQueue: Response[] = [];
  const calls: Array<{ method: string; args: unknown[] }> = [];

  const makeChain = (): ChainProxy => {
    const dequeue = (): Response => {
      const r = responseQueue.shift();
      if (r === undefined) {
        throw new Error(
          "[supabase-mock] no queued response — did the code call .from() more times than the test queued? " +
            `Call log so far: ${JSON.stringify(calls)}`,
        );
      }
      return r;
    };

    const proxy: any = new Proxy(() => {}, {
      get(_target, prop) {
        if (prop === "then") {
          // Awaited — dequeue and resolve.
          return (onFulfilled: (v: Response) => unknown) =>
            Promise.resolve(dequeue()).then(onFulfilled);
        }
        if (prop === "single" || prop === "maybeSingle") {
          return () => Promise.resolve(dequeue());
        }
        return (...args: unknown[]) => {
          calls.push({ method: String(prop), args });
          return proxy;
        };
      },
      apply() {
        return proxy;
      },
    });
    return proxy as ChainProxy;
  };

  const admin: MockAdmin = {
    from: (table: string) => {
      calls.push({ method: "from", args: [table] });
      return makeChain();
    },
    schema: (s: string) => {
      calls.push({ method: "schema", args: [s] });
      return admin; // chainable: admin.schema("x").from("y")
    },
    rpc: (fn: string, args?: unknown) => {
      calls.push({ method: "rpc", args: [fn, args] });
      return makeChain();
    },
    queueResponse: (r) => {
      responseQueue.push(r);
    },
    calls,
  };

  return admin;
}
```

## Common Pitfalls

### Pitfall 1: `vi.mock` hoisting + closure capture

**What goes wrong:** Variables referenced inside `vi.mock(...)`'s factory must exist at module-evaluation time. If the test imports a helper that internally calls `cookies()` at module load (NOT inside a function), the mock won't be in place yet.
**Why it happens:** Vitest hoists `vi.mock` calls above all imports, but the factory body still runs once when the mocked module is first required.
**How to avoid:** Keep the mock factory **purely synchronous** and reference only the closure'd state variables (`mockCookies`, etc.). Do **not** import anything inside the factory.
**Warning signs:** "`cookies` was called outside a request scope" persisting after the mock lands → the import is happening before the mock is registered.

### Pitfall 2: `cookies()` async in Next.js 15+ / 16

**What goes wrong:** Next 15 made `cookies()`, `headers()`, `draftMode()` return Promises. If the consuming code does `const c = await cookies()`, the mock returning a sync object causes `await` to succeed (Promise.resolve of an object) but `c.get(...)` works fine — actually safe by accident.
**Why it happens:** `await nonPromise` returns the non-promise value unchanged.
**How to avoid:** Verify on at least one stage-shell test before rolling out. If async-style access works with the sync mock, ship it. Otherwise wrap returns in `Promise.resolve(...)`.
**Warning signs:** A passing-but-suspicious test that never actually exercises the cookie state. Add one assertion that reads back a `__setMockCookies({foo: "bar"})` value to confirm the path is live.

### Pitfall 3: ESLint `globalIgnores` order matters in flat config

**What goes wrong:** If a later config block in `defineConfig([...])` specifies `files: ["**/*.ts"]` and a global rule, files in `lib/automations/**` may still be matched because flat-config ignore semantics are config-block-scoped, not global, **unless** you use `globalIgnores()`.
**Why it happens:** `eslint-config-next` may include blocks with explicit `files:` patterns.
**How to avoid:** Use `globalIgnores()` (already imported in `eslint.config.mjs`), not a per-block `ignores:` array. The current config does this correctly for `.next/**` etc. — just extend the same array.
**Warning signs:** `npm run lint` still reports errors in `lib/automations/**` after the config edit. Run `npm run lint -- --debug 2>&1 | grep automations` to see which config block is matching.

### Pitfall 4: React Compiler "set-state-in-effect" — easy to make worse

**What goes wrong:** Naïvely deleting the `setState` call breaks the component; naïvely wrapping in `setTimeout(0)` makes RC happy but introduces a real flicker; naïvely adding a guard (`if (!already) setState(...)`) defers but doesn't fix the cascade.
**Why it happens:** The compiler is pointing at a real anti-pattern: derived state. The right fix is usually to *derive* the value during render instead of *storing* it via `setState`.
**How to avoid:** For each of the 7 setState-in-effect errors, ask: "Can I compute this value from props/existing state during render?" If yes, replace `useEffect(() => setX(derive(...)))` with `const x = derive(...)`. If no, the value is genuinely derived from a side effect (subscription, network) and the effect stays — but the setState gets wrapped in the effect's update-guard.
**Warning signs:** Tests start failing in completely different files after a setState fix — sign that you converted a stored value to a derived one and downstream tests were asserting on the stored shape.

### Pitfall 5: Codegen drift in CI without local secrets

**What goes wrong:** The `codegen` CI job runs `npm run codegen && git diff --exit-code`. Locally, `npm run codegen` **fails** without `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` in `web/.env.local` (verified this session: `[gen-entity-types] FAILED: ... required`). CI has the secrets; local doesn't. So a regen drift can ship undetected locally.
**Why it happens:** `gen-entity-types.ts` queries Supabase live; without creds it bails.
**How to avoid:** Either (a) before opening the phase PR, confirm `web/.env.local` has the required keys and run `npm run codegen && git diff --exit-code` locally; OR (b) accept that codegen drift is a CI-only check and watch for it on the PR. The phase MUST NOT introduce any registry change that would alter `*.generated.ts` output — the SPEC explicitly forbids registry edits.
**Warning signs:** `codegen` CI job goes red on the phase PR. Triage: pull `web/.env.local`, run codegen locally, inspect the diff.

### Pitfall 6: Build-cache key includes every `.ts`/`.tsx`

**What goes wrong:** `pr-checks.yml#build` uses cache key `nextjs-...-${{ hashFiles('web/**/*.[jt]s', 'web/**/*.[jt]sx') }}`. Every source-file edit in waves W2/W3 invalidates the build cache → cold-cache build runs on every push.
**Why it happens:** Cache key is too granular for a burn-down phase that touches many files.
**How to avoid:** Accept the cold-cache hit during burn-down; measure warm-cache only on a post-merge follow-up PR (typo fix / doc tweak) that doesn't change source.
**Warning signs:** Req 5 warm-cache run on the phase PR exceeds 5 min. Acceptance permits the warm measurement on a **follow-up PR** — use that for the budget verification rather than the burn-down PR itself.
