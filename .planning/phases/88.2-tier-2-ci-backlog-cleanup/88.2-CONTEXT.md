<decisions>
## Implementation Decisions

### next/headers mock shape
- **D-01:** Mock lives **globally in `web/test-setup.ts`** via a single `vi.mock("next/headers", ...)`. Picked up by all 5 stage-shell tests and any future server-component test without per-file boilerplate.
- **D-02:** Default returns are **empty/safe**:
  - `cookies()` → `{ get: () => undefined, getAll: () => [], set: () => {}, delete: () => {} }`
  - `headers()` → `new Headers()`
  - `draftMode()` → `{ isEnabled: false }`
  Tests opt into auth/header state explicitly; the default does not pretend to be authenticated.
- **D-03:** Mock is **overridable per-test via exported helpers** from `test-setup.ts` (e.g. `__setMockCookies(map)`, `__setMockHeaders(record)`). Cleared in an `afterEach` that the setup file registers globally so tests don't have to remember.

### Supabase admin mock design
- **D-04:** Mock shape is a **chainable Proxy** — every method access returns the same proxy; terminal awaits (`.then`, `.single`, `.maybeSingle`) resolve to per-test-configured `{ data, error }`. Future chain methods (`.not()`, `.schema()`, `.order()`, `.range()`, `.ilike()` ...) work without code changes to the mock.
- **D-05:** Mock helper lives at **`web/test-utils/supabase-mock.ts`** (new shared directory). Sets the convention for future shared test utilities. Imported via `import { createSupabaseAdminMock } from "@/test-utils/supabase-mock"` and applied with `vi.mock("@/lib/supabase/admin", ...)` in the consuming test or test-setup if global is appropriate.
- **D-06:** Per-call response API is **queue-based**: `mock.queueResponse({ data, error })` pushes expected responses in order; terminal awaits dequeue them. Mirrors actual query order, predictable, easy to assert call counts. Empty queue at terminal → throw with a clear message ("no queued response — did the code call .from() more times than the test queued?").

### Fixture-vs-skip cutoff
- **D-07:** Per-test cutoff is **time-boxed at 15 minutes**. Executor spends up to 15 min on a fixture-drift test; if it can be repaired inside that window (regenerate snapshot, add missing key, adjust 1-2 assertions), fix it. Otherwise `.skip()` + todo. Bounds the phase against fixture archaeology spirals.
- **D-08:** Skipped-test todos are **minimal**: title = test name; body = failing line + one-line root-cause guess + which file/fixture likely needs work. One page max. Cheap to write, enough for whoever picks it up.
- **D-09:** Hard cap of **10 `.skip()`s total across the phase**. If we'd exceed 10, pause and reassess — likely means fixtures need their own dedicated phase, not a sub-task here. Cookies-mock fix and Supabase mock together should clear most of the 47 without needing skips; the cap exists to keep the "CI is green only because we skipped a lot" optics from biting.

### any-narrowing style for lib/inngest/ (and similar)
- **D-10:** Default narrowing strategy is **inline minimal interfaces** — for each `any`, write the smallest interface that satisfies the call sites. Keeps types co-located with usage. No attempt to build a full domain model in this phase.
- **D-11:** For Inngest `event.data` access specifically: **type per function via the registered event schema**. Inngest functions register schemas already; downstream agent should hook those into the function signature so `event.data` is typed inside `step.run` blocks. One-time investment that pays off across every future Inngest function. If a function has no registered schema, fall back to an inline interface for `event.data` only and add a `// TODO(phase-88.2-followup): register event schema` comment.
- **D-12:** `unknown` + boundary assertion is acceptable for genuine boundary code (HTTP response parsing, JSON.parse of external payloads) but **not** as a shortcut for narrowing internal calls. Reviewer guidance for the PR.

### Lint burn-down commit cadence
- **D-13:** **Per-directory commits** — one commit per top-level area (e.g. `fix(lint): narrow any in lib/inngest/functions/`, `fix(lint): require → import in components/v7/`). Reviewable diffs, clear blast radius, easy partial revert. NOT one big "fix 194 lint errors" commit, NOT per-rule sprawl.

### React Compiler errors
- **D-14:** Fix at source (per SPEC constraint — no rule disable). Specific patterns expected: extract render-time side effects into `useEffect`, lift conditional hook calls, wrap JSX-in-try/catch in a separate component boundary. Planner should produce a per-error breakdown when scanning the 12 RC errors.

### `lib/automations/` ignore mechanism
- **D-15:** Use ESLint flat-config **`globalIgnores()`** in `web/eslint.config.mjs` (not `.eslintignore` — deprecated in flat config). One-line entry: `globalIgnores(["lib/automations/**"])`. Justification comment inline citing SPEC §"Out of scope".

<specifics>
## Specific Ideas

- The "trusted boundary" framing for `lib/automations/` is the user's deliberate stance — these are tactical scripts, externally credentialed, reviewed by humans on commit. Don't treat the ignore as a workaround; it's the intended posture.
- Time-boxing (D-07, 15 min/test) and skip-cap (D-09, ≤ 10) are paired: the time-box governs individual cost, the cap governs aggregate optics. Both must hold.
- Empty-default `cookies()` (D-02) is intentional — tests that need a session should declare it explicitly. We're not building a "feels like prod" mock; we're building a green CI floor.
