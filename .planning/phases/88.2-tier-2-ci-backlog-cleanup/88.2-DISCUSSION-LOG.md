## next/headers mock shape

### Q1: Where should the next/headers mock live?

| Option | Description | Selected |
|--------|-------------|----------|
| Global in test-setup.ts | One vi.mock in test-setup.ts. All 5 stage tests + future server-component tests pick it up automatically. Risk: hides bugs where real request-scope is intended. | ✓ |
| Per-file vi.mock in 5 stage tests | Each stage-shell test declares its own mock. Explicit, but 5x duplication. | |
| Shared helper imported per-file | web/test-utils/mock-next-headers.ts called from each test's beforeAll. | |

### Q2: What should cookies() / headers() return by default in the mock?

| Option | Description | Selected |
|--------|-------------|----------|
| Empty defaults | cookies() → empty getters; headers() → new Headers(); draftMode() → { isEnabled: false }. | ✓ |
| Supabase-session-shaped fixtures | Pre-populated with sb-access-token / sb-refresh-token cookies. More realistic but couples mock to one auth library. | |
| Throw by default, opt-in per test | Forces every test to acknowledge it uses request-scoped APIs. Highest friction. | |

### Q3: Should the global mock be overridable per-test, and how?

| Option | Description | Selected |
|--------|-------------|----------|
| Export setCookies/setHeaders helpers | test-setup.ts exports __setMockCookies(map) / __setMockHeaders(obj). Tests import and call in beforeEach. | ✓ |
| Each test does vi.mocked(cookies).mockReturnValue(...) | Standard Vitest pattern, more boilerplate per test. | |
| No — don't build infrastructure for hypothetical | Keep the global mock dumb until a test actually needs overrides. | |

**Notes:** Empty-default + overridable helpers is the "green CI floor, opt-in realism" combo. Tests that need auth state declare it explicitly.

## Supabase admin mock design

### Q1: What's the mock shape for the Supabase admin client?

| Option | Description | Selected |
|--------|-------------|----------|
| Chainable Proxy | Single Proxy where every method access returns self, and terminal methods resolve to per-test-configured {data, error}. Handles all chain methods without code changes. | ✓ |
| Explicit builder with typed chain methods | Hand-written builder exposing every known method. Type-safe but every new prod method = mock update. | |
| Off-the-shelf community package | Less code we own; risk of maintenance + version drift. | |

### Q2: Where does the mock helper live?

| Option | Description | Selected |
|--------|-------------|----------|
| web/test-utils/supabase-mock.ts | New shared directory under web/. Imported by any test that needs the admin client. | ✓ |
| web/lib/supabase/__mocks__/admin.ts | Co-locate using Vitest's __mocks__ convention. Auto-picked-up by vi.mock. | |
| Inline in each test | Co-located behavior; high duplication. | |

### Q3: How do tests configure per-call responses on the Proxy mock?

| Option | Description | Selected |
|--------|-------------|----------|
| Queue-based: mock.queueResponse({data, error}) | Tests push expected responses in order; terminal awaits dequeue. Mirrors actual SQL execution. | ✓ |
| Path-keyed: mock.whenFrom('table').select('*').returns({data}) | Expressive but more code to maintain and learn. | |
| Default-resolve to {data: null, error: null}; tests stub at call sites | Lightest, least guidance. | |

**Notes:** Proxy + queue is the right shape for the failures we're fixing (chained calls with sequential terminal awaits). Path-keyed would be over-engineered for a CI-greening phase.

## Fixture-vs-skip cutoff

### Q1: What's the concrete rule for 'refresh fixture' vs '.skip() + todo'?

| Option | Description | Selected |
|--------|-------------|----------|
| Time-boxed: 15 min/test, then skip | Bounds the phase against fixture archaeology spirals. | ✓ |
| Diff-size threshold | Refresh if schema delta ≤ N fields. Mechanical but hard to apply uniformly. | |
| Per-failure-mode: small obvious = fix, semantic = skip | Requires executor judgment. | |

### Q2: How aggressive is the todo content for skipped tests?

| Option | Description | Selected |
|--------|-------------|----------|
| Minimal: test name + failure line + root cause guess | One-page todo. Enough for whoever picks it up. | ✓ |
| Full: investigation notes + suggested fix + estimated effort | Higher quality handoff, much more phase time. | |
| Auto-grouped by area | Less noise but loses granularity. | |

### Q3: Hard cap on total .skip() count for this phase?

| Option | Description | Selected |
|--------|-------------|----------|
| Up to 10 skips total | Generous ceiling; pause and reassess beyond that. | ✓ |
| Up to 5 skips total | Tighter; risks phase blowing time-box. | |
| No hard cap — time-box per test is the only governor | Phase still completes but optics suffer. | |

**Notes:** Time-box (per-test cost) + skip-cap (aggregate optics) are paired governors. Both must hold.

## any-narrowing style for lib/inngest/

### Q1: Default narrowing strategy when fixing no-explicit-any errors?

| Option | Description | Selected |
|--------|-------------|----------|
| Inline minimal interfaces | Smallest interface satisfying call sites. Co-located, low-friction. | ✓ |
| Concrete domain types from existing models | Highest fidelity but slow; may exceed phase budget. | |
| unknown + boundary assertions | Type-safe, fast, but pushes shape mismatches to runtime. | |

### Q2: Inngest event payload typing — event.data approach?

| Option | Description | Selected |
|--------|-------------|----------|
| Type per function via registered event-schema | Inngest functions register schemas; hook them in so event.data is typed inside step.run. | ✓ |
| Per-function inline interface for event.data only | Faster but repeats the pattern every function. | |
| Cast at the top: const data = event.data as MyType | Pragmatic; lies about safety. | |

### Q3: Commit cadence for the lint burn-down?

| Option | Description | Selected |
|--------|-------------|----------|
| Per-directory commits | One commit per top-level area. Reviewable diffs, clear blast radius. | ✓ |
| Per-rule commits | All no-require-imports together, all no-explicit-any together. Sprawls across files. | |
| One big lint commit | Fastest to write, hardest to review. | |

**Notes:** Event-schema typing is a one-time investment with compounding payoff. `unknown` + boundary assertion is still acceptable for genuine boundary code (HTTP parse, JSON.parse of external payloads) — see CONTEXT.md D-12.
