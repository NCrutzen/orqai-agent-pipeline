# Phase 999.5: Switch Stage 0/1 single-shot classifiers from Orq Router to Orq Deployments — Research

**Researched:** 2026-05-07
**Domain:** Orq.ai transport-layer migration (Router → Deployments) for cost tracking restoration
**Confidence:** MEDIUM-HIGH (Orq SDK request/response shapes verified in `@orq-ai/node@4.7.7` source; one critical assumption — per-call cost field — remains [ASSUMED] pending Wave 0 live smoke. Endpoint, request schema, JSON-schema config, and queue-product distinction all VERIFIED.)

## Summary

Phase 999.4 swapped Stage 0 (`stage-0-safety-classifier`) and Stage 1 (`stage-1-category-classifier`) onto `POST /v2/router/chat/completions` to escape the Orq Agents-product queue (6–17 min stuck-row tail). It worked — but Router is a raw model proxy and does NOT surface per-call billing in the response, so `automation_runs.result.cost_cents = 0` for every Stage 0/1 run. That breaks the Stage 0 budget gate (`web/lib/stage-0/budget-counter.ts` reads `automation_runs.result.cost_cents` over the last 30 days against a fixed ceiling) and zeros out the dashboard cost surface for the two highest-volume classifiers in the funnel.

Phase 999.5 migrates to **Orq Deployments** — a separate Studio-versioned product that:
- exposes the same `POST /v2/deployments/invoke` endpoint with full versioned config in Studio,
- carries `usage` (token counts) in the response, with `cost`/`billing` field [ASSUMED] surfaced (see A1, the canonical Wave 0 gate),
- uses a different runtime path from the Agents product, so it does NOT share the queue that bit us in 999.4,
- centralises model + system prompt + JSON schema + fallback chain in Studio (versioned, restorable, side-by-side comparison) — retiring `orq_agents.system_prompt` (added 999.4 Plan 03), one less drift surface.

**Primary recommendation:** Refactor `invokeOrqModel` in `web/lib/automations/orq-agents/client.ts` to POST to `/v2/deployments/invoke` with body `{ key, inputs, metadata?, extraParams? }`. Configure both deployments in Studio (model, system prompt, JSON schema, fallback chain). Keep the registry's `agent_key` as the lookup primary key — add a `deployment_key` column (or reuse `orqai_id` when `swarm_type='deployment'`). Wave 0 must perform a 5-minute live smoke against ONE deployment to confirm the response shape carries cost — this is the single point of [ASSUMED] knowledge in this research.

### Locked Decisions (provisional — to be confirmed in `/gsd-discuss-phase`)

- **L-01:** Switch transport from `POST /v2/router/chat/completions` to `POST /v2/deployments/invoke` for `stage-0-safety-classifier` and `stage-1-category-classifier`. No other agents change.
- **L-02:** Configure each deployment in Orq Studio with: model = Bedrock EU Haiku 4.5 (`aws/eu.anthropic.claude-haiku-4-5-20251001-v1:0`); fallback model(s) per Studio fallback chain UI (Sonnet 4.5 EU primary candidate); system prompt copied from current `orq_agents.system_prompt`; response_format = JSON Schema (`anyOf` for nullable per CLAUDE.md learning `3970bad9`); temperature = 0; max_tokens copied from `model_config.max_tokens`.
- **L-03:** Restore `cost_cents` end-to-end. Wire whichever cost field the Deployments response carries into `InvokeResult.cost_cents` and `result.billing.total_cost`. Stage 0 budget gate becomes accurate again. **Validation gate:** `cost_cents > 0` on a live trace post-deploy.
- **L-04:** Drop `orq_agents.system_prompt` column ONCE both Stage 0 + Stage 1 callers consume the new path AND no other caller reads the column. Migration is additive in the sense that we only stop *reading* it — drop via a second migration after a stabilisation window.
- **L-05:** Keep the 45 s `OrqClientTimeoutError` deadline at the fetch boundary (999.4 Fix B). Stage 0 D-02 fail-open coercion (`verdict='safe'` on timeout) and Stage 1 D-11 catch (`category_key='unknown'`) remain unchanged — only the URL and request body shape change inside `invokeOrqModel`.
- **L-06:** Keep the `stage-0/stale-sweeper` cron (999.4 Plan 04). Defense-in-depth; orthogonal to transport.
- **L-07:** Migration is two-step rollout, not big-bang: (a) Wave 0 — create deployments in Studio behind a feature flag, smoke against ONE deployment with curl/tsx; (b) Wave 1 — refactor `invokeOrqModel` body shape behind a feature flag (env var `ORQ_TRANSPORT=router|deployments` defaulting to `router` until Wave 2 flip); (c) Wave 2 — flip the flag for Stage 0 + Stage 1, verify `cost_cents > 0` on canary; (d) Wave 3 — remove the Router code-path branch and the `system_prompt` column.

### Claude's Discretion

- **Body field shape for `inputs`** — Deployments use `{{variable}}` template substitution at the deployment level. The current Stage 0 user message is the JSON body of `{ email_id, email_subject, email_body }`; the current Stage 1 user message is `{ subject, body_text, categories: [...] }`. Two options: (a) keep the Studio prompt's user-template as `{{json_input}}` with a single variable that we stringify into; (b) split each input field into its own template variable. Recommend (b) — better Studio observability, `inputs` payload is already structured. Researcher recommends but planner finalises.
- **Registry shape change** — Add a `deployment_key text` column on `public.orq_agents`, or repurpose `orqai_id` (which currently holds the Agents-product slug)? Recommend a NEW column `deployment_key` and leave `orqai_id` for the (now-stale-for-Stage-0/1) Agents-product slug. Cleaner audit trail. Planner decides.
- **Feature-flag mechanism** — env var `ORQ_TRANSPORT` (process-wide), or per-row `orq_agents.transport ∈ {agents,router,deployments}`? Recommend per-row column for sub-agent rollout granularity (one classifier at a time). Planner decides.
- **Retire `invokeOrqModel` entirely vs evolve it** — name is now misleading (the helper will hit Deployments, not the Router). Options: (a) keep the name, just change the body; (b) introduce `invokeOrqDeployment` and deprecate `invokeOrqModel`. Recommend (b) for clarity; (a) for lower churn. Planner decides after reading call-sites.

### Deferred Ideas (OUT OF SCOPE)

- Migrating other Orq agents (Stage 2 entity, Stage 3 coordinator, Stage 4 handlers) — only after Phase 999.5 proves Deployments handle our load.
- Enforcing `response_format` per-call via `extraParams` — Studio deployment-level config wins; per-call override is an anti-pattern that re-introduces drift.
- Implementing Orq's `deployments/{id}/metrics` endpoint to push custom evaluator scores — separate observability phase.
- Multi-deployment per agent_key (canary / A/B between deployment versions) — Studio supports it, but adds plan complexity. Defer until needed.
- A `cost_cents` reconciliation job that retroactively patches old Router-path rows from Orq's analytics API — out of scope; old rows stay 0.

### Pattern 2: `invokeOrqDeployment` (or refactored `invokeOrqModel`) — direct-path transport (REQ-DEP-02)

**What:** Sibling helper to today's `invokeOrqModel`, POSTs to `/v2/deployments/invoke`. Reuses `OrqClientTimeoutError` + 45 s `AbortController` (unchanged from 999.4). Reads `agent.deployment_key` (new column) from registry; nothing else from registry needed for transport.

**Example:**
```typescript
// Source: NEW — web/lib/automations/orq-agents/client.ts
// (sibling to invokeOrqModel; or replace invokeOrqModel body entirely depending on
//  Discretion item "Retire invokeOrqModel entirely vs evolve it")

const DEPLOYMENTS_INVOKE_URL = "https://api.orq.ai/v2/deployments/invoke";

export async function invokeOrqDeployment(
  agent_key: string,
  inputs: Record<string, unknown>,
  opts?: { metadata?: Record<string, unknown>; extraParams?: Record<string, unknown> },
): Promise<InvokeResult> {
  if (!ORQ_API_KEY) throw new Error("ORQ_API_KEY is not set");
  const agent = await loadAgentByKeyCached(agent_key);
  const deployment_key = (agent as { deployment_key?: string }).deployment_key;
  if (!deployment_key) {
    throw new Error(
      `Orq agent ${agent_key} registry row missing deployment_key — populate via migration before Wave 2 flip`,
    );
  }

  const requestBody = {
    key: deployment_key,
    inputs,
    // NB: no system prompt, no response_format, no model, no fallbacks here.
    // ALL of that is configured at the deployment level in Studio (Pattern 1).
    ...(opts?.metadata ? { metadata: opts.metadata } : {}),
    ...(opts?.extraParams ? { extra_params: opts.extraParams } : {}),
  };

  const ctrl = new AbortController();
  const timer = setTimeout(() => ctrl.abort(), CLIENT_DEADLINE_MS);
  let res: Response;
  try {
    res = await fetch(DEPLOYMENTS_INVOKE_URL, {
      method: "POST",
      headers: {
        "content-type": "application/json",
        authorization: `Bearer ${ORQ_API_KEY}`,
      },
      body: JSON.stringify(requestBody),
      signal: ctrl.signal,
    });
  } catch (err) {
    if ((err as { name?: string } | undefined)?.name === "AbortError") {
      throw new OrqClientTimeoutError(
        `Orq deployments deadline exceeded for ${agent_key} after ${CLIENT_DEADLINE_MS}ms`,
      );
    }
    throw err;
  } finally {
    clearTimeout(timer);
  }

  if (!res.ok) {
    const text = await res.text().catch(() => "");
    throw new Error(
      `Orq deployments invoke ${agent_key} failed: HTTP ${res.status} ${text.slice(0, 200)}`,
    );
  }

  // [VERIFIED: SDK type DeploymentInvokeResponseBody at
  //  web/node_modules/@orq-ai/node/src/models/operations/deploymentinvoke.ts:285-339]
  // Top-level fields: id, created, object, model, provider, isFinal, integrationId,
  // telemetry { traceId, spanId }, finalized, systemFingerprint, retrievals,
  // providerResponse, usage, choices[].
  // The SDK type does NOT explicitly type a top-level `cost` — see A1 + S-1.
  const json = (await res.json()) as {
    id: string;
    choices?: Array<{ index?: number; message?: { content?: string | null; role?: string } }>;
    usage?: {
      total_tokens?: number;
      prompt_tokens?: number;
      completion_tokens?: number;
    };
    // [ASSUMED A1] one of these may carry per-call cost. Wave 0 smoke prints raw
    // response keys; whichever exists, the parser maps to InvokeResult.cost_cents.
    cost?: number;          // candidate — likely absent
    billing?: { total_cost?: number; cost?: number; cost_cents?: number };  // candidate
    provider_response?: { usage?: { total_cost?: number } };  // candidate (Anthropic-style)
    telemetry?: { traceId?: string; spanId?: string };
  };

  const rawText = json.choices?.[0]?.message?.content ?? "";
  if (!rawText) {
    throw new Error(`Orq deployments invoke ${agent_key}: empty content`);
  }
  const stripped = rawText
    .trim()
    .replace(/^```(?:json)?\s*/i, "")
    .replace(/\s*```$/i, "");
  let raw: unknown;
  try {
    raw = JSON.parse(stripped);
  } catch (e) {
    throw new Error(
      `Orq deployments invoke ${agent_key}: JSON.parse failed: ${(e as Error).message}`,
    );
  }

  // Wave 0 smoke S-1 resolves which of these candidates is the actual cost field.
  // Until then, prefer the most-specific to least-specific candidate.
  const total_cost =
    json.billing?.total_cost
    ?? json.cost
    ?? json.provider_response?.usage?.total_cost
    ?? 0;
  const cost_cents = Math.round(total_cost * 100);

  return {
    raw,
    agent,
    usage: {
      prompt_tokens: json.usage?.prompt_tokens ?? 0,
      completion_tokens: json.usage?.completion_tokens ?? 0,
      total_tokens: json.usage?.total_tokens ?? 0,
    },
    billing: { total_cost },
    cost_cents,
  };
}
```

### Anti-Patterns to Avoid

- **Configuring response_format per-call via `extraParams`** when Studio already has it. Re-introduces the SSOT split that L-04 eliminates.
- **Hardcoding deployment keys in TS.** Use `orq_agents.deployment_key`. Same registry-as-SSOT principle that drove the Phase 56 client design.
- **Skipping Wave 0 smoke** because "the SDK types look right." The SDK type does NOT explicitly type a top-level cost field; only Wave 0 against the live API confirms which path returns cost. See Pitfall 1.
- **Big-bang transport flip without a feature flag.** L-07 mandates per-row flag for safe rollback.
- **Dropping `system_prompt` in the same migration that switches transport.** The flag-flip and the column drop must be ordered; dropping the column while a code-path still reads it triggers `loadAgentByKeyCached` to throw at first invocation. Drop in Wave 3 only.
- **Pasting the cron string into JSDoc** — non-applicable here (no new cron in this phase) but flagged because the planner may extend the sweeper. CLAUDE.md `*/N` rule.
- **Destructuring `inngest.send`** — non-applicable (no new emits) but Phase 65 learning still binds.

## Common Pitfalls

### Pitfall 1: Cost field path is unverified pre-Wave-0
**What goes wrong:** Code assumes `json.billing.total_cost` exists; in reality the field may be `json.cost`, `json.usage.cost`, `json.provider_response.cost`, or absent entirely.

**Why it happens:** Public Orq SDK type `DeploymentInvokeResponseBody` does not explicitly type a top-level cost field [VERIFIED: `web/node_modules/@orq-ai/node/src/models/operations/deploymentinvoke.ts:285-339` lists `usage` (tokens only) but no `cost`/`billing`/`price`]. Public docs confirm "each trace and event detail will hold usage and billing information" but don't enumerate the response schema.

**How to avoid:** Wave 0 hard gate (Pattern 4) — print full response JSON before locking the parser. Plan must NOT lock REQ-DEP-03's parser until smoke output is in `wave-0-orq-deployments-smoke.md`.

**Warning signs:** `automation_runs.result.cost_cents = 0` post-flip is the first signal — same failure mode 999.4 left us in.

### Pitfall 2: Deployment-creation tool may not exist
**What goes wrong:** Plan assumes `mcp__orqai-mcp__create_deployment` is callable; in reality the Orq remote MCP is a docs-search MCP only.

**Why it happens:** ROADMAP entry references the tool by name [CITED: `.planning/ROADMAP.md:1172`]. Public Orq docs MCP is at `https://docs.orq.ai/mcp` and exposes documentation tools [CITED: search result "Orq.ai's MCP server provides AI agents with documentation search capabilities"]. The Orq Node SDK has no `deploymentsCreate` function [VERIFIED: directory listing of `web/node_modules/@orq-ai/node/src/funcs/`].

**How to avoid:** Treat Studio UI as the primary creation path. Plan a manual Studio creation step in the deploy gate. If `mcp__orqai-mcp__create_deployment` is available in this project's MCP config (private/local MCP separate from `docs.orq.ai/mcp`), use it — but do NOT depend on it for Wave 0.

**Warning signs:** `MCP tool not found` error during plan execution. Pivot to Studio UI immediately.

### Pitfall 3: Studio deployment key must EXACTLY match `agent_key`
**What goes wrong:** Operator names the Studio deployment `Stage 0 Safety Classifier` (Title Case) while `orq_agents.agent_key = 'stage-0-safety-classifier'` (kebab-case). Lookup fails at runtime.

**Why it happens:** Studio's deployment-key field is alphanumeric+kebab and is a free-text input.

**How to avoid:** Either (a) make `orq_agents.deployment_key` an explicit column (recommended) with a CHECK constraint matching `^[a-z0-9][a-z0-9-]+[a-z0-9]$`, OR (b) document the convention in CLAUDE.md so operators always use the kebab `agent_key`.

**Warning signs:** "deployment_key not found" 4xx from `/v2/deployments/invoke`.

### Pitfall 4: JSON schema strict-mode rejects array-shorthand nullable
**What goes wrong:** Pasting Stage 1's `output_schema` (e.g. `{"reasoning": {"type": ["string", "null"]}}`) into Studio's JSON-schema tool fails validation.

**Why it happens:** [CITED: CLAUDE.md learning `3970bad9-4c97-4ccf-a2f5-46475313ed1a` — "OpenAI strict mode én Orq.ai's json_schema tool validator afgewezen ('Invalid JSON detected')"]. `anyOf` is required.

**How to avoid:** Convert every nullable field to `{"anyOf": [{"type": "string"}, {"type": "null"}]}` before pasting. Verify the existing `orq_agents.output_schema` already follows this — `20260506_phase74_stage1_classifier_agent.sql:47-50` confirms Stage 1 does. Stage 0 schema needs a check.

**Warning signs:** Studio UI shows red "Invalid JSON detected" on save.

### Pitfall 5: Template variables vs. raw JSON body
**What goes wrong:** Studio's user-message template uses `{{email_id}}` but the request `inputs` carries `email_id`. Mismatch silently substitutes empty string.

**Why it happens:** Studio supports three engines (Text, Jinja, Mustache) [CITED: docs.orq.ai/docs/deployments]. Variable name must match exactly between `inputs` keys and `{{...}}` references.

**How to avoid:** Two options: (a) one variable `{{json_input}}` + `inputs: { json_input: JSON.stringify(rawInputs) }` — robust against typos; (b) explicit per-field variables — better Studio observability, but requires a contract test that asserts `inputs` keys ↔ template variables. Researcher recommends (a) for v1, (b) as a follow-up.

**Warning signs:** LLM ignores user inputs and produces nonsense — verdict on every email is "safe" with reason like "no input provided".

### Pitfall 6: 999.4 `system_prompt` column drift if not synced before Wave 3
**What goes wrong:** Operations forgets to update `orq_agents.system_prompt` after a Studio prompt edit. The Router-direct path (still live as fallback transport) sees stale prompt; the Deployments path (live transport) sees fresh prompt. Different prompts on different rollback branches.

**Why it happens:** Wave 2 keeps Router code as a rollback path; Wave 3 garbage-collects. Between Wave 2 and Wave 3, the Studio prompt is the SSOT for Deployments but `orq_agents.system_prompt` is the SSOT for Router. Two SSOTs.

**How to avoid:** Either (a) shorten the Wave 2 → Wave 3 window to one deploy cycle, OR (b) on Wave 2 flip, freeze the prompt — no Studio edits between flip and Wave 3. Plan must call this out explicitly.

**Warning signs:** Differences in agent behaviour after a flag-flip back to Router transport.

### Pitfall 7: Streaming default
**What goes wrong:** SDK type `InvokeDeploymentRequest.stream` defaults `undefined` (not `false`). If a future caller assumes `undefined === false` and the Orq server defaults to `true`, the response shape changes (SSE vs JSON).

**Why it happens:** [VERIFIED: `web/node_modules/@orq-ai/node/src/models/components/invokedeploymentrequest.ts:1089` — `stream?: boolean | undefined`].

**How to avoid:** Pass `stream: false` explicitly in the request body. (Or omit and trust Orq's default — but explicit is safer.)

**Warning signs:** `res.body` is a stream, `res.json()` throws "unexpected end of JSON input".

### Pitfall 8: Per-row transport flag without DB-side default
**What goes wrong:** New `transport` column added without `DEFAULT 'router'`. Existing rows have NULL. Dispatcher `switch (agent.transport)` falls into `default` — undefined behaviour.

**How to avoid:** Migration MUST `ADD COLUMN transport text NOT NULL DEFAULT 'router' CHECK (transport IN ('agents','router','deployments'))`. Backfill is automatic via the default.

**Warning signs:** Stage 2 / Stage 3 / handler agents start throwing `unknown transport` after migration ship.
