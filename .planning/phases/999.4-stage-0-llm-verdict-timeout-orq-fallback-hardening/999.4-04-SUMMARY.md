# Phase 999.4 Plan 04: Stale-row sweeper (D-09)

Recurring Inngest cron that drains `automation_runs` rows stuck in `status='pending'`
from `triggered_by LIKE 'stage-0/%'` for >15 min. Marks them `status='failed'` with
`result.llm_reason='inngest_cancelled_stale'`, preserves existing JSONB keys via
per-row read-modify-write, and emits the per-swarm Bulk Review realtime channel
once per distinct `swarm_type`. Defense in depth on top of Plan 02 (45s deadline +
fail-open coercion) and Plan 03 (Router-direct transport).

## What changed

### New: `web/lib/inngest/functions/automation-runs-sweeper.ts`

- Exports `stage0StaleSweeper` Inngest function.
- `id: "stage-0/stale-sweeper"`, `retries: 0` (idempotent — already-failed rows are
  filtered out by `status='pending'`; next tick is 10 min away).
- Cron: `TZ=Europe/Amsterdam */10 6-19 * * 1-5` — every 10 min, business hours, Mon–Fri.
  Cron string lives ONLY in the `createFunction` config object; the JSDoc above
  describes the cadence in words ("every 10 minutes, hours 06-19 Amsterdam time, Mon-Fri")
  per CLAUDE.md rule (the `*/10` slash-N would close a `/** */` block).
- Steps (all side effects inside `step.run` — Phase 65 replay-id learning):
  1. `compute-cutoff` → `now - 15 min` ISO string.
  2. `find-stale` → SELECT id, swarm_type, result FROM automation_runs WHERE
     status='pending' AND triggered_by LIKE 'stage-0/%' AND created_at < cutoff.
  3. Early-return `{ swept: 0, channels: [] }` if no rows.
  4. `compute-completed-at` → ISO timestamp.
  5. `mark-failed` → per-row UPDATE merging `{ ...row.result, llm_reason:
     'inngest_cancelled_stale' }`, setting `status='failed'`, `completed_at`.
     Per-row (not bulk) so JSONB keys (`email_id`, `regex_matched`, etc.) survive.
  6. `compute-channels` → distinct sorted `${swarm_type}-review` list.
  7. `emit-stale-{ch}` (one step per channel) → calls `emitAutomationRunStale(admin, ch)`.

### Modified (test scaffold fix): `web/lib/inngest/functions/__tests__/automation-runs-sweeper.test.ts`

The Wave 0 RED scaffold from Plan 01 had three blocking defects that prevented GREEN:

1. `vi.mock` factory referenced top-level `inngestSend` / `emitAutomationRunStaleMock`
   variables — vitest hoists `vi.mock` calls and threw `ReferenceError: Cannot access
   'inngestSend' before initialization`. Fixed by wrapping in `vi.hoisted({ ... })`.
2. The `selectChain` mock object had `.eq` / `.lt` / `.in` but was missing `.like`,
   so the sweeper's `.like("triggered_by", "stage-0/%")` threw `is not a function`.
   Added `like: vi.fn(() => selectChain)`.
3. T-D4 asserted `emitAutomationRunStaleMock.mock.calls[i][0]` was the channel string,
   but the production helper signature is `emitAutomationRunStale(admin, automation)` —
   the channel is at `[1]`. Fixed the assertion.

These are scaffold defects, not contract changes — the four behaviour assertions
(T-D1 cutoff filter, T-D2 JSONB merge, T-D3 replay-safe cutoff, T-D4 distinct emit)
are intact and pass GREEN.

## Verification SQL (post-first-tick)

Run after the first cron tick that follows deploy. Should return >0 rows initially
(historical stuck-row drain) and 0 rows on subsequent ticks once the inventory clears:

```sql
SELECT id, swarm_type, status,
       result->>'llm_reason' AS llm_reason,
       completed_at
FROM automation_runs
WHERE status = 'failed'
  AND result->>'llm_reason' = 'inngest_cancelled_stale'
  AND completed_at > <deploy_timestamp>
ORDER BY completed_at;
```

The deploy-time stuck-row inventory count and the post-first-tick swept count
were not measured pre-deploy (no production Supabase access during this plan
execution); the operator running the deploy should capture them in the phase
verification log.
