# Phase 999.4: Stage 0 LLM-verdict timeout / Orq queue-wait hardening — Research

**Researched:** 2026-05-07
**Domain:** Inngest durable-function timeout hardening + Orq.ai transport-layer migration
**Confidence:** HIGH (in-tree code paths verified by Read; Orq SDK shape verified from `node_modules/@orq-ai/node@4.7.7`; CLAUDE.md + locked RFCs consulted)

## Summary

Phase 999.4 ships two surgical changes that together remove a class of stuck `automation_runs.status='pending'` rows caused by the Orq Agents-product orchestration layer queuing a chat-completion for 6–17 minutes (vs. the 1.6 s the actual `chat-completion` span takes once it fires).

**Fix B** — wrap `llmInjectionVerdict()` in `web/lib/inngest/functions/stage-0-safety-worker.ts` and the `invokeOrqAgent("stage-1-category-classifier", …)` call in `classifier-screen-worker.ts` with a 45 s client deadline. On timeout, throw a typed `OrqClientTimeoutError`; Stage 0 coerces `verdict='safe'` and Stage 1 coerces `category_key='unknown', confidence='low'` (Stage 1's existing D-11 catch already handles this — only the deadline is new).

**Fix C** — migrate the two single-shot JSON classifiers off `POST /v2/agents/{key}/responses` (Agents-product, queued) onto `POST /v2/router/chat/completions` (LLM Router, direct). Same Bedrock EU Haiku 4.5 primary, same prompt + JSON schema, same registry row in `public.orq_agents` as the SSOT for model+schema. The Orq Node SDK already exposes this as `client.router.chat.completionsCreate(...)`.

**Plus** — add one Inngest cron sweeper (`TZ=Europe/Amsterdam */10 6-19 * * 1-5`) that marks stale `automation_runs.status='pending'` rows from `triggered_by='stage-0/safety-worker'` as `status='failed'` with `result.llm_reason='inngest_cancelled_stale'`.

**Primary recommendation:** Single-source the system prompt in Studio (the agent definition stays as-is, untouched). Single-source the JSON schema + model_config in `public.orq_agents` (already there). The new direct-path helper reads the registry, fetches the system prompt from `get_agent` once at module load (cached), and POSTs to `/v2/router/chat/completions` with the merged shape. This avoids both prompt duplication in TS and runtime per-call dependency on `get_agent`.

| Capability | Primary Tier | Secondary Tier | Rationale |

| SSOT for JSON schema + model_config | `public.orq_agents` registry | — | `output_schema`, `model_config.primary`, `model_config.fallbacks`, `temperature`, `max_tokens` are already columns on the registry rows for both target agents (verified in `20260506_phase74_stage1_classifier_agent.sql` lines 40–58). ## User Constraints (from CONTEXT.md)

### Locked Decisions

**Fix B — Client-side deadline + graceful coercion**
- **D-01:** Wrap `llmInjectionVerdict()` call inside `step.run("llm-verdict", ...)` in `web/lib/inngest/functions/stage-0-safety-worker.ts` with an explicit **45 s** deadline (matches CLAUDE.md "45s client timeout (Orq.ai intern retry = 31s)" pattern; Inngest per-step ceiling is ~90 s, so 45 s leaves headroom for the persist + emit steps).
- **D-02:** On timeout or thrown error from `llmInjectionVerdict()`, **coerce `verdict='safe'`** and forward to Stage 1 (mirrors Phase 74 D-11 fail-open philosophy). Persist the run with `result.llm_reason = "timeout: client_deadline_exceeded"` (or the underlying error message). Do NOT route to `topic='safety_review'` — timeouts are infrastructure failures, not suspected injections.
- **D-03:** Stage 1 (`web/lib/inngest/functions/classifier-screen-worker.ts`) **already** has a D-11 catch block coercing `category_key='unknown'`, `confidence='low'`. Add a **45 s** client deadline around the same Orq call so the catch block actually fires on hangs.
- **D-04:** Implement the deadline as `AbortController` + `setTimeout` (or `Promise.race`) inside `web/lib/stage-0/llm-verdict.ts` and the equivalent helper used by classifier-screen-worker. Throw a typed `OrqClientTimeoutError` so worker catch blocks can distinguish timeout from parse / schema errors.

**Fix C — Bypass the Orq Agents-product queue**
- **D-05:** Trace evidence proves the latency is queue-wait inside the Agents-product orchestration layer, NOT fallback chain activation. All 3 traces used primary Bedrock EU Haiku 4.5; `chat-completion` span = 1.6 s every time; remaining 6–17 minutes is pre-LLM queue.
- **D-06:** Migrate `stage-0-safety-classifier` and `stage-1-category-classifier` from the Orq Agents product (`invokeOrqAgent` → `POST /v2/agents/{key}/responses`) to a direct chat-completion / LLM Router invocation that targets the same Bedrock EU Haiku 4.5 model with the same system prompt + JSON schema.
- **D-07:** Add a sibling helper `invokeOrqModel` (or extend `invokeOrqAgent` with a `mode: "agent" | "model"` parameter) in `web/lib/automations/orq-agents/client.ts`. Source the system prompt + JSON schema from the existing agent definition so we maintain a single source of truth.
- **D-08:** Direct-path target = chat-completion endpoint resolving to Bedrock EU Haiku 4.5 (`aws/eu.anthropic.claude-haiku-4-5-20251001-v1:0`). Same fallback semantics in-process: catch chat-completion error → fall to Bedrock EU Sonnet 4.5 (`aws/eu.anthropic.claude-sonnet-4-5-20250929-v1:0`) → fail. Two hops max, EU-only.

**Cleanup of historical stuck rows**
- **D-09:** Recurring **Inngest cron sweeper** finds `automation_runs` with `status='pending'`, `triggered_by='stage-0/safety-worker'` (or downstream), `created_at < now() - interval '15 minutes'`, marks `status='failed'` with `result.llm_reason='inngest_cancelled_stale'`. Cadence: `TZ=Europe/Amsterdam */10 6-19 * * 1-5`.

### Claude's Discretion

- Exact deadline-implementation shape (`AbortController` vs `Promise.race`) — choose whichever cleanly threads through the existing `invokeOrqAgent` / `fetch` boundary. → **Research recommends `AbortController`** (see Pattern 1).
- Exact name of the new helper (`invokeOrqModel` vs adding a `mode` flag to `invokeOrqAgent`) — pick the lower-churn option after reading the call sites. → **Research recommends `invokeOrqModel` as a sibling helper** (see Pattern 2).
- Sweeper's mark-failed vs re-emit policy — start with mark-failed for visibility (operator can manually re-emit from Bulk Review); upgrade to auto-re-emit later if the failure mode is proven transient. → **Research recommends mark-failed** for v1.

### Pattern 1: AbortController-as-deadline (proven in-tree)

**What:** Use `AbortController` + `setTimeout` to enforce a client-side deadline at the `fetch` boundary; throw a typed `OrqClientTimeoutError` on `AbortError`.

**When to use:** Every direct-path Orq invocation (Stage 0 + Stage 1 LLM calls). The existing Agents-product helper (`client.ts:130-147`) already uses this for the per-agent `timeout_ms` from the registry — the new direct-path helper mirrors the pattern with a fixed 45 s.

**Example:**
```typescript
// Source: web/lib/automations/orq-agents/client.ts:130-147 (existing pattern, mirrored)

export class OrqClientTimeoutError extends Error {
  constructor(message: string) {
    super(message);
    this.name = "OrqClientTimeoutError";
  }
}

const ctrl = new AbortController();
const timer = setTimeout(() => ctrl.abort(), 45_000);
try {
  const res = await fetch("https://api.orq.ai/v2/router/chat/completions", {
    method: "POST",
    headers: {
      "content-type": "application/json",
      authorization: `Bearer ${ORQ_API_KEY}`,
    },
    body: JSON.stringify({ /* … */ }),
    signal: ctrl.signal,
  });
  // … parse JSON, validate with Zod …
} catch (err) {
  if ((err as { name?: string }).name === "AbortError") {
    throw new OrqClientTimeoutError(
      `Orq direct-path deadline exceeded after 45s`,
    );
  }
  throw err;
} finally {
  clearTimeout(timer);
}
```

### Pattern 2: Sibling helper `invokeOrqModel` (direct-path transport)

**What:** Add a new exported function in `web/lib/automations/orq-agents/client.ts` next to `invokeOrqAgent`. Same registry lookup, same `loadAgent()` cache (60 s TTL — already there), but POSTs to `/v2/router/chat/completions` with `model`, `messages`, `response_format`, `fallback_models` constructed from the registry row.

**When to use:** Stage 0 + Stage 1 LLM calls only. Other agents stay on `invokeOrqAgent`.

**SSOT plumbing:**
- `model_config.primary` → request `model` field
- `model_config.fallbacks` → request `fallback_models` array (Orq Router supports this — verified in SDK `routerChatCompletionsCreate` request schema, see `models/operations/createchatcompletion.ts`)
- `model_config.temperature`, `model_config.max_tokens` → request `temperature`, `max_tokens`
- `output_schema` (the JSONB column already populated for both agents) → request `response_format = { type: "json_schema", json_schema: { name: agent_key, strict: true, schema: <output_schema> } }`
- **System prompt** → fetched once at module load via `GET /v2/agents/{orqai_id}` and cached. Prompt iteration in Studio remains the single edit surface; cache TTL = 60 s (matches `REGISTRY_CACHE_TTL_MS`).

**Example (sketch):**
```typescript
// Source: NEW — web/lib/automations/orq-agents/client.ts (sibling to invokeOrqAgent)

const ROUTER_URL = "https://api.orq.ai/v2/router/chat/completions";
const DIRECT_PATH_DEADLINE_MS = 45_000;

export async function invokeOrqModel(
  agent_key: string,
  inputs: Record<string, unknown>,
): Promise<InvokeResult> {
  if (!ORQ_API_KEY) throw new Error("ORQ_API_KEY is not set");
  const agent = await loadAgent(agent_key);
  const systemPrompt = await loadSystemPromptForAgent(agent.orqai_id); // cached

  const ctrl = new AbortController();
  const timer = setTimeout(() => ctrl.abort(), DIRECT_PATH_DEADLINE_MS);
  try {
    const res = await fetch(ROUTER_URL, {
      method: "POST",
      headers: {
        "content-type": "application/json",
        authorization: `Bearer ${ORQ_API_KEY}`,
      },
      body: JSON.stringify({
        model: (agent.model_config as { primary: string }).primary,
        fallback_models:
          (agent.model_config as { fallbacks?: string[] }).fallbacks ?? [],
        temperature:
          (agent.model_config as { temperature?: number }).temperature ?? 0,
        max_tokens:
          (agent.model_config as { max_tokens?: number }).max_tokens ?? 600,
        messages: [
          { role: "system", content: systemPrompt },
          { role: "user", content: JSON.stringify(inputs, null, 2) },
        ],
        response_format: {
          type: "json_schema",
          json_schema: {
            name: agent.agent_key.replace(/-/g, "_"),
            strict: true,
            schema: agent.output_schema,
          },
        },
      }),
      signal: ctrl.signal,
    });
    if (!res.ok) {
      const text = await res.text().catch(() => "");
      throw new Error(
        `Orq router invoke ${agent_key} failed: HTTP ${res.status} ${text.slice(0, 200)}`,
      );
    }
    const json = (await res.json()) as {
      choices?: Array<{ message?: { content?: string } }>;
      usage?: { prompt_tokens?: number; completion_tokens?: number; total_tokens?: number };
    };
    const rawText = json.choices?.[0]?.message?.content ?? "";
    if (!rawText) throw new Error(`Orq router invoke ${agent_key}: empty content`);
    const stripped = rawText.trim().replace(/^```(?:json)?\s*/i, "").replace(/\s*```$/i, "");
    const raw = JSON.parse(stripped);
    return {
      raw,
      agent,
      usage: {
        prompt_tokens: json.usage?.prompt_tokens ?? 0,
        completion_tokens: json.usage?.completion_tokens ?? 0,
        total_tokens: json.usage?.total_tokens ?? 0,
      },
      billing: { total_cost: 0 },
      cost_cents: 0,
    };
  } catch (err) {
    if ((err as { name?: string }).name === "AbortError") {
      throw new OrqClientTimeoutError(
        `Orq router deadline exceeded for ${agent_key} after ${DIRECT_PATH_DEADLINE_MS}ms`,
      );
    }
    throw err;
  } finally {
    clearTimeout(timer);
  }
}
```

### Pattern 3: Worker-side coercion on `OrqClientTimeoutError`

**What:** In `stage-0-safety-worker.ts`, wrap the `step.run("llm-verdict", ...)` body in try/catch and, on `OrqClientTimeoutError`, return a synthetic `StageZeroVerdict` with `verdict: 'safe'`, `reason: 'timeout: client_deadline_exceeded'`, zeroed usage. The downstream `persist-verdict` step writes this through the SAME insert path — no parallel write path.

**Why same insert path:** CONTEXT.md §code_context "`automation_runs` insert path — single insert point in `step.run("persist-verdict", ...)`. Timeout branch shares the same insert with `verdict='safe'` + a distinguishing `result.llm_reason`."

**Example:**
```typescript
// Source: NEW — modification to web/lib/inngest/functions/stage-0-safety-worker.ts:108-111

const llmResult = await step.run("llm-verdict", async () => {
```

### Pattern 4: Inngest cron sweeper (4 in-tree precedents)

**What:** New Inngest function with `{ cron: "TZ=Europe/Amsterdam */10 6-19 * * 1-5" }`. Reads stale `automation_runs` rows, marks them failed, emits the realtime stale-channel event so Bulk Review refreshes.

**When to use:** D-09 cleanup. Modeled after the 4 existing crons in `web/lib/inngest/functions/`:
- `debtor-email-bridge.ts` — `*/2 6-19 * * 1-5`
- `browserless-keepalive.ts` — `*/2 6-19 * * 1-5`
- `debtor-email-icontroller-cleanup-dispatcher.ts` — `*/5 6-19 * * 1-5`
- `classifier-promotion-cron.ts` — `0 6 * * 1-5`

**JSDoc rule (CLAUDE.md):** Describe the cron in words, NEVER paste the literal `*/N` string into a `/** */` block — `*/N` closes the comment.

```typescript
// Source: NEW — web/lib/inngest/functions/stage-0-stale-sweeper.ts

/**
 * Phase 999.4 D-09 — sweep stale automation_runs.
 *
 * Cron cadence (described, NOT pasted as a string per CLAUDE.md JSDoc rule):
 * every 10 minutes, hours 06–19 Amsterdam, Mon–Fri.
 *
 * Marks automation_runs rows that have been status='pending' for >15 minutes
 * with triggered_by starting 'stage-0/' as status='failed', llm_reason
 * 'inngest_cancelled_stale', completed_at=now(). Refreshes the realtime
 * channel so Bulk Review's stale rows clear.
 */
export const stage0StaleSweeper = inngest.createFunction(
  { id: "stage-0/stale-sweeper", retries: 0 },
  { cron: "TZ=Europe/Amsterdam */10 6-19 * * 1-5" },
  async ({ step }) => {
    const admin = createAdminClient();
    const cutoff = await step.run("compute-cutoff", async () =>
      new Date(Date.now() - 15 * 60_000).toISOString(),
    );
    const stale = await step.run("find-stale", async () => {
      const { data, error } = await admin
        .from("automation_runs")
        .select("id, swarm_type, automation")
        .eq("status", "pending")
        .like("triggered_by", "stage-0/%")
        .lt("created_at", cutoff);
      if (error) throw new Error(`stale select failed: ${error.message}`);
      return data ?? [];
    });
    if (stale.length === 0) return { swept: 0 };
    await step.run("mark-failed", async () => {
      const ids = stale.map((r) => r.id);
      const { error } = await admin
        .from("automation_runs")
        .update({
          status: "failed",
          completed_at: new Date().toISOString(),
          // NOTE: `result` is JSONB; this PATCH replaces the whole column.
          // Read-modify-write per row is safer; planner: implement per-row loop.
        })
        .in("id", ids);
      if (error) throw new Error(`mark-failed failed: ${error.message}`);
    });
    // Per-swarm channel refresh
    const channels = await step.run("compute-channels", async () =>
      Array.from(new Set(stale.map((r) => `${r.swarm_type}-review`))),
    );
    for (const ch of channels) {
      await step.run(`emit-stale-${ch}`, async () =>
        emitAutomationRunStale(admin, ch),
      );
    }
    return { swept: stale.length };
  },
);
```

### Anti-Patterns to Avoid

- **`Promise.race` for the deadline:** leaks the underlying `fetch`. Use `AbortController`. (Existing pattern.)
- **Destructuring `inngest.send`:** strips `this`-binding → runtime `TypeError` (CLAUDE.md, Phase 65 commit `dae6276`). The sweeper does not emit Inngest events, so this risk is contained — but if any planner-added emit appears, use the inline-cast pattern.
- **Generating UUIDs/timestamps outside `step.run`:** Phase 65 replay-id learning. The sweeper's `cutoff` and any per-row timestamps are generated inside `step.run("compute-cutoff", ...)` per Pattern 4.
- **JSDoc with `*/N` cron string:** CLAUDE.md rule. Always describe in words or use `//` single-line. Pattern 4 sketch follows this.
- **Parallel `automation_runs` write paths:** the timeout branch must write through `step.run("persist-verdict", ...)` — same code, same shape, only `result.llm_reason` differs.
- **Pasting the system prompt into TS:** breaks the Studio-as-SSOT iteration workflow. Fetch from `get_agent` once at module load, cache for 60 s.
- **Coercing on parse/schema errors:** D-02 only applies to `OrqClientTimeoutError`. JSON.parse failures and Zod-validation failures are real bugs — let them throw and surface in the worker's `onFailure` path (or the Stage 1 D-11 catch block, which already handles parse errors).

## Common Pitfalls

### Pitfall 1: Conflating `OrqClientTimeoutError` with parse / schema errors
**What goes wrong:** A loose `catch (err) { return { verdict: 'safe' } }` swallows real bugs (e.g., the agent returns malformed JSON because the prompt was edited in Studio without updating the schema). Result: every email silently passes Stage 0 with `verdict='safe'`, regardless of whether it's an injection attempt.

**Why it happens:** the easy implementation is `try { … } catch { coerce }`. The hard implementation distinguishes timeout from other errors.

**How to avoid:** check `err.name === "OrqClientTimeoutError"` (which is set on the typed error class). Re-throw everything else.

**Warning signs:** `automation_runs.result.llm_reason` containing `"JSON.parse failed"` or `"safeParse"` strings — these should be hitting the existing failure path, not the timeout-coercion path.

### Pitfall 2: `safety_overridden=true` short-circuit must not be touched
**What goes wrong:** A naïve refactor of the worker's flow control wraps the override branch in the same try/catch, changing operator-driven re-emit semantics.

**Why it happens:** the override branch (lines 84-103) returns early before reaching `step.run("llm-verdict", ...)` — but a planner might "consolidate" branches.

**How to avoid:** the try/catch goes ONLY around the `step.run("llm-verdict", ...)` body. The early-return for `safety_overridden` stays exactly as it is.

**Warning signs:** any operator action from Bulk Review producing a Stage 0 timeout-coerced row instead of a clean Stage 1 forward.

### Pitfall 3: Bulk UPDATE replaces JSONB `result` column
**What goes wrong:** `update({ result: { llm_reason: 'inngest_cancelled_stale' } })` replaces the entire `result` column, losing the partial `email_id`, `regex_matched`, etc. that the worker may have written before being cancelled.

**Why it happens:** PostgreSQL JSONB UPDATE replaces by default; merging requires `result || jsonb_build_object(...)` or per-row read-modify-write.

**How to avoid:** per-row loop in `step.run`: SELECT `result`, merge, UPDATE. Or use a server-side SQL function with `result = result || jsonb_build_object('llm_reason', 'inngest_cancelled_stale')`.

**Warning signs:** swept rows lose their `email_id` / `source_mailbox` and become un-debuggable in Bulk Review.

### Pitfall 4: Inngest replay regenerates the `cutoff` timestamp
**What goes wrong:** `const cutoff = new Date(Date.now() - 15 * 60_000).toISOString()` outside `step.run` regenerates on every replay, picking up different rows each tick → idempotency broken.

**Why it happens:** Phase 65 learning. `Date.now()` is non-deterministic.

**How to avoid:** wrap in `step.run("compute-cutoff", async () => ...)` (shown in Pattern 4 sketch).

**Warning signs:** sweeper sweeps different row counts on adjacent retries of the same tick.

### Pitfall 5: Direct-path response shape ≠ Agents-path response shape
**What goes wrong:** Agents path (`/v2/agents/{key}/responses`) returns `{ output: [{ parts: [{ text }] }], usage: { input_tokens, output_tokens } }`. Router path (`/v2/router/chat/completions`) returns OpenAI-style `{ choices: [{ message: { content } }], usage: { prompt_tokens, completion_tokens } }`. A naïve copy-paste of `client.ts:154-187` parsing breaks.

**Why it happens:** two different Orq endpoints, two different response shapes.

**How to avoid:** new helper has its own response parser. The `InvokeResult` shape (the contract callers see) stays identical — only the parser between fetch.json() and that contract is different.

**Warning signs:** `result.usage.total_tokens === 0` on every direct-path call (parser missed the field).

### Pitfall 6: `stage-1-category-classifier` agent is `enabled=false` in some envs
**What goes wrong:** The seed migration `20260506_phase74_stage1_classifier_agent.sql:60` sets `enabled=false` until the Studio activation ritual runs. If a deploy ships the Fix C swap before the agent is enabled, `loadAgent("stage-1-category-classifier")` throws.

**Why it happens:** Phase 74 activation is gated; not all envs may be activated.

**How to avoid:** verify the registry has both agents `enabled=true` in the target env before deploy. Could also add a graceful fallback: if registry lookup fails, log + skip LLM 2nd-pass (treat regex result as final).

**Warning signs:** any new "agent_key not found or disabled" errors in the Inngest logs after deploy.

### Pitfall 7: System-prompt cache TTL too long for Studio iteration
**What goes wrong:** if the cached system prompt is held for hours, a Studio prompt edit takes hours to take effect — surprising for the operator who's iterating.

**How to avoid:** use the same 60 s TTL as `REGISTRY_CACHE_TTL_MS` in `client.ts:17`. Document this in the helper.

**Warning signs:** "I changed the prompt 30 minutes ago and the LLM is still using the old version."
