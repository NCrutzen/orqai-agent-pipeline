# Phase 999.4 Plan 02: Fix B (Deadline + Graceful Coercion) Summary

Stage 0 LLM-verdict path now ships with a 45s client-side deadline and graceful fail-open: timeouts coerce to `verdict='safe'` and forward to Stage 1, while parse/schema errors still throw. Built on the Wave 0 RED scaffolds from Plan 01 — T-B2, T-B3, T-B4, and the OrqClientTimeoutError class test all GREEN; Wave 2 (Plan 03) `invokeOrqModel` tests stay RED by design.

## What Shipped

### Task 1 — `OrqClientTimeoutError` + 45s deadline (commit `a845cc5`)

`web/lib/automations/orq-agents/client.ts`:
- Exported `class OrqClientTimeoutError extends Error` with `name="OrqClientTimeoutError"`.
- Added `CLIENT_DEADLINE_MS = 45_000` constant (replaces per-agent `timeout_ms` for the OUTER abort).
- Wrapped the existing `await fetch(...)` in `invokeOrqAgent` with an explicit `AbortController + setTimeout(45_000)` pattern.
- On `AbortError`, the catch translates into `OrqClientTimeoutError` with a message naming the agent_key + deadline.
- `clearTimeout(timer)` runs in the `finally` so the timer never leaks past the request.
- Response parse / `res.json()` / billing math live OUTSIDE the timeout try/catch — JSON parse failures keep their original `SyntaxError`-derived shape (Pitfall 1 mitigation).

### Task 2 — Stage 0 worker fail-open coercion (commit `4f339b1`)

`web/lib/inngest/functions/stage-0-safety-worker.ts`:
- Imported `OrqClientTimeoutError`.
- Wrapped the `step.run("llm-verdict", ...)` callback body in a try/catch.
- On `OrqClientTimeoutError` (checked via `instanceof` AND `err.name` for cross-realm safety), the callback returns a synthetic verdict:
  ```ts
  {
    verdict: "safe",
    reason: `timeout: client_deadline_exceeded — ${err.message}`,
    matched_span: null,
    usage: { prompt_tokens: 0, completion_tokens: 0, total_tokens: 0, cost_cents: 0 },
  }
  ```
- All other errors (parse, schema, non-abort transport) propagate via `throw err` — no silent coercion (T-999.4-02 mitigation).
- `safety_overridden=true` short-circuit branch (lines 84–103) untouched (T-B4 regression covers this).
- The existing `persist-verdict` step already writes `result.llm_reason` from `llmResult.reason`, so the audit query Plan 04 sweeper test expects (`result->>'llm_reason'`) finds the synthetic `"timeout: client_deadline_exceeded"` reason without further changes.

## Deviations from Plan

### Minor adjustment — discriminator strengthened to `instanceof || name`

- **Found during:** Task 2 implementation review.
- **Plan said:** Use `err.name === "OrqClientTimeoutError"` discriminator.
- **What I did:** Used `err instanceof OrqClientTimeoutError || err.name === "OrqClientTimeoutError"` (logical OR — either one matches → coerce).
- **Why:** `instanceof` is the canonical typed-error check; `err.name` string check is the cross-realm fallback (e.g., if a future caller throws a re-instantiated error across an Inngest step boundary that loses the prototype chain). Using both belt-and-suspenders means the import is also non-decorative for tooling that flags unused imports.
- **Impact:** Strictly a superset of the plan's contract — both T-B2 (which throws a real `OrqClientTimeoutError`) and a hypothetical re-thrown clone are caught. T-B3 (`new Error("...parse failed...")`) still rethrows because neither branch matches.
- **Files:** web/lib/inngest/functions/stage-0-safety-worker.ts
- **Commit:** 4f339b1

### Note — no `llm_reason` ADD needed

- **Plan said:** "If existing code writes `result.reason` only, ADD a line that copies `reason → llm_reason`".
- **Reality:** The persist-verdict step already writes `llm_reason: llmResult.reason` as a first-class key (line 156). No change required. The synthetic verdict's `reason` field flows through unchanged and lands in `result.llm_reason` automatically, satisfying Plan 04's sweeper audit query.
