## Deviations from Plan

### [Rule 3 â€” Blocking issue] System prompt sourcing

**Found during:** Task 1 implementation against the RED test contract.

**Issue:** Plan said "fetch system prompt from Studio via one `get_agent` REST call cached for 60s". Test T-C3 asserts `adminMocks.single` is called exactly once across two `invokeOrqModel` invocations and the mock chain is `fromâ†’selectâ†’eqâ†’single` (a single Supabase row fetch). A separate Studio `get_agent` HTTP call would be invisible to that mock and the cache assertion would still need a Supabase fetch â€” making the registry-row fetch the de-facto SSOT for the cache count.

**Fix:** Source `system_prompt` from the registry row directly (new column added by migration `20260507e`). Studio remains the editorial SSOT; Operations syncs the column via UPDATE on prompt change. The 60s cache bounds prompt-drift, identical to the plan's intent.

**Consequence:** No second HTTP call per invocation (latency win). Operations now owns a one-line UPDATE on Studio prompt change â€” see deploy gate below.

**Files modified:** `web/lib/automations/orq-agents/client.ts`, new migration `20260507e_orq_agents_system_prompt_phase994.sql`.

**Commit:** `603c861` (helper) + `5aa8b02` (migration).

### [Rule 2 â€” Critical functionality] agent_runs.result column

**Found during:** Task 2 â€” T-B5 asserts `row.result.category_key` and `row.result.confidence`.

**Issue:** Worker code wrote `tool_outputs.error` on D-11 catch but the test contract requires a structured `result` field. No `result` column existed on `public.agent_runs`.

**Fix:** Migration `20260507d` adds `result jsonb default '{}'`. Worker writes the gated category + confidence on both success + D-11 catch paths.

**Consequence:** Downstream consumers can dispatch on `result.category_key` uniformly without parsing the (confidence column) + (tool_outputs.gated_to) seam. Backwards compatible â€” pre-Phase-999.4 rows keep an empty `{}`.

**Commit:** `5aa8b02`.

### [Rule 3 â€” Blocking issue] Test mock alignment

**Found during:** Task 2 â€” three pre-existing tests started failing because they mocked `invokeOrqAgent` but the worker now imports `invokeOrqModel`.

**Fix:** Updated `web/lib/stage-0/__tests__/llm-verdict.test.ts` and `web/lib/inngest/functions/__tests__/classifier-screen-worker.test.ts` to expose both names and route the renamed import through the existing test mock instance. Single `.mockResolvedValue` call drives both transports.
