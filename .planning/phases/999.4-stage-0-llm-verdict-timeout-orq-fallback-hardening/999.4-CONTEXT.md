# Phase 999.4: Stage 0 LLM-verdict timeout / Orq queue-wait hardening — Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate stuck `automation_runs.status='pending'` rows caused by Stage 0 (and Stage 1 LLM 2nd-pass) Orq.ai calls exceeding Inngest per-step timeout. Two-pronged fix:

1. **Fix B (in-tree, deterministic):** add an explicit client-side deadline to the Orq invocation in stage-0-safety-worker AND classifier-screen-worker, with graceful coercion on timeout/error so the pipeline never strands a row.
2. **Fix C (root-cause, Orq orchestration layer):** migrate the two single-shot JSON classifiers (`stage-0-safety-classifier`, `stage-1-category-classifier`) off the Orq Agents product onto a direct chat-completion / LLM Router invocation path that bypasses the Agents-product queue.

Out of scope: any other classifier or downstream agent (Stage 2 entity, Stage 3 coordinator, handlers). Out of scope: changing the actual prompt, JSON schema, or model choice (Haiku 4.5 EU stays).

</domain>

<decisions>
## Implementation Decisions

### Fix B — Client-side deadline + graceful coercion
- **D-01:** Wrap `llmInjectionVerdict()` call inside `step.run("llm-verdict", ...)` in `web/lib/inngest/functions/stage-0-safety-worker.ts` with an explicit **45 s** deadline (matches CLAUDE.md "45s client timeout (Orq.ai intern retry = 31s)" pattern; Inngest per-step ceiling is ~90s, so 45s leaves headroom for the persist + emit steps).
- **D-02:** On timeout or thrown error from `llmInjectionVerdict()`, **coerce `verdict='safe'`** and forward to Stage 1 (mirrors Phase 74 D-11 fail-open philosophy). Persist the run with `result.llm_reason = "timeout: client_deadline_exceeded"` (or the underlying error message) so the failure mode is auditable in `automation_runs`. Do NOT route to `topic='safety_review'` — timeouts are infrastructure failures, not suspected injections.
- **D-03:** Stage 1 (`web/lib/inngest/functions/classifier-screen-worker.ts`) **already** has a D-11 catch block coercing `category_key='unknown'`, `confidence='low'`. Add a **45 s** client deadline around the same Orq call so the catch block actually fires on hangs (today the catch only fires on thrown errors — a hang silently exceeds the Inngest step ceiling).
- **D-04:** Implement the deadline as `AbortController` + `setTimeout` (or `Promise.race` with a timeout promise) inside `web/lib/stage-0/llm-verdict.ts` and the equivalent helper used by classifier-screen-worker. Throw a typed `OrqClientTimeoutError` so worker catch blocks can distinguish timeout from parse / schema errors in the `result.llm_reason` / `tool_outputs.error` audit field.

### Fix C — Bypass the Orq Agents-product queue
- **D-05:** Trace evidence (3 spans, 2026-05-07 morning) proves the latency is **NOT fallback chain activation**. All 3 traces used primary `eu.anthropic.claude-haiku-4-5-20251001-v1:0` and the actual `chat-completion` span took **1.6 s** in every case. The 6–17 minute trace duration is queue-wait inside the Orq Agents-product orchestration layer between trace start and `chat-completion` start. See Trace Evidence section below.
- **D-06:** Migrate `stage-0-safety-classifier` and `stage-1-category-classifier` from the Orq Agents product (`invokeOrqAgent`) to a direct chat-completion / LLM Router invocation that targets the same Bedrock EU Haiku 4.5 model with the same system prompt + JSON schema. Both classifiers are single-shot, JSON-only, no tools, no multi-turn — they don't need the Agents product. The new path bypasses whatever Orq queue is responsible for the multi-minute pre-LLM wait.
- **D-07:** Add a sibling helper `invokeOrqModel` (or extend `invokeOrqAgent` with a `mode: "agent" | "model"` parameter) in `web/lib/automations/orq-agents/client.ts`. Source the system prompt + JSON schema from the existing agent definition so we maintain a single source of truth. Stage 0 + Stage 1 LLM helpers switch over; all other agents stay on the Agents-product path.
- **D-08:** Deployment target for the direct path: chat-completion endpoint that resolves to Bedrock EU Haiku 4.5 (model id `aws/eu.anthropic.claude-haiku-4-5-20251001-v1:0`, confirmed via `list_models`). Same fallback semantics handled in-process: catch chat-completion error → fall to Bedrock EU Sonnet 4.5 (`aws/eu.anthropic.claude-sonnet-4-5-20250929-v1:0`) → fail. Two hops max, EU-only. (Note: the current Agents-product fallback chain has never actually fired in observed traces, so collapsing it is no regression.)

### Cleanup of historical stuck rows
- **D-09:** Add a recurring **Inngest cron sweeper** that finds `automation_runs` with `status='pending'`, `triggered_by='stage-0/safety-worker'` (or downstream), and `created_at < now() - interval '15 minutes'`, then either (a) marks them `status='failed'` with `result.llm_reason='inngest_cancelled_stale'` or (b) re-emits `stage-0/email.received` to retry. Cadence: `TZ=Europe/Amsterdam */10 6-19 * * 1-5` (every 10 min, business hours, Mon–Fri) — matches CLAUDE.md cron defaults. Self-healing for any future Orq queue-wait spike. Re-emit vs mark-failed: planner's call (probably configurable per stage).

### Claude's Discretion
- Exact deadline-implementation shape (`AbortController` vs `Promise.race`) — choose whichever cleanly threads through the existing `invokeOrqAgent` / `fetch` boundary.
- Exact name of the new helper (`invokeOrqModel` vs adding a `mode` flag to `invokeOrqAgent`) — pick the lower-churn option after reading the call sites.
- Sweeper's mark-failed vs re-emit policy — start with mark-failed for visibility (operator can manually re-emit from Bulk Review); upgrade to auto-re-emit later if the failure mode is proven transient.

</decisions>

<specifics>
## Specific Ideas

- The user explicitly challenged the ROADMAP's "fallback chain" hypothesis with three trace IDs and the hint "maybe changing servers". The traces confirmed the hypothesis was wrong — the latency is queue-wait, not fallback activation. Fix C is therefore **migrate off the Orq Agents-product queue**, not "tune the fallback list".
- Both classifiers (`stage-0-safety-classifier`, `stage-1-category-classifier`) are simple single-shot JSON-only calls. They do not benefit from Agents-product orchestration features (no tools, no multi-turn). This makes them safe candidates for the direct chat-completion path.
- Operator-driven `safety_overridden` short-circuit (Pitfall 5) must continue to work unchanged — it bypasses Stage 0 entirely so the timeout fix and Fix C migration don't apply to that branch.

</specifics>
