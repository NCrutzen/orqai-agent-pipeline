### Pattern 1: Pure Helper Module Wrapping Third-Party Parser

**What:** A thin `strip-quoted-history.ts` that imports `EmailReplyParser` from `email-reply-parser`, calls `parser.read(body).getVisibleText()`, and returns the result. No I/O, no async, no state.

**When to use:** Any input-shaping step in Stage 0 / Stage 1 that needs to feed the LLM a transformed view of upstream data. Pattern matches `regex-screen.ts` and `regex-patterns.ts` already in the codebase.

**Example:**

```typescript
// web/lib/stage-0/strip-quoted-history.ts
//
// Phase 999.7 — strips quoted reply history from email bodies before the
// Stage 0 safety LLM sees them. Pure function, no I/O.
//
// Library: email-reply-parser@2.3.5 (Crisp). Supports Outlook NL/EN,
// Gmail NL/EN, German, French, Spanish, Italian, Portuguese, Polish,
// Korean, Japanese, Chinese reply patterns out of the box. Verified
// patterns include `Op <date> schreef <name>:` and `Van: <name> <email>`
// (Outlook NL).
//
// Defensive: returns the original body if the parser throws OR returns
// an empty string. Stage 0 must NEVER silently see an empty body — that
// would bypass the safety classifier.
//
// Replay-safe: pure function over input string. Same input → same output.
// Wrap call site in step.run() for telemetry parity with sibling steps.

// @ts-expect-error — email-reply-parser ships its own .d.ts but uses
// CommonJS-style export interop; default-import is the documented form.
import EmailReplyParser from "email-reply-parser";

export interface StripResult {
  /** Body with quoted reply history removed. */
  stripped: string;
  /** True if the parser actually changed the body (i.e. quoted history was found). */
  changed: boolean;
  /** Length delta in characters (negative = shrunk). Useful for telemetry. */
  delta_chars: number;
  /** Set if the parser threw or returned empty; stripped === input in that case. */
  fallback_reason?: "parser_threw" | "parser_returned_empty";
}

export function stripQuotedHistory(body: string): StripResult {
  if (!body || body.length === 0) {
    return { stripped: body, changed: false, delta_chars: 0 };
  }

  try {
    const parser = new EmailReplyParser();
    const email = parser.read(body);
    const stripped = email.getVisibleText();

    // Defensive: if the parser greedy-matched and ate the whole body,
    // we'd silently feed the LLM an empty string. Refuse that: fall
    // back to the original body so safety classification still runs.
    if (!stripped || stripped.trim().length === 0) {
      return {
        stripped: body,
        changed: false,
        delta_chars: 0,
        fallback_reason: "parser_returned_empty",
      };
    }

    return {
      stripped,
      changed: stripped.length !== body.length,
      delta_chars: stripped.length - body.length,
    };
  } catch {
    // Parser library never throws on our inputs in practice (pure regex),
    // but if it does, fail open: pass through the original body. The
    // safety LLM still gets to look at it; only the budget benefit is
    // lost on this email.
    return {
      stripped: body,
      changed: false,
      delta_chars: 0,
      fallback_reason: "parser_threw",
    };
  }
}
```

### Anti-Patterns to Avoid

- **Stripping in the ingest route instead of the worker.** Loses the deterministic step.run() telemetry; harder to roll back if regression appears.
- **Mutating `email_pipeline.emails.body` to the stripped value.** Violates D-09 spirit (no data migration) and breaks downstream stages that expect the full body.
- **Putting the helper under `web/lib/automations/debtor-email/`.** Stage 0 is cross-swarm by design (sales-email Phase 78 will use the same worker). Default to `web/lib/stage-0/` per CONTEXT discretion guidance.
- **Throwing on empty stripped output.** Silently bypasses safety. Always fall back to the original body and tag a `fallback_reason` for observability.
- **Hand-rolling NL regex patterns.** The crisp-oss library already has them; duplicating is rework with worse coverage.

## Common Pitfalls

### Pitfall 1: Greedy strip eats the whole body

**What goes wrong:** On certain edge-case threads (e.g. forwarded emails with no plain-text leader before the first divider), the parser's first-divider match consumes everything and `getVisibleText()` returns an empty string. If we naïvely pass that to the LLM, the safety classifier sees an empty body and silently issues `verdict: 'safe'`.

**Why it happens:** `email-reply-parser` matches the first reply divider in the body and treats everything below it as quoted. If the divider is at line 0, visible text is empty.

**How to avoid:** Defensive fallback in the helper (shown in Pattern 1 code): if `stripped.trim().length === 0`, return the original body with `fallback_reason: 'parser_returned_empty'`. The safety LLM gets the full email; only the budget benefit is lost on that email.

**Warning signs:** Operator reports a long email that "got through Stage 0 with verdict=safe but feels suspicious" + telemetry shows `prompt_tokens` of original-body magnitude despite strip step running. → Inspect `decision_details.strip_fallback_reason`.

### Pitfall 2: Bumping `BUDGET_CEILING_TOKENS` without bumping the test fixture

**What goes wrong:** `web/lib/inngest/functions/__tests__/stage-0-safety-worker.test.ts` line 283 has a fixture with `prompt_tokens: 5000, total_tokens: 5200`. After the constant change, that fixture stops being a meaningful "near-ceiling but safe" assertion (it's now nowhere near 16000).

**Why it happens:** Test fixtures encode magic numbers when constants change.

**How to avoid:** Rebase the fixture to `total_tokens: 15800` (just under 16000) so the "happy path with high token use" assertion remains meaningful. Add a separate fixture at `total_tokens: 16500` to assert the breach path still fires at the new ceiling.

**Warning signs:** Tests still green after constant change + fixture-only edit signals the tests aren't actually exercising the new boundary.

### Pitfall 3: Strip step runs OUTSIDE step.run() and replays unsafely

**What goes wrong:** If the helper is called inline (`const stripped = stripQuotedHistory(body)`) outside `step.run()`, it re-executes on every Inngest replay. Because the function is deterministic, this is technically harmless — but it breaks the per-step telemetry pattern of the worker and makes `decision_details.strip_delta_chars` non-trivial to record.

**Why it happens:** "It's pure, why bother wrapping it" reasoning.

**How to avoid:** Always wrap in `step.run("strip-quoted-history", () => stripQuotedHistory(body))`. Cost: zero (Inngest memoizes step output). Benefit: clean telemetry hook + pattern parity.

**Warning signs:** N/A — fix in code review.

### Pitfall 4: Forwarding the stripped body to Stage 1

**What goes wrong:** If `forward-to-classifier` (line 222 of worker) is changed to send `body_text: classifierBody` (stripped) to Stage 1, the Stage 1 noise filter loses signal — auto-reply / OOO emails are detected by the full body's "I am out of office" / "automatisch antwoord" markers, which often appear in the part `email-reply-parser` would lop off.

**Why it happens:** "Strip once at the top" feels DRY.

**How to avoid:** Stage 0 worker MUST forward the **original** `body` to Stage 1, not `stripped.stripped`. Document in the comment block above the forward step. Stage 1 (Phase 74 LLM 2nd-pass) does its own input shaping if needed.

**Warning signs:** Stage 1 noise classification accuracy drops post-deploy on auto-reply emails.

### Pitfall 5: Strip telemetry not surfaced in Bulk Review

**What goes wrong:** When operators see a Stage 0 row in Bulk Review and want to know "what did the LLM actually see," they should be able to. If we don't persist `delta_chars` and `fallback_reason` into `pipeline_events.decision_details`, we lose the audit trail.

**How to avoid:** Extend the `decision_details` jsonb the worker writes (line 207-213 of `stage-0-safety-worker.ts`) with `strip_changed`, `strip_delta_chars`, `strip_fallback_reason`. Phase 70 dual-write to `pipeline_events` automatically picks these up.
