# Phase 82.7.1 Plan 01: Per-Stage Submit Override Summary

Added a per-stage Submit override lime-pill button to the dirty branch of every `<StageStep>` on Stages 0/2/3/4, wired through `PipelineFlow.onSubmitDirty` into a new `detail-pane.tsx` useCallback that dispatches the existing `KEYBOARD_EVENTS.overrideSubmit` window event — preserving the W6 single-write contract by reusing the same dispatch path the Stage 1 footer Submit already invokes.

## What Was Built

- **Task 1 — Per-stage Submit override (single commit `b2a71d7`):**
  - **`stage-step.tsx`** — added `onSubmitDirty?: () => void` to `StageStepProps`, destructured it, and rendered a NEW sibling block IMMEDIATELY BEFORE the existing per-stage cancel-override row in the `stage.state === "dirty"` branch:
    - Left label: `Submit Stage {stage.n} override` (amber/mono/12px — mirrors the cancel-override pending label color).
    - Right action: real `<button>` styled as the StageFeedbackPanel Confirm lime-pill (`var(--v7-lime)` fill + `var(--v7-bg)` text + `var(--v7-radius-pill)` border-radius + `inline-flex` with 6px gap + 6/14 padding + font-mono 600 weight).
    - Inner content: `<span data-row="primary">Submit override</span>` + trailing `<kbd>⌘⏎</kbd>` (mono, 10px, opacity 0.7).
    - `onClick={() => onSubmitDirty?.()}` (optional-chained per pattern from 82.7-03 cancel-override).
    - `aria-label={\`Submit override on Stage ${stage.n}\`}`, `data-testid={\`stage-${stage.n}-submit-override\`}`.
  - **`pipeline-flow.tsx`** — added `onSubmitDirty: (stageN: number) => void` to `PipelineFlowProps` immediately after `onCancelDirty`, added to the destructure, and passed `onSubmitDirty={() => onSubmitDirty(s.n)}` to `<StageStep>` as a sibling to the existing `onCancelDirty` thread (rendered in both the regular and future-expanded branches — same code path).
  - **`detail-pane.tsx`** — added a new `onSubmitDirty` `useCallback` directly after `onCancelDirty`. Body:
    ```typescript
    const onSubmitDirty = useCallback((_stageN: number) => {
      window.dispatchEvent(new Event(KEYBOARD_EVENTS.overrideSubmit));
    }, []);
    ```
    Threaded into the existing `<PipelineFlow>` JSX as `onSubmitDirty={onSubmitDirty}` (sibling to `onCancelDirty={onCancelDirty}`).

## What Was NOT Touched

- **Footer `handlePrimary` (detail-pane.tsx L487-497)** — byte-identical pre/post edit. Still scoped to `activeStage === 1` per Phase 82.6 D-02b.
- **Footer rendering at L693 (`{activeStage === 1 ? ...}`)** — untouched. The Stage 1 footer Submit/Cancel/Approve buttons render exactly as before for `activeStage === 1`; nothing on Stages 0/2/3/4 footer changes.
- **`KEYBOARD_EVENTS.overrideSubmit` listeners in `stage-1-widget.tsx` and stage-3/stage-4 `client-shell.tsx`** — untouched. They handle the dispatched event identically whether it originated from the keyboard, the footer, or the new per-stage button. Single-write contract preserved.
- **Hard-separation contract** — zero `swarm_intents` references, zero `swarm_noise_categories` references, zero new dispatch paths, zero new Inngest events, zero DB writes. The per-stage button is pure UI chrome on top of the existing 4-axis override model. Each Stage widget continues to own its own POST per the canonical RFC (`docs/agentic-pipeline/README.md`).
- **`onCancelDirty` per-stage block (stage-step.tsx L294-322 post-edit)** — preserved verbatim; the new Submit block is a sibling above it, not a replacement.
- **`StageFeedbackPanel`, `fireFeedback`, `fireOverrideFeedback`** — untouched. The new button does NOT call these; it only dispatches the window event the listening Stage widget already handles.
