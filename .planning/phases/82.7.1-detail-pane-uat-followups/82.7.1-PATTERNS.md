### D-04 / D-05 — 150ms opacity fade-out on pendingRemoval rows

#### `_shell/row-list.tsx` — add fade transition before unmount

**Analog:** the existing `pendingRemovalIds` filter at L40-43 (binary

`rows`" and "filtered out").

**Exact analog excerpt** (row-list.tsx L34-44, ALREADY IN FILE):

  // (graceful default until Plan 06 page-level prefetch lands).
  const { selectedId, setSelected, pendingRemovalIds } = useSelection();

  const visible =
    pendingRemovalIds.size === 0
      ? rows
      : rows.filter((r) => !pendingRemovalIds.has(r.id));

**Existing per-row inline-style block** (L76-99) — D-04 adds two style props
inline here, no structural change:

```typescript
<li
  key={r.id}
  onClick={() => setSelected(r.id)}
  ...
  style={{
    padding: isSelected ? ... : ...,
    borderBottom: "1px solid var(--v7-border)",
    borderLeft: isSelected ? ... : ...,
    background: isSelected ? "var(--v7-bg-2)" : "transparent",
    cursor: "pointer",
    display: "grid",
    gridTemplateColumns: ...,
    alignItems: "center",
    gap: "var(--space-3)",
    position: "relative",
  }}
>
```

**Pattern for D-04/D-05 fade-out:**

Stop filtering `pendingRemovalIds` rows out of `visible` immediately.
Instead, keep them rendered for ~150ms with `opacity: 0` + `transition`,
then drop them. Two viable shapes:

**Option A — CSS transition + delayed unmount (recommended):** keep the
row in `visible` even when it's pending; mark it with `data-pending="true"`;
set inline style `opacity: pendingRemovalIds.has(r.id) ? 0 : 1` and
`transition: opacity 150ms ease-out`. Then add a local `useEffect` in the
provider (selection-context.tsx) or in row-list that, ~180ms after a row
enters `pendingRemovalIds`, also adds it to a `_hidden` set that the filter
DOES respect. The CSS transition runs while it's visible-but-fading; the
hidden filter then unmounts. Avoids stale-DOM jank, preserves a11y reads.

**Option B — pure CSS, no JS timer:** trim immediately as today (keep
existing filter logic) but render with `<TransitionGroup>` or a tiny custom
`useExitTransition(150)` hook. Higher complexity; not justified by the
small UX win. Reject in favour of A.

**Concrete edit shape (Option A):**

```typescript
// row-list.tsx — render block change
<li
  key={r.id}
  onClick={() => setSelected(r.id)}
  data-pending={pendingRemovalIds.has(r.id) ? "true" : undefined}
  style={{
    ...,
    opacity: pendingRemovalIds.has(r.id) ? 0 : 1,
    transition: "opacity 150ms ease-out",
    pointerEvents: pendingRemovalIds.has(r.id) ? "none" : "auto",
  }}
>
```

And replace the immediate filter with a `_unmount` set populated 180ms
after pending entry (state in row-list local `useState<Set<string>>` +
`useEffect([pendingRemovalIds])` that schedules `setTimeout(180, () => add to _unmount)`).

**D-05 consistency:** since the fade lives in row-list's render of every
row in `pendingRemovalIds`, both Approve auto-advance (Plan 04 markPendingRemovalAndAdvance)
AND Submit-override post-commit removal flow through the same opacity path
automatically — no special-casing.

**Easing curve choice (Claude's discretion per CONTEXT):** use `ease-out` —
operator perceives the row as "settling out" rather than "linear vanish."
Matches V7 motion convention (no project token for animation curves; just
inline the literal).

## Shared Patterns

### V7 OKLCH token convention
**Source:** every detail-pane file analyzed.
**Apply to:** every new style block.

- Never hardcode hex; use `var(--v7-*)` tokens.
- Lime fill primary: `background: var(--v7-lime); color: var(--v7-bg)`.
- Ghost / outline secondary: `background: transparent; border: 1px solid var(--v7-brand-secondary); color: var(--v7-brand-secondary)`.
- Muted placeholder: `color: var(--v7-muted)`, `opacity: 0.4`–`0.7`.
- Amber for override-pending state: `var(--v7-amber)` + `font-mono`.
- Pill radius for buttons: `border-radius: var(--v7-radius-pill)`.
- Card radius for containers: `border-radius: var(--v7-radius-sm)`.
- Spacing scale: `var(--space-1)` … `var(--space-6)`.

### Radix Tooltip pattern
**Source:** predicted-row.tsx L181-198.
**Apply to:** D-06 brand-color swatch.

```typescript
import { Tooltip, TooltipContent, TooltipProvider, TooltipTrigger } from "@/components/ui/tooltip";

<TooltipProvider>
  <Tooltip>
    <TooltipTrigger asChild>{/* trigger element with role+aria-label */}</TooltipTrigger>
    <TooltipContent>{tooltipText}</TooltipContent>
  </Tooltip>
</TooltipProvider>
```

Default Radix delay is 700ms (`delayDuration` on `TooltipProvider`). Match the
existing `↻` tooltip behaviour — leave defaults unless operator UAT flags
discoverability later (CONTEXT Claude's Discretion).

### Optional-chained callback pattern
**Source:** stage-step.tsx L253 (`onCancelDirty?.()`).
**Apply to:** any new optional prop callback (`onSubmitDirty?: () => void`).

Mirror the `?: () => void` prop signature + optional-chained call site so
callers without the flow can omit safely (matches how `stage-step.tsx`
already structures `onCancelDirty`).

### Single-write contract (W6)
**Source:** detail-pane.tsx L487-497 comment.
**Apply to:** D-01/D-02 per-stage Submit button — MUST fire the existing
`KEYBOARD_EVENTS.overrideSubmit` window event, NOT a direct POST. Otherwise
two writes hit `email_feedback` for the same override. The 82.6 single-write
contract is explicit: "A footer-level POST here would double-write (W6
single-write contract)."

### Hard-separation contract
**Source:** detail-pane.tsx L3-9 + CLAUDE.md.
**Apply to:** every file in this phase. Zero changes to Stage 1 ↔
`swarm_noise_categories` or Stage 3 ↔ `swarm_intents` dispatch paths. This
phase is pure UI chrome on top of the existing 4-axis override model.

### No-toast / silent UX (Phase 82.6 D-03)
**Source:** detail-pane.tsx L530-534 + Phase 82.6 D-03 lock.
**Apply to:** D-04/D-05 fade-out, D-01/D-02 Submit. No toasts, no banners;
the fade-out IS the visual confirmation.
