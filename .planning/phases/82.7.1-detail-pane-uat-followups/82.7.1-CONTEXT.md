## Defect inventory

### E-01 — Stage 0 override has no submit path (BLOCKER for Stage 0 ops)

**Observed:** On the Stage 0 (Safety) tab, when the operator clicks `override stage`
on Stage 0, the row enters dirty mode and shows "Override in progress" + a
`cancel override` link in the detail pane — but **there is no footer Submit
override button visible anywhere**. The operator cannot complete a Stage 0
override from the Stage 0 tab.

**Expected:** From the Stage 0 view, an operator who flags an email as
`injection_suspected` (or corrects a `safe`/`injection_suspected` verdict) must
be able to submit that override.

**Root cause hypothesis:** Phase 82.6 D-02b scoped the footer to `activeStage === 1`
only. That decision made sense when only Stage 1 had a meaningful
"approve all stages" action — but with override now wired on all stage tabs, the
Stage 0 tab has an interaction dead-end.

**Resolution options (to discuss in /gsd-discuss-phase 82.7.1):**

- (a) **Show the footer on every stage tab when `anyDirty`**, scoped to the
  active stage's submission. Submit/Cancel apply to dirty entries on the active
  stage only.
- (b) **Per-stage Submit button inside the dirty card** (next to / replacing the
  `cancel override` link), no footer change. Keeps the footer Stage-1-exclusive
  per Phase 82.6 D-02b.
- (c) Hybrid: per-stage Submit inside the dirty card on Stage 0/2/3/4; keep the
  existing footer behavior on Stage 1.

**Recommended:** (b) — keeps the footer contract intact and matches the
per-stage cancel link added in 82.7-03.

### E-02 — Approve is silent during processing (no loading feedback)

**Observed:** On Stage 1, clicking Approve immediately advances to the next row.
Works correctly but feels uncertain — operator doesn't know whether the verdict
actually landed.

**Expected:** A visual cue during the `recordVerdict` + advance cycle:
- Approve button enters loading state (spinner / disabled) for the duration of
  the optimistic + server cycle, OR
- Brief transient skeleton on the outgoing row, OR
- Approved row fades out as it leaves the list (animated pendingRemoval).

**Constraint:** Phase 82.6 D-03 locked silent UX (no toasts). This constraint
still applies — the cue must be in-place visual feedback, not a notification.

**Note:** Optimistic removal already runs (`markPendingRemovalAndAdvance` in
82.7-04) — the row should already disappear from the list. Operator may not be
noticing it because it's instant. A 150ms fade-out on pendingRemoval is the
lightest fix.

### E-03 — Orange dot is undocumented to operators

**Observed:** Operator asked "what is the orange dot?" on the Stage 1 noise tab.

**Root cause:** The 6×6px brand-color swatch on each row
(`predicted-row.tsx:97-107`, `style={{ background: var(brandColorToken(row.entity_brand)) }}`)
is the entity-brand identifier (Fire Control → amber/orange, Smeba → other,
etc.). The affordance has no legend, tooltip, or column header. Operators with
no codebase context cannot decode it.

**Expected:** Either
- (a) Tooltip on hover: "Fire Control" / "Smeba" / etc.
- (b) Add a small brand label/legend in the filter strip.
- (c) Move the swatch into the recipient column with the inbox name (where it
  already exists in `PredictedRow`) and verify the noise-view row has the same.

**Recommended:** (a) — minimum-cost fix, no layout disturbance.

### E-04 — Override form button positioning needs polish

**Observed (image 10):** When override is active on Stage 1, the form shows:
`auto_reply ▾` dropdown | `Notes REQUIRED` textarea | `Eval type: REG | CAP`
radio cards | bottom row `Discard changes Esc` (left) and `Submit override ⌘⏎`
(right).

The bottom button row looks cramped: Submit override button has an orange/amber
warning fill that reads as alert/error, the Discard "button" is plain text with
an Esc chip next to it (visually weak), and the spacing under the radio cards
feels tight.

**Expected:** Polish the action row:
- Submit override button: use brand-secondary lime fill (matches the Confirm /
  Approve primary precedent established in 82.7-01 + 82.7-02) rather than
  amber. Amber reads as a warning state, not a commit action.
- Discard changes: bump to a real ghost button with the `Esc` keychip as
  trailing affordance.
- Add `var(--space-4)` top padding above the row.

**Out of scope:** the REG / CAP radio card visual design — leave as-is (operator
did not flag it).

### E-05 — Some rows render without a category/safety chip

**Observed (operator, 2026-05-16):** Not all rows in the Stage 1 noise tab (or
Stage 0 safety tab — to be clarified) show a left-edge category chip. At least
one row was visibly missing its label chip.

**Expected:** Every row in the list view has a category/safety chip on the left
edge — even if the value is `unknown` or pre-cutover historical (in which case
show a muted placeholder, not nothing).

**Investigation needed before planning:**
1. Confirm which tab/view the unlabelled row appeared on (Stage 0 Safety, Stage 1
   Noise, or both).
2. Identify the row(s) — get the email_id or sender so we can trace the
   pipeline_events / automation_runs / classifier rows.
3. Determine root cause:
   - (a) Renderer bug: chip not drawn when category is `null` or a specific value
   - (b) Data bug: row has no Stage-1 classification at all (Stage 1 didn't run
     or didn't write a verdict)
   - (c) Pre-cutover historical row that legitimately has no predictor attribution
     and the placeholder logic is missing

**Resolution path depends on root cause** — defer to /gsd-discuss-phase 82.7.1.
