# Phase 82.4 Plan 02: Feedback POST API Route Summary

POST `/api/automations/debtor-email/feedback` ships as an auth-gated, zod-validated, synchronous-INSERT endpoint backing the Stage feedback panel (Plan 03) and override wiring (Plan 04).

## What Shipped

**Route:** `web/app/api/automations/debtor-email/feedback/route.ts`

- Auth via `createClient()` from `@/lib/supabase/server` (cookie-based session)
- Validation via zod (`FeedbackPayload`): `email_id` uuid, `stage` ∈ {0,1,2,3}, `verdict` ∈ {confirm, override, unclear}, optional `corrected_value` (≤500), optional `prose_notes` (≤4000)
- Insert via `createAdminClient()` directly into `public.email_feedback` (one round-trip, RLS-bypassing service role)
- Returns `{ ok: true, id }` on 200, structured errors on 401/400/500

**Tests:** `web/app/api/automations/debtor-email/feedback/__tests__/route.test.ts`

Six cases, all green:
1. unauthenticated → 401
2. invalid JSON → 400
3. invalid payload (stage=4) → 400
4. invalid payload (verdict='bogus') → 400
5. happy path → 200 + insert called with server-stamped operator_id
6. T-82.4-02-01: client-supplied `operator_id: 'evil'` is ignored — insert call carries `auth.uid()`

## Threat Mitigations

| Threat ID | Status | How |
|-----------|--------|-----|
| T-82.4-02-01 (operator_id spoof) | mitigated | Zod schema omits the field; route spreads from `parsed.data` and adds `operator_id: user.id` last. Test #6 asserts smuggle attempt produces session uid in insert payload. |
| T-82.4-02-02 (DoS via large payload) | mitigated | `prose_notes.max(4000)`, `corrected_value.max(500)` at zod layer. |
| T-82.4-02-03 (info disclosure) | mitigated | Response is `{ ok, id }` only — never echoes prose back. |
| T-82.4-02-04 (repudiation) | accepted | `created_at` + `operator_id` on the row are sufficient audit; no separate log needed. |

## Decisions Made

- **Synchronous INSERT, no Inngest hop.** Feedback rows have no replay risk (pure capture) and the ≤200ms p95 latency target rules out an event hop. Override route (Plan 71-02) uses Inngest because it triggers downstream replay; feedback does not.
- **Zod omits `operator_id` entirely** (rather than stripping or validating it). This makes the smuggle impossible at the type-system level — `parsed.data` has no slot for it, so spreading into the insert body can never carry a client-supplied value.
- **Test mocks `@/lib/supabase/admin` at the `from().insert()` boundary**, returning a chainable `{ select: () => { single: ... } }` stub. This keeps assertions on the actual insert payload (operator_id stamping is the security-critical assertion).
