# Phase 82.4 Plan 04: Wire override surface → email_feedback rows Summary

**One-liner:** Existing "override stage" link clicks on Stage 0–3 sidebar entries now write an additive `email_feedback` row (verdict='override' + prose_notes) via a fire-and-forget `fireFeedback` helper, leaving the Phase 71/82 Inngest dispatch path behaviourally untouched.

## What changed

Wave 4 of the 82.4 feedback-capture build closes the loop on the override side of the system: every override click now leaves both an Inngest event (existing, untouched) AND a feedback row (new, additive). The two paths are intentionally decoupled — the feedback POST is fire-and-forget and never throws, so a feedback-row failure can never block the authoritative override dispatch.

### Task 1 — `fireFeedback` helper

New file `web/lib/automations/debtor-email/feedback/fire-feedback.ts` exports:

```typescript
fireFeedback(args: {
  email_id: string;
  stage: 0 | 1 | 2 | 3;
  verdict: "confirm" | "override" | "unclear";
  corrected_value?: string;
  prose_notes?: string;
}): Promise<void>
```

Invariants (verified by 4 vitest cases):

1. Happy 2xx → fetch shape correct, no `console.warn`.
2. Non-2xx response → `console.warn` called, function resolves to `undefined` (no throw).
3. Network rejection → `console.warn` called, function resolves to `undefined` (no throw).
4. Both `corrected_value` and `prose_notes` propagate verbatim to the stringified body.

The helper is a pure HTTP-side-effect wrapper — operator-id stamping, validation, and persistence all live in the API route from Plan 02.

Commit: `c3cb1a2`.

### Task 2 — Wire fireFeedback into stage-step.tsx

Both `onClick={onMarkDirty}` link sites in `stage-step.tsx` (the `state === "ok"` and `state === "skipped"` branches, lines 124/153 pre-change) now wrap a small inline arrow that:

1. Calls `onMarkDirty()` **first and unchanged** — the existing Phase 71/82 path runs exactly as before.
2. Then fires `fireFeedback({ email_id, stage, verdict: "override", prose_notes? })` if `stage.emailId` is set AND `stage.n !== 4`.

To capture the textarea contents at click-time, the prose state was lifted out of `StageFeedbackPanel` and into `stage-step.tsx`. `StageFeedbackPanel` now accepts optional `value` / `onValueChange` props with a fallback to its internal uncontrolled state, so all Plan 03 invariants and existing callers stay green.

A convenience export `fireOverrideFeedback(emailId, stage, corrected_value, prose_notes?)` is exposed for future Stage-widget submit handlers — when a widget actually picks a corrected_value (e.g. user picks `spam` in Stage 1), the widget can call this to write the follow-up row carrying the actual decision value. V9.0 reads the latest row per `(email_id, stage)` so the 2-row sequence (intent → corrected_value) is harmless.

Commit: `0782dfb`.
