### `supabase/migrations/2026MMDDx_email_feedback.sql` (migration, CRUD-DDL)

**Analog:** `supabase/migrations/20260513b_pipeline_health.sql`

This migration is the closest recent additive table that also locks RLS, adds composite indexes, and exposes RPCs — all things 82.4 needs (table, `(email_id, stage)` index, `(operator_id, created_at DESC)` index, RLS service-role-only).

**Header / rationale comment pattern (lines 1-31):**
```sql
-- Phase 82.2 — Stage 0 telemetry coverage fix (v8.0 stabilisation)
-- Plan 82.2-03 / TELE-COV-02
--
-- Creates the durable write target for the daily Stage 0 coverage probe
-- (D-07/D-08, implemented by Plan 82.2-10 in
-- web/lib/inngest/functions/stage-0-coverage-probe.ts) plus two RPC helpers
-- called by the probe and the historical backfill (Plan 82.2-09).
--
-- Trust boundaries (see 82.2-03 threat_model):
--   T-82.2-03-01 Information Disclosure: pipeline_health rows carry only
--     mailbox name + counts (no email content). RLS denies anon and
--     authenticated entirely; service_role only.
-- Idempotency: CREATE TABLE/INDEX IF NOT EXISTS + DROP/CREATE POLICY + REPLACE
-- on functions make this migration safe to re-run.
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

**Table + indexes + RLS pattern (lines 38-74):**
```sql
CREATE TABLE IF NOT EXISTS public.pipeline_health (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  probe_run_id   uuid NOT NULL,
  mailbox        text NOT NULL,
  swarm_type     text NOT NULL,
  stage1_count   bigint NOT NULL DEFAULT 0,
  ...
  recorded_at    timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS pipeline_health_recorded_at_idx
  ON public.pipeline_health (recorded_at DESC);

ALTER TABLE public.pipeline_health ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS pipeline_health_service_all ON public.pipeline_health;
CREATE POLICY pipeline_health_service_all ON public.pipeline_health
  FOR ALL TO service_role USING (true) WITH CHECK (true);

GRANT ALL ON public.pipeline_health TO service_role;
```

**Mimic:**
- `IF NOT EXISTS` everywhere → re-runnable migration.
- `uuid PRIMARY KEY DEFAULT gen_random_uuid()` + `created_at timestamptz NOT NULL DEFAULT now()`.
- RLS enabled, service_role-only policy. Anon and authenticated denied entirely.
- `DROP POLICY IF EXISTS … ; CREATE POLICY …` pattern for re-runnable RLS.
- Composite index naming convention `{table}_{cols}_idx`.
- CHECK constraints inline on the column (mimic `stage smallint NOT NULL CHECK (stage BETWEEN 0 AND 3)` and `verdict text NOT NULL CHECK (verdict IN ('confirm','override','unclear'))`).
- Header comment block: phase, plan, trust-boundary, idempotency.

**Diverge:**
- 82.4 must ALSO allow read access for the **authenticated** role (the feedback POST writes via service_role, but the loader for Option Z lists reads server-side via admin client; if any client-side read is wanted, decide RLS in discuss). Default to service-role only and load via `createAdminClient()` (same as `pipeline_health`).
- Do **not** copy `pipeline_health`'s GENERATED columns — `email_feedback` has no derived fields.

### `web/app/api/automations/debtor-email/feedback/route.ts` (controller, request-response)

**Analog:** `web/app/api/automations/debtor-email/override/route.ts`

This is the canonical 82.x POST route: auth-first via `createClient()` then zod payload validation then a side-effect. 82.4 follows the same shape with one important divergence — the side effect is a direct Supabase INSERT (no Inngest fanout needed; feedback writes are pure data capture with no downstream pipeline replay risk).

**Imports + dynamic config (lines 17-24):**
```typescript
import { NextRequest, NextResponse } from "next/server";
import { z } from "zod";
import { createClient } from "@/lib/supabase/server";
import { inngest } from "@/lib/inngest/client";

export const dynamic = "force-dynamic";
export const maxDuration = 30;
```

**Zod schema (lines 26-41):**
```typescript
const OverridePayload = z.object({
  axis: z.enum([...]),
  email_id: z.string().uuid(),
  original_event_id: z.string().uuid(),
  decision: z.string(),
  reason: z.string().max(1000).optional(),
  ...
});
```

**Auth + parse + dispatch (lines 45-85):**
```typescript
export async function POST(req: NextRequest): Promise<NextResponse> {
  // D-13: server-side auth — never trust client-supplied operator_id.
  const sb = await createClient();
  const { data: { user } } = await sb.auth.getUser();
  if (!user) {
    return NextResponse.json({ error: "unauthenticated" }, { status: 401 });
  }

  let body: unknown;
  try { body = await req.json(); }
  catch { return NextResponse.json({ error: "invalid json" }, { status: 400 }); }

  const parsed = OverridePayload.safeParse(body);
  if (!parsed.success) {
    return NextResponse.json({ error: parsed.error.message }, { status: 400 });
  }

  try {
    await (inngest.send as unknown as SendFn)({
      name: "debtor-email/override.submitted",
      data: { ...parsed.data, operator_id: user.id },
    });
  } catch (err) {
    return NextResponse.json({ error: `…` }, { status: 500 });
  }
  return NextResponse.json({ ok: true });
}
```

**Mimic:**
- `createClient()` from `@/lib/supabase/server` for operator auth.
- `auth.getUser()` then 401 if missing. The user.id is server-stamped as `operator_id`; the zod schema must NOT include an `operator_id` field (operator-supplied values are silently impossible).
- Zod `.safeParse` → 400 with `error.message` on failure.
- `reason`/`prose_notes` capped via `.max(1000)`.
- Closed enum on `verdict` (`z.enum(["confirm", "override", "unclear"])`) and `stage` (`z.union([z.literal(0), z.literal(1), z.literal(2), z.literal(3)])` or `z.number().int().min(0).max(3)`).
- `export const dynamic = "force-dynamic"; export const maxDuration = 30;`.

**Diverge:**
- No Inngest fanout. Instead, use `createAdminClient()` (from `@/lib/supabase/admin`) inside the route and `await admin.from("email_feedback").insert({...})`. This is acceptable here because the write is single-row, idempotency is per-`created_at` (multiple revisions allowed by design), and the ≤200ms p95 latency target rules out an Inngest hop.
- Do NOT use `inngest.send` here. The override route uses Inngest because it triggers pipeline replay; feedback writes don't.

### `web/app/(dashboard)/automations/[swarm]/_shell/_lib/feedback-list-loader.ts` (loader, CRUD-read)

**Analog:** `web/app/(dashboard)/automations/[swarm]/_lib/kanban-loader.ts`

This is the closest existing loader for a paginated, server-side list view shaped around `automation_runs` + `email_pipeline.emails`. Option Z needs a per-stage list that joins `pipeline_events` (Stage N rows for the chosen swarm) with `email_pipeline.emails` (sender/subject/received_at) and LEFT JOINs `email_feedback` (own-history filter).

**Loader signature + ordering + limit cap pattern (lines 69-100):**
```typescript
export async function loadKanbanRows(
  admin: SupabaseClient,
  swarmType: string,
): Promise<KanbanRow[]> {
  const { data, error } = await admin
    .from("automation_runs")
    .select("id, swarm_type, topic, entity, created_at, result")
    .eq("swarm_type", swarmType)
    .eq("status", "pending")
    .not("result->>kanban_reason", "is", null)
    .order("created_at", { ascending: false })
    .limit(500);
  if (error) throw new Error(`loadKanbanRows: ${error.message}`);
  const rows = (data ?? []) as RawRow[];
  if (rows.length === 0) return [];

  const emailIds = Array.from(
    new Set(rows.map((r) => r.result?.email_id).filter(Boolean) as string[]),
  );
  ...
}
```

**Mimic:**
- Server-only loader function taking `(admin: SupabaseClient, swarmType: string, params: {...})`.
- Single `.order(..., { ascending: false }).limit(N)` query.
- Throw on `error` with prefixed message.
- Empty-set early return.
- For the email-metadata join, fetch in a second query keyed on the unique `email_id` set, then merge into a Map. This is the established 71/82 pattern and avoids deep PostgREST joins.

**Diverge:**
- Pagination: kanban-loader uses a hardcoded `limit(500)`. 82.4 needs proper pagination. Use **cursor-based** on `received_at` (CONTEXT open_questions notes this as preferred for live data): accept `before?: string` (ISO timestamp) in `params`, add `.lt("received_at", before)` when present, then `.order("received_at", { ascending: false }).limit(50)`. Return `{ rows, nextBefore }` where `nextBefore` is `rows[rows.length-1].received_at`.
- Filter chips: when "needs action" is on, add `.in("stage_state", [...needs-action variants...])`. When "show my feedback only" is on, INNER JOIN `email_feedback` on `(email_id, stage) AND operator_id = $current`.
- Sort secondary key per CONTEXT: needs-action-at-this-stage first, then auto-handled, then own-already-reviewed. Implement as a `CASE`-derived `sort_bucket` in the SELECT (or three separate UNION queries — decide during planning).

## Shared Patterns

### Authentication on POST routes
**Source:** `web/app/api/automations/debtor-email/override/route.ts` lines 45-53
**Apply to:** Every new POST route in 82.4.
```typescript
const sb = await createClient();
const { data: { user } } = await sb.auth.getUser();
if (!user) return NextResponse.json({ error: "unauthenticated" }, { status: 401 });
```
The `operator_id` must be `user.id`, server-stamped. Never trust client payload to provide it. The zod schema must NOT contain an `operator_id` field.

### V7 OKLCH inline-style tokens for audit-surface UI
**Source:** `web/components/automations/bulk-review/audit/StageDetailExpander.tsx` lines 41-83
**Apply to:** `StageFeedbackPanel`, `needs-action-chip.tsx`, any new client component in the bulk-review audit surface.
- Inline `style={{ background: "var(--v7-panel-2)", color: "var(--v7-text)", ... }}` — NOT Tailwind colour classes.
- 11px uppercase 600 section labels with `letterSpacing: "0.04em"`.
- 12-13px body, mono `var(--font-mono)` for counters/timestamps.
- Pill shape via `borderRadius: "var(--v7-radius-pill)"`.
- Focus ring via `outlineColor: "var(--v7-brand-secondary)"`.

### Supabase admin client for server-side writes
**Source:** `web/lib/inngest/functions/stage-0-coverage-probe.ts` lines 30, 74-108
**Apply to:** Feedback API route insert, snapshot cron read+upload.
```typescript
import { createAdminClient } from "@/lib/supabase/admin";
const admin = createAdminClient();
const { error } = await admin.from("email_feedback").insert(rows);
if (error) throw new Error(`email_feedback insert failed: ${error.message}`);
```

### Inngest replay-safety
**Source:** `web/lib/inngest/functions/stage-0-coverage-probe.ts` lines 69-72 + CLAUDE.md Phase 65 lock
**Apply to:** Snapshot cron.
Generate any non-deterministic id (UUIDs, `Date.now()` used as DB key, random nonces) INSIDE `step.run` — never outside. The CLAUDE.md note: "Buiten step.run regenereert Inngest de waarde op iedere replay → INSERT op key-A, UPDATE op key-B, `.eq()` matcht 0 rijen".

### Migration idempotency
**Source:** `supabase/migrations/20260513b_pipeline_health.sql` lines 38-74
**Apply to:** `email_feedback` migration.
- `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`.
- `DROP POLICY IF EXISTS … ; CREATE POLICY …`.
- `CREATE OR REPLACE FUNCTION` for any RPC.
- `GRANT … TO service_role` after policy.
