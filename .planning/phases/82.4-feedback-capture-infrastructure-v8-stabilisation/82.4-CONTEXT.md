<rationale>
## Why this is v8.0 scope, not V9.0

The debtor-person starts giving feedback on 2026-05-18. V9.0's full synthesis layer (T2 draft-proposer with held-out clusterer eval) is multi-week work. If V9.0 captures and synthesises together, week-1 feedback is lost.

Capturing prose without synthesising for 2–4 weeks gives V9.0:
1. Real captured data to design the clusterer against, not speculation
2. A clean handoff: V9.0 = "build synthesis on top of existing capture stream"
3. Operator habit formation — by the time V9.0 ships, prose feedback is part of the workflow

## Why Option Z (email-everywhere) and not stage-of-stuckness

Confirmed by operator: the debtor-person is new and we want him to **train by reviewing green rows too**, not only orange (needs-action). Stuckness-only would hide auto-handled emails entirely. Z surfaces them in every stage's list, sorted but reviewable. Future state: V9.0+ adds calibration sampling that surfaces ~10% of green rows for forced re-review to keep operator accuracy benchmarked.

</rationale>

<scope>
## In scope

### Data layer

- **`email_feedback` table** keyed by `(email_id, stage, operator_id, created_at)`:
  - `stage smallint NOT NULL CHECK (stage BETWEEN 0 AND 3)` — Stage 4 out of scope per operator decision
  - `verdict text NOT NULL CHECK (verdict IN ('confirm', 'override', 'unclear'))`
  - `corrected_value text NULL` — the new verdict value the operator picked via the existing override dropdown (mirrors what `pipeline_events.overridden` would record)
  - `prose_notes text NULL` — free-text, this is the 82.4 delta
  - `operator_id text NOT NULL`
  - `created_at timestamptz NOT NULL default now()`
  - Index on `(email_id, stage)` for sidebar fetch; index on `(operator_id, created_at desc)` for history view
- **Multiple feedback rows per (email_id, stage) allowed** — operator can revise their feedback as they learn. V9.0 reads the latest row per (email_id, stage) by default but has access to the full history.

### Sidebar additions (rides on 82.3's expander)

- **Prose-notes textarea** beneath each Stage 0–3 expander. Optional. Saves on blur OR explicit Save button (pick during discuss).
- **Operator action wiring** — when operator changes the existing override dropdown / clicks `override stage`, the action ALSO writes an `email_feedback` row with `verdict='override'`, the new `corrected_value`, and any prose typed at that moment.
- **"Confirm correct" affordance** — currently the sidebar's `Looks correct` is read-only display. 82.4 wires a click on it (or a thumbs-up chip next to it) to write `email_feedback` with `verdict='confirm'`. This is the audit signal Option Z exists to capture.

### Stage-tab list re-scope (Option Z)

- **Each Stage 0–3 tab lists every email that has a verdict at that stage.**
  - Stage 0 tab: all incoming emails (because every email triggers Stage 0 once 82.2 lands)
  - Stage 1 tab: all emails that reached Stage 1 (≈ all of them)
  - Stage 2 tab: ~25% that pass the noise filter
  - Stage 3 tab: same ~25%
- **Same email appears in 3–4 stage tabs.** Clicking opens the same detail-pane sidebar (showing the full pipeline trace). Tab focus is NOT used to emphasise a specific stage entry in the sidebar — all stages render with equal visual weight (Option 2B locked).
- **Sort:** needs-action-at-this-stage at top (orange-circle rows), then auto-handled rows below, then own-already-reviewed rows further down.
- **Pagination required** — Stage 0 will easily be 100+ rows/day. Default page size TBD during discuss (50?).

### Filter chip per tab

- **"Needs action" chip** on every Stage 0–3 tab. Default OFF on all tabs (audit-first culture; operator opts in to inbox-only view).
- Chip state persisted in URL params so refresh preserves it.

### Own-feedback-history view

- Read-only view filterable on the stage-tab lists: "Show my feedback only" toggle filters to rows where `email_feedback.operator_id = current operator`. Helps the debtor-person see what he's already reviewed.

### Read-only export

- Nightly Supabase snapshot of `email_feedback` to a JSON dump in Supabase Storage (or a versioned table). Gives V9.0 a frozen corpus to iterate the clusterer against without churn.

</scope>

<open_questions_for_discuss>
- Save UX: on-blur autosave vs. explicit Save button per stage entry? Autosave is friendlier but riskier if prose still being typed; explicit button is the safer default.
- Confirm-correct chip placement: replace the existing `Looks correct` status line with an interactive variant, OR add a separate `✓ Confirm` button alongside? UX consistency vs. clear affordance.
- `email_feedback` history depth: keep all revisions, or `ON CONFLICT (email_id, stage, operator_id) DO UPDATE`? The schema above implies "keep all" via `created_at` as part of the natural key — confirm.
- Stage-tab list ordering: needs-action at top is clear, but within auto-handled rows do we sort newest-first or by `pipeline_events.created_at`? Likely the same thing, but worth being explicit.
- Pagination: cursor-based on `email_pipeline.emails.received_at`, OR offset-based with page numbers? Cursor is friendlier for live data, offset is simpler.
- Should the confirm-correct action also auto-collapse the 82.3 expander to keep the operator moving? Tiny UX call.

</open_questions_for_discuss>
