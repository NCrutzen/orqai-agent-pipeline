# Phase 82.4: Feedback capture infrastructure (v8.0 stabilisation)

**Gathered:** 2026-05-12
**Status:** Stub — needs `/gsd-discuss-phase 82.4` before planning
**Type:** INSERTED stabilisation phase — debtor-person prerequisite, V9.0 dependency

<domain>
## Phase Boundary

Ship the **capture half** of V9.0 inside v8.0 so the debtor-person has somewhere to put feedback from day 1 (week of 2026-05-18). Synthesis (LLM clustering, draft proposing, immediate-apply on new intents) stays in V9.0.

Scope is the storage + form + history view. Nothing AI-driven, nothing that touches the live pipeline. Pure data capture.

</domain>

<rationale>
## Why this is v8.0 scope, not V9.0

The debtor-person starts giving feedback on 2026-05-18. V9.0's full synthesis layer (T2 draft-proposer with held-out clusterer eval) is multi-week work. If V9.0 captures and synthesises together, week-1 feedback is lost to whatever ad-hoc channel they use (Slack / Excel / verbal hand-off).

Capturing without synthesising for 2–4 weeks gives V9.0:
1. Real captured data to design the clusterer against, not speculation
2. A clean handoff: V9.0 = "build synthesis on top of existing capture stream"
3. Operator habit formation — by the time V9.0 ships, prose feedback is part of the workflow

</rationale>

<scope>
## In scope

- **`email_feedback` table** keyed by `(email_id, stage, operator_id)`:
  - `stage smallint` (2 or 3 only — Stage 0/1 use existing 4-axis overrides)
  - `verdict text` (correct / incorrect / unclear)
  - `corrected_value text NULL` (Stage 2: NXT customer ID hint; Stage 3: intent label, may be new)
  - `prose_notes text NULL` (free-text explanation)
  - `operator_id text NOT NULL`
  - `created_at timestamptz default now()`
- **Stage 2 form on Bulk Review per-row**: "incorrect mapping?" toggle + corrected NXT customer ID field + prose notes
- **Stage 3 form on Bulk Review per-row**: confirm-or-correct intent dropdown (existing intents from `swarm_intents` registry) + "type new intent" inline option + prose notes (optional)
- **Fuzzy-match guardrail on new-intent typing**: when operator types a label, fuzzy-match against existing `swarm_intents` and offer "did you mean X?" before accepting
- **Own-feedback-history view**: filter Bulk Review by `WHERE email_feedback.operator_id = ?` so operator can see what they've already reviewed
- **Read-only export**: nightly Supabase snapshot of `email_feedback` to a JSON dump (so V9.0 can iterate against a frozen corpus without churn)

</scope>

<out_of_scope>
- **No synthesis** — no LLM clustering, no `proposed_actions` table, no Learning Inbox tab. All V9.0.
- **No immediate-apply on new intents** — typed intents are captured but NOT added to the Stage 3 LLM's intent list. V9.0 turns this on with the fuzzy-match + dedup logic in place.
- **No promotion candidates** — `promotion_candidates` table is V9.0.
- **No `proposed_actions` reviewer UI** — V9.0.
- **No data-driven resolver-step refactor** — V9.0 (`resolveDebtor` stays hardcoded 4-layer for now).
- **No sales-email operator** — V10.0 (multi-operator handling). 82.4 ships single-operator-only.

</out_of_scope>

<must_haves>
## Acceptance criteria

- Debtor-person can open any Bulk Review row, see Stage 2 + Stage 3 verdicts, click an "incorrect" toggle, type a corrected value, type prose notes, click save → row persisted in `email_feedback`.
- Debtor-person can see their own feedback history filtered to their `operator_id`.
- Form latency ≤ 200ms p95 for save (no LLM calls in the hot path).
- New-intent typing triggers a fuzzy match against existing intents and offers a "did you mean X?" suggestion before accepting.
- ≥ 95% of Stage 3 rows shown to the operator are saveable (no blocking errors due to missing email metadata / null pipeline_events rows).
- Existing 4-axis override surface for Stage 0/1 (Phase 71/82) untouched.

</must_haves>

</volume_assumptions>
