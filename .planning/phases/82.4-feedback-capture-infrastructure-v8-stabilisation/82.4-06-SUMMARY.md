# Phase 82.4 Plan 06: Wire Option Z chips + loadStageFeedbackList into stages 0/1/2/3 Summary

Two URL-param-driven toggle chips ("Needs action", "My feedback only", both default OFF) plus a per-stage Option Z audit list are now wired into Stage 0/1/2/3 pages. Stage 4 untouched. All filtering flows through `loadStageFeedbackList` (Plan 05); pagination is server-side cursor-based on `pipeline_events.created_at`.

## What landed

- `_shell/needs-action-chip.tsx` — two pure presentation chips (`NeedsActionChip`, `MineOnlyChip`) with V7 OKLCH inline-style tokens, `role="switch"` + `aria-checked`.
- `_shell/stage-list-chips.tsx` — client wrapper that owns the URL-param plumbing: `useSearchParams` + `router.replace`; toggling clears `?before` to keep cursor pagination consistent with the active filter set. Uses `replace` (not `push`) so back-button doesn't accumulate filter toggles.
- `_shell/__tests__/needs-action-chip.test.tsx` — 6 RTL tests (aria-checked, testid, click handler, role=switch, source-grep for V7 tokens). All pass.
- `stage-0/page.tsx`, `stage-2/page.tsx` — full Option Z mount (chips + RowList sourced from loader + Load more). These stages were placeholder surfaces; Option Z becomes their primary row list.
- `stage-3/page.tsx` — additive "All Stage 3 verdicts" section above the live Kanban shell.
- `stage-1/page.tsx` — additive "All Stage 1 verdicts" section above the existing chip strip + predicted-row shell. Hidden when `?sub=pending` (candidate-rule workflow branch).

## Hard-separation contract (RFC docs/agentic-pipeline/README.md)

`loadStageFeedbackList` accepts an explicit `stage` parameter; each page passes its own value (0/1/2/3) so the loader's `pipeline_events.stage = N` filter prevents any blur between Stage 1 (`swarm_noise_categories`) and Stage 3 (`swarm_intents`). The Option Z row badges echo the per-stage `pipeline_events.decision` value with stage-appropriate variant labels (`safety` / `noise` / `placeholder` / `intent`).
