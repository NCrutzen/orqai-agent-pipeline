### Destructive Actions

| Action | Confirmation Approach |
|--------|----------------------|
| Reject changes | No confirmation dialog. Reject is a single-click action. The optional comment field provides the friction point. Rejected changes skip the proposed prompt update; they do not delete anything. The pipeline continues with the original prompt. |

Rationale: Rejecting is non-destructive -- it preserves the current state. Approving is the action that mutates state. Neither requires a confirmation dialog because both are easily reversible in the next iteration cycle.

## Layout Patterns

### Approval Panel (inline within StepLogPanel)

When a pipeline step has status "waiting", the StepLogPanel expands and renders the ApprovalPanel in place of the log content area.

```
[StepLogPanel -- auto-expanded when status is "waiting"]
  [Timeline dot: amber-500 (filled), animate-pulse]
  [Step display name + "Waiting for Approval" badge (animate-pulse)]

  [ApprovalPanel -- replaces log/error content]
    [Explanation Card (bg-muted/50 rounded-lg p-4)]
      [Heading: "What changed and why" (text-sm font-semibold)]
      [Explanation text (text-sm, body weight)]

    [Separator (my-4)]

    [Split/Unified Toggle (top-right of diff area, size sm, variant outline)]
    [ApprovalDiffViewer]
      [Left: "Current Prompt" | Right: "Proposed Changes"]
      [Word-level diff with green/red highlighting]
      [Line numbers in gutter]
      [Monospace font (Geist Mono)]
      [Max height: 400px with overflow-y scroll]

    [Separator (my-4)]

    [Comment field]
      ["Comment" label + "Optional" hint (right-aligned)]
      [Textarea (3 rows, resize-y, placeholder text)]

    [Action bar (flex justify-end gap-3 pt-4)]
      [Reject button (variant outline, text-destructive): XCircle + "Reject Changes"]
      [Approve button (variant default): CheckCircle + "Approve Changes"]
```

Max width: The approval panel fills the available width of the StepLogPanel content area (ml-5 from the timeline connector). On the run detail page within the Sheet drawer, this is approximately 340px (400px sheet minus padding and timeline). When the Sheet is closed and the approval is shown in a full-width layout, it expands to fill available space up to max-w-3xl (768px) for comfortable diff reading.

### Approval in Timeline Sheet Drawer

The existing Phase 36 Sheet drawer (400px right side) shows the StepLogPanel per step. When a step is waiting, the ApprovalPanel renders inline. Because 400px is narrow for a diff viewer:

- Default to unified view (not split) when rendered inside the Sheet drawer (width < 600px)
- Default to split view when rendered on wider layouts
- User can toggle between split/unified regardless of container width

### Graph Node "Waiting" State

The existing AgentNode component gains a "waiting" visual state:

```
[AgentNode with status "waiting"]
```

```
[Card (variant outline)]
```

## Interaction Patterns

### Approval Flow (Primary)

1. Pipeline reaches a HITL stage and creates an `approval_requests` row with status "pending".
2. Inngest function calls `broadcastStepUpdate()` with status "waiting" and `approvalId`.
3. Run detail page receives Broadcast event. StepLogPanel for the waiting step auto-expands and renders ApprovalPanel.
4. Graph node for the relevant stage transitions to "waiting" state (amber border, animate-pulse, PauseCircle overlay).
5. Run status badge on the page header shows "Waiting for Approval" (amber badge).
6. User reads the explanation, reviews the diff, optionally writes a comment.
7. User clicks "Approve Changes" or "Reject Changes".
8. Button shows loading state: disabled, icon becomes Loader2 with animate-spin, text becomes "Submitting...".
9. Server action writes decision to DB and sends Inngest event.
10. Broadcast delivers the decision update. StepLogPanel transitions from ApprovalPanel to the decided state (approved/rejected badge + comment).
11. Pipeline resumes. Next step transitions to "running".

### Deep Link from Email

1. User receives email and clicks "Review and Approve" link.
2. URL contains `?approval={approvalId}` query parameter.
3. Run detail page opens. If the approval step is not visible (collapsed or scrolled away), the page auto-scrolls to the waiting step and opens the Sheet drawer if needed.
4. The `approvalId` query param is consumed on mount and removed from the URL via `router.replace()` to prevent stale bookmarks.

### Double-Submit Prevention

- Client-side: Button disables immediately on click. React state tracks `isSubmitting`.
- Server-side: UPDATE query uses `.eq("status", "pending")` so only the first write succeeds.
- If server action returns an error (already decided), show toast: "This approval has already been decided."

### Optimistic UI After Decision

After the user clicks Approve or Reject:

1. Immediately transition the ApprovalPanel to a read-only "decided" state showing the decision badge and comment.
2. Do not wait for the Broadcast confirmation to update the local UI.
3. If the server action fails, revert to the interactive state and show the error toast.

### Real-Time Updates (extends Phase 36 Broadcast)

- Existing `run:{runId}` Broadcast channel is reused.
- `step-update` events now include `approvalId` field when status is "waiting".
- New event type `approval-decided` carries `{ approvalId, decision, decidedBy, comment }` for multi-user scenarios (another project member decides while you are viewing).
- If another user decides the approval while you are viewing the ApprovalPanel, the panel transitions to read-only decided state via Broadcast.

### Comment Input Behavior

- Textarea: 3 visible rows, resize-y enabled, max 500 characters.
- Character count shown below textarea when user has typed (e.g., "142 / 500").
- Comment is optional for both approve and reject actions.
- Comment is submitted with the decision in a single server action call.
