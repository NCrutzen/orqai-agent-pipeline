# Phase 80.1 Plan 01 — Disambiguate triage helpers in swarm-bridge sync

## Context

Surfaced by Phase 80 UAT (Issue #2). `agent_runs.status='predicted'` is
written by three workers — Stage 1 screen-worker, Stage 2 invoice-copy
handler, and Stage 3 classifier (Phase 80) — each carrying a distinct
`tool_outputs` marker. `web/lib/automations/swarm-bridge/sync.ts`
currently calls `triageAgentFromStatus(row.status)` and
`triageStageFromStatus(row.status)` with only the status string, so the
single `case "predicted"` arm forces one label across all three stages.
After Phase 80, every `predicted` row gets attributed to "Stage 3
Dispatcher" in the v7 KanbanBoard (`/swarm/[swarmId]`). For Stage 1 and
Stage 2 rows this is wrong.

Pre-Phase-80 the same single-case ambiguity returned "Copy-Document
Agent" via the switch default — wrong but in a different direction.
This plan fixes the underlying single-status switch by threading the
agent run's `tool_outputs` into the triage helpers.

## Goal

Both triage helpers branch on `tool_outputs` markers to produce the
correct stage and agent label per row:

| `tool_outputs` marker key | Stage | `triageAgentFromStatus` returns | `triageStageFromStatus` returns |
| -- | -- | -- | -- |
| `intent_first_pass` | Stage 3 | `"Stage 3 Dispatcher"` | `"progress"` |
| `handler_output` | Stage 2 | `"Copy-Document Agent"` | `"progress"` |
| `stage1_category` | Stage 1 | `"Screen Worker"` | `"progress"` |
| (none of the above) | unknown | `"Pipeline"` (catch-all) | `"progress"` |

The `progress` lane is correct for all three stages — `predicted` always
means "carries a prediction, awaiting next step", which is "in progress"
on the kanban. The label split is the bug.

## Tasks

### Task 1 — RED test
Create `web/lib/automations/swarm-bridge/__tests__/sync.test.ts` (if absent)
and add three failing tests asserting:

1. `triageAgentFromStatus({ status: "predicted", tool_outputs: { intent_first_pass: {} } })` returns `"Stage 3 Dispatcher"`.
2. `triageAgentFromStatus({ status: "predicted", tool_outputs: { handler_output: {} } })` returns `"Copy-Document Agent"`.
3. `triageAgentFromStatus({ status: "predicted", tool_outputs: { stage1_category: "out_of_office" } })` returns `"Screen Worker"`.

Plus the inverse: `triageStageFromStatus({ status: "predicted", tool_outputs: {...} })` returns `"progress"` for all three.

If a sync.test.ts already exists, append the new cases; otherwise create a
minimal scaffold with the project's existing test pattern (vitest +
mocked admin client where needed).

Commit: `test(80.1-01): RED scaffold for triage helper disambiguation`

### Task 2 — Thread tool_outputs into both helpers and update call sites
Change both helper signatures:

```ts
function triageStageFromStatus(row: { status: string; tool_outputs: Record<string, unknown> | null }): TriageStage
function triageAgentFromStatus(row: { status: string; tool_outputs: Record<string, unknown> | null }): string
```

Inside each helper, when `status === "predicted"`, switch on
`row.tool_outputs?.intent_first_pass ? "stage3" : row.tool_outputs?.handler_output ? "stage2" : row.tool_outputs?.stage1_category ? "stage1" : "unknown"` and return per the table above.

Update the four call sites in sync.ts (lines ~339, ~340, ~420, ~422) to
pass the full row instead of `row.status`. `AgentRunRow` already has
`tool_outputs` so no schema change is needed.

Add a short comment on each helper explaining the marker-based branching
and pointing to this plan path.

Run `npm run test -- sync.test.ts` and confirm the three new tests pass.
Run `tsc --noEmit` on web/lib/automations/swarm-bridge/sync.ts to confirm
no regressions.

Commit: `fix(80.1-01): disambiguate predicted-row triage labels by tool_outputs marker`

## Out of scope

- Stage 1 sub-attribution (Screen Worker vs Label Resolver vs Spotcheck
  Sampler). Label Resolver and Spotcheck Sampler write to `automation_runs`,
  not `agent_runs`, so they never reach these helpers. A future
  follow-up can add finer-grained Stage 1 splits when more `tool_outputs`
  markers exist.
- Stage 0 safety worker — writes `automation_runs.status='predicted'`,
  not agent_runs, so unaffected.
- Backfilling `swarm_jobs` rows generated before this fix. The
  swarm-bridge sync runs continuously; affected rows refresh on the
  next sync tick. No DB migration needed.
