# Phase 82.7.3: Detail-pane + Stage-1 list UAT follow-ups (round 2) — Specification

**Created:** 2026-05-18
**Ambiguity score:** 0.145 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

Eliminate the 6 operator-visible defects surfaced during 2026-05-18 UAT of the Stage 1 detail-pane and noise-tab list: collapse three submit/cancel surfaces to one canonical footer, replace engineering shorthand with plain-English copy, stop the footer button row from truncating, surface mailbox identity on every row, fix the misleading "All N" counter semantics, and reduce filter-row visual noise by hiding advanced filters behind a single control.

## Background

Operator UAT 2026-05-18 (immediately after 82.7.2 merge to main, commit `89f0616`) surfaced 6 net-new defects spanning two surfaces:

- **Stage 1 detail-pane** (`web/app/(dashboard)/automations/[swarm]/_shell/detail-pane.tsx` + `_shell/components/stage-1-widget.tsx` + `stage-1/components/stage-step.tsx` + `stage-1/components/eval-type-radio.tsx`): three independent submit/cancel surfaces render simultaneously when the dirty-branch is active, REG/CAP pills use engineering shorthand the operator does not parse intuitively, and the footer's `Submit override (Stage N) · Cancel override · Reject` row truncates `Reject` at current pane width.
- **Stage 1 noise list** (`web/app/(dashboard)/automations/[swarm]/_shell/row-list.tsx` + `stage-1/page.tsx` + `stage-1/noise-category-chip-strip.tsx` + `stage-1/predictor-confidence-chip-strip.tsx`): mailbox identity is not visible on rows (data exists — `Row.mailbox_id` is populated and `_lib/load-mailbox-labels.ts` resolves names — but the renderer does not surface it), the top filter chip "All 459" counts all rows in the current view (not the human-review subset the operator expected), and the always-visible Predictor + Confidence chip strips dominate the top of the list for filters used infrequently.

All six defects come from a single UAT round, single operator (Nick), 2026-05-18. No production telemetry yet.

**Codebase-scout findings (locked):**
- `stage-1-widget.tsx:378` renders a `Discard changes` button.
- `stage-step.tsx:253-309` renders an inline `cancel override + Submit override` action row (introduced by 82.7.2-01).
- `detail-pane.tsx:495,732,744` renders the global footer `Submit override (Stage N) + Cancel override + Reject` row.
- `eval-type-radio.tsx:61,68` carries the `Used to work` / `New pattern` subhead copy and the REG/CAP pill markup.
- `Row` type (`_shell/_lib/types.ts:17`) already carries `mailbox_id: number | null` — render-side gap only.
- `noise-category-chip-strip.tsx:103` defines `{ key: "all", label: "All", count: totalCount }` — the count semantic the operator wants changed.
- `predictor-confidence-chip-strip.tsx` is a standalone always-visible component owning both filter rows.

## Requirements

1. **G-01 — Single canonical submit/cancel surface (footer-only)**: After this phase, exactly one Submit/Cancel pair is visible to the operator while the dirty-branch is active — the detail-pane footer (`Submit override (Stage N) · Cancel override · Reject`). The inline `cancel override + Submit override` pair inside `stage-step.tsx` and the `Discard changes + Submit override` pair inside `stage-1-widget.tsx` are removed from the rendered output when the canonical footer is present.
   - Current: three submit/cancel surfaces render simultaneously when the dirty branch is active (stage-1-widget pair + stage-step inline pair + detail-pane footer).
   - Target: only the detail-pane footer pair renders. The two inline pairs are removed (or conditionally hidden when the canonical footer is mounted).
   - Acceptance: in the dirty-branch state on Stage 1, only `data-testid="stage-N-submit-override"`-equivalent buttons present on screen come from the footer; no `Discard changes` button is rendered; no inline `cancel override + Submit override` pair is rendered inside `stage-step.tsx`. Lime Submit fill (`var(--v7-lime)`) preserved on the surviving Submit. Keyboard shortcuts (`Esc` for discard, `⌘⏎` for submit) continue to dispatch through the canonical footer handlers.

2. **G-02 — Plain-English copy on eval-type radio cards (meaning preserved)**: The two radio cards inside `eval-type-radio.tsx` use plain-English titles/subheads instead of REG/CAP shorthand. The underlying `eval_type` payload to `/api/automations/debtor-email/override` (`capability` | `regression`) is unchanged.
   - Current: card 1 = red `REG` pill + "Used to work" subhead; card 2 = lime `CAP` pill + "New pattern" subhead.
   - Target: card 1 framing communicates "recent regression — used to work correctly"; card 2 framing communicates "new case — never handled correctly before". Exact wording is operator-locked in discuss-phase, but the framing semantic is fixed.
   - Acceptance: rendered card titles and subheads contain no occurrence of the literal strings `REG` or `CAP` as user-facing copy; both cards have a sentence-case title and an explanatory sub-line ≥ 4 words; clicking each card still dispatches `evalTypeRegression` / `evalTypeCapability` events; existing keyboard shortcuts (`c` = capability, `g` = regression) still work; API payload assertion in `web/app/api/automations/debtor-email/override/__tests__/route.test.ts` continues to pass (eval_type=capability/regression unchanged).

3. **G-03 — Footer action row fully visible at full pane width**: The detail-pane footer (`Submit override (Stage N) · Cancel override · Reject`) renders all three buttons fully visible — no truncation, no overflow — at the post-fix detail-pane width. Fix is column-level (widen detail pane), not button-level.
   - Current: at the current detail-pane width, "Reject" is truncated past the right edge; operator can see only `Reje…`.
   - Target: at the post-fix pane width, all three buttons render entirely within the pane viewport on a single line, with no horizontal scrollbar inside the pane.
   - Acceptance: in DOM at the post-fix width, the right edge of the `Reject` button's bounding box is strictly less than the right edge of the detail-pane container's content box. Verified on all four stage tabs (Stage 0, 1, 3, 4 — Stage 2 detail-pane is per Phase 82.3, out of scope here) that the footer fits. List column compression is acceptable; total page width stays within the existing browser-viewport breakpoints used in `_shell` layout.

4. **G-04 — Mailbox label on every row, all stage tabs**: Every row in `row-list.tsx` renders a small mailbox label/chip identifying which mailbox the row belongs to. Applies to all stage tabs (Stage 0, 1, 3, 4 — wherever `row-list.tsx` is used).
   - Current: `Row.mailbox_id` is populated on every row, but no mailbox label is rendered. Operator must open the detail-pane to find out which mailbox a row belongs to.
   - Target: each row displays a mailbox name resolved through `_lib/load-mailbox-labels.ts`. Resolution is best-effort — rows whose `mailbox_id` cannot be resolved degrade to a neutral placeholder (e.g. dash) without erroring.
   - Acceptance: on the Stage 1 noise tab with the current 459-row dataset, ≥ 95% of rows render a non-empty mailbox label. Same holds for Stage 0, Stage 3, Stage 4 row-list mounts. No console errors when `mailbox_id` is null. Layout of existing columns (recipient, from/subj, stage cells, cost, tag) is not regressed — visual review on a standard desktop width.

5. **G-05 — "All" filter chip count reflects rows-needing-review subset**: The top filter chip's count reflects the count of rows that need human review in the current view, not the count of all rows.
   - Current: `noise-category-chip-strip.tsx:103` — `count: totalCount` (= all rows in the current view). Operator reads "All 459" as a global database count.
   - Target: the count shown on the "All" chip equals the number of rows in the current view that are still actionable for human review (not yet handled, overridden, or skipped). Label may be renamed in discuss-phase (e.g. "Needs review · N") — the semantic is locked: count = review-needed rows in current view.
   - Acceptance: when no per-category filter is active, the "All" chip's count equals the row-count rendered by `RowList` for the review-needed subset. When a category filter is active, the "All" chip's count still equals the review-needed subset (i.e. it acts as a "reset to all reviewable rows" button). Visual verification: the count visibly decreases compared to the pre-fix display on the same dataset (because handled/overridden/skipped rows are subtracted).

6. **G-06 — Predictor + Confidence filter rows hidden by default behind a collapsed control**: The two chip strips in `predictor-confidence-chip-strip.tsx` are hidden on initial page load. A single visible control ("Advanced filters" toggle, or equivalent) reveals both rows when activated. State persistence across navigations is not required.
   - Current: both `PREDICTOR_CHIPS` and `CONFIDENCE_CHIPS` rows always render at the top of the list, occupying ~2 rows of vertical space above the row strip.
   - Target: both rows collapsed by default. A control (operator-visible label + (i) tooltip or similar) toggles visibility. When collapsed, the filter chips do not render in the DOM (or render with `hidden`/`display: none` — both acceptable).
   - Acceptance: on first page load with no URL params for predictor/confidence, the predictor and confidence chip rows are NOT visible to the operator. The "Advanced filters" control IS visible. Clicking the control reveals the chips. When predictor/confidence URL params ARE present on load (e.g. operator linked from another tool), the chips render expanded (otherwise the filter state would be hidden from the operator). Existing filter logic — `?predictor=` / `?confidence=` query params, click handlers, count badges — is unchanged.
