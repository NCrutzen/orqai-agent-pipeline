## Summary

Phase 82.9 takes the Stage 2 audit panel from the current 2-chip render (`identifier_source`, `confidence`) + a count-only candidate list to a full evidence surface: per-method `INPUTS`, a richer `CANDIDATES` list (id + name + contact_person + recent_invoices), and a `REASONING` block for the LLM tiebreaker path. The data is shaped end-to-end by a **TS discriminated union on `decision_details.method`** that travels from `resolve-debtor.ts` → `classifier-label-resolver.ts` writer → `pipeline_events.decision_details` jsonb → `build-stage-audit-map.ts` mapper → `Stage2EvidencePanel.tsx`. Forward-only; legacy rows degrade gracefully via `inputs == null`.

Stage 3's equivalent expansion shipped in commit `ff12322` and is the visual + structural template. The mechanical work (types, mapper, panel) is straightforward; the **single hard external question** is whether `nxt.candidate_details` can be extended in-place to return `contact_person` + `recent_invoices` or whether a sibling NXT-Zap tool is needed — this is investigated in Plan 01 below and concretely answered HIGH-confidence: in-place extension is the right path (one row update + Zap edit, no new endpoint).

**Primary recommendation:** Slice into 4 plans as CONTEXT.md suggests (Zap+registry → resolver+writer → types+mapper → panel UI). Build the discriminated union in `resolve-debtor.ts` first so the writer can stop assembling structure inline; reuse the Stage 3 visual treatment 1:1 for INPUTS/REASONING/CANDIDATES sections; add a brand-new `build-stage-audit-map.test.ts` covering all 5 method branches + legacy fallback (it does not exist yet — confirmed Wave 0 gap).

### Locked Decisions

**D-01 · Decision-details schema: per-path discriminated union**

`decision_details.method` is the discriminant. TS discriminated union in the writer ensures only valid combinations exist. Mapper exhaustively branches on `method`.

```ts
type Stage2DecisionDetails =
  | { method: "thread_inheritance"; customer_account_id: string; customer_name: string | null;
      inputs: { prior_email_label_id: string; conversation_id: string } }
  | { method: "sender_match"; customer_account_id: string; customer_name: string | null;
      inputs: { sender_email: string; candidates: Candidate[] } }
  | { method: "identifier_match"; customer_account_id: string; customer_name: string | null;
      inputs: { matched_identifiers: string[]; candidates: Candidate[] } }
  | { method: "llm_tiebreaker"; customer_account_id: string; customer_name: string | null;
      inputs: { sender_email: string | null; matched_identifiers: string[]; candidates: Candidate[]; llm_reason: string } }
  | { method: "unresolved"; customer_account_id: null; customer_name: null;
      inputs: { sender_email: string | null; matched_identifiers: string[] } };
```

`customer_account_id` and `customer_name` stay at top level (back-compat).

**D-02 · LLM tiebreaker reproducibility: reason only**

Persist `inputs.llm_reason: string` only. Do NOT store model/agent_key/temperature/prompt version — `agent_runs` row + Orq.ai trace IDs cover forensic replay elsewhere.


**D-04 · Historical-row treatment: forward-only**

New rows: full evidence. Legacy rows: chip-only render + muted "Legacy run — limited evidence captured" line. Mapper detects via `decision_details.inputs == null`. No backfill.

### Open Question Resolved: Zapier-side schema change vs sibling endpoint

**Question (D-03):** Does extending `lookupCandidateDetails` to return `contact_person` + `recent_invoices` require a Zapier-side schema change to the existing Zap, OR a brand-new sibling endpoint?

**Answer: In-place schema change to the existing Zap + one-row update to `zapier_tools.input_schema`/`output_schema`. No new endpoint, no new registry row.**

Evidence `[VERIFIED: codebase + migration grep]`:

1. The registry row for `nxt.candidate_details` (migration `20260429_zapier_tools_registry.sql:156-193`) defines `output_schema` as a jsonb document. Adding two optional fields (`contact_person`, `recent_invoices`) is a column-level update — same pattern as the `auth` field changes in `20260429c` and `20260429e`.

2. The Zod schema (`nxt-zap-client.ts:135-145` — `CandidateDetailMatchSchema`) already has 5 nullable-optional fields (`brand_id`, `country_id`, `city`, `classification`, `email`). Adding `contact_person: z.string().nullable().optional()` and `recent_invoices: z.array(z.string()).max(5).optional()` follows the same pattern with zero breaking change on legacy responses (Zod treats missing optional fields as absent).

3. The auth + transport (POST body shape, `lookup_kind=candidate_details`, callback URL, async wait) is unchanged — only the SQL in the Zap's "Find Records" / "Run Code" step needs to JOIN `contact_persons` + `invoices` and project the two extra columns.

4. Same `auth_secret_env` (`DEBTOR_FETCH_WEBHOOK_SECRET`), same callback route (`/api/automations/debtor/nxt-lookup/callback`), same `allowed_for_intents` (`unknown`). Zero new env vars, zero new Vercel routes.

**Plan 01 scope (concrete):**
- Edit existing Zap "candidate_details" in Zapier: extend the SQL step to JOIN contact_persons + recent invoices, project `contact_person` (single string — likely first/primary contact name) and `recent_invoices` (top 5 invoice numbers newest-first, comma-joined or array).
- Confirm Zap "Find Multiple Rows" / "Run Code" output shape matches the new schema (use postman-style trigger from Zapier).
- One-row UPDATE to `public.zapier_tools.output_schema` for `tool_id = 'nxt.candidate_details'`.
- Update Zod `CandidateDetailMatchSchema` to add the two new optional fields.
- Production smoke against acceptance NXT (`nxt_benelux_prod` is fine — read-only SELECT, no writes).

### Pattern 1: Discriminated union end-to-end

**What:** `decision_details.method` is the discriminant at the writer; at the mapper it's the switch key; at the panel it gates conditional sections (REASONING only when `method === "llm_tiebreaker"`).

**When to use:** Any time downstream consumers need to render conditionally on a "kind" field, and the legal combinations of fields are not arbitrary.

**Example (writer — classifier-label-resolver.ts):**

```ts
// Source: web/lib/inngest/functions/classifier-label-resolver.ts (extension pattern)
type Stage2DecisionDetails =
  | { method: "thread_inheritance"; customer_account_id: string; customer_name: string | null;
      inputs: { prior_email_label_id: string; conversation_id: string } }
  | { method: "sender_match"; customer_account_id: string; customer_name: string | null;
      inputs: { sender_email: string; candidates: Candidate[] } }
  // ... 3 more arms

function buildStage2DecisionDetails(
  r: ResolveResult,
  ctx: { conversation_id: string | null; sender_email: string | null },
): Stage2DecisionDetails {
  switch (r.method) {
    case "thread_inheritance":
      return {
        method: "thread_inheritance",
        customer_account_id: r.customer_account_id!,
        customer_name: r.customer_name,
        inputs: {
          prior_email_label_id: r.inputs.prior_email_label_id,
          conversation_id: ctx.conversation_id!,
        },
      };
    // ...exhaustive arms; TS narrows on r.method
  }
}
```

**Example (mapper — build-stage-audit-map.ts):**

```ts
// Source: web/app/(dashboard)/automations/[swarm]/_shell/_lib/build-stage-audit-map.ts (extension pattern)
```

### Pattern 2: Optional re-fetch on single-hit `sender_match`

**What:** Layer 2 (sender_match) currently short-circuits on single-hit WITHOUT calling `lookupCandidateDetails` (`resolve-debtor.ts:104-112`). To persist a rich `Candidate` for the chosen match (INPUT-02), the resolver must call `lookupCandidateDetails([m.top_level_customer_id])` on the single-hit path too.

**When to use:** Required to fulfill INPUT-02 with full candidate detail. Alternative is to ship slim `{id, name}` on single-hit `sender_match` only — but that contradicts D-03's intent ("operator wants forensic candidate detail without a NXT round-trip during review").

**Cost:** ~1 extra Zapier task per Stage-2 `sender_match` single-hit — by far the most common path. Acceptable given operator UAT explicitly asked for it.

**Implementation note:** Wrap the new call in `step.run("fetch-sender-candidate-detail", ...)` if it lives inside the Inngest function, OR keep it inside the existing `step.run("resolve-debtor")` since the resolver function is already entirely wrapped in one `step.run` (`classifier-label-resolver.ts:98-128`). The latter is simpler.

### Pattern 3: Mapper exhaustiveness assertion

```ts
const _exhaustive: never = method as never;
throw new Error(`unknown stage-2 method: ${method}`);
```

In the `default:` arm of the mapper switch — catches an unhandled `method` value at runtime AND lets TS flag a missing arm at compile time when a new method is added.

### Anti-Patterns to Avoid

- **Persisting `llm_model` in `decision_details`** — D-02 forbids this. Trace IDs in `agent_runs` cover replay.
- **Hand-rolling JSONB shape in the writer** — use the discriminated union type as input to `emitPipelineEvent` so TS catches malformed payloads at compile time.
- **Dropping legacy-row support from the mapper** — D-04 mandates forward-only; legacy rows must still render the existing chip-only payload. Detect via `decision_details.inputs == null`.
- **`as any` casts in mapper input narrowing** — use the existing `isRecord`/`asString`/`asNumber` helpers (build-stage-audit-map.ts:72-105) which are tolerant of malformed JSONB and never throw.
- **Adding a new Zapier tool row when extending the existing one works** — duplicate registry rows fragment the operator surface and double the auth/secret rotation surface.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSONB → typed payload narrowing | Custom `as` casts + ad-hoc type guards | Existing `isRecord/asString/asNumber/asStringArray` helpers in `build-stage-audit-map.ts` | These are already tolerant of malformed JSONB and never throw — required because legacy rows have unpredictable shape. |
| NXT-Zap response validation | Inline `typeof === "string"` checks | Zod schema in `nxt-zap-client.ts` (`CandidateDetailMatchSchema`) | Zod already handles nullable optional fields, string-vs-number coercion via `.transform(String)`, and surfaces parse errors as `Error`. Extending the schema is one-line per field. |
| Inngest replay safety on the new single-hit `lookupCandidateDetails` call | Top-level `await` outside `step.run` | Stays inside the existing `step.run("resolve-debtor", ...)` block | CLAUDE.md mandates side-effects in `step.run`. ## Common Pitfalls

### Pitfall 1: Forgetting to widen `ResolveResult` before writing decision_details

**What goes wrong:** Writer references `result.inputs.prior_email_label_id` but `ResolveResult` only carries `customer_account_id` + `customer_name` + `method` + `confidence`. Either the writer rebuilds inputs from scratch (duplicating logic and risking drift) or it has no access to the data.

**Why it happens:** The current resolver throws away the per-layer raw data after deciding the customer. E.g. Layer 1 SELECTs the prior `email_labels` row but only keeps `customer_account_id` — discards the row's `id` (= `prior_email_label_id`).

**How to avoid:** Widen `ResolveResult.inputs` as a discriminated field FIRST (Plan 02 / step 1). Resolver populates `inputs` at each `return`. Writer then has a clean pipe-through.

**Warning signs:** Writer code starts re-querying Supabase or re-running `extractInvoiceCandidates` to reconstruct what the resolver just discarded.

### Pitfall 2: Forgetting Inngest replay-safety on the new NXT-Zap call

**What goes wrong:** `lookupCandidateDetails` makes an HTTP POST + waits on Supabase Realtime. If called outside `step.run`, Inngest replays it on every retry → duplicate Zap calls, duplicate rows in `debtor.nxt_lookup_requests`.

**Why it happens:** Phase 65 learning (CLAUDE.md): "Alle side effects in `step.run()`."

**How to avoid:** Keep the new sender-side candidate-detail fetch inside the existing `step.run("resolve-debtor", ...)` wrapper at `classifier-label-resolver.ts:98-128`. The resolver function is called from inside the step.run, so any HTTP it does is already replay-safe.

**Warning signs:** Test runs show >1 row in `debtor.nxt_lookup_requests` per Stage-2 single-hit invocation.

### Pitfall 3: Legacy row detection by wrong key

**What goes wrong:** Mapper detects legacy via `decision_details.method == null`. But legacy rows DO have `method` (it's been written since Phase 56). They lack `inputs`. So `method == null` matches almost nothing.

**Why it happens:** Drift between D-04's prescription (`decision_details.inputs == null`) and a casual reading of the existing payload.

**How to avoid:** Use exactly the contract: legacy iff `!isRecord(decision_details.inputs)`. Test fixture: a row with `{method: "sender_match", customer_account_id: "X", customer_name: "Y"}` (no `inputs`) MUST hit the legacy branch.

**Warning signs:** Mapper test for legacy row falls into a method-arm and produces a half-empty `Stage2AuditPayload`.

### Pitfall 4: Stage2AuditPayload type churn breaking the page-level fetch

**What goes wrong:** Replacing the flat `Stage2AuditPayload` with a discriminated union breaks call sites that destructure `payload.identifier_source` directly (the current panel does this).

**Why it happens:** Discriminated unions require narrowing before field access.

**How to avoid:** Either (a) keep `identifier_source` and `confidence` at the top level of every union arm (back-compat), then narrow only for the new fields, OR (b) rewrite the panel to switch on `payload.method` first. Recommend (a) — minimal panel churn, and the current chips don't depend on per-method logic.

### Pitfall 5: `nxt.candidate_details` Zap output_schema enforcement (or lack thereof)

**What goes wrong:** Updating the Zod schema before the Zap is updated produces parse failures in production (resolver throws, falls through to `unresolved`).

**Why it happens:** The Zap is the source of truth for what comes back; the Zod schema is just a guard.

**How to avoid:** Ship Plan 01 (Zap edit + registry UPDATE) BEFORE Plan 02 (resolver/writer changes). New Zod fields are `.optional()` so the resolver still parses old responses if the Zap update lags. Verify with one smoke call against the Zap before shipping the resolver change.

### Pitfall 6: `recent_invoices` size unbounded from the SQL side

**What goes wrong:** Zap SQL forgets the `LIMIT 5` and returns 200 invoice numbers — explodes `decision_details` JSONB size, slows queries.

**Why it happens:** D-03's `recent_invoices.length <= 5` is a contract on the read side; the Zap SQL must enforce it.

**How to avoid:** Bound IN THE SQL (`ORDER BY invoice_date DESC LIMIT 5`). Add a Zod `.max(5)` as backup defense-in-depth. Add a mapper-side `.slice(0, 5)` as third-tier defense.

## Open Questions

1. **Should `sender_match` single-hit re-fetch candidate details to populate the rich `Candidate` field?**
   - What we know: D-03 wants forensic detail without re-querying NXT during review. Single-hit `sender_match` currently does NOT call `lookupCandidateDetails`. Persisting `{id, name}` only would technically satisfy INPUT-02 textually, but operator intent is fuller.
   - What's unclear: Cost vs value of one extra Zapier task per Stage-2 sender-match invocation.
   - Recommendation: Re-fetch (one `lookupCandidateDetails([id])` call). Acceptable cost on the most common Stage-2 path; matches operator intent. Documented in Pattern 2.

2. **Naming: `inputs.kind` (TS discriminant) vs reusing `method`?**
   - What we know: CONTEXT.md D-01 spec uses `method` at the outer level AND implicitly inside `inputs`. Less DRY but works.
   - What's unclear: Whether to also key the inner discriminated union by `kind` (clearer) or `method` (matches outer).
   - Recommendation: Use `method` at outer level only; nest plain `inputs` (no inner discriminant). TS narrowing on `r.method` propagates to `r.inputs` via the union literal — cleaner. Aligns with D-01 spec.

3. **Does the panel render a "Legacy run — limited evidence captured" line in a new INPUTS section, or replace the entire panel content?**
   - What we know: D-04 says "panel renders the existing chip-only render plus a muted line in the INPUTS section."
   - Recommendation: Add INPUTS section that, for legacy rows, contains ONLY the italic muted line (mirroring `Stage3EvidencePanel.tsx:194-197` "Legacy run — classifier inputs were not captured." pattern). Existing EVIDENCE + SCREENSHOTS sections continue to render below.
