# Phase 82.9 Plan 03: Stage 2 Audit Types + Mapper Bridge Summary

Widens `Stage2AuditPayload` to the discriminated `inputs` shape from D-01, extends the Stage 2 branch of `build-stage-audit-map.ts` with an exhaustive `method` switch (+ tolerant default + legacy fallback), and adds 7 new vitest cases — bridging the runtime payload landed by Plan 02 to the Stage2EvidencePanel rewrite that Plan 04 will ship.

## What was built

### 1. `audit-types.ts` — discriminated `Stage2InputsView`

```typescript
export type CandidateView = {
  id: string;
  name: string;
  contact_person: string | null;
  recent_invoices: string[];
};

export type Stage2InputsView =
  | { kind: "thread_inheritance"; prior_email_label_id: string; conversation_id: string }
  | { kind: "sender_match"; sender_email: string; candidates: CandidateView[] }
  | { kind: "identifier_match"; matched_identifiers: string[]; candidates: CandidateView[] }
  | { kind: "llm_tiebreaker"; sender_email: string | null; matched_identifiers: string[]; candidates: CandidateView[]; llm_reason: string }
  | { kind: "unresolved"; sender_email: string | null; matched_identifiers: string[] };

export interface Stage2AuditPayload {
  stage: 2;
  // EXISTING — DO NOT REMOVE (Pitfall 4 back-compat):
  identifier_source: "thread" | "sender" | "identifier" | "unresolved" | null;
  confidence: "high" | "medium" | "low" | null;
  top_candidates: Array<{ account_id: string; name: string; score: number }>;
  screenshot_paths: { before: string | null; after: string | null };
  raw: Record<string, unknown>;
  // Phase 82.9 — null/absent on legacy rows.
  inputs?: Stage2InputsView | null;
  candidates?: CandidateView[];
  reasoning?: string | null;
}
```

All existing top-level fields preserved per Pitfall 4 (back-compat with the current chip-only panel).

### 2. `build-stage-audit-map.ts` — Stage 2 mapper extension

- New `asCandidates(v: unknown): CandidateView[]` helper next to `asTopCandidates`, enforcing the Tier-3 `.slice(0, 5)` bound on `recent_invoices` (defense-in-depth alongside Tier-1 SQL `LIMIT 5` and Tier-2 Zod `.max(5)`).
- Existing Stage 2 narrowing (`ctxSource`, `methodMapped`, `confidence`, `top_candidates`, `extractScreenshotPaths(result)`) preserved unchanged.
- Discriminated-union assembly added after existing narrowing:
  - `inputsRaw = isRecord(d.inputs) ? d.inputs : null` (Pitfall 3 — detect legacy by `inputs`, NOT by `method`).
  - 5-arm switch on `methodKey` populating the matching `Stage2InputsView` arm.
  - `default:` arm sets `stage2Inputs = null` (tolerant — never throws on unknown future `method` value).
  - `candidates` populated only for `sender_match` / `identifier_match` / `llm_tiebreaker`.
  - `reasoning` populated only for `llm_tiebreaker`.
- Final `Stage2AuditPayload` construction includes the three new fields alongside the existing back-compat ones.
- Zero new `as any`, zero new `throw` — uses existing `isRecord` / `asString` / `asStringArray` tolerant helpers only.

### 3. `build-stage-audit-map.test.tsx` — 7 new test cases

New `describe("Stage 2 — Phase 82.9 discriminated inputs", ...)` block at end of file with:

All 16 tests in the file pass (9 baseline + 7 new). No regressions.
