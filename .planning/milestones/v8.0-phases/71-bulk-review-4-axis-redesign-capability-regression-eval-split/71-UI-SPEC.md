# Phase 71 — UI Design Contract

> Visual and interaction contract for the Bulk Review 4-axis override redesign.
> Inputs locked by sketch 004 (production V7 tokens, recipient chip strip, N-stage data-driven detail pane).
> Generated by gsd-ui-researcher 2026-05-05; revised after gsd-ui-checker feedback (typography 4-role collapse, copywriting noun-scoped CTAs, spacing exception note, primary-focal-point declaration).

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialised in `web/components/ui/`) |
| Preset | not applicable — V7 namespace tokens in `web/app/globals.css` are the source of truth |
| Component library | Radix primitives via shadcn (Dialog, Select, Tooltip, Sheet, RadioGroup) |
| Icon library | `lucide-react` (already in use across the dashboard) |
| Font | Satoshi (sans), Cabinet Grotesk (display headings), Geist Mono (mono code/values) |

**Token namespace:** all colour, radius, and spacing references use the `--v7-*` namespace (e.g. `var(--v7-bg)`, `var(--v7-brand-primary)`). Do NOT introduce raw hex values in components — token-only.

**Theme:** dark-first (`[data-theme="dark"]`). Phase 71 ships the dark variant; light variant inherits the same V7 tokens automatically (no per-phase work).

**Min-width:** desktop-only operator workflow. Phase 71 sets `min-width: 1280px` on the `/automations/[swarm]/review` page; below that we render a "Bulk Review is desktop-only — please open on a wider display" placeholder. CONTEXT D-16 explicitly defers mobile/tablet layouts.

## Spacing Scale

V7 4px grid. Multiples of 4 only.

| Token | Value | Usage |
|-------|-------|-------|
| `--v7-space-1` | 4px | Icon gaps, dot offsets, tight inline padding (recipient dot ↔ address) |
| `--v7-space-2` | 8px | Compact element spacing (chip internal padding, stage cell padding) |
| `--v7-space-3` | 12px | Default row spacing inside a card, control padding |
| `--v7-space-4` | 16px | Card padding, default element spacing |
| `--v7-space-5` | 20px | Section padding (detail-pane sections, list outer) |
| `--v7-space-6` | 24px | Large section breaks (between recipient strip and list) |
| `--v7-space-7` | 32px | Page-level layout gaps (between toolbar regions) |

### Spacing exception note

The token series `--v7-space-1..7` (4·8·12·16·20·24·32) extends the standard 4px grid set {4, 8, 16, 24, 32, 48, 64} with intermediate values **12px and 20px**. These two values are established V7 production spacing measurements verified in the live `web/app/(dashboard)` surface (card padding 12px, panel gap 20px) and are required to keep the predicted-row strip and detail-pane drill-in dense without breaking the 4px subdivision. They are not arbitrary — they are V7 design-system precedent. All values remain multiples of 4 (the grid invariant); the standard set is broadened, not violated.

Other exceptions:
- Pipeline node circle is 30×30px — keeps the connecting line visually centred. Locked by sketch 004; not a re-usable token.
- Recipient dot is 6px (row) and 8px (chip) — both deliberate visual hierarchies; not promoted to tokens.

## Typography

Satoshi (sans), Cabinet Grotesk (display H1-H4), Geist Mono (values, addresses, IDs, timestamps).

**4-role typography contract.** Operators do NOT see any other font sizes inside the review surface. Differentiation between Label-tier and Pill-tier within `--v7-fs-xs` is achieved via uppercase + letter-spacing + pill chrome (background, radius, padding) — not via a fifth size.

| Role | Token / Size | Weight | Line Height | Notes |
|------|--------------|--------|-------------|-------|
| Label / Pill / Mono-small | `--v7-fs-xs` 11px | 600 (label, pill) / 400 (mono-small) | 1.3 | Column headers, "Inbox", stage labels (S1/S2/S3/S4), `cap` / `reg` eval-type pills, chip row-count badges. Labels & pills: `text-transform: uppercase`, `letter-spacing: 0.04em`. Pills additionally get `--v7-bg-panel` background + 999px radius + 4px×6px padding to differentiate from labels. |
| Mono / Value | `--v7-fs-sm` 12px | 400 | 1.3 | Stage decision values, recipient addresses, cost (€0.024), timestamps, email IDs. Geist Mono. |
| Body | `--v7-fs-base` 13px | 400 | 1.5 | Default page text, row metadata, helper copy. |
| Section + Page Heading | `--v7-fs-md` 14px | 600 | 1.3 | "Pipeline overrides", detail-pane H3, toolbar "Bulk Review" (Cabinet Grotesk for the page heading; Satoshi for section headings). |

## Color

V7 dark theme. 60/30/10 split.

| Role | Value | Token | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `#0c1117` | `--v7-bg` | Page background |
| Secondary (30%) | `#101722` / `#151d29` | `--v7-bg-2`, `--v7-panel-2` | Detail pane background, toolbar, row cards |
| Accent (10%) | `#ff6a34` | `--v7-brand-primary` | Primary CTA only ("Submit override") |
| Selected / Info | `#69a8ff` | `--v7-brand-secondary` | Selected row border, focus rings, active filter chip, link colour |
| Override semantic | `#ffb547` | `--override` (= `--v7-amber`) | Override-only signal — `↻` glyph, dirty stage cells, dirty pipeline nodes |
| Success | `#8ad05e` | `--v7-lime` | Pipeline node "ok" border, capability pill |
| Destructive | `#ff6b7a` | `--v7-red` | Regression pill, error states, destructive confirmations |

**Accent reserved for:**
1. Primary "Submit override" button (filled).
2. Selected eval-radio card (border + soft fill).
3. Hovered "override stage" inline link (transitions secondary → primary on hover; sketch 004 confirmed).

Brand-primary is NOT used for: row selection (that's `--v7-brand-secondary`), filter chip active state (also secondary), regular hover states (those use `--v7-bg-hover`).

**Override accent (amber) is reserved EXCLUSIVELY for the "stage was overridden" semantic.** Never used as chrome, decoration, or generic warning. Operator must learn: amber ↔ override. This is locked by sketch 004 § "What Changed" row.

**Recipient brand dot mapping (deterministic, registry-driven):**

The dot colour next to each recipient address (both in the chip strip and per-row) is **deterministically derived from `swarms.entity_brand`**. The mapping lives in a single helper `web/lib/swarms/brand-color.ts` (NEW in Phase 71):

| `entity_brand` code | Colour token | Hex |
|---------------------|--------------|-----|
| `smeba` | `--v7-lime` | `#8ad05e` |
| `smeba-fire` | `--v7-pink` | `#ff78cf` |
| `berki` | `--v7-amber` | `#ffb547` (NB: clash with `--override`; see below) |
| `sicli-noord` | `--v7-blue` | `#69a8ff` |
| `sicli-sud` | `--v7-teal` | `#3ac7c9` |

> **Amber clash mitigation:** `berki` resolves to `--v7-amber` which is also the override semantic colour. Because the recipient dot is 6px and only ever appears in the recipient column / chip strip (never inside a stage cell), the contexts do not collide visually. If a 6th brand lands and the palette runs out, swap berki to a new dedicated token (e.g. `--v7-violet` to be added) — DO NOT introduce a hash-derived palette. Determinism via registry > determinism via hash.

The helper signature:
```ts
export function brandColorToken(brand: Entity): string;  // returns CSS var name e.g. "--v7-lime"
```
The mapping table lives next to `ENTITY_BRANDS` in `entity.generated.ts` (or a sibling `entity-colors.ts`). New brands MUST have a colour assigned at registry-INSERT time; CI gate (`npm run codegen && git diff --exit-code`) catches drift.

**Contrast verification:**
- `--v7-text` (`#edf3fb`) on `--v7-amber-soft` (`rgba(255,181,71,0.13)` over `#0c1117`): contrast ratio ≈ 14.2:1 ✓ WCAG AAA.
- `--v7-muted` (`#9fadc4`) on `--v7-amber-soft`: contrast ratio ≈ 7.8:1 ✓ WCAG AAA.
- `--override` (`#ffb547`) used only for the `↻` glyph and small text on `--v7-amber-soft` background: 8.2:1 ✓ WCAG AAA.

### eval_type radio (REVW-05)

Two cards, mutually exclusive. Default = `regression` (CONTEXT D-08 safety bias).

| Option | Heading | Subhead/desc |
|--------|---------|--------------|
| `regression` (default) | `Regression` (red `reg` pill + "Used to work") | `Worked correctly before the most recent model/prompt change.` |
| `capability` | `Capability` (green `cap` pill + "New pattern") | `Never handled correctly before, or a brand-new case.` |

Tooltip on the section heading "Eval type — applied to every overridden stage": `Tags this override so model swaps don't silently break previously-correct decisions. Defaults to "regression" — the higher-cost failure mode.`

### Empty / loading / error states

| Surface | State | Copy |
|---------|-------|------|
| Predicted-row list | Empty (no rows for selected inbox filter) | Heading: `Inbox is clear`. Body: `No emails currently in predicted state for {inbox}. New arrivals appear here automatically.` |
| Predicted-row list | Empty (no rows at all, no filter) | Heading: `Nothing waiting for review`. Body: `The pipeline has caught up. New emails will appear here when classifier confidence is below auto-approve threshold.` |
| Predicted-row list | Loading | Skeleton rows (3 placeholder rows using `Skeleton` from `web/components/ui/skeleton.tsx`); recipient strip stays interactive. |
| Predicted-row list | Error | `Couldn't load review queue. {error message}` + Retry button (ghost). |
| Detail pane | No selection | Heading: `Pick an email to review`. Body: `Use ↑/↓ or click a row in the list to start reviewing.` |
| Detail pane | Loading single email | Skeleton: header strip, 4 step rows, eval-bar. |
| Detail pane | Error loading detail | `Couldn't load this email's pipeline events. Try refreshing the page.` |
| Override submit | In-flight | Submit button shows spinner + label `Submitting override…`. Form fields disabled. |
| Override submit | Success | Toast: `Override recorded for {short subject}.` Row optimistically marked, removed from list after 400ms. |
| Override submit | Error | Toast (destructive): `Override failed: {message}. Please try again.` Form stays editable. |
| Inbox filter chip | Loading row counts | Counts render as `—` (em-dash, mono) until count loads. |

### Confirmation modal (CONTEXT D-16)

**Trigger conditions** (modal fires; otherwise submit silently):
1. Stage 2 override with `re_run_downstream = true` (re-runs cost LLM tokens).
2. Stage 3 override (always — replaces the chosen handler; original Stage 4 output is discarded).
3. Any submit when ≥ 2 axes are dirty in the same submission (compound override; safety check).

**Modal copy:**

| Field | Copy |
|-------|------|
| Title | `Confirm override` |
| Body — Stage 2 re-run | `Re-running downstream stages will spend additional LLM tokens (~€0.02 per email) and replace the existing draft. The original pipeline events stay in audit history. Continue?` |
| Body — Stage 3 | `This dispatches a new {handler_key} handler. The original Stage 4 output will be kept as audit but not re-used. The iController draft (if any) is NOT auto-updated — you'll need to update it separately.` |
| Body — multi-axis | `You're overriding {N} stages in one submission ({axis list}). Each axis emits its own learning signal. Continue?` |
| Confirm button | `Yes, submit override` (primary, brand-primary fill) |
| Dismiss button | `Keep editing` (ghost — communicates that the in-progress form is retained, not discarded) |

### Stage-4 info banner (CONTEXT D-15, displayed AFTER successful submit)

Only fires for Stage 3 or Stage 4 overrides where an iController draft exists for the email.

```
ⓘ Override recorded — please update the draft in iController separately.
   We didn't auto-modify draft #{iC_id} because operator workflows vary.
```

Banner uses `--v7-blue-soft` background, `--v7-blue` border, `ⓘ` glyph. Dismissible.

### Destructive confirmations

No destructive actions in Phase 71 — overrides are additive (new `pipeline_events` row, original preserved). The "compound override" modal is precautionary, not destructive. No "Are you sure you want to delete?" surfaces.

## Layout & Interaction Contracts

### Page structure (locked by sketch 004)

```
┌─────────────────────────────────────────────────────────────────┐
│ Toolbar                                              [4|6 toggle]│  ← --v7-space-3 vert padding
├─────────────────────────────────────────────────────────────────┤
│ Recipient chip strip (horizontal scrollable)                    │  ← --v7-space-3 vert padding
├──────────────────────────────────────────┬──────────────────────┤
│ Predicted-row list                       │ Detail pane (480px)  │
│  - col header (uppercase labels)         │  - email head        │
│  - rows (recipient | from/subj | stages) │  - vertical pipeline │
│    - 4 stage cells per row               │    (N-stage)         │
│    - cost + cap/reg pill                 │  - eval-bar          │
│                                          │  - submit-bar        │
└──────────────────────────────────────────┴──────────────────────┘
```

- Page grid: `grid-template-columns: 1fr 480px` (detail pane fixed; list flexes).
- Recipient chip strip overflow-x: auto; chips never wrap.
- List rows: `grid-template-columns: 200px 280px 1fr 84px 70px` (recipient | from/subj | stages | cost | tag).
- Stage cells inside row: `grid-template-columns: repeat(4, 1fr)` — when N stages > 4, the column count adjusts but list-row stage cells STAY at 4 (lockstep with debtor-email's actual stage count); only the **detail pane** scales N. Future swarms with 5–6 stages will get their own row layout when Phase 73+ lands; Phase 71 hardcodes 4 in the list and N in the detail pane.

**Primary focal point:** the `Submit override` button (brand-primary `--v7-brand-primary` fill, fixed at the bottom of the detail pane). The operator's goal at every drill-in is to complete or dismiss an override; the brand-orange button is the only chrome element that draws attention away from the pipeline timeline. Selected list-row uses blue (`--v7-brand-secondary`) to avoid two oranges competing for the eye.

### Detail-pane vertical pipeline (N-stage)

Rendered from a data array `Stage[]` so the layout works for 4 (Phase 71) and 5–6 (future swarms). Each step has:

- **Node circle** (30px, left-anchored): contains the stage number; border colour = state (`ok` → `--v7-lime`, `dirty` → `--override`, neutral → `--v7-line`).
- **Title row**: `Stage {N} — {title}` + monospace axis tag (`stage_{N}_{slug}`).
- **Current value**: muted text, mono, prefixed `Current:`.
- **Control area**:
  - If `state === 'ok'`: `✓ Looks correct` + inline link `override stage` (right-aligned, `--v7-brand-secondary` → `--v7-brand-primary` on hover).
  - If `state === 'dirty'`: per-stage form widget (see Copywriting § per-stage table).

Connecting line: 2px wide, `--v7-line`, runs from top of first node to bottom of last node, 15px from left edge. Pure CSS pseudo-element.

### Recipient chip strip behaviour

- Single-select (CONTEXT-confirmed YAGNI on multi-select).
- "All" chip is the default active state on page load.
- Active chip: `--v7-brand-secondary-soft` background, `--v7-brand-secondary` border, `--v7-brand-secondary` text. Inactive: `--v7-bg-panel` / `--v7-border` / `--v7-text-muted`.
- Row count badge inside chip: monospace 11px (`--v7-fs-xs`), `rgba(255,255,255,0.06)` background, mutates to `rgba(105,168,255,0.18)` when active.
- Filtering is URL-driven: clicking a chip mutates `?inbox=debiteuren@smeba.nl` (or `?inbox=all`). State survives reload + supports operator deep-links.

### Row interactions

- Click anywhere on a row → selects it (URL `?id={email_id}`). Selected row: `--v7-brand-secondary` border + `--v7-bg-selected` background.
- Hover → `--v7-bg-hover` background, `--v7-border-strong` border.
- Stage cell with `↻` glyph (overridden) → cursor stays default (drill-in is via row click, not stage click). Tooltip on hover: `Stage {N} overridden by {operator email} on {date}` (read from `override` jsonb).
- Skipped stage cells (`opacity: 0.4`) — when Stage 1 routes to noise/archive and Stages 2–4 don't fire — render `—` em-dash; no tooltip.

### Focus order (accessibility)

Tab order on detail pane:
1. Recipient chip strip (left → right).
2. List row currently selected (focus ring on row).
3. Stage 1 override link / dirty form.
4. Stage 2 → Stage 3 → Stage 4.
5. eval_type radios (regression then capability).
6. Reason textarea (Stage 4 only — only present when Stage 4 dirty).
7. `Discard changes` button.
8. `Submit override` button.

Focus rings: `outline: 2px solid var(--v7-brand-secondary)`, `outline-offset: 2px`. ALL interactive elements get a visible focus ring (no `outline: none` without replacement).

ARIA:
- Chip strip: `role="tablist"`, each chip `role="tab"` + `aria-selected`. Filtered list: `role="tabpanel"`.
- Pipeline steps: `<ol>` with `<li>` per step; node circle `aria-hidden="true"` (decorative); state announced via visually-hidden span (`Stage 2 — overridden` or `Stage 2 — ok`).
- eval-radio cards: `role="radiogroup"`, each card `role="radio"` + `aria-checked`.
- Override submit button: `aria-describedby` pointing to a live-region `<span>` that announces dirty count: `1 stage dirty. Press Cmd-Enter to submit.`
