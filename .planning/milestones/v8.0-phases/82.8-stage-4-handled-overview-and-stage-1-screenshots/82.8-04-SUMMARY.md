# Phase 82.8 Plan 04: 30-day Stage 4 backfill + screenshot-path extract — Summary

One-shot migration that retroactively (a) inserts a Stage 4 `auto_archived_noise` `pipeline_events` row for every tagged `debtor.email_labels` row in the last 30 days, and (b) extracts the storage path out of any already-stored 1-hour signed URL into the new `screenshot_*_path` columns from Plan 01. Both passes idempotent.

## What changed

- **New migration** `supabase/migrations/20260518b_backfill_stage4_auto_archived_noise.sql` wrapping two passes inside a single `BEGIN; ... COMMIT;`:

### Pass A — Stage 4 telemetry backfill

```sql
INSERT INTO public.pipeline_events (swarm_type, stage, email_id, decision, decision_details, automation_run_id, triggered_by, created_at)
SELECT 'debtor-email', 4, el.email_id, 'auto_archived_noise',
       jsonb_build_object(
         'noise_category',         pe1.decision,
         'noise_category_unknown', (pe1.decision IS NULL),
         'source_stage',           1,
         'archived_at',            el.labeled_at,
         'backfilled',             true
       ),
       NULL, 'backfill', el.labeled_at
FROM debtor.email_labels el
LEFT JOIN public.pipeline_events pe1
  ON pe1.email_id = el.email_id AND pe1.stage = 1 AND pe1.swarm_type = 'debtor-email'
WHERE el.icontroller_tag_status = 'tagged'
  AND el.labeled_at >= NOW() - INTERVAL '30 days'
  AND el.email_id IS NOT NULL
ON CONFLICT ON CONSTRAINT pipeline_events_one_per_stage_email DO NOTHING;
```

### Pass B — screenshot path extract from existing URLs

```sql
UPDATE debtor.email_labels
   SET screenshot_before_path = substring(screenshot_before_url FROM 'object/sign/automation-screenshots/(.+?)\?')
 WHERE screenshot_before_url IS NOT NULL
   AND screenshot_before_path IS NULL
   AND labeled_at >= NOW() - INTERVAL '30 days';

-- same shape for screenshot_after_path / screenshot_after_url
```

| Migration is re-runnable | Pass A `ON CONFLICT ON CONSTRAINT pipeline_events_one_per_stage_email DO NOTHING`; Pass B `screenshot_*_path IS NULL` guards |

## Idempotency notes

- **Pass A** leans on the Phase-82.2 partial-unique constraint `pipeline_events_one_per_stage_email (email_id, swarm_type, stage) WHERE email_id IS NOT NULL`. Any second run of the migration produces zero inserts because every candidate row already has a `(email_id, debtor-email, 4)` event.
- **Pass B** writes only when the destination column is NULL. After the first run there are no NULL `screenshot_*_path` rows in the 30-day window left to update.

## Risk / rollback

- INSERT-only into `pipeline_events` and UPDATE-only on `debtor.email_labels.screenshot_*_path` (column added by Plan 01) — no schema changes, no row deletions.
- Rollback (if ever needed): `DELETE FROM public.pipeline_events WHERE stage = 4 AND decision = 'auto_archived_noise' AND decision_details->>'backfilled' = 'true';` and `UPDATE debtor.email_labels SET screenshot_before_path = NULL, screenshot_after_path = NULL WHERE labeled_at >= NOW() - INTERVAL '30 days';` — but the marker is permanent and intentional, so this should never run in production.
