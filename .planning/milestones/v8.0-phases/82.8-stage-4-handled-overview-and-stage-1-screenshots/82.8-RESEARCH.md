# Phase 82.8 — Research

**Researched:** 2026-05-18
**Confidence:** HIGH on verified facts; planner-blocking ambiguities flagged inline.

## Summary

Most of the heavy lifting already exists. `ScreenshotThumb`, `Collapsible`, `Dialog`, the signed-URL API route, `emitPipelineEvent`, and `loadKanbanRows` are all in tree. The two surprises that change CONTEXT.md's plan:

1. **D-06 conflicts with an existing unique index from Phase 82.2.** A unique index on `(email_id, swarm_type, stage)` already exists. CONTEXT's proposed `(email_id) WHERE stage=4 AND decision='auto_archived_noise'` is redundant — and worse, the existing one means *any* second Stage 4 row per email (e.g. future handler writing `decision='needs_review'`) will be rejected. Planner must reconcile.
2. **D-03 read path is broken as drafted.** `email_labels.screenshot_before_url` stores a **1-hour signed URL** captured at tag-time (`screenshots.ts:49`). It is stale within an hour of capture. The detail-pane cannot use it directly. Reuse the existing `ScreenshotThumb({ path })` pattern (Stage 2 already does this) — but D-03 needs storage *paths*, not URLs, and the paths are **not currently persisted to `email_labels`**.

Additional concrete gaps: `debtor.email_labels.automation_run_id` does not exist (D-07 SQL fails). Canonical `noise_category` source is `pipeline_events.stage=1 decision_details.final_category_key` — not `email_pipeline.emails`.

- **Redundant** for the auto_archived_noise insert path — the broader index already rejects it.

- **(A) Drop D-06 entirely.** Worker catches `23505` from the existing index. Simplest, works today, but blocks ALL future second-Stage-4 rows per email — including legitimate "handler retried after error and now succeeded" scenarios.

- **(C) Encode the decision distinction differently** — e.g. accept that one email gets exactly one Stage 4 terminal row and pick the most-recent on conflict (worker `upsert` rather than `insert`).

**Recommended: (A)** for tomorrow's deadline. Carry "Stage 4 multi-row support" as an explicit deferred item.

### 1.5 `captureScreenshot` storage — VERIFIED

`web/lib/browser/screenshots.ts:25-52`:
- Bucket: `automation-screenshots`
- Path format: `{automation}/{label}-{timestamp}.png` (e.g. `debtor-email-labeling/before-1715812345678.png`)
- Returns `{ path, url }` where `url` is **`createSignedUrl(path, 3600)`** → **1-hour TTL**
- Caller (`label-email-in-icontroller.ts:88-92`, `158-162`) discards `path` and persists only `url`

So `debtor.email_labels.screenshot_before_url` / `.screenshot_after_url` store **expired signed URLs** the moment the operator opens the detail-pane more than an hour after the tag ran.

**This breaks D-03 as drafted.** Three options:

- **(O1) Plumb `path` instead of `url`.** Change `label-email-in-icontroller.ts` to return `{ before_path, after_path }`; change the tagger to write into new columns `screenshot_before_path` / `screenshot_after_path` (additive migration); `StageScreenshotStrip` passes path to `ScreenshotThumb`. Existing `_url` columns stay populated for back-compat / new rows. **Recommended.** ~30 min work.
- **(O2) Reverse-engineer path from URL.** Supabase signed URLs contain the path in the query string (`/object/sign/automation-screenshots/<path>?token=...`). Brittle; couples UI to Supabase URL format. Skip.
- **(O3) Re-sign at read time using a regex-extracted path.** Same brittleness as O2. Skip.

Volume note: ~7000 historical labeled rows (estimate; see §4) — if O1, the backfill cannot recover historical paths from the stale URL strings without parsing them. Pragmatic: **for historical rows, parse the path out of the stored URL once during the D-07 backfill migration** and write it to the new path columns. New rows fill it natively from then on.

### 1.6 `noise_category` source for D-07 backfill — VERIFIED

There is **no `noise_category` column on `email_pipeline.emails`**. The canonical Stage 1 noise classification lives in:

- **`pipeline_events.decision`** for `stage=1` rows — this is the `finalCategoryKey` (verified `classifier-screen-worker.ts:342`).
- Mirrored in `pipeline_events.decision_details.final_category_key` for the same row (line 351).

Backfill query should JOIN `pipeline_events stage=1` on `email_id`, not look for a column on `emails`:

```sql
SELECT pe1.decision AS noise_category
FROM debtor.email_labels el
JOIN public.pipeline_events pe1
  ON pe1.email_id = el.email_id
 AND pe1.stage = 1
 AND pe1.swarm_type = 'debtor-email'
WHERE el.icontroller_tag_status = 'tagged'
  AND el.labeled_at >= NOW() - INTERVAL '30 days'
```

Pre-Phase 82.2 emails may lack a Stage 1 pipeline_events row (Phase 82.2 was the coverage backfill). For those, `noise_category` is NULL; the Stage 4 row should still be inserted with `decision_details.noise_category = null` and an additional `decision_details.noise_category_unknown = true` marker. Operator can still find the email by sender/subject in the Auto-archived section.

### 3.3 Replay-safety reminders (CLAUDE.md)

- **D-01:** `emit-stage-4-handled` MUST be its own `step.run()`. Do NOT colocate inside `mark-completed` — those are two different DB targets and Inngest replay semantics requires atomic-per-step. Per Phase 70 contract this is the one exception: when dual-writing the legacy table + telemetry, both go in the SAME step. But D-01 has no legacy table to dual-write with (the `automation_runs.status='completed'` already happens in `mark-completed`). Decision tree:
  - Pure Stage 4 emit, no legacy mate → own `step.run("emit-stage-4-handled")` before `mark-completed`.
  - If you want atomicity with `mark-completed` → merge them into one step. Cleaner, fewer replay boundaries. **Recommended.**
- **D-01:** Re Phase 65 lesson — no random IDs generated outside `step.run()`. emit payload uses `emailRow.id` (already-deterministic from earlier step) and `automation_run_id` (from event data) — both safe. The `archived_at` ISO timestamp **must be generated inside the step** (`new Date().toISOString()` outside step would drift on replay).
- **D-07 backfill migration** is a SQL file, not Inngest. No replay concern. Idempotent via `LEFT JOIN ... WHERE pe.id IS NULL` per CONTEXT — keep that. Wrap in `BEGIN; ... COMMIT;` like Phase 82.2 migration did.

## 5. Risks + Landmines

### R1 — D-06 unique index conflict (HIGH)
Already covered §1.2. Without resolution, the D-06 migration either fails to add (existing broader index satisfies the requirement) or silently passes while the broader index does the actual blocking. Either way the worker's `try/catch 23505` works. The **real** risk: when future handlers want to write a second Stage 4 row per email (e.g. `decision='needs_review'`), they'll silently fail to insert. **Mitigation:** add a code-comment in D-01 worker noting the broader-index constraint; carry "Stage 4 multi-row support" as a deferred item; add a Vitest case asserting insert of a second Stage 4 row with a different decision returns 23505 (so the constraint is documented in tests).

### R2 — Stale signed URLs for historical rows (HIGH — operator-visible)
Already covered §1.5. Without O1 + path backfill, **every backfilled auto-archived row in D-07 shows "no screenshots" even though screenshots exist** because the persisted URLs have expired. Operator will think the system is broken. **Mitigation:** O1 + path-extraction backfill is mandatory for D-03 to be useful on historical data.

### R3 — `loadKanbanRows`-derived sections share a single 500-row cap (MEDIUM)
`loadKanbanRows` caps at 500 (`kanban-loader.ts:81`). It's filtered down to two sections client-side. If handler-error + handler-needs-review combined exceeds 500 pending rows, pagination breaks. Today's volume is low — but if it ever spikes, the Auto-archived section won't see it (different loader) but the other two will silently truncate. **Mitigation:** add a `console.warn` (or telemetry) when 500-row cap hits; document in the loader.

### R4 — `swarm_type` literal in classifier-verdict-worker D-01 (LOW)
The worker is registry-driven (no swarm literals). D-01's snippet uses `swarm_type ?? "debtor-email"` — that fallback is fine (same as `classifier-invoice-copy-handler.ts:308`). Just don't hardcode `"debtor-email"` — use the event field.

### R5 — Realtime broadcast for new Stage 4 emits (LOW but operator-visible)
`pipeline_events` is on the `supabase_realtime` publication (Phase 70 migration). When D-01 emits, any Bulk Review tab subscribed to `pipeline_events` will see the row instantly. Stage 4 page currently subscribes to `${swarmType}-kanban` channel via `AutomationRealtimeProvider` (page.tsx:168) — which is `automation_runs`-driven, NOT pipeline_events. **So the Auto-archived section won't live-update unless the planner adds a pipeline_events subscription.** For tomorrow's ship, fine — operator can refresh. Flag as nice-to-have follow-up.

### R6 — Phase 82.7.1 may also touch `pipeline_events` reads in detail-pane (LOW)
82.7.1 is detail-pane UAT followups (per file list in CONTEXT). If it changes the pipeline_events read shape in the detail-pane, D-04 join might merge-conflict. **Mitigation:** D-03 mount is already gated on 82.7.1 ship; D-04 loader work happens after merge — no extra mitigation needed beyond what CONTEXT already locks.

### R7 — `noise_category` from Stage 1 is the LLM/regex pass output, NOT the Outlook label (LOW, semantic)
`pipeline_events.stage=1 decision` is the `finalCategoryKey` from `swarm_noise_categories`. This is what the operator wants ("which noise bucket was it?"). Just confirming: NOT confusing this with the Outlook folder label the email was filed under. They map 1:1 today (the verdict worker uses `category.outlook_label` to do the actual Outlook categorize), but semantically `noise_category` in the Stage 4 row is the **registry key**, not the Outlook string. Document this in the decision_details JSDoc.
