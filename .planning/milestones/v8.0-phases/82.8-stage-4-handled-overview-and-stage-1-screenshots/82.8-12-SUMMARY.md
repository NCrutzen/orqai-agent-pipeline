# 82.8-12 — Source Auto-archived screenshots from automation_runs

## Outcome

Stage 4 page loader (`web/app/(dashboard)/automations/[swarm]/stage-4/page.tsx`)
now reads cleanup-worker screenshots from `public.automation_runs.result.screenshots`
in addition to the intent-tagger paths on `debtor.email_labels`.

Lookup chain implemented:

1. `pipeline_events stage=4` → `email_id` (UUID)
2. `email_pipeline.emails` → `(id, source_id)` mapping (Outlook message_id ↔ email_id)
3. `automation_runs` filter `automation='debtor-email-cleanup' AND result->>message_id IN (sourceIds)`
4. Extract `result.screenshots.before.path` + `result.screenshots.after.path`
5. Merge into `screenshotPathsByEmailId[email_id]` — but only if not already
   populated by the email_labels pass (intent paths take precedence)

## Why this was needed

Phase 82.8 was designed against `debtor.email_labels.screenshot_*_path`
(intent-tagger path). UAT 2026-05-18 surfaced that noise-archived emails
(the Auto-archived section's actual data) go through
`debtor-email-icontroller-cleanup-worker.ts` instead, which writes
screenshots to `automation_runs.result.screenshots`. The two pipelines
write to disjoint tables; the loader needed to read both.

Side discovery: the cleanup-worker has no `dry_run` gate. It runs against
iController live regardless of `debtor.labeling_settings.dry_run`. The 867
cleanup runs in this env are real iController delete operations with real
screenshots, despite SMEBA being officially in dry-run mode for intent
labeling. Worth flagging operationally — the "dry_run" flag has a narrower
scope than its name suggests. Out of scope for this plan to widen the
gate; tracked separately if the operator wants noise-cleanup also gated.

## Non-goals

- Not unifying the two screenshot storage shapes (email_labels columns vs
  automation_runs JSON blob). Both stay; loader normalises at read time.
- Not adding dry-run gate to cleanup-worker. Separate decision.
- Not backfilling email_labels with cleanup-derived paths.

## Refresh expectations

After Vercel deploy (or local dev refresh), clicking an Auto-archived row
whose email_id was processed by cleanup-worker will mount the
`StageScreenshotStrip` below the pipeline with real Before/After thumbs
from `automation-screenshots/debtor-email-cleanup/...`. Rows that came
exclusively from the backfill migration (no cleanup-run history) will
still empty-state because the source data doesn't exist.
