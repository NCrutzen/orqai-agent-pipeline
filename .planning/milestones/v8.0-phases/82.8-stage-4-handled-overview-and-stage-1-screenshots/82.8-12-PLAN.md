# 82.8-12: Source Auto-archived screenshots from automation_runs

## Problem

Phase 82.8 wired the Stage 4 Auto-archived screenshot strip to read
paths from `debtor.email_labels.screenshot_*_path`. That table is the
**intent-labeling** path (`debtor-email-icontroller-tagger.ts`). Auto-
archived emails are the **noise-cleanup** path
(`debtor-email-icontroller-cleanup-worker.ts:146`) — which writes
screenshots to `automation_runs.result.screenshots`, not to email_labels.

Verified 2026-05-18: this env already has 867 `debtor-email-cleanup`
runs with the screenshots blob populated, shape:
```json
{ "before": {"url": "...", "path": "<storage key>"}, "after": {...} }
```

Newest row 13:08 today — cleanup-worker is actively writing screenshots
even in this otherwise dry-run env (the cleanup-worker delete action
runs unconditionally; only the iController-tagger label step is
dry-run-gated).

## Fix

Extend the page-level `screenshotPathsByEmailId` build in
`stage-4/page.tsx` with a 4th lookup against `public.automation_runs`
where `automation = 'debtor-email-cleanup'`, filtered to the
Auto-archived email_id set, and merged into the existing map.

Join chain: `pipeline_events stage=4 .email_id` → `email_pipeline.emails.id`
(already in scope as `emailIds`) → `emails.source_id` → `automation_runs.result->>message_id`.

The cleanup-worker result payload carries `message_id` (Outlook id) but
not `email_id` (UUID), so we do a 2-step lookup: fetch `id, source_id`
from `email_pipeline.emails` for our auto-archived IDs (already done at
line 220-228 of page.tsx), then use the `source_id` set to filter
automation_runs.
