### Index decision — `labeling-flip-cron`'s new group-by

The cron's existing query (verified at `labeling-flip-cron.ts:94-105`):

```ts
admin
  .from("agent_runs")
  .select("human_verdict")
  .eq("swarm_type", "debtor-email-labeling")
  .not("human_verdict", "is", null)
  .filter("context->>icontroller_mailbox_id", "eq", String(mailbox.id))
  .order("verdict_set_at", { ascending: false })
  .limit(50);
```

Adding `.eq("predictor", ...)` does NOT need a dedicated multi-column index. The current `agent_runs_needs_verdict_idx` is already conditioned on `WHERE human_verdict IS NULL`; this query reads the **complement** (`human_verdict IS NOT NULL`). The dominant filter on a per-mailbox slice is the `context->>icontroller_mailbox_id` jsonb path — for which there is no index today either. Adding an index on `predictor` alone would be low-selectivity (2 values + NULL).

**Recommendation:** No new index in Phase 999.8. If the cron starts timing out (today it runs Mon-Fri 06:00 with N≤50 per mailbox so latency is low), a follow-on phase can add `create index agent_runs_swarm_verdict_predictor_idx on public.agent_runs (swarm_type, verdict_set_at desc) where human_verdict is not null and predictor is not null;` — but this is premature for the locked traffic shape. [CITED: PostgreSQL docs on partial indexes; not verified against production EXPLAIN.]

### Pitfall — `swarm_type` reconciliation

Verified at `labeling-flip-cron.ts:97`: the cron filters `.eq("swarm_type", "debtor-email-labeling")`. But the verdict-write at `stage-1/actions.ts:134` writes `swarm_type: parsed.swarm_type` — which for debtor-email is `"debtor-email"`, NOT `"debtor-email-labeling"`. This is a pre-existing inconsistency (not introduced by Phase 999.8) and the cron's Wilson-CI math has historically only seen `debtor-email-labeling` rows written by a separate code path. **Planner must verify** by counting `agent_runs` rows by `swarm_type` and confirming whether the verdict-side `recordVerdict` rows (`swarm_type="debtor-email"`) are visible to the cron or not. ### Current emit-verdict shape (verified at `classifier-screen-worker.ts:312-330`)

```ts

```

### Gate insertion point

The LLM result is already in scope at `classifier-screen-worker.ts:278-285`:
`llmInvoked`, `llmConfidence`, `llmCategoryKey`, `finalCategoryKey`, `agentRunId`. The gate is a single conditional branch around the existing `emit-verdict` step:

```ts
// Pseudocode — to be expanded in PLAN.md
const gateClearsForAutoArchive =
  !llmInvoked ||                          // regex path: always clears
  llmConfidence === "high" ||              // high LLM confidence: clears
  finalCategoryKey === "unknown";          // 'unknown' label-only-skip path

if (gateClearsForAutoArchive) {
  await step.run("emit-verdict", () => /* existing send */);
} else {
  await step.run("emit-requires-review", () =>
    (inngest.send as unknown as SendFn)({
      name: "classifier/screen.requires_review",
      data: {
        automation_run_id,
        swarm_type,
        message_id,
        source_mailbox,
        predicted_category: finalCategoryKey,
        llm_confidence: llmConfidence,
        agent_run_id: agentRunId,
      },
    }),
  );
}
```

### Why this preserves `status=predicted` on `automation_runs`

The `automation_runs` row is created upstream (Stage 0 → Stage 1 seam) with `status="predicted"` by the ingest path. The current verdict-worker (`classifier-verdict-worker.ts:60-67`) flips it to `"pending"` then `"completed"` via its `flip-to-pending` step. If we never fire `classifier/verdict.recorded`, the verdict-worker never runs, and the row remains at `status="predicted"` — which is exactly the value Phase 81's `loadPageData` filters for at `stage-1/page.tsx:587`:

```ts
admin.from("automation_runs").select("id, result")
  .eq("swarm_type", swarmType)
  .eq("status", "predicted");
```

So the row appears in the Stage 1 row list automatically with **zero** changes to `loadPageData` for the medium/low surfacing.

### `pipeline_events` row still emitted

The existing `emit-pipeline-event` step at `classifier-screen-worker.ts:289-310` is **upstream** of the gate and stays unconditional. The Stage 1 prediction event lands in `pipeline_events` regardless of confidence, so the per-email `pipeline_events_email_summary` view at `stage-1/page.tsx:561` still shows the row with `stage_1_decision=<category>` even when the gate held it back.

## §3. D-03 high-conf FP alert — concrete proposal

### Survey of existing alert/drift patterns

The codebase has **no existing calibration-drift alert pattern**. Searched `web/lib/inngest/functions/` and `web/scripts/`:

- `pipeline/budget_breached` event (`events.ts:286-293`) + handler `budget-breach-handler.ts` — closest precedent: an Inngest event triggered by a guard, consumed by a handler that files a Kanban human-review row at `automation_runs.topic='budget_breach'`. [VERIFIED: file read]
- No "drift alert" or "FP rate threshold" event exists.
- No periodic stats job exists today; the only periodic backend job for Stage 1 is `labeling-flip-cron` itself.

### Recommendation: piggy-back on `labeling-flip-cron`

The cron already runs Mon-Fri 06:00 Amsterdam, already aggregates per `(mailbox, predictor)`, already writes audit rows to `classifier_rule_evaluations`. Adding a **second pass** inside the same cron run to compute high-conf FP rate per mailbox is essentially free and reuses the existing rolling-50 window machinery. No new cron, no new event subscriber to maintain in Phase 999.8.

### Proposed threshold + emission shape

| Knob | Proposed value | Rationale |
|------|----------------|-----------|
| Window | Rolling 50 **high-conf LLM** predictions per mailbox | Mirrors `FLIP_N_MIN=50` in `labeling-flip-cron.ts:24` |
| Threshold (warn) | >2% FP rate (i.e. ≥2 wrong out of 50) | "Wrong" = `human_verdict IN ('rejected_other','rejected_*')` or `corrected_category IS NOT NULL` |
| Threshold (alarm) | >5% FP rate (≥3 wrong out of 50) | At 5% the LLM is materially worse than its claimed 95% — actionable signal |
| Emission | New Inngest event `classifier/calibration_drift.detected` (warn or alarm) + audit row to `classifier_rule_evaluations` with `rule_key = "llm_calibration:<mailbox>"` | Audit-row reuse matches existing cron pattern at `labeling-flip-cron.ts:115-122` |
| Subscriber | NONE in Phase 999.8 | Mirror D-10's "no subscriber yet" pattern. Future learning-loop worker can subscribe. Phase 999.8 just emits + audit-logs. |

Sketch of the cron extension (planner expands in PLAN.md):

```ts
// inside evaluateMailbox, after the existing Wilson-CI block:
if (predictor === "llm_2nd_pass") {
  const highConfRows = list.filter(r => r.confidence === "high");
  const highConfFp = highConfRows.filter(r =>
    r.human_verdict?.startsWith("rejected_") ||
    r.corrected_category !== null
  ).length;
  const highConfN = highConfRows.length;
  if (highConfN >= 50 && highConfFp / highConfN > 0.02) {
    await admin.from("classifier_rule_evaluations").insert({
      swarm_type: "debtor-email-labeling",
      rule_key: `llm_calibration:${mailbox.source_mailbox}`,
      n: highConfN,
      agree: highConfN - highConfFp,
      ci_lo: 1 - (highConfFp / highConfN),
      action: highConfFp / highConfN > 0.05 ? "calibration_alarm" : "calibration_warn",
    });
    await inngest.send({ /* classifier/calibration_drift.detected */ });
  }
}
```

```ts
  return "no_change";
}
if (wouldPromote) return "shadow_would_promote";
if (wouldDemote) return "shadow_would_demote";
return "no_change";
```

`n < FLIP_N_MIN` (i.e. <50 verdicts in the window) yields `"no_change"` for both `mutate` and shadow paths. So a cold-start bucket (e.g. a new `(mailbox, predictor='llm_2nd_pass')` slice with only 12 verdicts) naturally yields `no_change` until N reaches 50. **This is already a graceful cold-start floor** — no special handling is required at the per-predictor split.

### Recommendation: keep mutate enabled, rely on N>=50 gate

The cron does **not** need to enter shadow mode during cold-start. The existing N≥50 gate provides exactly the same protection. Promotion/demotion will simply not fire on a sparse bucket; that is the desired behavior.

**Concrete sequencing decision (recommended for planner to lock):**

> Phase 999.8 keeps `LABELING_CRON_MUTATE` at its current value (whatever it is in prod today). The cron's per-predictor split is hard-cut: from the deploy moment forward, `evaluateMailbox` iterates over both `predictor IN ('regex','llm_2nd_pass')` values and runs the existing pickAction logic on each. Buckets below N=50 yield `no_change` and never mutate `labeling_settings.dry_run` — exactly the existing behavior, no special shadow mode required.

### Pitfall — the OLD per-mailbox math is dead post-cutover

Per CONTEXT.md D-09 note: "During cold-start, the *old* per-mailbox math is dead (gone with the cron change)". This means: at cutover, every `(mailbox, predictor)` bucket starts at N=0 (because `predictor IS NOT NULL` filter rejects historical rows). The old per-mailbox aggregate that today drives Wilson-CI is replaced — not augmented. For a mailbox that is currently auto-promoted (`dry_run=false`), the cron will see two empty buckets, conclude `no_change`, and the mailbox stays auto-promoted indefinitely or until the new buckets fill. This is acceptable behavior (no incorrect demotion) but the planner should document it as a known cold-start trade-off.

## Pitfalls

### Pitfall 1 — `swarm_type` mismatch between verdict-write and cron
The verdict-side writes `swarm_type="debtor-email"`; `labeling-flip-cron` filters `swarm_type="debtor-email-labeling"`. **Planner MUST verify** today's row counts by `swarm_type` and decide whether to (a) loosen cron filter to IN-list, or (b) change verdict-write to use the labeling-suffix value, or (c) accept that two different agent_runs populations exist and `predictor` only attaches to the verdict-side one. This is a pre-existing inconsistency unrelated to Phase 999.8 but is fully load-bearing for whether the per-predictor Wilson-CI math actually splits.

### Pitfall 2 — `regex.matchedRule === "no_match"` is NOT a rule_key
When regex abstains, `decision_details.regex` is `{invoked: true, matchedRule: "no_match", category: "unknown"}`. The verdict-side `recordVerdict` MUST read `decision_details.llm_invoked` to decide between `predictor='regex'` and `predictor='llm_2nd_pass'`. Treating `matchedRule==="no_match"` as a regex prediction would mis-attribute every LLM-2nd-pass row as `regex`.

### Pitfall 3 — `pipeline_events_email_summary` view does NOT expose predictor today
The view aggregates per-email Stage 0..4 decisions but does not project `decision_details` JSON paths. If the chip filter goes through the view (current `loadPageData` path), the predictor filter needs either (a) a view migration to add a `predictor` projected column, or (b) the JOIN-back filter pattern already used for `topic`/`entity`/`mailbox` at `page.tsx:478-503`. The JOIN-back path is the lower-risk choice for Phase 999.8.

### Pitfall 4 — `create_agent` vs `update_agent` for Orq calibration
CLAUDE.md learning `cba7352b`: `create_agent` silently drops `model.parameters.response_format`. **Phase 999.8 is an `update_agent` (PATCH) op only** — the agent already exists. PATCH persists correctly. But the planner should still re-supply the FULL `model.parameters` block in the PATCH so that any future `update_agent`-via-Studio overwrites land with a complete payload.

### Pitfall 5 — `automation_runs.status` after the new "requires_review" branch
The `automation_runs` row is created upstream with `status="predicted"`. When the gate skips `emit-verdict`, the verdict-worker never fires its `flip-to-pending` step (`classifier-verdict-worker.ts:60-67`) — so the row stays at `"predicted"`, which is what we want for Stage 1 row-list surfacing. **But** the operator must still be able to issue `recordVerdict` on these rows. Verified: `recordVerdict` at `stage-1/actions.ts:107-118` already updates `automation_runs.status="feedback"` directly — it does NOT depend on the row currently being at any specific status (no precondition guard). So the operator review flow works for both auto-archived rows (today) and gated rows (Phase 999.8) identically.

### Pitfall 6 — Replay-safe id generation
If the new `classifier/screen.requires_review` event ever grows a payload field that's used as a DB key (it currently does not — `agent_run_id` is generated inside `step.run("llm-call")` per the existing pattern at `classifier-screen-worker.ts:185`), the new id must be generated INSIDE the `step.run("emit-requires-review")` block. Phase 65 replay-id learning applies (CLAUDE.md).

### Pitfall 7 — Cold-start gives the impression of a stuck cron
Post-deploy, the audit table `classifier_rule_evaluations` will start logging rows with `n < 50` and `action="no_change"` for every `(mailbox, predictor)` bucket. To an operator monitoring the cron this looks like the cron broke. Add a comment in the cron audit row payload making the cold-start visible: `action: "no_change_cold_start"` or include a `reason` jsonb field. Avoids a false-positive incident report in the days after cutover.

### Pitfall 8 — UI sequencing
Per CONTEXT.md `**Depends on:**` line, the UI half (D-05 chips, D-08 row chip) MUST sequence after Phase 81. Backend half (gate D-01, attribution D-07, migration D-09) is independent. If the UI half lands before Phase 81 closes, the chip strip changes will be made on the old `/review/` directory and re-broken by the Phase 81 rename. **Planner: lock Phase 81 status before Wave 5.**

### Pitfall 9 — `email_id` vs `automation_run_id` confusion in `recordVerdict`
At `stage-1/actions.ts:136`, the existing code writes `email_id: parsed.automation_run_id` — i.e. the email_id column literally stores the automation_run_id value. This is a pre-existing bug/quirk. Adding the `predictor` lookup requires a REAL email_id to query `pipeline_events`. Planner should plumb the actual email_id through `VerdictInput` rather than relying on the broken `email_id===automation_run_id` aliasing.
