# Phase 999.8 Plan 06: Wave 4 — labeling-flip-cron per-predictor Wilson-CI split + D-03 calibration drift signal

Splits the Wilson-CI promotion/demotion math by predictor so LLM 2nd-pass mistakes don't drag the regex stream's accuracy down (and vice versa). Same cron, same N>=50 floor, two streams per mailbox. Adds the D-03 high-conf LLM FP calibration-drift signal as a second pass inside the same cron run.

### 2. Per-predictor Wilson-CI split + D-03 alert in `labeling-flip-cron`

File: [web/lib/inngest/functions/labeling-flip-cron.ts](../../../../web/lib/inngest/functions/labeling-flip-cron.ts)

**Restructure:** Introduced a private `evaluatePredictorStream(admin, mailbox, predictor, mutate)` helper. The public `evaluateMailbox` keeps its 3-arg signature and now iterates over `PREDICTORS = ["regex", "llm_2nd_pass"] as const`, invoking the helper twice per mailbox.

**Per-stream query shape:**

```typescript
admin
  .from("agent_runs")
  .select("human_verdict, predictor, confidence, corrected_category")
  .eq("swarm_type", "debtor-email")
  .eq("predictor", predictor)
  .not("predictor", "is", null)          // D-09 forward-only
  .not("human_verdict", "is", null)
  .filter("context->>icontroller_mailbox_id", "eq", String(mailbox.icontroller_mailbox_id))
  .order("verdict_set_at", { ascending: false })
  .limit(FLIP_N_MIN);
```

**Audit-row write per stream:**

```typescript
await admin.from("classifier_rule_evaluations").insert({
  swarm_type: "debtor-email",                                  // reconciled (Plan 02 deferral)
  rule_key: `mailbox_flip:${mailbox.source_mailbox}:${predictor}`, // two streams per mailbox
  n, agree, ci_lo, action,
  reason: n < FLIP_N_MIN ? { cold_start: true } : null,        // RESEARCH Pitfall 7
});
```

**D-03 calibration drift check (llm_2nd_pass stream only):**

```typescript
if (predictor === "llm_2nd_pass") {
  const highConfRows = list.filter((r) => r.confidence === "high");
  const highConfN = highConfRows.length;
  if (highConfN < FLIP_N_MIN) return;  // cold-start, no D-03 output

  const highConfFp = highConfRows.filter(
    (r) =>
      (r.human_verdict ?? "").startsWith("rejected_") ||
      r.corrected_category != null,
  ).length;
  const fpRate = highConfFp / highConfN;
  if (fpRate < CALIBRATION_WARN_FP_RATE) return;

  const tier: "warn" | "alarm" =
    fpRate > CALIBRATION_ALARM_FP_RATE ? "alarm" : "warn";

  // audit row (both tiers)
  await admin.from("classifier_rule_evaluations").insert({
    swarm_type: "debtor-email",
    rule_key: `llm_calibration:${mailbox.source_mailbox}`,
    n: highConfN,
    agree: highConfN - highConfFp,
    ci_lo: 1 - fpRate,
    action: `calibration_${tier}`,
  });

  // event (alarm tier only)
  if (tier === "alarm") {
    const detectedAt = new Date().toISOString();
    await (inngest.send as unknown as SendFn)({
      name: "classifier/calibration_drift.detected",
      data: { /* see event declaration */ },
    });
  }
}
```

**Thresholds (planner-locked):**

| Tier   | FP-rate gate | Audit row                          | Event |
| ------ | ------------ | ---------------------------------- | ----- |
| silent | <= 2%        | none                               | no    |
| warn   | > 2%, <= 5%  | `action='calibration_warn'`        | no    |
| alarm  | > 5%         | `action='calibration_alarm'`       | YES   |

Strict `>` on both gates so 5.00% lands in `warn`, 2.00% lands silent — matches plan wording.

**Audit-row swarm_type reconciliation:** The audit row insert at the old line 122-129 wrote `swarm_type: "debtor-email-labeling"`. Plan 02 left this as Plan 06's call — Plan 06 aligns it to `"debtor-email"` so the cron's read filter and write filter use the same label end-to-end.

**Replay safety:** `detected_at` is generated inside `evaluatePredictorStream`, which runs inside the parent handler's `step.run("eval-<mailbox>", ...)`. The step.run boundary is the parent; no additional step.run is needed inside evaluateMailbox (which would complicate the 3-arg test signature). Phase 65 replay-id rule preserved.

**inngest.send pattern:** Inline `(inngest.send as unknown as SendFn)({...})` cast; never destructured (CLAUDE.md learning `dae6276`).

## Deviations from Plan

- **[Rule 4 → resolved inline] D-03 confidence filter — string vs numeric.** The plan's `<action>` block sketched `list.filter(r => r.confidence >= 0.85)` (numeric). The Plan 01 RED test fixture uses `confidence: "high"` as a STRING. Plan 05 (recordVerdict) writes the verdict-side `agent_runs` row from `pipeline_events.decision_details.llm_confidence` which is the string label. The numeric variant in the plan was a research-§3 inheritance — the test contract wins. Switched to `r.confidence === "high"` string equality. No user check needed: this is Rule 1 (bug — would fail tests) caught at write time.

- **[Rule 3 — internal refactor] Split into `evaluatePredictorStream` helper.** The plan said "restructure evaluateMailbox". Implementing the entire two-stream loop inside one function got ugly and duplicated the side-effect blocks. Extracted a private helper so each stream is a single linear call. `evaluateMailbox`'s public signature (3 args) is unchanged so the Plan 01 RED tests bind cleanly.

- **[design choice — documented]** D-03 alarm-tier strict `>` on 5% threshold. The plan uses ambiguous language ("> 5% → alarm" but also "On n_high_conf >= 50 AND fp_rate > 0.05"). Settled on strict `>` to match the textual ">5%" — at exactly 5.00% FP, the row counts as warn.

- **[Rule 2 — observability]** Added `reason: { cold_start: true }` JSONB hint to the audit-row insert when n<FLIP_N_MIN. The plan called for "a cold-start marker"; this is the cheapest queryable form. Existing `classifier_rule_evaluations.reason jsonb` column accepts arbitrary metadata.

No architectural deviations. No authentication gates encountered.
