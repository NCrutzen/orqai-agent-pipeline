# Phase 82.1 Plan 04: Stage 1 override picker port Summary

JWT-style atomic port of the Stage 1 override picker logic from the deleted `stage-1-override-pane.tsx` into a new shell-local `_shell/components/stage-1-widget.tsx` cell widget, with a renamed `extrasBelowPipeline` slot for the tagging-failure artifacts.

## What shipped

- **New widget** `_shell/components/stage-1-widget.tsx` (~330 LOC). Carries the category dropdown (re-uses the Phase 71-04 `Stage1CategorySelect`), notes textarea, `EvalTypeRadio`, `OverrideConfirmDialog` mount, `fetch('/api/automations/debtor-email/override', POST)` axis payload, `recordVerdict` fallback, optimistic-removal via `useSelection().markPendingRemoval`, and the eval-type / submit / discard keyboard events. The widget owns ONLY the stage_1 axis flow — multi-axis aggregation stays on `UnifiedDetailPane`.
- **UnifiedDetailPane** (`_shell/detail-pane.tsx`) — import flipped to the new widget. `taggingFailuresSection` slot renamed to `extrasBelowPipeline` (renders below the 5-cell PipelineFlow). New optional `predictedRow?: PredictedRow | null` prop so the Stage 1 cell receives the full `PredictedRow` shape it needs for `email_id` / `automation_run_id` / `result.message_id` args. `onMarkDirty` lifted above `stagesData` `useMemo` so the Stage 1 cell's `onChange` can fire it.
- **Stage1ClientShell** (`stage-1/client-shell.tsx`) — drops the `Stage1OverridePane` mount, derives `selectedPredictedRow` from `predictedRows`, passes new `TaggingArtifactsSection` via `extrasBelowPipeline` and the `predictedRow` to `UnifiedDetailPane`.
- **New helper** `stage-1/components/tagging-artifacts-section.tsx` — extracted from the old override pane's tagging-failure surface.
- **Stage4ClientShell** — followed the slot rename (`taggingFailuresSection` → `extrasBelowPipeline`; semantics unchanged) so Stage 4's HandlerErrorWidget still renders below the pipeline cells.
- **Deletions** — `stage-1-override-pane.tsx` (~1276 LOC) gone; `tests/queue/detail-pane.test.tsx` (Phase 61-02 legacy test that imported the deleted component) gone.
- **New tests** `stage-1/__tests__/stage-1-widget.test.tsx` — 4 tests, all green:
  1. Category dropdown renders with current value pre-selected.
  2. Selecting a new category + clicking Submit dispatches POST `/api/automations/debtor-email/override` with `axis: "stage_1_category"`.
  3. After a successful override POST, `useSelection().markPendingRemoval(row.id)` fires.
  4. Hard-sep grep gate inline assertion: the source file contains no `swarm_intents` / `SwarmIntentRow` references.
