# Phase 56.7-03: Generic Queue Route Summary

**One-liner:** Replaced the debtor-email-specific queue UI with a dynamic-segment route at `/automations/[swarm]/review` that resolves swarm config (tree levels, override categories, drawer fields) from the registry shipped in 56.7-01 — adding a new swarm is now a single `swarms` row INSERT.

## What shipped

1. **Move + genericize** (10 files). The entire `debtor-email-review/` directory was relocated to `[swarm]/review/`. Page-level data loader signature became `loadPageData(params, admin, swarmType)`, and every call to Supabase that previously carried the `'debtor-email'` literal now threads `swarmType` from `await params`. The page calls `notFound()` when `loadSwarm(admin, swarmType)` returns null/disabled. The override-category dropdown is built from `loadSwarmCategories(admin, swarmType)` (filtered by `action !== 'reject'`), removing the need for a per-swarm static enum.

2. **`next.config.ts` redirects**. The legacy URL `/automations/debtor-email-review` (and any sub-path) now 307s to `/automations/debtor-email/review` so operator bookmarks don't break. The `permanent: false` flag is intentional — D-07 calls for the flip to 308 in a follow-up commit after one deploy cycle.

3. **Tests rewritten** for the new contracts. `page.test.tsx` asserts that `loadPageData` threads `swarmType` into both the `classifier_queue_counts` RPC AND the `automation_runs` filter, with a separate test for `'sales-email'` to prove the literal isn't sneaking in anywhere. `actions.test.ts` mocks `loadSwarmCategories` so post-validation can be exercised — `'payment'` succeeds (in registry) while `'bogus-not-seeded'` throws. The `'sales-email-review'` broadcast channel is asserted end-to-end.

4. **60-05 PLAN amended**. Frontmatter `depends_on` now includes 56.7; all `files_modified` paths point at `[swarm]/review/`; the body has a prominent `<phase_56_7_dependency_note>` block explaining the registry-driven shape. The `VerdictInput` interface in the plan's `<interfaces>` was updated to match the shipped 56.7-03 signature (swarm_type required, free-form override_category).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Test imports across 5 additional files migrated**
- **Found during:** Task 1 (Step H sweep)
- **Issue:** Plan's `files_modified` lists only `web/tests/queue/page.test.tsx` and `web/tests/queue/actions.test.ts`, but the directory move broke imports in 5 other queue test files (`detail-pane.test.tsx`, `race-cohort.test.tsx`, `rule-filter.test.tsx`, `keyboard-shortcuts.test.tsx`, `fetch-review-email-body.test.ts`). Without the path migration the entire test suite fails to even compile.
- **Fix:** Mechanical `sed` of `@/app/(dashboard)/automations/debtor-email-review/` -> `@/app/(dashboard)/automations/[swarm]/review/` in all 5. Plus `loadPageData` callers updated to pass the new third arg `"debtor-email"`. Plus `RaceCohortBanner` test usages now pass `swarmType="debtor-email"`. Plus `detail-pane.test.tsx` got a thin `DetailPane` wrapper that injects test-default `swarmType` / `categories` / `drawerFields` props so existing assertions stay terse.
- **Files modified:** 5 test files plus the wrapper helper in detail-pane.test.tsx
- **Commit:** `96398ae` (Task 1)

**2. [Rule 2 - Critical] Type coercion for `override_category` in Inngest event payload**
- **Found during:** typecheck after Task 1
- **Issue:** Schema accepts `string | null`, Inngest event type is `override_category?: string`. Passing `null` would fail TS.
- **Fix:** `parsed.override_category ?? undefined` at the event-send call site.
- **Files modified:** `web/app/(dashboard)/automations/[swarm]/review/actions.ts` line 163
- **Commit:** `96398ae` (Task 1)

**3. [Rule 3 - Blocking] `categories.ts` retained as type-only sibling**
- **Found during:** Task 1 design
- **Issue:** The plan instructs "categories.ts is deleted, types come from `@/lib/swarms/types`". But `actions.ts` is a `"use server"` file — Next 15 / Turbopack codegen emits a `module.exports.X = X` for every export, including type exports, causing ReferenceError at module-eval time (per a 60-06 hotfix comment in the original code). The two helper types (`VerdictInput`, `ReviewEmailBodyResult`) need a non-`use server` host file.
- **Fix:** Stripped `categories.ts` of the `OVERRIDE_CATEGORIES` const + `OverrideCategory` derived type (those are exactly what 56.7 replaces with the registry). Kept the two helper types. The OVERRIDE_CATEGORIES const is gone — registry IS the source of truth.
- **Files modified:** `web/app/(dashboard)/automations/[swarm]/review/categories.ts` (new, slim)
- **Commit:** `96398ae` (Task 1)
