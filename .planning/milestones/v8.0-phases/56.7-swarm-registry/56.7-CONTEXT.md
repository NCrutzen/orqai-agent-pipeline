# Phase 56.7: Swarm Registry — Context

**Gathered:** 2026-04-29
**Status:** Ready for planning
**Source:** spawned from Phase 56 architectural review on review-screen reusability

<domain>
## Phase Boundary

Promote the queue-review pattern from "debtor-email-specific" to a swarm-agnostic system so any new automation/agent-swarm (Sales email next, then Planning, Order Entry, …) plugs in via DB rows instead of new code/routes.

Two new tables (`public.swarms`, `public.swarm_categories`) drive a generic review surface, generic verdict-worker dispatch, and self-onboarding into the existing `/automations/classifier-rules` dashboard.

**Cross-swarm by design (continuation of Phase 60 D-00).** Phase 60 already keyed every classifier table by `swarm_type`; Phase 56.7 finishes the job for the queue UI + verdict actions.

**Explicit out-of-scope:** building any new swarm's actual rules/categories beyond migrating the existing debtor-email set; reorganizing `automation_runs` (already swarm_type-keyed); replacing the Inngest event-trigger model.

</domain>

<decisions>
## Implementation Decisions

### Schema (architectural anchor)

- **D-00:** Two tables in `public.`:
  - `swarms (swarm_type pk, display_name, description, review_route, source_table, enabled, ui_config jsonb, side_effects jsonb, created_at, updated_at)`
  - `swarm_categories (swarm_type fk, category_key, display_label, outlook_label nullable, action enum, swarm_dispatch nullable, display_order, enabled, primary key (swarm_type, category_key))`
- **D-01:** `swarm_categories.action` is a **strict CHECK enum** with values `('categorize_archive','reject','manual_review','swarm_dispatch')`. Adding a new action kind requires a code change + DB migration — explicit choice for type-safety + exhaustive switch checks. Free-form jsonb rejected.
- **D-02:** `swarm_categories.swarm_dispatch` is the Inngest event name fired when `action='swarm_dispatch'` (e.g. `orq.copy_document` for Phase 62 invoice-copy automation). Ignored otherwise.
- **D-03:** `swarms.ui_config` is **jsonb** (not typed columns). Shape: `{ tree_levels: string[], row_columns: [{key, label, width}], drawer_fields: string[], default_sort: 'created_at desc' }`. Adding a new swarm with a different column layout = JSON edit, no migration.
- **D-04:** RLS: `swarms` + `swarm_categories` enable RLS. Service-role full access. Authenticated read-only (so dashboards can render without admin-client per request).
- **D-05:** Both tables seeded with one row each in the migration: `swarm_type='debtor-email'` plus 6 category rows (matching today's `OVERRIDE_CATEGORIES` + `CATEGORY_LABEL`). Phase 56's existing review page becomes the first consumer of the registry on the same deploy.

### Caching

- **D-06:** Mirror Phase 60's classifier_rules cache: **60s in-memory TTL, keyed by swarm_type**. New module `web/lib/swarms/registry.ts` exports `loadSwarm(swarmType)` and `loadSwarmCategories(swarmType)`. Cron + verdict-worker + UI all read through this layer.

### Generic queue UI

- **D-07:** Route: `web/app/(dashboard)/automations/[swarm]/review/page.tsx`. Catches all swarm_types via dynamic segment. Old `/automations/debtor-email-review` becomes a redirect (`/automations/debtor-email/review`) for backward-compat; phased out after one deploy cycle. Operator bookmarks updated.
- **D-08:** Tree-nav structure (`ui_config.tree_levels`): debtor-email uses `['topic','entity','mailbox_id']`. Sales swarm could use `['topic','entity']` (no mailbox). Generic `<QueueTree>` component reads the array, builds dynamic nesting. Reuses Phase 60's tree drawer.
- **D-09:** Counts RPC stays cross-swarm: `classifier_queue_counts(p_swarm_type text)` already accepts the swarm_type parameter. No changes.

### Verdict-worker generalization

- **D-10:** Single `classifier-verdict-worker` Inngest function. On approve: load `swarm_categories` row by `(event.swarm_type, finalCategoryKey)`. Switch on `action`:
  - `categorize_archive` → existing flow (Outlook label + archive)
  - `reject` → mark complete, no side-effects
  - `manual_review` → no side-effects, leave row in `feedback`
  - `swarm_dispatch` → fire Inngest event named in `swarm_dispatch` column (Phase 62 hook)
- **D-11:** `outlook_label` nullable: when null, the worker does NOT call Outlook even if `action='categorize_archive'` (graceful for non-Outlook-sourced swarms). Explicit-not-null is operator's job in the seed.
- **D-12:** Side-effects beyond Outlook (iController delete, etc.) stay code-resident for now — `swarms.side_effects jsonb` reserved for declarative future expansion (Phase 56.8+). Today: code-coupled per swarm via the dispatch event.

### Adding a new swarm (operational contract)

- **D-13:** Adding Sales swarm = (a) `INSERT swarms` row, (b) `INSERT swarm_categories` rows, (c) one new Inngest function listening on the swarm-dispatch event for any non-trivial actions. **No new route, no new component, no Vercel deploy for routing.**
- **D-14:** Self-onboarding: `/automations/classifier-rules` dashboard already cross-swarm; adding rows to `classifier_rules` with new `swarm_type` auto-appears. Adding `swarms` row enables the new `/automations/{swarm}/review` page.

### Migration ordering

- **D-15:** **Additive-first, ship 56.7 BEFORE Phase 56 Wave 2/3 (60-05 queue UI rewrite).** The Phase 56 plan is amended so the queue rewrite is implemented against the registry from day 1, not refactored after the fact. Concretely: 56-05's PLAN.md gets a frontmatter dependency on 56.7 completion; 56-05's component code uses `loadSwarm('debtor-email')` from the registry rather than hardcoded `OVERRIDE_CATEGORIES`.

### Naming + scope-fence

- **D-16:** `swarms.swarm_type` is the canonical key. Existing `classifier_rules.swarm_type`, `agent_runs.swarm_type`, `automation_runs.swarm_type` reference it logically (no FK enforced; foreign keys to small lookup tables add lock contention without enforcement value).
- **D-17:** Phase 56.7 ships ONE seeded swarm (`debtor-email`). Sales swarm provisioning (its rows + dispatch function + Inngest worker) is the Sales swarm phase's job — Phase 56.7 just makes that work a 30-line INSERT instead of a multi-day refactor.

### Claude's Discretion

- Generic component naming (`<QueueTree>`, `<QueueRowList>`, `<QueueDrawer>`) — planner picks file boundaries.
- Whether to inline-cache via module-scope Map or use a Supabase Realtime invalidation (probably module-scope for Phase 56.7; Realtime is overkill until the registry mutates faster than 60s).
- Migration path for the redirect at `/automations/debtor-email-review` (Next.js redirect config vs route stub).
- Test scaffolding: extend existing vitest queue tests OR new directory `web/tests/swarm-registry`.

</decisions>

## Existing Code Insights

### Reusable Assets

- **Phase 60 cross-swarm schema** — every relevant table is already keyed by `swarm_type`. Phase 56.7 only adds `swarms` + `swarm_categories`; doesn't touch existing schema.
- **Classifier registry pattern** (`zapier_tools` from 56-01b) — proven precedent for "INSERT one row to enable a thing". Phase 56.7 directly mirrors the loader/cache approach.
- **Phase 60 cron** — already iterates all swarms via the telemetry view; no changes needed.
- **`/automations/classifier-rules` dashboard** — already swarm-agnostic; new swarms appear automatically.
- **Existing tree-drawer + counts RPC** — both already accept `swarm_type` parameter; just need to be wired through `[swarm]` segment.

### Established Patterns

- 60s in-memory cache (`classifier_rules`, `zapier_tools`) — same TTL applied here.
- Service-role writes, authenticated reads (RLS posture).
- jsonb config columns for "shape that changes per consumer" (mirrors `classifier_rules.input_schema`).
- Strict CHECK enums for actions (mirrors `classifier_rule_evaluations.action`, `classifier_rules.status`).

</code_context>

<specifics>
## Specific Ideas

- The first seeded `swarms` row should set `display_name='Debtor Email'`, `review_route='/automations/[swarm]/review'`, `ui_config.tree_levels=['topic','entity','mailbox_id']`.
- The 6 seeded `swarm_categories` rows match today's OVERRIDE_CATEGORIES (post-`invoice_copy_request` add): payment, auto_reply, ooo_temporary, ooo_permanent, invoice_copy_request, unknown. Outlook labels match today's CATEGORY_LABEL.
- Phase 56's existing 60-05 plan needs amending to depend on 56.7 — the queue UI rewrite consumes the registry instead of hardcoded categories.
- Sales-swarm Phase (whenever it happens) ships its own swarm/category INSERTs + a `sales-email/dispatch.run` Inngest worker for any custom side-effects. Zero changes to swarms-registry code expected.

</specifics>

<deferred>
## Deferred Ideas

- **Declarative side_effects beyond Outlook** — `swarms.side_effects jsonb` reserved field; today the verdict-worker hardcodes Outlook + iController. Phase 56.8 (or whenever a third swarm needs different actions) generalizes.
- **Foreign-key enforcement** between `agent_runs.swarm_type` → `swarms.swarm_type` — adds lock contention; defer until orphan-row-cleanup becomes a real problem.
- **Realtime registry invalidation** — module-scope cache + 60s TTL is enough at current scale. Add Supabase channel listener if the registry grows or mutates frequently.
- **Tooling**: a CLI / admin UI to add a swarm. INSERT works fine for now; UI is premature.
- **Sales swarm itself** — its own phase, gated on actual Sales-team scoping work.
