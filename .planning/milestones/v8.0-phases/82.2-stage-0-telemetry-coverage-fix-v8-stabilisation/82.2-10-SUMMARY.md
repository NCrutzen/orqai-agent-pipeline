# Phase 82.2 Plan 10: Stage 0 Coverage Probe Summary

Daily Inngest cron at `TZ=Europe/Amsterdam 0 9 * * 1-5` measures trailing-24h Stage 0 coverage per active mailbox via the Plan 03 `stage0_coverage_24h` RPC and writes one row per (probe_run, mailbox) to `public.pipeline_health`; rows breach when `stage1_count > 0 AND coverage < 0.99`.

## Behaviour

1. **Trigger** — `{ cron: "TZ=Europe/Amsterdam 0 9 * * 1-5" }` — every weekday at 09:00 Amsterdam.
2. **resolve-run-id** step mints `probe_run_id = crypto.randomUUID()` (Phase 65 replay-safety).
3. **measure-coverage** step loops the 5 active mailboxes and calls `admin.rpc('stage0_coverage_24h', { mailbox_arg, swarm_arg })`. Per-mailbox `coverage = stage1 === 0 ? 1 : stage0/stage1`; `breached = stage1 > 0 && coverage < 0.99`.
4. **write-pipeline-health** step bulk-INSERTs 5 rows. DB recomputes `coverage_pct` and `breached` as generated columns (migration 20260513b); supplied values are advisory for the function return value/tests.
5. **Return** — `{ probe_run_id, mailboxes: 5, breaches: N }`.

## Decisions

- **Risk #8 / 6th-mailbox.** Hardcoded list with explicit TODO referencing `swarms_labeling_settings`. Defer the registry migration until a 6th mailbox actually lands.

- **CLAUDE.md Phase 65 rule.** `probe_run_id` minted inside `step.run("resolve-run-id", …)` so replays group their 5 rows under a single id rather than fanning out to 5 different ids per replay.
