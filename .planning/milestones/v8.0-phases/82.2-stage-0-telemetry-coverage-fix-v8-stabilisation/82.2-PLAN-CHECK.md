### BLOCKER 1 — Research Gate: RESEARCH.md open questions not formally resolved

**Dimension:** Research Resolution (Dimension 11)
**Severity:** BLOCKER
**Plan:** Phase-level (affects Plans 06, 07)

RESEARCH.md ends with `### Open Questions (planner must resolve)` — the section heading does NOT carry the `(RESOLVED)` suffix required for planning to proceed. The three questions are:

1. Should D-01 scope be expanded to cover the debtor-email ingest gated-fire bug? — **The plans answer YES** (Plans 06+07 implement D-A). This resolution is implicit in the plan structure but not formally recorded in RESEARCH.md.
2. STATE.md file-write vs pipeline_health table? — **Plans answer: table** (Plan 10 explicitly documents this). Again, implicit.
3. Worker-level vs ingest-level D-06 test? — **Plans answer: worker-level** (Plan 05). Implicit.

The concern is not that the decisions were made wrong — they are all correct. The concern is that the RESEARCH.md `## Open Questions` section remains open, which means the section heading rule (Dimension 11) is violated. Any future reader or automated checker would correctly flag this as unresolved.

**Fix:** The planner should append the `(RESOLVED)` suffix to the section heading and add a brief inline resolution for each of the three questions, referencing the plan numbers that implement each answer. This is a 3-line edit to RESEARCH.md, not a re-planning exercise.

### BLOCKER 2 — Plan 06: Stage 1 pipeline_events emit ownership is unresolved

**Dimension:** Task Completeness / Key Links
**Severity:** BLOCKER
**Plan:** 82.2-06, Task 2

Plan 06 Task 2 Step 7 says: "Stage 1 pipeline_events emit — NOTE: today's ingest writes Stage 1 pipeline_events at L347–362. The Stage 1 worker today already emits its own pipeline_events row at the end of its noise-filter pass (verify by reading classifier-screen-worker.ts L1–end). Decision: which is canonical?"

This is an unresolved branch inside the plan's action description — the executor is asked to decide at runtime whether to add or remove a Stage 1 emit. This is not executable as written. Two bad outcomes:

- Executor adds a duplicate Stage 1 emit to the worker when one already exists → double-write Stage 1 rows post-deploy, breaking TELE-COV-01's "exactly one pipeline_events row" guarantee.
- Executor leaves the existing worker emit and also leaves the ingest route emit → Plan 07's deletion of ingest Stage 1 emit could remove the only canonical emit, silently dropping Stage 1 coverage.

The executor cannot safely make this architectural call mid-execution. The planner must inspect `classifier-screen-worker.ts` now and resolve this before Plan 06 executes.

**Fix:** Planner reads `classifier-screen-worker.ts` and determines whether the Stage 1 worker today emits `pipeline_events`. One of two outcomes:
- (a) Worker already emits Stage 1 pipeline_events: Plan 06 Task 2 Step 7 should say "existing emit is canonical; drop Plan 07 ingest-side emit in Task 2." Plan 07 Task 2 must explicitly include removal of the L347–362 ingest emit.
- (b) Worker does NOT emit Stage 1 pipeline_events: Plan 06 Task 2 Step 7 must add the emit with a specific payload shape (mirroring L347–362 of the ingest route). Plan 07 Task 2 must still remove the ingest-side emit.

Either way, the plan must state the resolved answer and give the executor a concrete action, not a runtime question.

### HIGH — Plan 07: Stage 0 placeholder automation_runs insert may conflict with Phase 64 SAFE-01..03 pattern

**Dimension:** Key Links
**Severity:** HIGH (WARNING)
**Plan:** 82.2-07, Task 2

CONTEXT.md calls out that Phase 64 introduced the pattern: ingest route pre-creates `automation_runs` row with `stage='stage_0_safety_pending'` (SAFE-01..03). Plan 07 preserves this: it explicitly creates the Stage 0 placeholder row and captures `stage0RunId` before firing `stage-0/email.received`.

However, Plan 06 Task 2 Step 4 says "DROP the `stage_0_safety_pending` placeholder write — that placeholder is now created by the Plan 07 thin ingest BEFORE Stage 0 runs." This correctly resolves the ordering. But Plan 06 does NOT verify whether the Stage 0 worker's UPDATE-by-id semantics (L217–252 in the worker) depend on finding this placeholder row. If the worker UPDATE path expects to find the placeholder row created by the ingest, and Plan 06 removes the alternative placeholder-creation path (the `stage_0_safety_pending` write inside the `r.category === "unknown"` branch), then post-deploy the worker will always find the placeholder because Plan 07 creates it unconditionally. This is actually correct, but:

**The gap:** Plan 06 does not explicitly verify that the Stage 0 worker's UPDATE path does not also do a fallback INSERT if the placeholder is missing. If it does, Plans 06+07 need a step confirming the INSERT-fresh vs UPDATE-existing semantics remain consistent. Plan 04 (which refactors the worker) doesn't audit this either — it only fixes the emit sites, not the automation_runs write logic.

**Fix (MEDIUM priority):** Plan 07 Task 2 or Plan 04 Task 3 should grep-verify the worker's UPDATE/INSERT branch logic (`step.run("persist-verdict")` lines L217–252) and document which path fires post-Plan-07. A one-line addition to Plan 04's Task 3 audit is sufficient.

### MEDIUM — Plan 05 depends on Plan 04 but plan 05 is Wave 2 and depends on Plan 01 only

**Dimension:** Dependency Correctness
**Severity:** WARNING
**Plans:** 82.2-05

Plan 05 frontmatter: `depends_on: [82.2-01]`, Wave 2.

Plan 05's objective says: "a NEW, more comprehensive test that verifies the FULL pipeline_events row shape on the **post-refactor (Plan 04) single-emit path**, including the new `decision_details.emit_source='main-path'` field."

Plan 05's Test 2 explicitly asserts `decision_details.emit_source='main-path'`. This field only exists after Plan 04's `VerdictSource`-based refactor. If Plan 05 runs concurrently with Plan 04 (both are Wave 2), the test would fail RED against the current worker — which is correct for TDD's RED phase. But the GREEN state requires Plan 04 to have landed first.

Plan 05's Task 1 `<done>` says "All assertions pass after Plan 04 worker refactor is in place." This is an implicit dependency that is not reflected in `depends_on`. In wave-parallel execution, Plan 05's GREEN verification step would race against Plan 04.

**Fix:** Change Plan 05's `depends_on` to `[82.2-01, 82.2-04]` and either bump it to Wave 3 (after Plan 04 is complete) or document that Plan 05 Task 1 runs after Plan 04 closes.

Alternatively: keep Plan 05 in Wave 2 but change the `<done>` gate to say "RED verified; GREEN deferred to after Plan 04 lands."

### MEDIUM — Plan 08: smoke gate tests only "normal email" path; whitelist-match path is conditional

**Dimension:** Scope Sanity / Test Coverage
**Severity:** WARNING
**Plan:** 82.2-08

Plan 08 Task 2's "Expected per mailbox" section requires "at least 2 automation_runs rows" and "at least 2 pipeline_events rows" (stage=0, stage=1). This is the happy path for non-whitelisted emails.

Plan 08 Task 3 (operator checkpoint) asks the operator to "confirm at least one whitelist-match debtor email also passed through Stage 0 (i.e. the D-A bug is genuinely closed)." This is the right check, but it depends on the test emails happening to match the whitelist. The plan offers no fallback if no whitelist-match email arrives naturally during the 60-second window.

The D-A bug (whitelisted emails bypassed Stage 0) is the dominant root cause of the 26–45% gap. The smoke gate should explicitly test a whitelist-match email — not hope one arrives organically.

**Fix:** Plan 08 Task 2 should include an explicit sub-step: "Send one synthetic email that matches a known whitelist rule (operator selects a subject/sender that the whitelist is known to match based on Plan 02 diagnostic output). Verify this email also shows stage=0 in pipeline_events." This converts a conditional check into a deterministic gate.

### LOW — Plan 10: probe_run_id resolution inside step.run is in the interfaces spec but `stamp` is gone

**Dimension:** CLAUDE.md Compliance (replay-safety)
**Severity:** LOW (INFO)
**Plan:** 82.2-10

The RESEARCH.md pseudocode for the coverage probe had `const stamp = await step.run("resolve-stamp", () => new Date().toISOString())` to capture the run timestamp. Plan 10's interfaces spec replaces this with `probe_run_id = await step.run("resolve-run-id", () => crypto.randomUUID())` (Phase 65 pattern). This is correct.

However, Plan 10's `step.run("write-pipeline-health")` bulk INSERT uses `recorded_at` with a DB default (`now()`). Per RESEARCH §Critical replay-safety note: "The DB Default is `now()` so the INSERT must specify the column." For the probe's `recorded_at`, DB default is fine because probe rows are not being time-corrected (unlike backfill rows which need the historical timestamp). This is intentionally different from the backfill case and is correct.

No fix required; flagged here for executor awareness to avoid applying the backfill's `created_at` pattern to the probe.
