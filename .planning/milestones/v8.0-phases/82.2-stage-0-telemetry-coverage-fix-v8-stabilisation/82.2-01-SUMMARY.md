## Operator escalation

The partial UNIQUE index `pipeline_events_one_per_stage_email` cannot be created until the 266 existing duplicate groups are resolved. The operator must decide:

1. **Dedupe first** — collapse each duplicate group to the canonical row (e.g. `MIN(created_at)` or `MIN(id)`), then re-run Plan 82.2-01.
2. **Scope the index to stage=0 only** — change the partial WHERE clause to `WHERE email_id IS NOT NULL AND stage = 0`, which would still cover 41 conflicting groups at stage 0 (still requires dedupe of those 41).
3. **Defer the partial UNIQUE index entirely** — drop the index requirement; D-01 worker refactor and D-03 backfill rely on application-level idempotency instead.

Plans 82.2-04 (worker single-emit) and 82.2-09 (backfill) depend on this index to safely use `ON CONFLICT` semantics — they cannot proceed until the operator picks a path.

The note above this section in CONTEXT.md says scope is "telemetry — the safety verdict itself is correct (no operational incidents), only the `emitPipelineEvent` call is missing on certain branches." The 41 stage-0 dupes confirm the worker is currently capable of emitting duplicates (consistent with the Phase 64 SAFE-01..03 pre-create + worker emit pattern). The 155 stage-1, 35 stage-2, 35 stage-3 dupes are out of this plan's stated scope but the operator should be aware they exist.
