### Task 2 — GREEN dispatch block (commit `acd7634`)

### Task 3 — Idempotency + double-execution audit

Findings:

1. **Outlook categorize replay-safety**: `step.run("categorize", ...)` dedupes per (run_id, step_name). Additionally the `idempotency-precheck` step runs FIRST and short-circuits when `msg.categories` intersects `DEBTOR_MR_LABELS` — so a Zapier re-delivery of the same `message_id` cannot double-tag. (T-82.2-06-01 mitigated.)

2. **Outlook archive replay-safety**: `archiveEmail` PATCHes the message into the Archive folder. Re-running the move on an already-archived message returns success (no-op) — Graph silently accepts moves to the current folder. With `retries:0` on the worker (line 114) and step.run caching, the archive call fires at most once per event delivery.

3. **`debtor-email-cleanup` queue dedupe**: NO unique constraint exists on `automation_runs(result->>'message_id') WHERE automation='debtor-email-cleanup' AND status IN ('pending','completed')`. With `retries:0` + the cleanup dispatcher's `concurrency:{limit:1}` (`debtor-email-icontroller-cleanup-worker.ts:60`), practical double-queue risk is bounded to a manual re-trigger of the same email through the ingest pipeline. **Decision**: do NOT add the index in this plan. The cleanup-worker's first action is to flip the row to `status='pending'` with `processed_by` (line 94-101), so concurrent picks self-serialize per row. **Follow-up flagged**: partial UNIQUE index on `(result->>'message_id', automation)` to harden against operator-driven double-trigger — defer to a hardening plan in a later phase. (T-82.2-06-02 mitigated by accept-with-rationale.)

4. **Single producer**: `grep -rn "classifier/screen.requested" web/lib/inngest/functions/ web/app/` confirms only `stage-0-safety-worker.ts` emits this event (two emit sites: happy-path forward + operator override). No direct emitter in any API route or other worker. (T-82.2-06-03 closed.)

5. **Cross-swarm leak guard**: the new dispatch block is gated on `swarmRow.stage1_regex_module === '@/lib/debtor-email/classify'`. Sales-email's `stage1_regex_module` is `null` (per Phase 74 D-03), so a sales-email event falls through to the existing Step 5 verdict path with zero risk of triggering debtor-email's Outlook helpers. (T-82.2-06-04 mitigated; registry-driven check is stronger than the originally-planned literal because adding a new swarm requires explicit module registration in `STAGE1_REGEX_MODULES`.)

6. **Retries**: `{ id: "classifier/screen-worker", retries: 0 }` — same Phase 65 pattern as the rest of the Stage workers. Worst-case retry behavior: none. A failed dispatch surfaces as either a `failed` audit row (for Outlook errors with their own `step.run("write-*-failure")`) or an Inngest run-level failure with no DB row.

## Deviations from Plan

**1. [Rule 3 — Architectural correction within plan scope] Gate dispatch on `swarmRow.stage1_regex_module` instead of `swarm_type === 'debtor-email'`**
- **Found during**: Task 2 GREEN run — REQ-6 static-grep test ("zero `swarm_type === 'X'` literal branches") failed because the plan's task 2 step 1 explicitly directed using `swarm_type === 'debtor-email'`.
- **Resolution**: switched to `swarmRow.stage1_regex_module === '@/lib/debtor-email/classify'`. This is registry-driven (Phase 74 D-04 pattern), semantically equivalent (sales-email has `stage1_regex_module=null`), and stronger because onboarding a new auto-action swarm requires explicit module registration in `STAGE1_REGEX_MODULES` — no risk of a typo-mismatched swarm_type silently bypassing the gate.
- **Plan-text note**: the plan's `<interfaces>` block explicitly mentions REQ-6 in the existing worker; the swarm_type literal in Task 2 step 1 is an inconsistency with REQ-6 that I resolved in REQ-6's favor.
- **Files modified**: `classifier-screen-worker.ts`. Commit: `acd7634`.

**2. [Rule 2 — Missing critical functionality, additive] Failure-path automation_runs writes for categorize/archive errors**
- **Found during**: Task 2 GREEN. The pre-deletion ingest route (L478-505, L508-535) wrote `automation_runs.status='failed'` rows for Outlook errors before returning. I preserved that behavior in the reconstructed dispatch via `step.run("write-categorize-failure")` and `step.run("write-archive-failure")`. Without these, replay-loss would leave failed Outlook calls invisible.
- **Files modified**: `classifier-screen-worker.ts`. Commit: `acd7634`.
