# Phase 82.6: Footer Approve → recordVerdict wiring - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning
**Depends on:** 82.5

<domain>
## Phase Boundary

Wire the Phase 82.5 footer **"✓ Approve (Stages X+Y)"** to actually advance Stage 1 rows out of the queue.

Today the footer posts `verdict='confirm'` to `/api/automations/debtor-email/feedback` per ok stage — a Phase 82.4 learning/observation log. It does NOT call `recordVerdict` (Phase 56.7-03), which is the action that:
1. Flips `automation_runs.status: predicted → feedback`
2. Writes `agent_runs` telemetry
3. Fires the `verdict-recorded` Inngest event → registry-driven `classifier-verdict-worker` performs the swarm dispatch
4. Emits broadcast → queue UI invalidates → row disappears

**Operator symptom verified 2026-05-15 (Browserless probe on commit 7112c78):** Click "✓ Approve" → 2 POSTs to `/api/automations/debtor-email/feedback` succeed (200 OK, rows written) → row stays visible in Stage 1 list.

**Scope: Stage 1 only.** Stage 0/2/3/4 progression flows are explicitly out of scope (see Deferred Ideas).

</domain>

<decisions>
## Implementation Decisions

### Wiring approach

- **D-01:** New server-action wrapper `approvePrediction(row_id, decision)`. Takes minimal args (row_id + decision), looks up `automation_run_id` + `message_id` + `source_mailbox` + `rule_key` + `predicted_category` server-side, then calls `recordVerdict`. Detail-pane calls a single function from `handlePrimary`. Cleanest decoupling — keeps the shared `_shell/detail-pane.tsx` free of Stage-1-specific fetch logic. Mirrors the email_feedback POST shape (`row_id, decision/verdict`).

  **Implementation detail (planner decides):** wrapper location is either `stage-1/actions.ts` (extends the existing module) or `_shell/_lib/approve-prediction.ts` (shared shell util). Planner choice.

### Per-stage scope

- **D-02:** **Stage 1 only.** Footer Approve on Stage 0/2/3/4 is out of scope for this phase.
- **D-02b:** **Hide the footer Approve button** on Stage 0/2/3/4. Operator only sees Approve where it does something. Reject + Skip still show. Stage 3 keeps its own Phase 80 dispatcher path via inline widget actions — not the footer.
- **D-02c:** Stages 2 (confidence gate + learning loop) and Stage 3 (approve/reject with notes) are owned by **V9.0 Promotion Recommender + Learning Inbox** synthesis layer + Phase 82.4 capture infra (already shipped). Do not duplicate that work here.

### Failure UX

- **D-03:** Match existing override-widget pattern — **non-fatal silent recovery on `recordVerdict` failure.** Precedent: `stage-1-widget.tsx:188-191`. Feedback row is durable; status flip will reconcile on next server roundtrip via the realtime broadcast. No toast on failure (consistency with the override path's existing behavior).

### Removal UX

- **D-04:** **Optimistic local removal via `pendingRemovalIds`.** Add `row.id` to `pendingRemovalIds` (selection-context) on approve click → row vanishes instantly. Existing precedent: `row-list.tsx:38` already filters by `pendingRemovalIds.size`. If `recordVerdict` somehow rolls back the status the next server roundtrip restores. Best perceived performance for the operator.

### Claude's Discretion

- Wrapper file location (`stage-1/actions.ts` vs `_shell/_lib/approve-prediction.ts`) — planner picks based on existing import boundaries.
- Hide-vs-disable visual for the Stage 0/2/3/4 footer (D-02b) — planner picks based on `<footer>` layout impact.
- Toast on success — none specified; planner can decide whether silent removal needs any confirmation feedback (precedent in `stage-1-widget.tsx:193-197` toasts on override success, so a `toast.success("Approved")` would be consistent).
- Ordering inside `handlePrimary`: feedback POST then approvePrediction in parallel, or sequential? Either is fine; recordVerdict failure is non-fatal either way per D-03.

</decisions>

- **Single integration site:** `detail-pane.tsx:438-465` `handlePrimary`. The approve branch (`!anyDirty` path) gets an additional `approvePrediction` call per ok stage, plus a `pendingRemovalIds` write. No other component touched.
