# Phase 82.3: Per-stage audit surface (v8.0 stabilisation)

**Gathered:** 2026-05-12 (rescoped after image-1/image-2 review)
**Status:** Stub — needs `/gsd-discuss-phase 82.3` before planning
**Type:** INSERTED stabilisation phase — debtor-person prerequisite, blocks Phase 82.4 prose-notes capture

<domain>
## Phase Boundary

Add **read-only detail** to each stage entry in the existing Bulk Review detail-pane sidebar so the operator can see *why* a stage produced its verdict before deciding whether to override or comment on it.

The override controls (dropdowns, `override stage` links, Approve/Reject/Skip) **already exist** in the sidebar — verified from production screenshots 2026-05-12. 82.3 does NOT touch them. 82.3 adds an expandable info area next to each existing stage entry that surfaces the underlying evidence.

Scope is plumbing: the data already exists in `pipeline_events.decision_details`, `agent_runs.context`, `agent_runs.tool_outputs`, and `automation_runs.result`. Screenshots are in Supabase Storage bucket `automation-screenshots`. Phase 82.3 surfaces them.

</domain>

<rationale>
## Why this lands in v8.0 closure, not V9.0

V9.0's whole premise is *operator gives prose feedback on what the pipeline did*. The existing sidebar shows **verdicts** (dropdown values like `auto_reply`, `credit_request`) but not **evidence**: which regex matched, what the LLM JSON said, which resolver layer hit, what screenshots were captured. Without the evidence, operator feedback in 82.4 is "I think Stage 2 was wrong" without specificity, and V9.0's clusterer drowns in low-signal data.

The 2026-05-11 smoke-test note asked for this:
> *"For emails that got auto-handled, user wants to be able to drill into the details of what each stage did — including screenshots taken by the browser automation."*

</rationale>

<existing_surface>
## What already exists in the detail-pane sidebar (verified from screenshots 2026-05-12)

Per stage entry, the sidebar already shows:
- Stage number badge (0–4 circle)
- Stage name + `stage_N_*` tag chip
- `Current: <verdict>` line (e.g., `Current: auto_reply`, `Current: unresolved`)
- Status line (`Looks correct` ✓ green / `override to set` / `Stage didn't run`)
- Override dropdown (when applicable, e.g., Stage 1 category, Stage 3 intent)
- `override stage` link on every stage
- "Show full email" button at top of pane
- Approve / Reject / Skip buttons at bottom

82.3 layers on top of this — it does NOT redesign or replace.

</existing_surface>

<scope>
## In scope (80/20 cut)

Add a **details expander** to each Stage 0–3 entry in the sidebar. Click reveals an in-place panel (or hover-on-detail popover — pick during discuss) showing the verdict + reasoning summary + evidence + screenshots for that stage. Default closed; click to expand. Operator gets the evidence without leaving the row context.

### Per-stage payload to render

| Stage | Data sources (already in DB) | Renders inside the details expander |
|---|---|---|
| **0 Safety** | `pipeline_events.decision_details` | Regex patterns triggered, LLM injection verdict + reasoning, budget headroom remaining |
| **1 Noise** | `pipeline_events.decision_details` + `agent_runs.tool_outputs` (Phase 999.8 predictor attribution) | Which rule fired (regex Pass-1 OR LLM Pass-2), confidence label (high/medium/low), predictor source chip |
| **2 Customer** | `agent_runs.context` (resolver layer that matched: thread / sender / identifier / unresolved) + `automation_runs.result` (iController screenshots from Phase 67 DOM tagging) | Identifier source, confidence, candidate list (top 3), iController before/after screenshots via signed URLs |
| **3 Intent** | `agent_runs.tool_outputs` (ranked-intent JSON from `debtor-intent-agent` Phase 65) | Ranked intent list with confidences, coordinator one-line reasoning, why-selected summary |

**Stage 4 is OUT of scope** — confirmed by operator. v8.0 feedback covers Stages 0–3 only; Stage 4 detail surfacing waits for V10.0 / V11.0 when handlers themselves become first-class concerns.

### Screenshots

- Supabase Storage bucket: `automation-screenshots`, path `{automation}/{label}-{timestamp}.png`
- 1h signed URL expiry — **refresh on every expander open** (1 Storage API call per click; cheap; avoids 401 retries)
- Reachable via JSONB introspection on `automation_runs.result` and `agent_runs.tool_outputs` — no new column needed

### Data-quality fix (display layer)

The 2026-05-11 smoke flagged `sender_name = "Planning"` / `subject = "(no subject)"` rows on Stages 1 and 3. **Confirmed-by-user as a row-mapper display bug, NOT an ingest persistence bug.** Fix lives in the per-row component that feeds the audit surface (likely in `web/lib/automations/swarm-bridge/configs.ts` or the Bulk Review row mapper). No data backfill needed.

<must_haves>
## Acceptance criteria

- Each Stage 0–3 entry in the sidebar has a details expander (closed by default).
- Expanding renders within 200ms for cached data; ≤500ms when fetching a fresh signed URL for screenshots.
- For an email with complete `pipeline_events` coverage, each expanded stage shows:
  - Verdict + 1–3 line reasoning summary
  - Evidence chips/badges where structured (regex match label, confidence chip, identifier source)
  - Screenshots (when present) via fresh signed URL — no broken images, no 401s on first open
  - "Show raw JSON" sub-toggle reveals the source `decision_details` / `tool_outputs` JSON for the 20% case
- Existing override dropdowns + Approve/Reject/Skip behaviour unchanged.
- Stage 4 entry shows existing minimal state ("Stage didn't run — override to set"); no expander added in v8.0.
- Sender / subject metadata renders correctly on every row across every Bulk Review tab — the `Planning` / `(no subject)` display bug fixed.
- Works for `debtor-email` swarm (Stages 0–3 all populated for most rows after 82.2 ships).
- Works for `sales-email` swarm where data exists (Stage 0 + Stage 1 expand; Stage 2/3 show "not yet implemented for this swarm" placeholder — confirms V10.0 is the work).

</must_haves>
