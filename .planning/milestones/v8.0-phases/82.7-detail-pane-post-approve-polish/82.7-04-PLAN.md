<objective>
Land D-01 (auto-advance after Approve). The operator should be able to approve 10 rows in succession without clicking on the list — each Approve auto-advances to the next visible row, or shows the empty state if the list is exhausted.

Two coordinated changes:

1. **`selection-context.tsx`:** Add `markPendingRemovalAndAdvance(id: string, nextId: string | null)` — an atomic helper that adds `id` to `pendingRemovalIds` AND calls `setSelected(nextId)` in one render cycle. Existing `markPendingRemoval` is preserved (Plan 03 / Plan 04 of Phase 82.6 still use it on the non-Approve paths).

2. **`detail-pane.tsx`:** Accept a new prop `visibleRowIds: string[]` reflecting the operator's filtered visible list (Stage 1 page already computes this for `row-list.tsx`; thread it). In `handlePrimary` on the Approve branch (`!anyDirty`), compute `nextId` by finding the position of `row.id` in `visibleRowIds`, advancing one index, and falling back to `null` if at the end. Call `markPendingRemovalAndAdvance(row.id, nextId)` instead of `markPendingRemoval(row.id)`. Override-branch path keeps the existing `markPendingRemoval(row.id)` call (no advance).

If the caller (Stage 1 page or `client-shell.tsx`) doesn't already have the visible-list array readily available as the same value passed to `row-list.tsx`, thread it via a parent prop. The PATTERNS.md note: `row-list.tsx` L40-43 is the canonical visible filter — the page passes the SAME list to detail-pane (don't recompute, don't filter twice).

Purpose: Operator's #1 UX defect from 2026-05-16 UAT — "Side panel stays on the email you see it go on the left."

Output: Two modified files (selection-context, detail-pane). One caller wiring update on Stage 1 page (file determined during execution — `stage-1/page.tsx` or `stage-1/client-shell.tsx` per PATTERNS.md note on visible-list ownership). NO server, NO DB, NO Inngest changes.
