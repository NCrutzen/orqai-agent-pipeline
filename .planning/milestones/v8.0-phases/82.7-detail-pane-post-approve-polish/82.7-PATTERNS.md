### `detail-pane.tsx` — D-01 (auto-advance after Approve)

**Analog (in-file):** existing `markPendingRemoval(row.id)` call at L510 inside `handlePrimary`. The new code goes immediately AFTER it.

**Current site to extend** (L505-511):
```typescript
// D-04: Optimistic removal regardless of approvePrediction outcome.
// selection-context.tsx auto-trims this set when the server's next
// fetch omits the id (see selection-context.tsx:66-79), so a failed
// recordVerdict that left the row present will naturally restore
// visibility on the next refresh.
markPendingRemoval(row.id);
```

**What planner must add:**
1. `detail-pane.tsx` already receives `selectedId` indirectly via `useSelection()` (line 237 — currently destructures only `markPendingRemoval`). Extend to also pull `setSelected`.
2. `detail-pane.tsx` does NOT currently receive the visible-list ordering as a prop. **Critical gap:** the visible list is owned by `row-list.tsx`, which filters by `pendingRemovalIds` (L40-43). Either:
   - **Option A (recommended per CONTEXT D-01):** Add a `visibleRowIds: string[]` prop to `UnifiedDetailPaneProps`. The Stage 1 page (`stage-1/page.tsx` or `stage-1/client-shell.tsx`) already computes the visible list (same filter chain that feeds `row-list.tsx`); thread it through. Compute `nextId` in `handlePrimary` BEFORE the `markPendingRemoval` call (the call mutates `pendingRemovalIds` which would shift the index).
   - **Option B:** Add the helper to `selection-context.tsx` as `markPendingRemovalAndAdvance(currentId, nextId)` that does both setSelectedId + add-to-pending atomically. Caller still computes nextId.

**Pattern for atomic context update** — extend `selection-context.tsx` (matches the existing `useCallback` + `setPendingRemovalIds` shape at L90-97):
```typescript
const markPendingRemovalAndAdvance = useCallback(
  (id: string, nextId: string | null) => {
    setPendingRemovalIds((prev) => {
      if (prev.has(id)) return prev;
      const next = new Set(prev);
      next.add(id);
      return next;
    });
    // setSelected also updates URL via history.replaceState (see L81-88)
    setSelected(nextId);
  },
  [setSelected],
);
```

**Note on filtering ownership:** `row-list.tsx` L40-43 is the canonical visible filter. The page must compute `nextId` using the SAME filter chain (active stage tab + mailbox + needs-action). Per CONTEXT.md D-01, this is the operator's filtered view, not the raw `rows` prop.
