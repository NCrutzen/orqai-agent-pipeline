# Phase 82.7 Plan 04: Auto-Advance After Approve Summary

Operator can now Approve 10 rows in succession on Stage 1 without ever touching the left-rail list — each Approve click silently advances the detail pane to the next visible row, or shows the empty/idle state when the list is exhausted. Closes D-01, the #1 UX defect from the 2026-05-16 operator UAT ("Side panel stays on the email you see it go on the left").

## What Was Built

### Task 1 — `markPendingRemovalAndAdvance` helper (commit `74e2bb5`)

`web/app/(dashboard)/automations/[swarm]/_shell/selection-context.tsx`:

- Added new `useCallback`:
  ```typescript
  const markPendingRemovalAndAdvance = useCallback(
    (id: string, nextId: string | null) => {
      setPendingRemovalIds((prev) => {
        if (prev.has(id)) return prev;
        const next = new Set(prev);
        next.add(id);
        return next;
      });
      setSelected(nextId);
    },
    [setSelected],
  );
  ```
- Extended `SelectionContextValue` type with the new method signature.
- Extended the context `value` object + memo deps.
- Existing `markPendingRemoval` is untouched and still exposed (preserved for non-Approve callers).

### Task 2 — Wire visibleRowIds + nextId computation + advance call (commit `593eb78`)

`web/app/(dashboard)/automations/[swarm]/_shell/detail-pane.tsx`:

- `UnifiedDetailPaneProps` gains `visibleRowIds?: string[]` (optional; defaults to `[]` at the use site).
- Both the outer `UnifiedDetailPane` wrapper and the inner `DetailPaneInner` destructure the new prop and thread it through.
- `useSelection()` destructure replaced: `const { markPendingRemovalAndAdvance } = useSelection();` (dropped the bare `markPendingRemoval` — was the only call site).
- `handlePrimary` Approve branch (`!anyDirty`, after the existing `approvePrediction` try/catch):
  ```typescript
  const ids = visibleRowIds ?? [];
  const idx = ids.indexOf(row.id);
  const nextId =
    idx >= 0 && idx + 1 < ids.length ? ids[idx + 1] : null;
  markPendingRemovalAndAdvance(row.id, nextId);
  ```
- Override branch (`anyDirty === true`) is unchanged — it dispatches the `bulk-review:override-submit` window event and returns early. No auto-advance there (operator verification per plan must-have).

`web/app/(dashboard)/automations/[swarm]/stage-1/client-shell.tsx`:

- `<UnifiedDetailPane>` mount now passes `visibleRowIds={visibleUnified.map((r) => r.id)}` — the same operator-filtered array (post-mailbox filter) that drives `<RowList>` above.

The 2026-05-16 operator UAT recorded four UX defects in the Phase 82.6 detail pane:

| D-ID | Defect | Resolved in |

The operator had to click the next row in the list to continue. Plan 04 closes that loop: Approve → next row auto-appears.

## Deviations from Plan

### Deviation 1 (Rule 3 — defensive) — `visibleRowIds` made OPTIONAL instead of required

**Found during:** Task 2, when grepping for all `<UnifiedDetailPane` call sites.

**Issue:** Plan recommended `visibleRowIds: string[]` (required) with caller updates in step 4-5. Reality: 9 call sites exist — Stage 1 client-shell (1), Stage 3 client-shell (1), Stage 4 client-shell (1), option-z-detail-pane (1), and 6 test mounts. Making it required would require pure-ceremony updates to 8 non-Stage-1 callers passing `visibleRowIds={[]}` — none of those surfaces reach the Approve branch (Phase 82.6 D-02b hides the footer outside Stage 1).

**Fix:** Made the prop optional with `?? []` fallback at the use site inside `handlePrimary`. Plan explicitly allowed this fallback ("If TypeScript strict mode complains about callers that don't yet pass it, add `?:` then handle `undefined` with `?? []` at the use site"). Behavior is identical at runtime: empty array → `indexOf(row.id)` returns `-1` → `nextId = null` → selection clears, which is the correct semantic for surfaces where Approve is unreachable.

**Files modified:** detail-pane.tsx (one extra line of `const ids = visibleRowIds ?? [];`).

**Commit:** 593eb78 (rolled into Task 2's main commit).

### Deviation 2 (Rule 2 — correctness) — Dropped `markPendingRemoval` from detail-pane's destructure

**Found during:** Task 2, when replacing the Approve-branch call.

**Issue:** Plan said "extend destructure: `const { markPendingRemoval, markPendingRemovalAndAdvance } = useSelection();`". But after the Approve-branch replacement, `markPendingRemoval` had ZERO call sites in detail-pane.tsx — it would be flagged as an unused destructured variable under strict-mode `noUnusedLocals` (which the project has enabled). The Override branch never called it directly (it dispatches a window event for stage widgets to handle).

**Fix:** Removed `markPendingRemoval` from the destructure entirely. Only `markPendingRemovalAndAdvance` is pulled in. The context still exports `markPendingRemoval` for any future caller.

**Files modified:** detail-pane.tsx.

**Commit:** 593eb78.
