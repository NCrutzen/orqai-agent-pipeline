# Phase 82.7 Plan 02: Detail-Pane Label Fix + Footer Cancel Override Summary

Single-file batched edit in `_shell/detail-pane.tsx`: removed the duplicate `✓ ` prefix from the primary button label (D-02), and added a footer-level `Cancel override` escape hatch plus a per-stage `onCancelDirty` callback threaded into PipelineFlow (D-03), giving Plan 03 the receiving end of the prop chain.

## What Was Built

- **D-02 — label de-dup:** All three `primaryLabel` template literals (`Approve (Stages X+Y)`, `Submit override (Stage N)`, `Submit overrides (Stages X, Y)`) now start with the action verb. The lucide `<Check />` icon inside the Button JSX is the surviving single checkmark.
- **D-03 footer half:**
  - `Undo2` added to the lucide-react import (alphabetical position).
  - `resetAllStageFeedback` useCallback declared (sibling to `onMarkDirty`) — body `setDirty({})`, empty deps.
  - Footer JSX gains a `<Button variant="outline" data-testid="detail-pane-cancel-override">` rendered conditionally on `anyDirty`, inside the existing `activeStage === 1` ternary (wrapped in a React fragment alongside the primary Button). Click clears the entire dirty map; no network call.
- **D-03 per-stage half (prep for Plan 03):**
  - `onCancelDirty(stageN)` useCallback declared (sibling to `onMarkDirty`) — body omits the `stageN` key from the dirty map.
  - Threaded into `<PipelineFlow onCancelDirty={onCancelDirty} ... />` with a single `@ts-expect-error` directive (PipelineFlowProps does not declare the prop yet; Plan 03 extends the interface and removes the directive in the same edit).

## What Was NOT Touched

- `handlePrimary`, `approvePrediction`, `markPendingRemoval`, `dispatchAction` — all dispatch logic untouched.
- `dirty` state shape and its `useEffect` reset on `row.id` change — preserved.
- Reject / Skip footer buttons.
- Hard-separation invariant: zero `swarm_intents` references, zero `swarm_noise_categories` references, zero new Inngest events, zero new DB writes.
- Stage-step.tsx and pipeline-flow.tsx — Plan 03 lands those edits.

## Deviations from Plan

None — plan executed exactly as written. The `@ts-expect-error` directive was used as the plan explicitly allowed (PipelineFlowProps lacks `onCancelDirty` until Plan 03).
