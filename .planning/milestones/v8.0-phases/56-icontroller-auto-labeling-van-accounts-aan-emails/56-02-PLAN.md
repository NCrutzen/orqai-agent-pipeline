**A) Generate `NXT_ZAPIER_WEBHOOK_SECRET`** — operator runs `openssl rand -hex 32` locally; record it in 1Password / Bitwarden under "MR Automations / NXT Zapier Webhook".

**B) Build the Zap** in Zapier:

1. Trigger: **Webhooks by Zapier — Catch Hook** (POST). Capture the resulting URL — this is `NXT_ZAPIER_WEBHOOK_URL`.
2. Action 1: **Code by Zapier (Python or JavaScript)** — validate `Authorization` header equals `Bearer <secret>`. If not, throw to short-circuit. Use Zapier's "Storage by Zapier" or environment to hold the secret without committing it to the Zap config that's exported.
3. Action 2: **Paths (Switch on `lookup_kind`)** — three branches:
   - `sender_to_account` → NXT SQL `SELECT ... FROM contactperson WHERE email = {{from_email}}`. Use `nxt_database` field to switch the SQL connection (per-brand routing — D-07). If the contactperson schema differs per brand (per Task 1 output), each branch hits its own NXT connection and Zapier's per-brand SQL templates.
   - `identifier_to_account` → NXT SQL `SELECT ... FROM <invoice_table> WHERE invoice_number IN ({{invoice_numbers}})` joined back to customer table.
   - `candidate_details` → NXT SQL fanout: for each `customer_account_ids[]`, fetch `{customer_name, contactperson_name, recent_invoices (last 5), last_interaction}`.
4. Action 3: **Formatter** — normalize each branch's response to the `NxtZapResponse` shape: `{ matches: [ { customer_account_id, customer_name, contactperson_name?, invoice_numbers?, recent_invoices?, last_interaction?, confidence_signal? } ] }`.
5. Action 4: **Webhooks by Zapier — Custom Response** — return JSON 200 with the normalized body.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Vercel → Zapier (NXT-Zap) | New outbound transport; Bearer-secret leakage = NXT read access |
| Zapier → NXT databases | Whitelisted IP; per-brand connections |
| operator → Zapier dashboard | Zap config visible to all org admins |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-56-02-01 | Spoofing | Anyone with the URL POSTs requests | mitigate | Zap Action 1 validates `Authorization: Bearer <secret>`; reject otherwise (smoke-tested in Task 2 step E) |
| T-56-02-02 | Information Disclosure | Bearer secret leaks via logs | mitigate | Vercel env vars only; NEVER committed; `nxt-zap-client.ts` does not log Authorization header |
| T-56-02-03 | Tampering | Cross-brand data leak (sender resolves to wrong DB) | mitigate | Pitfall 5 — Task 2 step D includes explicit cross-brand sanity test (Smeba sender against Berki DB → 0 matches) |
| T-56-02-04 | Replay | Same sender lookup replayed | accept | Idempotent read; replay produces same result; no side-effects in Zap branches |
| T-56-02-05 | DoS | Many concurrent NXT lookups slow Zapier | accept | Volume bounded by inbound email rate (~50/day per mailbox); Zapier task limit handles this |
</threat_model>
