## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator → MCP `apply_migration` | Operator authorizes each migration call; SQL contents reviewable before paste. |
| Migration → live DB | Atomic transaction; PL/pgSQL guard provides idempotency. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-69-05 | Tampering | Wrong migration applied (operator paste error) | mitigate | Operator-gated checkpoint with file path read aloud before paste; post-apply verification queries cross-check shape against expected. |
| T-69-06 | Denial of Service | Half-applied migration (PL/pgSQL block fails mid-update) | accept | `do $$ ... end$$` block is atomic; failure rolls back. Subsequent apply re-runs the entire block (idempotency guard handles re-entry). |
| T-69-07 | Repudiation | Migration log diverges from in-tree file | mitigate | Supabase MCP records migration name; in-tree file is PR-reviewed; `name` parameter to `apply_migration` matches filename for traceability. |
| T-69-08 | Information Disclosure | Service-role key exposure during MCP call | accept | MCP runs server-side under operator credentials; key not echoed to chat. |
</threat_model>
