# Phase 999.7: Stage 0 budget-breach failures on long emails — Context

**Gathered:** 2026-05-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Stop the Stage 0 safety classifier from rejecting (and stranding) emails whose total LLM token usage exceeds the current 5000-token ceiling. Today such emails are marked `failed` with `error_message: "budget breach: token_count NNNN > 5000"` and filed as a `budget_breach` Kanban row, never advancing past Stage 0.

In scope:
- The Stage 0 budget ceiling and where it lives.
- Email-body input shaping at Stage 0 ingest (before the LLM call).
- Documentation of the runaway-loop-guard intent vs the long-email guard intent.

Out of scope (deferred):
- Pre-flight prompt-token estimation.
- Splitting prompt vs completion token budgets.
- Per-run iteration counters.
- Automated replay/backfill of already-stranded rows.

</domain>

<decisions>
## Implementation Decisions

### Approach
- **D-01:** Hybrid approach — raise the token ceiling AND strip quoted reply history at ingest before the Stage 0 LLM call. Neither half on its own; we want both cost discipline and tolerance for the realistic long-email tail.

### Ceiling
- **D-02:** Set `BUDGET_CEILING_TOKENS = 16000` (was 5000). Covers the observed 12358 / 12362-token samples (2026-05-07 14:46) with ~30% headroom. Still tight enough to fire on a real runaway loop, which would blow far past 16k.
- **D-03:** `BUDGET_CEILING_CENTS = 15` is unchanged. It remains the runaway-loop guard. The token ceiling's role is now "long-email guard," and this role split MUST be reflected in the comments on both constants in `web/lib/stage-0/budget-counter.ts` so future readers don't undo it.

### Input shaping (truncation)
- **D-04:** Strip quoted reply history at Stage 0 ingest, BEFORE the LLM call. Targets the standard email reply blocks: `>` quoted lines, `On <date> <person> wrote:`-style dividers, Outlook/Gmail reply separators. This is the bulk of the bloat in long debtor threads.
- **D-05:** Do NOT strip signatures or legal disclaimers in this phase. Smaller payoff and higher risk of stripping content that is genuinely part of the reply.
- **D-06:** Do NOT add a hard tail-truncate safety net. If a stripped body still exceeds 16k tokens, the budget check still fires and the row goes to `budget_breach` — that is acceptable because it's already an extreme outlier and the existing failure path is clean (see D-09).

### Budget-check timing
- **D-07:** Keep the budget check post-hoc, after the LLM call, against `total_tokens` from the Orq response — exactly as today. No pre-flight prompt-token estimation. Reasoning: with 16k + quoted-history strip, breach becomes rare, the wasted-token cost on a breach is bounded, and adding a second tokenizer would diverge from Orq's count.

### Runaway-loop guard
- **D-08:** No new loop-guard mechanism is added. `BUDGET_CEILING_CENTS = 15` already independently fires on runaway cost regardless of token count. The token ceiling stays a long-email safeguard. This MUST be documented in `budget-counter.ts` constant comments.

### Backfill
- **D-09:** No automated replay of already-stranded `budget_breach` rows. The `budget-breach-handler` already files a Kanban human-review row with `topic = 'budget_breach'`; operators handle those via Bulk Review. Confirmed during scout that the rejection path emits a clean failure status — these rows are NOT a source of stale `pending` rows (which closes ROADMAP open question #2).

### Claude's Discretion
- Exact regexes / parser approach for quoted-history stripping (e.g. existing libs like `email-reply-parser`, `mailparser`'s reply detection, vs hand-rolled regex). Researcher should evaluate.
- Test fixture coverage (Outlook NL, Gmail NL, mobile clients, threaded vs non-threaded).
- Whether the strip helper lives as `web/lib/stage-0/strip-quoted-history.ts` or under `web/lib/automations/debtor-email/` — choose based on whether other swarms need it. Stage 0 is cross-swarm, so default to `web/lib/stage-0/`.
- Migration of the Phase 64 D-14 test assertion in `web/lib/stage-0/__tests__/budget-counter.test.ts` (currently expects `BUDGET_CEILING_TOKENS === 5000`).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `BUDGET_CEILING_TOKENS` / `BUDGET_CEILING_CENTS` constants and `check(state)` — the only edit point for the ceiling change. Single-line constant change is sufficient for D-02.
- `pipeline/budget_breached` event + handler — already produces clean `failed` rows and `budget_breach` Kanban rows; nothing to change in the failure path.

### Established Patterns
- Stage 0 worker uses Inngest `step.run("check-budget", ...)` — any new ingest-shaping step should follow the same pattern (deterministic, side-effect-free, replay-safe).
- Phase 65 replay-id learning (CLAUDE.md) — applies if any new IDs are minted in a step; current scope doesn't introduce new IDs.

### Integration Points
- Quoted-history strip must run inside Stage 0 worker BEFORE the LLM call (i.e. as a `step.run("strip-quoted-history", ...)` before the existing LLM step), so the LLM sees the stripped body and `total_tokens` reflects post-strip usage.
- The strip is content-shaping only — the original `email_pipeline.emails.body` row is NOT mutated. We pass the stripped body to the LLM, but downstream stages that re-read the row keep the full body (this is a safety classifier input shape, not a data migration).

</code_context>

## Specific Ideas

- Concrete breach evidence (ROADMAP):
  - `budget breach: token_count 12358 > 5000` (smeba-fire, 2026-05-07 14:46:45)
  - `budget breach: token_count 12362 > 5000` (smeba.nl, 2026-05-07 14:46:44)
- Both samples are ~2.5× the current ceiling, well within the new 16k ceiling.
- Phase 64 commit history (`dd2583a`, `dae6276`) for the original BUDG-01 work — useful context for why the constants exist where they do.

</specifics>

<deferred>
## Deferred Ideas

- **Pre-flight prompt-token estimation** — add a tokenizer (tiktoken or similar) to short-circuit before the LLM call on outliers. Deferred: post-hoc check is sufficient with 16k + strip, and a second tokenizer adds drift risk vs Orq.
- **Split prompt vs completion budgets** — cap `completion_tokens` separately (~2k) so completion-side runaways are caught precisely. Deferred: cents ceiling already covers this.
- **Per-run iteration counter** — track LLM-call count per run. Deferred: Stage 0 is single-call by design.
- **Signature / disclaimer stripping** — incremental win but separate scope; revisit if 16k still breaches in production.
- **Hard tail-truncate safety net** — head+tail window after stripping. Deferred until we see post-strip data; the rare residual outlier is acceptable on the existing `budget_breach` failure path.
- **Automated backfill of stranded `budget_breach` rows** — explicitly out of scope. Operators handle via Bulk Review.

</deferred>
