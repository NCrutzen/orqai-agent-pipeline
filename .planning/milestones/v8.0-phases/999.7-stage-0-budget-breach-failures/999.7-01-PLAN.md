## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.7-01 | Tampering / Spoofing | Stage 0 LLM input (post-strip) | accept (residual) | Prompt-injection content hidden inside a quoted reply block can be removed by the strip step before reaching the Stage 0 LLM. Accepted because (a) the operator already saw the prior round of the thread when they replied, so the operator-visible payload is the same view the strip helper produces; (b) the FULL original body is forwarded to Stage 1 (regex pass-1 + LLM pass-2 noise filter) and downstream stages — so any injection content in a quoted block is still subject to the rest of the funnel. Documented in plan-03 worker integration; threat surfaces ONLY at the Stage 0 LLM, not the pipeline as a whole. (RESEARCH.md Pitfall 4 + Open Question 3) |
| T-999.7-02 | Tampering | strip helper output | mitigate | Strip helper returns empty string on greedy parser match → safety LLM would silently see empty body and rubber-stamp `safe`. Mitigated in Plan 02 via defensive fallback: if `stripped.trim().length === 0`, helper returns `{ stripped: <original body>, fallback_reason: "parser_returned_empty" }`. Test covers this path (Task 2 STRIP-04). Telemetry field surfaces fallback to operators (Plan 03). |
| T-999.7-03 | DoS | email-reply-parser regex execution | mitigate | Library uses `safe-regex` + `redos-detector` in CI. Worker enclosing this step has 60s Vercel timeout + Inngest `retries: 0` → bounded blast radius. Optional `re2` peer-dep available if ReDoS becomes a documented concern; not installed in this phase per RESEARCH.md Environment Availability. |
| T-999.7-04 | Tampering (supply chain) | email-reply-parser package | mitigate | Pin to exact version `2.3.5` (no caret); commit `package-lock.json`; review npm advisory feed on every dependency bump. |
