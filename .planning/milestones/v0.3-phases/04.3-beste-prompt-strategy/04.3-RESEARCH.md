## User Constraints (from CONTEXT.md)

### Locked Decisions

**Prompt Architecture:**
- Use **XML tags** (`<instructions>`, `<constraints>`, `<output_format>`, etc.) for structuring generated agent instructions -- clear section boundaries, model-native parsing
- Target **heuristic-first altitude** -- encode how skilled humans approach the task with decision heuristics, not rigid rules. Add specific rules only for observed failure modes
- **Always include 1-2 canonical few-shot examples** in every generated agent's instructions -- Anthropic calls examples "pictures worth a thousand words"
- Handle edge cases **via diverse examples**, not laundry-list rules in the prompt -- Anthropic explicitly warns against stuffing edge case lists

**Multi-agent Patterns:**
- Orchestrator prompts include **explicit delegation frameworks**: how to decompose tasks, assign to sub-agents, define output expectations per sub-agent -- not vague instructions like "research X"
- **Effort scaling guidelines embedded in orchestrator prompts**: simple tasks = 1 agent, 3-10 tool calls; comparisons = 2-4 subagents; complex research = 10+ subagents with clear domain division
- **Division of labor encoded at both levels**: orchestrator has the full agent map, each worker knows its own scope + neighboring agents' responsibilities -- Anthropic found collaborative framework prompts outperform individual-behavior-only prompts

**Tool Descriptions:**
- Tool descriptions are **self-contained + minimal**: no cross-references, minimal viable parameter set, no functional overlap between tools
- Tool return values designed for **token efficiency always** -- structured JSON with only necessary fields, no verbose narratives
- **Tool overlap detection at two stages**: during individual spec generation AND cross-validation across the full swarm after all specs are generated
- Auto-refinement pattern (agents rewriting their own tool descriptions) **deferred to v2** -- noted as future improvement

**Context Management:**
- All generated agents include **context budget awareness** in their system prompts -- directives to summarize, discard low-value info, prioritize high-signal tokens
- Long-horizon task handling uses **all three Anthropic patterns**: structured note-taking (via Orq.ai Memory Store built-in tool), compaction directives, AND sub-agent summary patterns (1-2k token condensed returns)
- **Progressive disclosure via tools** -- agents retrieve context just-in-time rather than pre-loading everything, matching Orq.ai's tool-based architecture
- **Orq.ai Memory Store recommended when applicable** -- agents handling multi-turn or long-running tasks get explicit Memory Store tool recommendations with write/read patterns

### Claude's Discretion
- Whether to include extended/interleaved thinking directives -- determined per agent role based on task complexity (orchestrators and complex reasoners likely yes, simple formatters likely no)

### Deferred Ideas (OUT OF SCOPE)
- Tool description auto-refinement loop (agents testing and rewriting their own tool descriptions) -- v2 improvement
- Prompt optimization loop using eval results to suggest improvements -- already in v2 requirements (ADV-02)

## Summary

This phase upgrades how the orq-agent tool generates agent instructions, orchestrator prompts, and tool descriptions by applying Anthropic's context engineering and multi-agent design principles across all generation subagents (spec generator, researcher, orchestration generator, tool schema generator). The scope is limited to improving what existing subagents produce -- no new subagents or capabilities are added.

The research identified a clear gap between how the current subagent prompts instruct Claude to generate Orq.ai agent specs and what Anthropic's latest guidance recommends. The current prompts are rule-heavy with extensive anti-pattern bullet lists, use markdown headers instead of XML tags, lack heuristic decision frameworks, and do not embed context management directives in generated agent instructions. The upgrade is primarily about restructuring what the spec generator, orchestration generator, and researcher produce -- making the generated prompts/instructions follow Anthropic's patterns, not just the orq-agent's own prompts.

There are two layers of change: (1) the orq-agent subagent prompts themselves (architect.md, researcher.md, spec-generator.md, orchestration-generator.md, dataset-generator.md, readme-generator.md, and the orchestrator orq-agent.md) can benefit from XML structuring and example-driven edge case handling, and (2) the instructions/content these subagents GENERATE for end-user Orq.ai agents must embed the Anthropic patterns (XML tags, heuristic-first, few-shot examples, context budget awareness, delegation frameworks, etc.).

**Primary recommendation:** Apply a systematic transformation across all generation subagents, prioritizing the spec generator (which produces the most critical output -- agent instructions) and the orchestration generator (which produces delegation frameworks). Use Anthropic's official guidance as the authoritative pattern source.

## Architecture Patterns

### Current State: What Exists Today

The orq-agent has 7 prompt files that produce Orq.ai agent specifications:

```
orq-agent/
  commands/orq-agent.md       # Orchestrator (Step 0-7 pipeline)
  agents/
    architect.md               # Swarm topology designer
    researcher.md              # Domain best practices investigator
    spec-generator.md          # THE KEY FILE -- generates agent instructions
    orchestration-generator.md # Generates ORCHESTRATION.md with delegation patterns
    dataset-generator.md       # Generates test datasets
    readme-generator.md        # Generates setup READMEs
```

**Current prompt pattern (across all subagents):**
- Markdown headers (`##`, `###`) for section structure
- Extensive rule-based anti-pattern lists ("Do NOT...", "NEVER...")
- Few-shot examples present but not XML-wrapped
- No XML tags for section boundaries
- No heuristic decision frameworks
- Edge cases handled via explicit rules, not diverse examples
- No context budget awareness directives in generated agent instructions
- No delegation frameworks in generated orchestrator instructions
- No Orq.ai Memory Store integration guidance in generated agent instructions

### Target State: What the Phase Produces

After this phase, the subagent prompts will instruct Claude to generate agent specs that follow Anthropic's patterns:

#### Pattern 1: XML-Tagged Agent Instructions
**What:** Generated agent instructions use XML tags for clear section boundaries
**When to use:** Every agent instruction generated by the spec generator
**Example of what the spec generator should produce:**
```xml
<instructions>
You are a [role] responsible for [responsibility].

<task_handling>
When you receive a [input type]:
1. [First heuristic -- how a skilled human would approach this]
2. [Decision point -- what determines the path]
3. [Action based on decision]
</task_handling>

<constraints>
- [Boundary 1 with WHY it matters]
- [Boundary 2 with WHY it matters]
</constraints>

<output_format>
[Expected structure with field descriptions]
</output_format>

<examples>
<example>
<input>[Realistic input]</input>
<output>[Complete expected output]</output>
</example>
<example>
<input>[Edge case input]</input>
<output>[How to handle gracefully]</output>
</example>
</examples>
</instructions>
```
**Source:** Anthropic Prompting Best Practices -- "XML tags help Claude parse complex prompts unambiguously", "Use consistent, descriptive tag names", "Nest tags when content has a natural hierarchy"

#### Pattern 2: Heuristic-First Altitude
**What:** Instructions encode how skilled humans approach the task, not rigid if-else rules
**When to use:** Every generated agent's task handling section
**Current anti-pattern in spec-generator.md:**
The spec generator's "deep instructions" example uses a rigid step-by-step process. While structured, it reads more like a flowchart than heuristics.
**Target pattern:**
```
When processing a customer query:
- Start by understanding the customer's emotional state and urgency
- Match their concern to the closest knowledge domain you have access to
- If the concern spans multiple domains, address the most urgent aspect first
- When uncertain, acknowledge uncertainty honestly rather than guessing
- Prefer citing specific policies over general reassurance
```
**Source:** Anthropic Context Engineering -- "specific enough to guide behavior effectively, yet flexible enough to provide the model with strong heuristics", "Start with the simplest prompt using your best model, then iteratively add instructions based on observed failures"

#### Pattern 3: Few-Shot Examples as Primary Calibration
**What:** 1-2 canonical examples in every generated agent's instructions, showing complete input/output pairs including edge cases
**When to use:** Every agent instruction produced by the spec generator
**Key insight:** Anthropic recommends 3-5 examples for best results but the CONTEXT.md decision says 1-2. The spec generator currently includes 2 examples in its own prompt (deep vs shallow) but only mandates "1-2 example interactions" in generated instructions. The change is to make examples the PRIMARY edge-case handling mechanism rather than supplementary.
**Source:** Anthropic Prompting Best Practices -- "Include 3-5 examples for best results", "Wrap examples in `<example>` tags"

#### Pattern 4: Explicit Delegation Framework (Orchestrator Prompts)
**What:** Generated orchestrator agent instructions include how to decompose tasks, assign to sub-agents, define output expectations
**When to use:** Multi-agent swarms where one agent orchestrates others
**Example of what orchestration-generator.md should produce in the orchestrator's instructions:**
```xml
<delegation_framework>
You coordinate [N] specialized agents. Before delegating:
1. Assess task complexity:
   - Simple (single-domain, factual): handle directly, 3-10 tool calls
   - Moderate (multi-source, comparison): delegate to 2-4 agents with clear domain boundaries
   - Complex (open-ended research, creative): delegate to specialized agents with clear output expectations per agent
2. For each delegation, specify:
   - The agent's specific objective (not "research X" but "find the top 3 pricing strategies for [domain] with source URLs")
   - Expected output format (structured data, not prose)
   - Clear task boundaries (what is in scope, what is not)
3. After receiving results, synthesize rather than concatenate
</delegation_framework>
```
**Source:** Anthropic Multi-Agent Research System -- "Each subagent needs an objective, an output format, guidance on the tools and sources to use, and clear task boundaries"

#### Pattern 5: Context Budget Awareness Directives
**What:** Every generated agent includes directives for managing its context window
**When to use:** All generated agents, with intensity proportional to task duration
**Example directives to embed in generated agent instructions:**
```xml
<context_management>
Your context window is a finite resource. Manage it actively:
- Retrieve information just-in-time via tools rather than requesting everything upfront
- When accumulating data across multiple tool calls, summarize findings before proceeding
- Prioritize high-signal tokens: specific facts, decisions, and actions over verbose descriptions
- For multi-turn conversations, track key decisions in memory rather than relying on conversation history
</context_management>
```
**For agents with Orq.ai Memory Store tools:**
```xml
<memory_patterns>
Use Memory Store for persistent context across turns:
- write_memory_store: Save key decisions, user preferences, task progress after each significant interaction
- query_memory_store: Retrieve relevant context before responding to new inputs
- Pattern: Read first, act second, write last
</memory_patterns>
```
**Source:** Anthropic Context Engineering -- "find the smallest set of high-signal tokens that maximize the likelihood of your desired outcome", "maintain lightweight identifiers and load data dynamically via tools"

#### Pattern 6: Self-Contained Tool Descriptions
**What:** Every tool description in generated specs is self-contained with no cross-references, minimal parameters, clear single-purpose
**When to use:** All function tool and built-in tool descriptions in generated agent specs
**Key principle:** "If humans can't pick the right tool, agents won't either"
**Anti-pattern:** Tool descriptions that say "Use this in conjunction with X" or "After calling Y, use this to..."
**Target:** Each tool description explains its complete purpose, inputs, and outputs independently
**Source:** Anthropic Context Engineering -- "Tools should be self-contained, robust to error, extremely clear about intended use", "Minimize tool set overlap"

### Anti-Patterns to Avoid

- **Stuffing edge case lists:** Current prompts list many "Do NOT..." rules. Replace the majority with diverse examples that demonstrate correct behavior implicitly. Keep explicit rules only for critical failure modes (security, data leakage).
- **Rule-heavy vs heuristic:** A prompt with 15 constraint bullets and 2 examples should become a prompt with 5 heuristic guidelines and 4 diverse examples.
- **Pre-loading context:** Generated agents should not request "all available information" upfront. They should use tools to retrieve context incrementally.
- **Vague delegation:** Orchestrator instructions like "delegate to sub-agents" without specifying objective, output format, and task boundaries.
- **Ignoring context limits:** Generated agents with no awareness that their context window is finite and degrades with length.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Agent instruction structure | Custom section headings per agent | XML tag template pattern (Pattern 1 above) | Consistent parsing, model-native, Anthropic-recommended |
| Edge case coverage | Bullet-list rules in constraints section | Diverse few-shot examples in `<examples>` tags | Anthropic explicitly warns against rule lists; examples generalize better |
| Context management | Nothing (currently omitted) | Context budget awareness directives (Pattern 5) | Agents without context awareness degrade on long conversations |
| Delegation instructions | Vague "coordinate sub-agents" | Explicit delegation framework (Pattern 4) | Anthropic found 90.2% improvement with structured delegation |
| Memory patterns | Ad-hoc memory tool recommendations | Standardized Memory Store patterns | Orq.ai Memory Store maps directly to Anthropic's structured note-taking pattern |

**Key insight:** This phase is not about writing better prompts from scratch -- it is about establishing a template transformation that the spec generator, orchestration generator, and researcher apply systematically to every generated agent. The patterns are well-documented by Anthropic; the implementation is applying them consistently.

## Common Pitfalls

### Pitfall 1: Upgrading orq-agent prompts instead of what they generate
**What goes wrong:** Developer restructures the architect.md, researcher.md, etc. with XML tags but does not change what the spec generator PRODUCES for end-user agents.
**Why it happens:** Confusion between the two layers -- the orq-agent's own prompts vs. the prompts it generates.
**How to avoid:** The primary target is what the spec generator, orchestration generator, and researcher produce. Upgrading the orq-agent's own prompts is secondary and should follow the same principles but is less impactful.
**Warning signs:** Changes only in the `orq-agent/agents/*.md` files with no changes to the template patterns and few-shot examples those files contain.

### Pitfall 2: Over-engineering the XML tag structure
**What goes wrong:** Creating deeply nested, complex XML structures that add verbosity without clarity.
**Why it happens:** Treating XML tags as a rigid schema rather than a readability tool.
**How to avoid:** Use 4-6 top-level tags maximum per agent instruction: `<instructions>`, `<task_handling>`, `<constraints>`, `<output_format>`, `<examples>`, optionally `<context_management>` and `<delegation_framework>`. Nest sparingly.
**Warning signs:** Generated instructions have 3+ levels of XML nesting or 10+ distinct tag types.

### Pitfall 3: Removing all rules in favor of examples
**What goes wrong:** Critical safety constraints (no PII exposure, no system prompt leakage) get replaced by examples and the agent violates them.
**Why it happens:** Over-applying the "examples over rules" principle.
**How to avoid:** Keep explicit rules for: security boundaries, data leakage prevention, scope enforcement. Use examples for: tone calibration, output format, edge case handling, ambiguity resolution.
**Warning signs:** Generated agent instructions have zero `<constraints>` content, relying entirely on examples.

### Pitfall 4: Breaking the spec generator's self-validation checklist
**What goes wrong:** Restructuring the spec generator's output format invalidates the 12-point validation checklist that ensures completeness.
**Why it happens:** Changing the output structure without updating the validation criteria.
**How to avoid:** Update the validation checklist alongside any structural changes. Add new checks for XML tag presence, example count, heuristic vs. rule ratio.
**Warning signs:** Spec generator produces specs that fail its own validation.

### Pitfall 5: Context management directives that conflict with Orq.ai's architecture
**What goes wrong:** Generated agents include directives about compaction, multi-window workflows, or context truncation that only apply to Claude Code, not Orq.ai agents.
**Why it happens:** Blindly copying Anthropic's context engineering guidance without adapting for Orq.ai's runtime model.
**How to avoid:** Orq.ai agents run in a managed runtime with `max_iterations` and `max_execution_time` constraints. Context management should focus on: (a) using tools for just-in-time retrieval, (b) using Memory Store for cross-turn persistence, (c) summarizing tool results before reasoning. Do NOT include compaction directives or multi-window patterns.
**Warning signs:** Generated agent instructions reference "context window compaction", "save state before context refresh", or similar Claude Code-specific patterns.

## Code Examples

### Example 1: Spec Generator Template Transformation

**Current pattern in spec-generator.md** (what it tells Claude to generate for the Instructions field):
```markdown
## Instructions

You are a [Role] for [Domain]...

### Behavioral Guidelines
- Maintain a professional tone...
- Address the customer by name...
- Never express frustration...

### Task Handling Process
1. Identify intent
2. Categorize query
3. Retrieve information
...

### Constraints
- NEVER process refunds...
- NEVER share internal document IDs...
- NEVER make promises...

### Edge Case Handling
- Empty input: Ask a clarifying question
- Profanity: Acknowledge frustration, continue helping
- Non-English: Respond in detected language
- PII shared: Warn customer

### Examples
**Example 1:** [input/output pair]
**Example 2:** [input/output pair]
```

**Target pattern** (what the spec generator should produce after this phase):
```xml
<instructions>
You are a [Role] for [Domain]. You help [users] with [core task] by [method].

<task_handling>
When you receive a [input type], approach it the way a skilled [role] would:
- First, understand the person's actual need -- what they are trying to accomplish, not just the words they used
- Match their need to your available tools and knowledge
- When the need is clear, act decisively; when ambiguous, ask one focused clarifying question
- Prefer citing specific sources over general reassurance
</task_handling>

<constraints>
These boundaries exist because [reason]:
- You have read-only access -- you cannot modify accounts, process transactions, or change data
- Internal system details (document IDs, knowledge base names, configuration) must never appear in responses
- When uncertain about a policy, acknowledge the uncertainty and offer escalation rather than guessing
</constraints>

<output_format>
Structure every response as:
- **Acknowledgment**: One sentence recognizing the person's situation
- **Answer**: The substantive response with specific details
- **Next step**: What happens next (further help, escalation, or action they should take)
</output_format>

<context_management>
Retrieve information via tools as needed rather than requesting everything upfront. After multiple tool calls, summarize what you have learned before composing your response.
</context_management>

<examples>
<example>
<input>Hi, I ordered some headphones last week. Order number ORD-2026-44821. When will they arrive?</input>
<output>Thank you for reaching out about your order.

I have looked up order #ORD-2026-44821. Your headphones are currently in transit with an estimated delivery date of March 2, 2026. Tracking number TRK-HDP-99012 is available on the carrier's website for real-time updates.

Is there anything else I can help you with?</output>
</example>
<example>
<input>I want to return the keyboard I bought 2 months ago. It is not working properly.</input>
<output>I am sorry to hear your keyboard is not working as expected.

Our standard return window is 30 days from purchase, and your purchase is outside that window. However, since you are experiencing a defect, this may qualify for a warranty claim with different terms. I was not able to find specific warranty details for your product.

I recommend escalating this to a specialist who can review your warranty eligibility. Would you like me to do that?</output>
<note>This example demonstrates handling an out-of-policy request with empathy, offering an alternative path rather than a flat rejection.</note>
</example>
</examples>
</instructions>
```

**Source:** Anthropic Prompting Best Practices, Anthropic Context Engineering Guide

### Example 2: Orchestrator Delegation Framework

**Current pattern** (orchestration-generator.md generates ORCHESTRATION.md but does not embed delegation heuristics in the orchestrator agent's instructions):
The orchestration generator produces a document ABOUT the orchestration, but the orchestrator agent's instructions (produced by spec-generator) lack explicit delegation heuristics.

**Target pattern** (spec generator produces orchestrator instructions with embedded delegation framework):
```xml
<delegation_framework>
You coordinate a team of [N] specialized agents:
- [agent-a]: [role] -- call when [condition]
- [agent-b]: [role] -- call when [condition]

When a request arrives:
1. Assess complexity: Can you answer directly (simple lookup, quick classification)? Handle it yourself.
2. For multi-faceted requests, decompose into sub-tasks with clear boundaries.
3. When delegating, always specify:
   - What to find or produce (objective)
   - How to structure the response (output format)
   - What is out of scope (boundaries)
4. After receiving sub-agent results, synthesize a coherent response -- do not concatenate raw outputs.

<effort_scaling>
- Simple queries (single-domain, factual): Handle directly with 3-10 tool calls
- Moderate queries (comparison, multi-source): Delegate to 2-4 agents with clear domain division
- Complex queries (open-ended, creative): Delegate to multiple agents with distinct responsibilities
</effort_scaling>
</delegation_framework>
```

**Source:** Anthropic Multi-Agent Research System -- "Each subagent needs an objective, an output format, guidance on the tools and sources to use, and clear task boundaries"

### Example 3: Memory Store Integration Pattern

**Current state:** Researcher recommends Memory Store tools when applicable, but without usage patterns.

**Target pattern** (embedded in generated agent instructions when Memory Store tools are present):
```xml
<memory_patterns>
You have access to persistent memory via Memory Store. Use it for cross-turn context:
```

## Open Questions

1. **Thinking directives per agent role**
   - What we know: CONTEXT.md marks this as "Claude's Discretion." Anthropic recommends adaptive thinking for orchestrators and complex reasoners.
   - What's unclear: Orq.ai's runtime may not expose thinking parameters. The generated instructions can recommend thinking but cannot enforce it at the API level.
   - Recommendation: Include a `<thinking_recommendation>` section in generated specs as advisory guidance for Orq.ai Studio configuration. Orchestrators and complex reasoners get "extended thinking recommended", simple classifiers and formatters get "standard mode sufficient."

2. **Tool overlap detection cross-validation**
   - What we know: CONTEXT.md requires two-stage detection -- during individual spec generation AND cross-validation across the full swarm.
   - What's unclear: The current pipeline has no cross-validation step after all specs are generated. Wave 3 runs post-generation tasks (orchestration, datasets, README) but no cross-validation.
   - Recommendation: Add a lightweight tool overlap check in the orchestration generator, which already reads all generated specs. This avoids adding a new pipeline stage while achieving cross-swarm visibility.

3. **Memory Store description field configuration**
   - What we know: Orq.ai Memory Store saves based on the memory store's description field. What gets saved depends on how you prompt the agent AND how the memory store is configured.
   - What's unclear: Whether the spec generator should recommend specific memory store description text.
   - Recommendation: Include memory store description guidance in generated agent specs when Memory Store tools are recommended. Example: "Configure the memory store description to: 'Store user preferences, key decisions, and task progress for this support interaction.'"
