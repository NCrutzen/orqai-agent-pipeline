<task type="auto">
  <name>Task 1: Upgrade spec-generator.md to produce XML-tagged, heuristic-first agent instructions</name>
  <files>orq-agent/agents/spec-generator.md</files>
  <action>
Transform the spec-generator.md prompt so that Claude generates agent instructions following Anthropic's patterns. This is about changing WHAT the spec generator produces, not just how the spec generator prompt itself is structured.

**Changes to the Instructions field generation guidance:**

1. **XML-tagged output structure:** Replace the current markdown-header-based instruction format with XML tags. The spec generator must instruct Claude to produce agent instructions wrapped in:
   - `<instructions>` (top-level wrapper)
   - `<task_handling>` (heuristic approach to the agent's core work)
   - `<constraints>` (boundaries with WHY they matter -- keep rules only for security/data-leakage/scope-enforcement; convert all other rules to examples)
   - `<output_format>` (expected response structure)
   - `<examples>` with nested `<example>` tags containing `<input>` and `<output>` pairs (1-2 canonical examples per agent)
   - `<context_management>` (context budget awareness -- always include)
   - `<memory_patterns>` (only when Memory Store tools are present)
   - `<delegation_framework>` (only for orchestrator agents -- coordinate with Plan 02 which handles this in detail)

2. **Heuristic-first altitude:** Update the "deep instructions" guidance and example. Replace rigid step-by-step flowcharts with heuristic approaches: "When you receive a [input type], approach it the way a skilled [role] would: first understand the actual need, match to available tools, act decisively when clear, ask one clarifying question when ambiguous." See RESEARCH.md Pattern 2 and Example 1 for the exact transformation pattern.

3. **Few-shot examples as primary calibration:** Update the instructions field guidance to make examples the PRIMARY mechanism for edge case handling. Current prompt mandates "1-2 example interactions" -- expand this to explain that examples should demonstrate: (a) a happy-path interaction showing complete input/output, (b) an edge case showing graceful handling of ambiguity or out-of-scope requests. Each example wrapped in `<example>` tags with `<input>`, `<output>`, and optional `<note>` explaining what the example demonstrates.

4. **Context budget awareness directives:** Add a new section in the spec generator prompt requiring every generated agent to include a `<context_management>` block. The content should instruct agents to: retrieve info just-in-time via tools, summarize findings before proceeding after multiple tool calls, prioritize high-signal tokens, and track key decisions in memory for multi-turn conversations. See RESEARCH.md Pattern 5 for exact directive text.

5. **Memory Store integration:** When the spec generator identifies that an agent needs Memory Store tools (multi-turn, long-running tasks), it must also embed a `<memory_patterns>` section in that agent's instructions with read-first/act-second/write-last pattern and specific memory store description guidance. See RESEARCH.md Example 3 for the pattern.

6. **Anti-pattern reduction:** Convert the current extensive "Do NOT..." and "NEVER..." rule lists into diverse examples where possible. Keep explicit rules ONLY for: security boundaries, data leakage prevention, scope enforcement, and Orq.ai field format requirements. Move all tone/style/edge-case rules into the few-shot examples. See RESEARCH.md "Anti-Patterns to Avoid" section.

7. **Self-validation checklist update:** Update the existing 12-point validation checklist to include new checks:
   - Instructions field uses XML tags (`<instructions>`, `<task_handling>`, `<constraints>`, `<output_format>`, `<examples>`)
   - At least 1 example with `<input>` and `<output>` pairs
   - `<context_management>` section present
   - Heuristic approach (not rigid flowchart) in `<task_handling>`
   - Rules in `<constraints>` have WHY explanations
   - Memory Store patterns included if Memory Store tools are recommended

8. **Thinking recommendation:** Per Claude's Discretion from CONTEXT.md, add advisory guidance: include a brief `<thinking_recommendation>` advisory section in generated specs. Orchestrators and complex reasoners get "extended thinking recommended", simple classifiers and formatters get "standard mode sufficient."

**What NOT to change:**
- The 18 Orq.ai field coverage (all fields still generated)
- The one-agent-per-invocation pattern
- The template placeholder system
- The file reading/output format
