**Step 6: Execute Generation Pipeline**

Structure as three waves with GSD-style banners per wave.

**Wave 1: Research (if not skipped)**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► RESEARCH (Wave 1)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
- If researcher was classified as SKIP: display "Skipped — user provided sufficient domain context" and proceed to Wave 2
- If researcher was classified as RUN:
  - For swarms with 1-3 agents: invoke ONE researcher for the entire swarm (current researcher design — one invocation, per-agent sections in output). Per Phase 2 decision: "One researcher for entire swarm with per-agent sections."
  - For swarms with 4+ agents: invoke MULTIPLE researcher instances in parallel, each handling a subset of agents (the researcher prompt supports this naturally since each agent gets its own section). Per RESEARCH.md open question 1.
  - Pass to researcher: architect blueprint file path, original user input
  - Researcher reads its own reference files via `<files_to_read>` — do NOT load them in the orchestrator
  - Display spawning indicators per agent being researched
  - On completion: display success count

**Wave 2: Spec Generation**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► SPEC GENERATION (Wave 2)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
- Spawn ONE spec generator per agent, all in parallel
- Each spec generator receives: architect blueprint path, research brief path (or note that research was skipped), the specific agent key to generate
- Spec generator reads its own reference files and templates — do NOT load them in the orchestrator
- Per Phase 2 decision: "One-agent-per-invocation: spec generator processes single agent for focused, high-quality output"
- Output: one `[agent-key].md` file per agent in the `agents/` subdirectory of the output directory
- Display spawning indicators per agent

**Wave 3: Post-Generation**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► POST-GENERATION (Wave 3)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
All of these run in parallel after all agent specs are complete:
- **Orchestration Generator** (multi-agent only): receives blueprint path + all generated agent spec paths. Produces `ORCHESTRATION.md` in output root.
- **Dataset Generator**: ONE invocation per agent, each receiving blueprint path + research brief path + agent spec path. Produces `[agent-key]-dataset.md` and `[agent-key]-edge-dataset.md` in `datasets/` subdirectory.
- **README Generator**: receives all generated file paths (blueprint, specs, orchestration doc, dataset file list). Produces `README.md` in output root. Per Phase 2: "README as final pipeline step: reads all outputs before generating" — this still works in parallel because it reads from file paths and is spawned simultaneously.

**Error handling for all waves (per CONTEXT.md locked decision):**
- If a subagent fails or times out:
  - Write a marker file: `[output-path].incomplete` with error details
  - Log failure in pipeline metadata
  - Continue with remaining subagents in current wave
  - Skip dependent stages for the failed agent ONLY (e.g., if researcher fails for agent X, spec generator for agent X still runs but with a note that research was unavailable; if spec generator fails for agent X, dataset generator for agent X is skipped)
  - At pipeline end: report all failures with retry option

**Step 7: Final Summary**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► COMPLETE ✓
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Display:
1. **Directory tree view** of the output directory (use `tree` or ASCII representation)
2. **Stats**: agent count, stages run/skipped, one-liner per agent
3. **Failures** (if any): list failed stages with error details and offer retry
4. **Next steps**: suggest review priorities and Orq.ai Studio deploy steps (per CONTEXT.md)

**Metadata file (`pipeline-run.json`):**
Write to the output directory root. Include (per RESEARCH.md Example 3):
- `pipeline_version`: "1.0"
- `swarm_name`: from architect blueprint
- `output_directory`: full path
- `started_at` / `completed_at` / `duration_seconds`
- `input_classification`: detail level, per-stage decisions with reasoning, any user overrides
- `agents_generated`: list with key and status (complete/incomplete)
- `stages_completed`: list with stage name, duration, agent count, status
- `failures`: list of any failed stages with error details

**Style guidance:**
- Keep orchestrator lean: pass file paths, don't load subagent outputs
- Each subagent invocation should reference the agent file by path
- GSD-style banners for each wave transition
- Spawning indicators: `◆ Spawning N generators in parallel...` then per-agent `→ [agent-key]`
- Completion indicators: `✓ Complete: N/M agents processed`
