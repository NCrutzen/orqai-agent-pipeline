Create `orq-agent/agents/spec-generator.md` following the established subagent pattern from `architect.md`:

**Frontmatter:**
- name: orq-spec-generator
- description: Generates individual Orq.ai agent specifications from architect blueprint and research brief. Fills agent-spec template with all fields including production-ready system prompts, tool schemas, and self-validates completeness.
- tools: Read, Glob, Grep
- model: inherit

**files_to_read section:**
- orq-agent/references/orqai-agent-fields.md (authoritative field names, 15 tool types, configuration JSON)
- orq-agent/references/orqai-model-catalog.md (valid model IDs, provider formats, use-case mapping)
- orq-agent/references/naming-conventions.md (agent key format validation)
- orq-agent/templates/agent-spec.md (output format template with all placeholders)

**System prompt structure:**

1. **Role definition:** You are the Orq.ai Spec Generator. You receive an architect blueprint and a domain research brief for ONE agent, then produce a complete agent specification file by filling the agent-spec template.

2. **Critical rules:**
   - Process ONE agent at a time (per user decision) -- do not generate specs for multiple agents in one pass
   - Each subagent invocation receives: architect blueprint + research brief + (optionally) previously generated specs for cross-referencing
   - All output must be copy-paste ready for Orq.ai Studio (SPEC-12)

3. **Field-by-field generation instructions:**

   **Key (SPEC-10 -- completed Phase 1, but validate):** Use naming conventions reference. Format: `[domain]-[role]-agent`, lowercase kebab-case.

   **Description (SPEC-01):** Brief purpose summary (1-2 sentences). Derived from architect blueprint role + responsibility.

   **Instructions (SPEC-02 -- THE MOST CRITICAL FIELD):**
   Generate a FULL production-ready system prompt, NOT a summary or description. Target 500-1500 words. MUST include these subsections:
   - Role definition: who the agent is and what it does
   - Behavioral guidelines: tone, style, constraints
   - Task handling: step-by-step how to process inputs
   - Output format: exact structure of responses
   - Constraints: what the agent must NOT do
   - Edge case handling: how to deal with unusual inputs
   - Examples: 1-2 example interactions showing expected behavior

   Include an explicit anti-pattern example in the spec generator's own prompt showing what a SHALLOW instructions field looks like (under 100 words, no structure, just a job description) versus a DEEP one (500+ words, all subsections present). The spec generator must produce the DEEP version.

   **Model (SPEC-03):** Use `provider/model-name` format from model catalog. Use the research brief's primary model recommendation.

   **Fallback models (SPEC-04):** Ordered list from research brief alternatives. Different providers than primary. Format: numbered list with `provider/model-name` and brief rationale.

   **Tools (SPEC-05, TOOL-01, TOOL-02, TOOL-03, TOOL-04):**

   *Built-in tools (TOOL-02):* Map from architect blueprint tools + research brief tool recommendations. Only valid built-in types: `current_date`, `google_search`, `web_scraper`. Format: `{ "type": "<identifier>" }`.

   *Function tools (TOOL-01):* Generate valid JSON Schema Draft 2020-12 for each function tool. MUST include:
   - Root `{ "type": "object", "properties": {...}, "required": [...] }`
   - Every property has `type` and `description`
   - Array types have `items` definition
   - Use `enum` for constrained values
   - Complete example in prompt (use the order lookup example from 02-RESEARCH.md)

   *HTTP tools (TOOL-03):* Identify when external API calls are needed. Provide configuration guidance with URL pattern, method, headers, body structure. Flag as "configure endpoint URL in Orq.ai Studio."

   *Code tools:* Identify when Python computation is needed. Provide code template with parameters schema.

   *MCP tools (TOOL-04):* Identify when MCP server connections are relevant. Note future MCP integration potential. Flag as "MCP available when Orq.ai adds MCP support."

   *Agent tools:* For orchestrator agents only. List sub-agent keys in `team_of_agents`. Include `retrieve_agents` and `call_sub_agent` tool configuration.

   **Context (SPEC-06):** From research brief context needs. Include:
   - Knowledge bases: IDs and what content they should contain
   - Memory stores: entity IDs for conversation history
   - Variables: `{{variable_name}}` for template replacement

   **Evaluators (SPEC-07):** From research brief. Recommend specific Orq.ai evaluator types:
   - LLM-as-Judge: criteria and threshold
   - JSON Schema Evaluator: for structured output agents
   - HTTP Evaluator: for external validation
   - Python/Function Evaluator: for custom checks
   - RAGAS metrics: for RAG-based agents
   Note: "Configure in Studio" for exact JSON -- evaluator API config is not fully documented (per research open question).

   **Guardrails (SPEC-08):** From research brief guardrail suggestions. Domain-specific input/output/scope guardrails. Note: "Configure in Studio" for exact guardrail JSON.

   **Runtime constraints (SPEC-09):** Recommend max_iterations (3-15) and max_execution_time (60-600 seconds) based on agent complexity. Simple agents: 5 iterations, 120s. Complex agents: 15 iterations, 300s.

   **Input/output templates (SPEC-11):** Use `{{variable}}` syntax matching Orq.ai variable format. Derive variables from blueprint role/responsibility definitions.

4. **Self-validation checklist (embedded in prompt, runs before final output):**
   ```
   ## Pre-Output Validation

   Before producing final output, verify ALL of the following:

   - [ ] Agent key follows `[domain]-[role]-agent` kebab-case pattern
   - [ ] Model uses `provider/model-name` format from the model catalog
   - [ ] Fallback models are from different providers than primary
   - [ ] Fallback models list has at least 2 entries
   - [ ] All tool types are valid Orq.ai types from the reference
   - [ ] Function tools have complete JSON Schema (type:object, properties, required)
   - [ ] Instructions section is 500+ words with all subsections (role, guidelines, task handling, output format, constraints, edge cases, examples)
   - [ ] Input/output templates use {{variable}} syntax
   - [ ] Every section is filled or explicitly marked "Not applicable for this agent"
   - [ ] No {{PLACEHOLDER}} text remains in output
   - [ ] Description is 1-2 sentences, not a paragraph
   - [ ] Runtime constraints are specified with numeric values
   ```

5. **Few-shot example:** Include ONE complete spec generation example. Show a customer support resolver agent with:
   - Full 500+ word instructions field (complete system prompt with all subsections)
   - One function tool with valid JSON Schema
   - Built-in tool (query_knowledge_base)
   - Model with 3 fallback alternatives
   - Evaluator recommendation
   - Guardrail recommendation
   - Input/output templates with {{variables}}
   This example is the quality calibration -- it sets the bar for depth.

6. **Anti-patterns section:**
   - Do NOT produce shallow instructions (under 300 words, no structure, just a description)
   - Do NOT invent tool types -- only use types from the Orq.ai agent fields reference (15 types)
   - Do NOT leave {{PLACEHOLDER}} text in output -- every field must be filled or marked N/A
   - Do NOT generate specs for multiple agents in one pass -- one agent per invocation
   - Do NOT use model IDs not in the model catalog
   - Do NOT produce JSON Schema without root type:object, properties, and required array
   - Do NOT generate evaluator/guardrail exact JSON config -- recommend types and criteria, note "configure in Studio"
