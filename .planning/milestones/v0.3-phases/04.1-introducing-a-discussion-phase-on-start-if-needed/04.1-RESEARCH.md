## Summary

Phase 4.1 replaces the current Step 2 (Classify Input Depth) and Step 3 (Confirm with User checkpoint) in the `/orq-agent` orchestrator with a structured discussion step that follows the GSD discuss-phase pattern. The discussion always runs, analyzes the user's use case to surface domain-specific gray areas, lets the user select which to discuss, and produces an enriched input that feeds directly into the architect (Step 4).

The implementation is entirely within the orchestrator command file (`orq-agent/commands/orq-agent.md`). No new subagent files, templates, or reference files are needed. The orchestrator prompt itself contains the discussion logic — it is the LLM executing the discussion, not a spawned subagent. This is consistent with the existing architecture where the orchestrator handles input capture (Step 1) and classification (Step 2) directly.

**Primary recommendation:** Rewrite Step 2 and Step 3 in `orq-agent/commands/orq-agent.md` as a single new Step 2 ("Discussion") that follows GSD discuss-phase patterns: analyze use case, generate gray areas, multi-select, 4-questions-per-area cadence, produce structured discussion summary. Keep researcher skip classification as an internal decision after discussion completes (no user-facing checkpoint).

## Architecture Patterns

### Pattern 1: Orchestrator-Inline Discussion (Recommended)

**What:** The discussion logic is embedded directly in the orchestrator command prompt, not delegated to a subagent. The orchestrator LLM conducts the discussion itself.

**When to use:** When the discussion requires back-and-forth user interaction and the results feed directly into the next orchestrator step. Spawning a subagent would lose conversational context.

**Why this pattern:** The current orchestrator already handles Step 1 (Capture Input) and Step 2 (Classify Input Depth) inline. The discussion step is a natural extension — it replaces classification with a richer interaction but stays within the same conversational flow.

**Structure of the new Step 2:**

```
Step 2: Discussion
  2.1 — Analyze use case → identify 3-4 domain-specific gray areas
  2.2 — Present gray areas as multi-select to user
  2.3 — For each selected area: ask 4 questions with concrete options
  2.4 — After each area: "more or next?"
  2.5 — After all areas: compile discussion summary
  2.6 — (Internal) Classify researcher skip decision based on enriched context
  2.7 — Proceed to Step 3 (renumbered: Run Architect)
```

### Pattern 2: Discussion Summary as Structured Document

**What:** The discussion output is formatted as a structured summary document (not raw conversation) that the architect can consume reliably.

**When to use:** Always — the architect needs structured input, not a conversation transcript.

**Recommended format:**

```markdown
## Discussion Summary

### Use Case
[Original user input]

### Decisions Made
- **[Gray Area 1]:** [Decision or preference]
- **[Gray Area 2]:** [Decision or preference]

### Additional Context Gathered
- [Key detail surfaced during discussion]
- [Another detail]

### Open Areas (Claude's Discretion)
- [Areas not discussed — architect can decide]
```

**Why this format:**
1. The architect prompt already accepts "use case description" as input — the summary extends this naturally
2. Structured sections are parseable by the architect LLM
3. "Decisions Made" maps directly to implementation constraints
4. "Open Areas" signals where the architect has flexibility

### Pattern 3: Gray Area Generation by Domain Analysis

**What:** The orchestrator analyzes the use case description to identify domain-specific gray areas rather than using a fixed checklist.

**How it works:**

1. Parse the use case for domain indicators (e.g., "FAQ bot" → customer support domain)
2. Identify what kind of system is being built:
   - User-facing conversational → tone, escalation, scope, knowledge sources
   - Data processing → input formats, error handling, output structure, validation
   - Multi-step workflow → handoff criteria, failure modes, human checkpoints, state management
   - Content generation → style, format, guardrails, personalization
3. Generate 3-4 gray areas specific to that domain
4. Each gray area gets a label + 1-2 example questions it covers

**The key insight from GSD discuss-phase:** Gray areas should be "implementation decisions the user cares about — things that could go multiple ways and would change the result." They should NOT be technical decisions (those are Claude's job).

### Pattern 4: Step Renumbering Strategy

**What:** After replacing Steps 2-3 with the discussion, all subsequent steps shift down by one number.

**Current flow:**
```
Step 1: Capture Input
Step 2: Classify Input Depth          ← REMOVE
Step 3: Confirm with User             ← REMOVE
Step 4: Run Architect                 ← becomes Step 3
Step 5: Blueprint Review              ← becomes Step 4
Step 6: Execute Generation Pipeline   ← becomes Step 5
Step 7: Final Summary                 ← becomes Step 6
```

**New flow:**
```
Step 1: Capture Input                 (unchanged)
Step 2: Discussion                    (NEW — replaces Steps 2+3)
Step 3: Run Architect                 (was Step 4)
Step 4: Blueprint Review              (was Step 5)
Step 5: Execute Generation Pipeline   (was Step 6)
Step 6: Final Summary                 (was Step 7)
```

**Impact:** All internal references to step numbers in the orchestrator command must be updated. The architect spawn logic, blueprint review, generation pipeline, and final summary sections reference step numbers in their instructions and comments.

### Anti-Patterns to Avoid

- **Over-interrogating detailed inputs:** When the user provides a detailed brief, the discussion should be short. The gray area analysis should naturally find fewer ambiguities. Do NOT force questions when the user has already answered them.
- **Generic gray areas:** "UI design" or "Error handling" are too vague. Gray areas must be domain-specific: "Knowledge source selection" for a FAQ bot, "Escalation triggers" for a support triage system.
- **Losing the original input:** The discussion summary must include the original user input verbatim. The architect needs both the original description and the discussion decisions.
- **Replacing the Blueprint Review:** The user explicitly said Step 5 (Blueprint Review checkpoint) stays. The discussion replaces only Steps 2-3.

## Common Pitfalls

### Pitfall 1: Discussion That Never Ends
**What goes wrong:** The discussion keeps generating new gray areas and the user gets stuck in a loop.
**Why it happens:** No clear exit path after initial areas are discussed.
**How to avoid:** Follow GSD pattern strictly: after all selected areas are done, offer "done" or "explore more." If "explore more," generate 2-3 additional areas max, then force completion. Cap at 2 rounds of gray area discovery.
**Warning signs:** More than 8 total gray areas being presented across rounds.

### Pitfall 2: Discussion Output Too Verbose for Architect
**What goes wrong:** The discussion produces a 2000-word summary that crowds the architect's context window.
**Why it happens:** Including full conversation history instead of distilled decisions.
**How to avoid:** The summary format (Pattern 2 above) captures decisions in 100-300 words. The orchestrator compiles the summary, not the raw conversation.
**Warning signs:** Discussion summary exceeding 500 words.

### Pitfall 3: Gray Areas That Are Actually Technical Decisions
**What goes wrong:** The discussion asks users about model selection, tool types, or prompt patterns — things the user should not decide.
**Why it happens:** Confusing "implementation decisions the user cares about" with "technical implementation details."
**How to avoid:** Gray areas must be about WHAT the system should do and HOW it should behave from a user perspective, not HOW it's technically built. "Should the bot escalate to humans?" = good. "Should we use gpt-4o or claude-sonnet?" = bad.
**Warning signs:** Gray area options referencing Orq.ai-specific features, model names, or tool types.

### Pitfall 4: Researcher Skip Logic Regression
**What goes wrong:** Removing Steps 2-3 accidentally removes the researcher skip classification entirely.
**Why it happens:** The old Step 2 contained the classification logic for whether to skip the researcher.
**How to avoid:** Preserve the classification logic as an internal step (Step 2.6 in the pattern above). After the discussion compiles the summary, the orchestrator internally evaluates whether the enriched input provides sufficient detail to skip research. This is NOT exposed to the user as a checkpoint — it's an internal decision.
**Warning signs:** Researcher always running even when user provided extremely detailed input with model/tool/guardrail specifics.

### Pitfall 5: Inconsistent Banner Styling
**What goes wrong:** The new discussion step uses different formatting than the rest of the orchestrator.
**Why it happens:** Not following the existing banner convention.
**How to avoid:** Use the same banner style as existing steps:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► DISCUSSION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

This is the core prompt text that replaces the current Steps 2-3 in `orq-agent.md`:

```markdown
## Step 2: Discussion

Analyze the user's use case to surface implementation decisions before the pipeline proceeds. This discussion ensures the architect receives enriched context regardless of how brief or detailed the original input was.

Display the discussion banner:

` ` `
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

` ` `

### 2.1: Analyze Use Case

Read the user's input and determine:

1. **Domain:** What kind of system is being built? (customer support, content generation, data processing, workflow automation, etc.)
2. **Gray areas:** 3-4 domain-specific implementation decisions that would change the result. These are decisions the USER cares about — not technical decisions.

**Gray area generation rules:**
- Each area must be specific to the use case domain (not generic like "UI" or "Error handling")
- Each area represents a decision that could go multiple ways
- Each area should have 1-2 example questions it covers
- Areas should NOT ask about technical implementation (models, tools, prompts — those are Claude's job)

**Examples by domain:**
- FAQ bot → "Knowledge sources", "Tone & personality", "Escalation behavior", "Scope boundaries"
- Customer support triage → "Routing criteria", "Priority detection", "Handoff protocol", "Response style"
- Content generation pipeline → "Output format & style", "Quality guardrails", "Source handling", "Personalization depth"
- Data processing workflow → "Input validation rules", "Error recovery strategy", "Output structure", "Notification triggers"

### 2.2: Present Gray Areas

Display the domain analysis and gray areas as a multi-select:

` ` `
Domain: [identified domain]

Which areas should we discuss before building?
(This ensures the agents match your expectations.)

☐ [Area 1] — [1-2 example questions]
☐ [Area 2] — [1-2 example questions]
☐ [Area 3] — [1-2 example questions]
☐ [Area 4] — [1-2 example questions]

──────────────────────────────────────────────────────
→ Select areas to discuss (comma-separated numbers, or "all")
→ Type "skip" to proceed directly to architect
──────────────────────────────────────────────────────
` ` `

Wait for user selection. If "skip": proceed to Step 2.5 with no discussion decisions.

### 2.3: Discuss Selected Areas

For each selected area:

1. Announce: "Let's discuss [Area]."
2. Ask 4 questions, one at a time:
   - Each question offers 2-3 concrete options
   - Include a "You decide" option where reasonable
   - Each answer should inform the next question
3. After 4 questions, ask: "More questions about [area], or move to next?"
   - "More" → ask up to 4 more, then force move
   - "Next" → proceed to next area

### 2.4: Compile Discussion Summary

After all selected areas are discussed, compile a structured summary:

` ` `
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► DISCUSSION SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Use Case
[Original user input — verbatim]

### Decisions Made
- **[Area 1]:** [Decision summary]
- **[Area 2]:** [Decision summary]

### Additional Context
- [Key details surfaced during discussion]

### Open Areas (Claude's Discretion)
- [Areas not discussed or where user said "you decide"]
` ` `

### 2.5: Internal Classification (Not User-Facing)

Silently evaluate the enriched input (original + discussion decisions) against the researcher skip criteria from the old Step 2. Apply the same 5-dimension check:
1. Model selection with rationale
2. Tool requirements (specific Orq.ai tool types)
3. Guardrail requirements (concrete rules)
4. Prompt strategy (role definition, constraints)
5. Context needs (knowledge bases, variables)

Store the researcher decision (RUN or SKIP) for use in Step 5 (generation pipeline). Do NOT display this to the user.

Proceed to Step 3 (Run Architect).
```

## Open Questions

1. **"Skip" option in gray area presentation**
   - What we know: User said discussion always runs. But there is value in allowing a user who provides a very detailed input to skip the discussion entirely with a single "skip" command.
   - What's unclear: Does "always runs" mean "always presented" (user can skip) or "always mandatory" (user must answer at least one area)?
   - Recommendation: Allow "skip" option in Step 2.2. The discussion step always runs (is always presented), but the user can choose to skip it. This respects the spirit of "always runs" while avoiding frustrating experienced users with detailed inputs.

2. **Researcher classification after discussion**
   - What we know: The old Steps 2-3 handled classification AND user confirmation. User said no user-facing classification gate.
   - What's unclear: Should the internal classification still exist, or should discussion output make it unnecessary?
   - Recommendation: Keep internal classification (Step 2.5). The discussion enriches the input but does not guarantee all 5 researcher dimensions are covered. Internal classification is lightweight and invisible to the user.

3. **Pipeline metadata schema change**
   - What we know: `pipeline-run.json` currently records `input_classification` with the old stage structure.
   - What's unclear: Exact schema for recording discussion metadata.
   - Recommendation: Add a `discussion` sub-object to `input_classification` (see Example 3 above). Remove the old `stages.classifier` entry. This is a metadata-only change with no functional impact.
