# Phase 4.1: Introducing a Discussion Phase on Start if Needed - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a GSD-style structured discussion step to the `/orq-agent` orchestrator that always runs at the start of every invocation. The discussion analyzes the use case, surfaces domain-specific gray areas, and gathers implementation decisions before the pipeline proceeds. This replaces the current Step 2 (Classify Input Depth) and Step 3 (Confirm with User checkpoint) in the orchestrator workflow.

</domain>

<decisions>
## Implementation Decisions

### Trigger criteria
- Discussion phase always runs on every `/orq-agent` invocation — no skip logic, no classification gate
- Applies equally to brief inputs ("build me a FAQ bot") and detailed inputs (the discussion will naturally be shorter when the user has already provided detail)

### Discussion format
- GSD-style structured discussion: analyze the use case, generate 3-4 domain-specific gray areas, let user select which to discuss
- Gray areas are generated per use case, not from a fixed template (e.g., FAQ bot gets "Knowledge sources", "Tone & personality", "Escalation behavior", "Scope boundaries")
- 4 questions per selected area, then offer "more or next?" — same cadence as GSD discuss-phase

### Pipeline integration
- Replaces current Step 2 (Classify Input Depth) and Step 3 (Confirm with User checkpoint) entirely
- Discussion output becomes the enriched input fed to the architect in Step 4
- The existing Step 5 (Blueprint Review checkpoint) remains unchanged — user still reviews the architect's blueprint before generation

### Claude's Discretion
- How to identify domain-specific gray areas from the use case description
- How to format the discussion output so the architect can consume it
- Whether to keep any lightweight classification logic internally (without exposing it as a user-facing checkpoint)

</decisions>

<specifics>
## Specific Ideas

- "If the user says 'build me a basic FAQ Bot' this is not enough context to work from" — the discussion must ensure there is more than enough context before proceeding
- Follow GSD discuss-phase patterns: multi-select gray areas, structured questions with concrete options, scope guardrails

</specifics>

<deferred>
## Deferred Ideas

None — discussion stayed within phase scope

</deferred>
