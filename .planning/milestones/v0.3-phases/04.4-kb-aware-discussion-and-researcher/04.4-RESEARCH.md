# Phase 04.4: KB-Aware Discussion & Researcher - Research

**Researched:** 2026-02-25
**Domain:** Knowledge base design guidance within an LLM-agent pipeline (prompt engineering, not infrastructure)
**Confidence:** HIGH

## Summary

Phase 04.4 makes the existing pipeline knowledge-base-aware across three touchpoints: the Discussion step (surfaces KB-specific questions), the Researcher subagent (produces KB design guidance), and the Spec Generator + Orchestration Generator (includes actionable KB setup instructions in output). This is a prompt-engineering phase -- no new files are created, no libraries are added. All work modifies existing `.md` files: `orq-agent/commands/orq-agent.md` (orchestrator), `orq-agent/agents/architect.md`, `orq-agent/agents/researcher.md`, `orq-agent/agents/spec-generator.md`, `orq-agent/agents/orchestration-generator.md`, `orq-agent/templates/orchestration.md`, and `orq-agent/agents/readme-generator.md`.

The approach follows a two-pass design: Phase 04.4 produces design recommendations (chunking heuristics, metadata schema, document prep guidance), while Phase 04.5 (deferred) handles automated provisioning with actual data analysis. The researcher cannot see real data -- all recommendations are data-type heuristics, not content-tuned.

**Primary recommendation:** Modify five existing prompt files to add KB-awareness at each pipeline stage, using the architect's existing `needs_kb` classification as the gate that determines whether KB sections appear or are skipped.

## Architecture Patterns

### Pattern 1: Architect KB Classification Gate

**What:** The architect subagent adds `needs_kb: true/false` and `kb_type` fields to each agent in the blueprint output. This acts as the gate for all downstream KB processing.

**When to use:** Every pipeline invocation. The architect already analyzes the use case -- KB classification is a natural extension of its existing analysis.

**How it works in the current blueprint format:**

The architect blueprint currently outputs per-agent sections like:

```markdown
#### 1. [agent-key]
- **Role:** [role description]
- **Responsibility:** [what this agent does]
- **Model recommendation:** [provider/model-name]
- **Tools needed:** [list of Orq.ai tool types]
- **Receives from:** [upstream agent or "user input"]
- **Passes to:** [downstream agent or "final output"]
```

Add two fields per agent:

```markdown
#### 1. [agent-key]
- **Role:** [role description]
- **Responsibility:** [what this agent does]
- **Model recommendation:** [provider/model-name]
- **Tools needed:** [list of Orq.ai tool types]
- **Knowledge base:** [none | documents | faq | product-data | policy | ...]
- **KB description:** [what the KB should contain, or "N/A"]
- **Receives from:** [upstream agent or "user input"]
- **Passes to:** [downstream agent or "final output"]
```

The architect determines KB need via LLM reasoning about the use case -- not keyword matching. An FAQ bot clearly needs a KB. A data transformation pipeline clearly does not. The architect's existing complexity gate logic already evaluates use case requirements; KB classification extends this naturally.

**Confidence:** HIGH -- this follows the existing blueprint extension pattern used by previous phases (e.g., Phase 04.3 added prompt strategy fields).

### Pattern 2: Conditional Discussion KB Section

**What:** A dedicated KB section appears in the discussion flow ONLY when the architect identifies KB needs. This requires a structural change: the architect must run BEFORE the discussion step for KB classification, OR the discussion step must do its own preliminary KB detection.

**Critical design issue:** In the current pipeline, the Discussion step (Step 2) runs BEFORE the Architect (Step 3). But the CONTEXT.md says "appears after standard gray areas, only when architect flags KB need." This creates a sequencing problem -- the architect has not yet run when discussion happens.

**Resolution options:**

1. **Discussion does its own lightweight KB detection** (recommended): The discussion step already analyzes the use case to identify gray areas. It can simultaneously detect KB-relevant signals (mentions of documents, policies, FAQs, data retrieval, knowledge lookup, etc.) without full architect analysis. This keeps the pipeline order unchanged. The architect later confirms and refines the KB classification.

2. **Move architect before discussion**: Breaks the current flow where discussion enriches architect input. Not recommended.

3. **Two-pass discussion**: Discussion runs, architect runs, KB questions presented as a follow-up. Adds user friction.

**Recommended approach:** Option 1. The discussion step uses heuristic signals (mentions of documents, knowledge, policies, FAQs, data lookup, retrieval) to decide whether to show the KB section. The architect later produces the formal `needs_kb` classification in the blueprint. If the discussion showed KB questions and the architect disagrees, no harm -- the KB answers are simply not used. If the discussion skipped KB questions but the architect flags KB need, the researcher uses defaults.

**Confidence:** HIGH -- option 1 is the simplest approach that preserves pipeline order.

### Pattern 3: Researcher KB Design Section

**What:** The researcher subagent adds a "Knowledge Base Design" section to its research brief for each agent that has `needs_kb: true` in the blueprint. This section provides data-type heuristic recommendations.

**Structure of the KB Design section per agent:**

```markdown
### Knowledge Base Design: [kb-name]

**Source type:** [PDFs | web pages | database records | API responses | mixed]
**Data structure:** [structured | semi-structured | unstructured]

**Chunking Strategy:**
- Recommended approach: [semantic chunking | fixed-size | paragraph-based]
- Chunk size: ~[N] tokens with [N] token overlap
- Rationale: [why this strategy for this data type]

**Metadata Fields:**
- `source`: [document name/URL/identifier]
- `timestamp`: [last updated date]

**Document Preparation:**
- [Step 1: e.g., "Extract text from PDFs, remove headers/footers/page numbers"]
- [Step 2: e.g., "Normalize encoding to UTF-8"]
- [Step 3: e.g., "Split multi-topic documents at heading boundaries"]

**KB Architecture:**
- [shared | per-agent] with rationale
```

**Chunking heuristic defaults by data type (training knowledge -- verified as industry-standard practice):**

| Data Type | Strategy | Chunk Size | Overlap | Rationale |
|-----------|----------|------------|---------|-----------|
| PDFs (long docs) | Semantic chunking at heading boundaries | ~512 tokens | 50-100 tokens | Preserves section coherence; headings are natural boundaries |
| FAQs | One chunk per Q&A pair | Varies (typically 100-300 tokens) | None | Each Q&A is a self-contained unit |
| Web pages | Paragraph-based chunking | ~300-500 tokens | 50 tokens | Web content is already structured by paragraphs |
| Database records | One chunk per record | Varies | None | Each record is a self-contained entity |
| API responses | JSON field-based chunking | Varies | None | Preserve field relationships within each object |
| Policy documents | Section-based chunking at numbered sections | ~512 tokens | 100 tokens | Policy sections are self-contained; overlap captures cross-references |

**Confidence:** MEDIUM -- chunking defaults are well-established best practices but exact numbers vary by use case. These are heuristics, not prescriptions, which aligns with the two-pass design (Phase 4.5 refines with real data).

### Pattern 4: Spec Generator KB Context References

**What:** The spec generator already produces a Context section with `knowledge_bases`, `memory_stores`, and `variables`. For KB-needing agents, it references named KBs with descriptive names.

**Current context section pattern:**
```json
{
  "knowledge_bases": [
    { "knowledge_id": "company-faq-kb", "description": "Company FAQ and policy documents" }
  ]
}
```

**Enhanced pattern for KB-aware agents:**
```json
{
  "knowledge_bases": [
    {
      "knowledge_id": "product-docs-kb",
      "description": "Product documentation and user guides. See ORCHESTRATION.md KB Design section for setup details."
    }
  ]
}
```

The key change: KB names are descriptive (e.g., `product-docs-kb`, `hr-policy-kb`) rather than generic (`kb-1`). Each references ORCHESTRATION.md for setup details. This is a minimal change to the spec generator -- it already handles knowledge_bases.

**Confidence:** HIGH -- this follows the existing pattern exactly, just with more descriptive naming.

### Pattern 5: ORCHESTRATION.md KB Design Section

**What:** A new section in ORCHESTRATION.md (and its template) that consolidates KB design information for the entire swarm. This is the authoritative location for KB setup instructions.

**Template addition:**

```markdown
## Knowledge Base Design

{{KB_DESIGN}}

> KB design recommendations for this swarm. Each named KB is referenced by agent specs in their Context section.
> Includes: source type, chunking strategy, metadata fields, and document preparation steps per KB.
> Not applicable if no agents need knowledge bases -- omit this section.
```

**Filled example:**

```markdown
## Knowledge Base Design

### product-docs-kb

**Used by:** `product-support-resolver-agent`
**Source type:** PDFs and web pages (product documentation, user guides)
**Data structure:** Semi-structured (headings, sections, tables)

**Chunking Strategy:**
- Approach: Semantic chunking at heading boundaries
- Chunk size: ~512 tokens with 50-100 token overlap
- Rationale: Product docs have clear heading structure; chunking at headings preserves topic coherence

**Metadata Fields:**
| Field | Description |
|-------|-------------|
| `source` | Document filename or URL |
| `timestamp` | Date the document was last updated |

**Document Preparation:**
1. Extract text from PDFs (remove headers, footers, page numbers)
2. Normalize encoding to UTF-8
3. Preserve heading hierarchy for semantic chunking boundaries
4. Remove duplicate content across document versions

**Access Control:** Internal-only -- restrict to authenticated users

---

### company-faq-kb

**Used by:** `product-support-resolver-agent`, `product-support-triage-agent`
**Source type:** Structured Q&A pairs
**Data structure:** Structured

**Chunking Strategy:**
- Approach: One chunk per Q&A pair
- Chunk size: Varies (typically 100-300 tokens per pair)
- Rationale: Each Q&A is a self-contained retrieval unit

**Metadata Fields:**
| Field | Description |
|-------|-------------|
| `source` | FAQ category or section name |
| `timestamp` | Date the Q&A was last reviewed |

**Document Preparation:**
1. Export Q&A pairs with clear question/answer separation
2. Categorize by topic for metadata tagging
3. Remove internal-only Q&As if KB is customer-facing

**Access Control:** Public
```

**Confidence:** HIGH -- this follows the existing ORCHESTRATION.md section pattern and template placeholder convention.

### Anti-Patterns to Avoid

- **Keyword matching for KB detection:** The user explicitly decided the architect should use LLM reasoning, not keyword matching. Do not add regex or keyword lists for KB detection.
- **Embedding model recommendations in this phase:** Explicitly deferred to Phase 4.5. The researcher must NOT recommend specific embedding models.
- **Separate KB design file:** The user decided KB design goes in ORCHESTRATION.md, not a separate file.
- **Generic KB names:** Use descriptive names like `product-docs-kb`, not `kb-1` or `knowledge-base`.
- **Overly precise chunking numbers:** These are heuristic defaults. Do not present them as optimal values -- they are starting points for Phase 4.5 to refine.

## Common Pitfalls

### Pitfall 1: Pipeline Ordering -- Discussion Before Architect

**What goes wrong:** The CONTEXT.md says "appears after standard gray areas, only when architect flags KB need." But the discussion step runs BEFORE the architect in the current pipeline (Step 2 before Step 3).

**Why it happens:** The user envisioned the architect flagging KB need, but the pipeline order puts discussion first.

**How to avoid:** The discussion step must do its own lightweight KB signal detection (mentions of documents, policies, FAQs, data retrieval). The architect later formalizes this as `needs_kb: true/false`. Both detection mechanisms should broadly agree for typical use cases.

**Warning signs:** If the discussion shows KB questions for a use case where the architect later says no KB needed, or vice versa.

### Pitfall 2: Blueprint Format Breaking Downstream Parsers

**What goes wrong:** Adding `needs_kb` and `kb_type` fields to the architect blueprint could break downstream subagents that parse the blueprint format if they are not updated to expect these fields.

**Why it happens:** The researcher, spec generator, orchestration generator, and other subagents all read the blueprint. They parse it based on expected structure.

**How to avoid:** The new fields are purely additive (new lines in the existing per-agent section). Downstream subagents that do not care about KB fields will simply ignore them -- markdown parsing is tolerant of additional fields. However, the researcher and spec generator prompts must be updated to explicitly look for and use these fields.

**Warning signs:** Downstream subagents producing output that ignores KB information even when `needs_kb: true`.

### Pitfall 3: KB Section Shown for Non-KB Use Cases

**What goes wrong:** The discussion step shows KB questions for a use case that clearly has no knowledge base needs (e.g., a data transformation pipeline, a code generator).

**Why it happens:** Overly broad KB signal detection in the discussion step.

**How to avoid:** Keep KB signal detection conservative. Only trigger on clear indicators: mentions of documents, policies, FAQs, knowledge lookup, data retrieval, reference materials. Do NOT trigger on generic data processing, computation, or generation tasks.

**Warning signs:** Users consistently skipping or dismissing KB questions.

### Pitfall 4: Researcher Recommending Embedding Models

**What goes wrong:** The researcher recommends specific embedding models despite this being deferred to Phase 4.5.

**Why it happens:** The researcher prompt already recommends models for agents; it is natural to extend this to embedding models.

**How to avoid:** Add an explicit constraint in the researcher's KB Design section: "Embedding model selection deferred to Phase 4.5. Do not recommend specific embedding models."

**Warning signs:** Researcher output containing embedding model names like `text-embedding-3-small` or `embed-v3`.

### Pitfall 5: ORCHESTRATION.md KB Section for Single-Agent Swarms

**What goes wrong:** The KB Design section is omitted for single-agent swarms because the orchestration generator simplifies output for single agents.

**Why it happens:** The orchestration generator has a "Single-Agent Swarm Handling" section that strips most content.

**How to avoid:** The KB Design section should be included even for single-agent swarms if that agent needs a knowledge base. Update the single-agent handling logic to include KB Design when relevant.

**Warning signs:** Single-agent swarms with KB needs missing setup instructions.

## Open Questions

1. **Discussion KB detection vs. Architect KB detection divergence**
   - What we know: Discussion runs before Architect. Both need to detect KB needs independently.
   - What is unclear: How often will they disagree? What happens when they disagree?
   - Recommendation: Treat discussion KB detection as "best effort" -- if it shows KB questions that the architect later ignores, no harm done. If it misses KB questions, the researcher falls back to defaults from the architect's `needs_kb` field. Document this as expected behavior.

2. **Shared vs. Per-Agent KB decision**
   - What we know: User left this as Claude's discretion. The researcher decides based on use case complexity.
   - What is unclear: What heuristics should drive the shared-vs-per-agent decision?
   - Recommendation: Default to shared KB when all agents in the swarm need the same information domain. Use per-agent KBs when agents have distinct knowledge domains (e.g., one agent needs product docs, another needs support scripts). The researcher prompt should include a decision heuristic.

3. **README KB setup steps**
   - What we know: README generator should include KB setup steps when KBs are present.
   - What is unclear: How detailed should README KB setup be vs. ORCHESTRATION.md KB Design section?
   - Recommendation: README includes high-level setup steps ("Create knowledge base `product-docs-kb` in Orq.ai Studio with the content described in ORCHESTRATION.md"). ORCHESTRATION.md has the detailed design. Avoid duplicating design details in README.
