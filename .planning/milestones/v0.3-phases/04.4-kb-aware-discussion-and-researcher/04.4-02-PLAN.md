  <action>
Modify the researcher subagent prompt to add a "Knowledge Base Design" section to its research brief output for each agent that has `Knowledge base` set to something other than `none` in the architect blueprint.

**Where to add:** After the existing per-agent research sections (model recommendation, prompt strategy, tool recommendations, guardrail suggestions, context needs), add a conditional KB Design section that only appears for agents where `Knowledge base` is not `none`.

**KB Design section structure per agent:**

```markdown
### Knowledge Base Design: [descriptive-kb-name]

**Source type:** [PDFs | web pages | database records | API responses | mixed]
**Data structure:** [structured | semi-structured | unstructured]

**Chunking Strategy:**
- Approach: [semantic chunking at heading boundaries | one chunk per Q&A pair | paragraph-based | record-based | field-based]
- Chunk size: ~[N] tokens with [N] token overlap
- Rationale: [why this strategy for this data type]

**Metadata Fields:**
| Field | Description |
|-------|-------------|
| `source` | [document name/URL/identifier] |
| `timestamp` | [last updated date] |

**Document Preparation:**
1. [Data-type-specific preparation step]
2. [Additional steps as needed]

**KB Architecture:** [shared | per-agent] with rationale

**Note:** Embedding model selection deferred to Phase 4.5. Chunking defaults are heuristic starting points.
```

**Embed chunking heuristic defaults as training knowledge in the researcher prompt** (so the researcher knows what to recommend without web search):

| Data Type | Strategy | Chunk Size | Overlap | Rationale |
|-----------|----------|------------|---------|-----------|
| PDFs (long docs) | Semantic chunking at heading boundaries | ~512 tokens | 50-100 tokens | Preserves section coherence |
| FAQs | One chunk per Q&A pair | Varies (100-300 tokens) | None | Each Q&A is self-contained |
| Web pages | Paragraph-based chunking | ~300-500 tokens | 50 tokens | Web content structured by paragraphs |
| Database records | One chunk per record | Varies | None | Each record is self-contained |
| API responses | JSON field-based chunking | Varies | None | Preserve field relationships |
| Policy documents | Section-based at numbered sections | ~512 tokens | 100 tokens | Sections are self-contained |

**Shared vs per-agent KB decision heuristic:** Default to shared KB when all agents need the same information domain. Use per-agent KBs when agents have distinct knowledge domains (e.g., one agent needs product docs, another needs HR policies). This is Claude's discretion per CONTEXT.md.

**KB name convention:** Use descriptive, reusable names like `product-docs-kb`, `hr-policy-kb` -- NOT generic names like `kb-1`.

**Explicit constraint:** "Embedding model selection deferred to Phase 4.5. Do NOT recommend specific embedding models (e.g., text-embedding-3-small, embed-v3)."

**Metadata fields:** Only `source` and `timestamp` per user decision. Do not add additional metadata fields.

**If discussion provided KB context** (source type, freshness, access control answers), use those answers to inform the recommendations. If no discussion KB context, fall back to heuristic defaults based on the architect's `kb_type`.
