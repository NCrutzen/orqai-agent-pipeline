  <action>
Modify the orchestrator prompt (orq-agent.md) to add a conditional KB discussion section in Step 2 (the Discussion step).

**Location:** After the existing gray area discussion flow, add a dedicated KB section that appears ONLY when the use case involves knowledge retrieval signals.

**KB signal detection (heuristic, NOT keyword matching):** The discussion step should detect KB-relevant signals by reasoning about whether the use case involves agents that need to look up information from a corpus. Signals include: mentions of documents, policies, FAQs, data retrieval, knowledge lookup, reference materials, manuals, guides, databases of information to search. Do NOT trigger on generic data processing, computation, code generation, or orchestration tasks.

**KB section format when triggered:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ORQ ► KNOWLEDGE BASE SETUP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Your use case involves knowledge retrieval. Let's clarify the data sources:

1. **Source type & format:**
   What types of documents will agents need access to?
   a) PDFs (manuals, reports, policies)
   b) Web pages / HTML content
   c) Database records / structured data
   d) API responses / live data feeds
   e) Mixed sources

2. **Data freshness:**
   How often does this content change?
   a) Rarely (policies, reference docs -- updated quarterly or less)
   b) Weekly (reports, summaries)
   c) Daily (news, inventory, pricing)
   d) Real-time (live feeds, current status)

3. **Access control:**
   Who should be able to query this knowledge?
   a) Public (anyone)
   b) Internal-only (authenticated employees)
   c) Per-user rules (different access levels)

──────────────────────────────────────────────────────
→ Answer each question (e.g., "1a, 2a, 3b")
→ Type "skip" to use defaults
──────────────────────────────────────────────────────
```

**When KB section is shown:** Include the user's KB answers in the discussion summary that feeds into the architect. The summary should include a "Knowledge Base Context" sub-section with the user's source type, freshness, and access control answers.

**When KB section is skipped (no KB signals):** The discussion summary should NOT mention knowledge bases at all. The pipeline proceeds normally.

**Important:** This heuristic detection runs BEFORE the architect. The architect later formalizes KB classification in the blueprint. If discussion shows KB questions but architect says no KB needed, no harm -- answers are simply unused. If discussion misses KB questions but architect flags KB need, the researcher falls back to defaults.

Follow existing patterns: GSD banner format (ORQ ► [STAGE]), separator lines, user prompt format matching the existing discussion step style.
