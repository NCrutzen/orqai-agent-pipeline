# Phase 4.4: KB-Aware Discussion & Researcher - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the pipeline knowledge-base-aware end-to-end. The Discussion step surfaces KB source questions when relevant, the Researcher produces actionable KB design guidance, and the Spec Generator includes KB setup instructions in its output. Phase 4.5 handles automated KB provisioning — this phase produces the design and instructions only.

</domain>

<decisions>
## Implementation Decisions

### KB Detection Logic
- Architect subagent classifies KB need via LLM reasoning (not keyword matching) as part of its existing use case analysis
- Output includes `needs_kb: true/false` plus `kb_type` per agent (e.g., 'documents', 'faq', 'product-data')
- Supports multiple knowledge bases per swarm (different agents can reference different KBs)

### Claude's Discretion
- Whether to use per-agent KBs or a shared KB per swarm — researcher decides based on use case complexity

### Discussion KB Questions
- Separate dedicated KB section in the discussion flow (appears after standard gray areas, only when architect flags KB need)
- Asks about source type + format (PDFs, web pages, database, API; structured/unstructured)
- Asks about data freshness with simple categories: 'rarely', 'weekly', 'daily', 'real-time'
- Asks about basic access control: public, internal-only, or per-user access rules
- Pipeline correctly skips entire KB section when architect says no KB needed

### Researcher KB Output
- Best-practice chunking defaults per data type (e.g., "For PDFs, semantic chunking at ~512 tokens with overlap")
- These are heuristic recommendations — refined later when phase 4.5 analyzes actual data
- Minimal metadata fields only: source and timestamp
- Embedding model selection deferred entirely to phase 4.5
- Document preparation guidance per source type included

### Spec Generator KB Instructions
- KB design section added to ORCHESTRATION.md (not a separate file)
- Actionable setup guide per KB: source type, format, chunking defaults, metadata fields, document preparation steps
- Each agent spec references its KBs by name in the context section (e.g., "Uses: product-docs-kb")
- ORCHESTRATION.md defines what each named KB contains; agent specs link to them
- Includes document preparation guidance per source type (e.g., "PDFs: extract text, remove headers/footers")

</decisions>

<specifics>
## Specific Ideas

- Researcher can't see actual data — recommendations are data-type heuristics, not content-tuned. Phase 4.5 refines when ingestion pipeline analyzes real data
- Two-pass approach: design now (4.4), refine with actual data later (4.5)
- KB names should be descriptive and reusable across phases (e.g., `product-docs-kb` not `kb-1`)

</specifics>

<deferred>
## Deferred Ideas

- Selectable KB provider (not just Supabase) — Orq.ai has its own MCP for KB management. Make KB backend configurable → Phase 4.5
- Fetch existing KBs from Supabase/provider to prevent overlap when creating new ones → Phase 4.5
- Embedding model selection → Phase 4.5 (needs actual data analysis)
- Chunking refinement based on real data analysis → Phase 4.5

</deferred>
