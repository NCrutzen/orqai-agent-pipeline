# Phase 4.2: Tool Selection and MCP Servers - Research

**Researched:** 2026-02-24
**Domain:** Tool resolution pipeline stage, MCP server catalog, Orq.ai tool configuration
**Confidence:** HIGH

## Summary

This phase adds a new "tool resolver" subagent to the pipeline that runs after the architect and before the domain researcher. Its job is to analyze the architect blueprint, consult a curated tool catalog plus web search, and produce a TOOLS.md file per swarm with verified, copy-paste-ready tool configurations. The key technical domains are: (1) Orq.ai's native tool configuration formats (especially MCP), (2) the MCP server ecosystem for curating 15-25 servers, and (3) the subagent prompt design that drives two-phase resolution (swarm-wide landscape, then per-agent assignment).

Research confirms that Orq.ai supports MCP tools via a two-step process: first create the MCP tool (with `server_url` and `connection_type`), then reference it by `tool_id` in agent settings. The MCP ecosystem has matured significantly -- there are official reference servers and thousands of community servers covering every major integration category. The curated catalog should focus on the ~20 most common enterprise integrations with verified GitHub sources.

**Primary recommendation:** Build the tool resolver as a self-contained subagent with WebSearch capability, consuming the architect blueprint and a new unified tool catalog reference file, producing TOOLS.md with capability-first organization and Orq.ai-native configuration JSON.

### Pattern 1: Two-Phase Tool Resolution

**What:** The tool resolver operates in two phases within a single invocation:
1. **Swarm-wide tool landscape** -- Analyze ALL agent roles from the blueprint, identify all integration needs across the swarm, consult catalog + web search
2. **Per-agent assignment** -- Map each tool to specific agents, identify shared tools, provide per-agent config

**When to use:** Always -- this is the locked decision from CONTEXT.md.

**Why it works:** Phase 1 prevents duplicate tool recommendations (e.g., two agents both needing Slack but getting different MCP server recommendations). Phase 2 ensures each agent gets only the tools it needs with role-appropriate configuration.

### Pattern 2: Capability-First Organization in TOOLS.md

**What:** TOOLS.md organizes by CAPABILITY needed (e.g., "Send Slack messages") first, then maps to tool type (MCP/HTTP/function). Not organized by tool type.

**Structure:**
```markdown
# Tools: [swarm-name]

## Swarm Tool Landscape
[Table of all capabilities needed across the swarm, mapped to tool types]

## Shared Tools
[Tools used by multiple agents -- configure once in Orq.ai Studio]

## Per-Agent Tool Assignments

### [agent-key-1]
[Tools for this agent with config JSON]

### [agent-key-2]
[Tools for this agent with config JSON]

## Setup Instructions
[Credential acquisition and configuration steps per external service]
```

### Pattern 3: MCP-First Resolution Chain

**What:** For each identified capability need, the resolver follows a priority chain:
1. **Built-in tool** -- If it is a generic capability (web search, date, memory), use the built-in
2. **MCP server** -- If an external integration, check curated catalog first, then web search for MCP server
3. **HTTP tool** -- If no MCP server exists, recommend HTTP tool with endpoint template
4. **Function tool** -- If custom business logic, provide JSON Schema scaffold
5. **Code tool** -- If computation needed, recommend Python code tool

**When to use:** Every capability identified during resolution.

### Pattern 4: Orq.ai MCP Tool Configuration

**What:** Orq.ai's MCP tool configuration is a two-step process:

**Step 1 -- Create the MCP tool (one-time setup in Orq.ai Studio or via API):**
```json
{
  "key": "github-mcp",
  "description": "GitHub repository management and code operations",
  "type": "mcp",
  "path": "Default",
  "mcp": {
    "server_url": "https://api.githubcopilot.com/mcp/",
    "connection_type": "http"
  }
}
```

**Step 2 -- Reference the tool in agent settings:**
```json
{
  "settings": {
    "tools": [
      {
        "type": "mcp",
        "tool_id": "{{GITHUB_MCP_TOOL_ID}}"
      }
    ]
  }
}
```

**Key fields:**
- `server_url`: The remote MCP server endpoint (Streamable HTTP transport)
- `connection_type`: `"http"` (Orq.ai uses HTTP-based MCP connections)
- `tool_id`: Returned by Orq.ai when the tool is created -- used for agent references

**Confidence:** HIGH -- verified from Orq.ai official docs at docs.orq.ai.

### Anti-Patterns to Avoid

- **Recommending MCP servers without verifying they exist:** The resolver must use web search to confirm GitHub/npm existence before recommending. Hallucinated server URLs are worse than no recommendation.
- **Duplicating tool schemas between TOOLS.md and agent specs:** TOOLS.md provides the landscape and config; spec generator consumes it. Spec generator should reference TOOLS.md recommendations, not regenerate tool configs from scratch.
- **Over-specifying MCP server URLs that will change:** Use `{{PLACEHOLDER}}` for server URLs that are deployment-specific. Only include concrete URLs for well-known hosted MCP servers.
- **Catalog bloat:** Keep the reference catalog at 500-1000 words. Each MCP entry should be 2-3 lines max. The resolver uses catalog as a starting point and web search for depth.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MCP server discovery | Custom scraping or hardcoded list that rots | Web search + curated catalog combination | MCP ecosystem changes rapidly; web search catches new servers, catalog provides baseline reliability |
| Tool configuration JSON | Freestyle JSON in prompts | Template with Orq.ai-native format from docs | Orq.ai has specific field names and structure; freestyle leads to invalid configs |
| Capability-to-tool mapping | Hardcoded if/else logic | LLM reasoning with catalog as context | The mapping is nuanced (e.g., "send email" could be MCP, HTTP, or function depending on context) |
| Server existence verification | Trust training data | WebSearch to GitHub/npm during resolution | MCP servers appear and disappear; only live verification is reliable |

**Key insight:** The tool resolver is fundamentally an LLM reasoning task with structured output, not a deterministic lookup. The catalog provides grounding; web search provides verification; the LLM provides the judgment to map capabilities to tools.

## Common Pitfalls

### Pitfall 1: MCP Server URL Rot
**What goes wrong:** Recommended MCP server URLs become stale as servers move, rebrand, or shut down.
**Why it happens:** The MCP ecosystem is young and volatile; server projects frequently change hosting.
**How to avoid:** Always include the GitHub source URL alongside the server URL. Use `{{PLACEHOLDER}}` for server URLs that users must configure themselves. Only hardcode URLs for well-established hosted servers (e.g., official Notion MCP at `https://mcp.notion.so/`).
**Warning signs:** TOOLS.md links returning 404s; users reporting connection failures.

### Pitfall 2: Tool Resolver Hallucinating MCP Servers
**What goes wrong:** The LLM invents plausible-sounding MCP server names and URLs that do not exist.
**Why it happens:** Training data contains many hypothetical MCP server examples from blog posts and tutorials.
**How to avoid:** Mandatory web search verification step in the resolver prompt. The resolver must search GitHub for any MCP server it recommends and confirm it exists before including it. If web search cannot confirm, fall back to HTTP/function tool type.
**Warning signs:** MCP server names not found in any search results; URLs that look plausible but lead nowhere.

### Pitfall 3: Overloading Agents with Tools
**What goes wrong:** Every agent gets recommended 10+ tools, making the agent confused and slow.
**Why it happens:** The resolver identifies all possible tools without filtering for relevance.
**How to avoid:** Per-agent assignment phase must be strict: each agent gets ONLY tools that directly serve its stated responsibilities from the architect blueprint. If a tool is "nice to have" but not core, omit it.
**Warning signs:** Agents with more than 5-6 tools; tools that duplicate capabilities.

### Pitfall 4: Catalog Context Window Bloat
**What goes wrong:** The unified tool catalog grows beyond 1000 words and consumes too much of the subagent's context window.
**Why it happens:** Each new MCP server entry tempts adding more detail.
**How to avoid:** Strict word budget (500-1000 words). Each catalog entry is 2-3 lines: name, URL, capability keywords, auth complexity. The resolver uses web search for details when needed.
**Warning signs:** Catalog exceeding 1000 words; resolver producing worse results because prompt is too long.

### Pitfall 5: TOOLS.md and Spec Generator Conflict
**What goes wrong:** TOOLS.md recommends one tool configuration, but the spec generator produces a different one, confusing the user.
**Why it happens:** Spec generator has its own tool logic that does not reference TOOLS.md.
**How to avoid:** Update spec generator prompt to explicitly reference TOOLS.md when available. Spec generator should use TOOLS.md recommendations as authoritative for tool selection and configuration. Spec generator adds tool schemas for function tools but respects TOOLS.md for MCP/HTTP/built-in choices.
**Warning signs:** Differences between TOOLS.md tool list and agent spec tool sections.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| SSE (Server-Sent Events) transport for remote MCP | Streamable HTTP transport | MCP spec 2025-03-26 | All new MCP servers use HTTP; Orq.ai uses `connection_type: "http"` |
| Stdio-only MCP servers | Remote HTTP MCP servers | 2025 | MCP servers can now be hosted remotely, no local process needed -- enables Orq.ai cloud integration |
| Generic tool recommendations | MCP-first with verified servers | 2025-2026 | MCP ecosystem matured to 7000+ servers; reliable options exist for most enterprise integrations |
| Each platform invents tool format | Orq.ai native MCP tool type | 2025 | `{ "type": "mcp", "server_url": "...", "connection_type": "http" }` is the standard Orq.ai format |

**Deprecated/outdated:**
- SSE transport: Deprecated in MCP spec 2025-03-26 in favor of Streamable HTTP
- Local-only MCP servers: Still supported but remote HTTP servers are preferred for cloud agent platforms like Orq.ai

## Open Questions

1. **Remote vs self-hosted MCP server URLs**
   - What we know: Orq.ai MCP tools use `server_url` pointing to a remote HTTP endpoint. Some MCP servers (Notion, Brave Search) offer hosted endpoints; most require self-hosting.
   - What's unclear: Which servers offer stable hosted endpoints vs requiring user deployment. This changes frequently.
   - Recommendation: The catalog should indicate "hosted" vs "self-host" per server. For self-hosted, TOOLS.md uses `{{PLACEHOLDER}}` for the URL.

2. **Spec generator update scope**
   - What we know: The spec generator currently handles tool recommendations independently. With TOOLS.md available, it should defer to TOOLS.md for tool selection.
   - What's unclear: How much to modify the spec generator prompt vs keeping it independent with TOOLS.md as optional input.
   - Recommendation: Minimal spec generator change -- add a clause saying "If TOOLS.md is available for this swarm, use its tool recommendations and configurations as authoritative. Add function tool JSON schemas as needed but do not override TOOLS.md MCP/built-in/HTTP tool selections."

3. **Researcher interaction with TOOLS.md**
   - What we know: The researcher runs after the tool resolver. TOOLS.md provides the tool landscape.
   - What's unclear: Should the researcher skip its own tool recommendation section entirely, or refine TOOLS.md recommendations?
   - Recommendation: Researcher should still produce tool recommendations in its brief (since TOOLS.md might be incomplete), but note "See TOOLS.md for authoritative tool configuration." This prevents the researcher from contradicting TOOLS.md while still allowing it to surface tool needs the resolver missed.
