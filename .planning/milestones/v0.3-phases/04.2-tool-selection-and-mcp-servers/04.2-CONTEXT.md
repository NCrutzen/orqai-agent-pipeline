# Phase 4.2: Tool Selection and MCP Servers - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a tool resolver pipeline stage and unified tool catalog so the generated Orq.ai agent specs include accurate, verified tool recommendations with copy-paste-ready configuration. The tool resolver is a new subagent that runs after the architect and before the domain researcher, producing a TOOLS.md per swarm.

</domain>

<decisions>
## Implementation Decisions

### Tool type prioritization
- MCP servers preferred for external integrations (GitHub, Slack, databases, etc.)
- Built-in tools (web search, date, memory store) for generic capabilities only
- Every tool recommendation includes a rationale explaining why that tool type was chosen
- Primary recommendation + alternatives listed in an "Also possible" section when multiple tool types could work

### Tool resolver architecture
- Separate pipeline stage (new subagent), NOT embedded in the spec generator
- Runs after architect, before domain researcher — both researcher and spec generator benefit from knowing tools upfront
- Two-phase resolution: swarm-wide tool landscape first, then per-agent assignment/refinement
- Always consults both curated catalog AND web search (not either/or)
- Web search focuses on common sources like GitHub for finding real MCP servers
- Verified recommendations only — web search confirms server existence before recommending
- When no MCP server exists for a use case, recommend the next-best alternative (don't flag gaps)
- Output is a separate TOOLS.md file in the swarm output directory (not part of architect blueprint)
- Output is final — no interactive confirmation step. User edits TOOLS.md manually if needed

### Tool resolver output format
- Capability-first organization: describe what capabilities are needed, then map each to the closest Orq.ai tool type
- Map every recommendation to a specific Orq.ai tool type (built-in, function, HTTP, Python, MCP)

### MCP server catalog
- Curated reference covering 15-25 common MCP servers across major integration categories
- Unified tool catalog: built-in tools, MCP servers, and common HTTP/function patterns in one reference file
- Organized by integration category (Search, Code/DevOps, Communication, Database, File Storage, etc.)
- Each entry includes: server name + source URL, capabilities summary, use case triggers (keywords/scenarios), auth/setup complexity level

### Tool configuration depth
- Full Orq.ai-native config JSON where possible — directly pasteable into Orq.ai Studio
- MCP server config uses Orq.ai's native MCP connection format
- Secrets use `{{PLACEHOLDER}}` syntax (consistent with existing Orq.ai variable convention) with a setup instructions section explaining where to get each credential
- No difficulty ratings or estimated setup times — keep it clean

### Agent-tool matching
- Per-agent tool assignment driven by agent role from architect blueprint
- Shared tools section in TOOLS.md noting which tools serve multiple agents (configure once)
- Novel integrations get a function tool JSON schema scaffold as starting point

</decisions>

<specifics>
## Specific Ideas

- Tool resolver needs web search capability to verify MCP server existence on GitHub
- The curated catalog should follow the same word-limit convention as other reference files (~500-1000 words) to preserve subagent context window
- TOOLS.md structure: swarm-wide tool landscape → shared tools → per-agent tool assignments → setup instructions with config JSON

</specifics>
