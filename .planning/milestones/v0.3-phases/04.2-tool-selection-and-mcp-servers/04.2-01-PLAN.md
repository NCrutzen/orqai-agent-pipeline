  <action>
Create the tool resolver subagent prompt file following the established subagent prompt pattern from existing agents (see architect.md, researcher.md, spec-generator.md for conventions: YAML frontmatter, files_to_read block, role section, decision framework, output format, examples, anti-patterns).

**Frontmatter:**
```yaml
---
name: orq-tool-resolver
description: Resolves tool needs for each agent in a swarm by consulting a curated catalog and web search, producing a TOOLS.md with verified, copy-paste-ready Orq.ai tool configurations.
tools: Read, Glob, Grep, WebSearch, WebFetch
model: inherit
---
```

**files_to_read block:**
- `orq-agent/references/tool-catalog.md`
- `orq-agent/references/orqai-agent-fields.md`
- `orq-agent/templates/tools.md`

**Sections to include:**

1. **Role introduction** -- You are the Orq.ai Tool Resolver subagent. You receive an architect blueprint for a swarm and produce a TOOLS.md file with verified tool recommendations and copy-paste-ready Orq.ai configuration JSON.

2. **Critical Rules:**
   - MCP servers preferred for external integrations; built-in tools for generic capabilities only (per user decision)
   - Every tool recommendation includes a rationale explaining why that tool type was chosen
   - Primary recommendation + alternatives in an "Also possible" section when multiple tool types could work
   - Verified recommendations only -- web search confirms server existence before recommending
   - When no MCP server exists, recommend the next-best alternative (do NOT flag gaps)
   - Output is final -- no interactive confirmation step
   - Per-agent tool limit: aim for 3-5 tools per agent max, only tools that directly serve the agent's stated responsibilities

3. **Two-Phase Resolution Process:**
   - **Phase 1: Swarm-Wide Tool Landscape** -- Read the full architect blueprint. For each agent, extract responsibilities and identify capability needs. Deduplicate across agents (e.g., two agents needing Slack = one Slack tool). Consult the curated catalog (from files_to_read) for known tools. Use WebSearch to find MCP servers for capabilities not in the catalog (search GitHub for "[capability] mcp server"). Verify every MCP recommendation exists via web search (GitHub/npm).
   - **Phase 2: Per-Agent Assignment** -- Map each identified tool to specific agents based on their roles. Identify shared tools (used by 2+ agents). Generate Orq.ai-native config JSON for each tool. For function tools (novel integrations), generate a JSON Schema scaffold. For MCP tools, include both the tool creation config AND the agent reference config. Use `{{PLACEHOLDER}}` syntax for secrets with descriptions in setup instructions.

4. **Resolution Priority Chain** (reference the chain from tool-catalog.md):
   1. Built-in tool (if generic capability like web search, date, memory, knowledge base)
   2. MCP server from curated catalog
   3. MCP server found via web search (must verify on GitHub)
   4. HTTP tool (if API endpoint exists, provide endpoint template)
   5. Function tool (if custom business logic, provide JSON Schema scaffold)
   6. Code tool (if computation needed, recommend Python code tool)

5. **Web Search Protocol:**
   - For EVERY capability not covered by a built-in tool, search for MCP servers first
   - Search patterns: `"[capability] mcp server github"`, `"[service-name] mcp"`, `"modelcontextprotocol [service]"`
   - Use WebFetch on GitHub pages to confirm the server exists and is maintained
   - If web search finds no MCP server: fall back to HTTP/function/code tool
   - NEVER recommend an MCP server you cannot find via web search -- hallucinated server URLs are worse than no recommendation

6. **Output Format:**
   - Fill the TOOLS.md template (from files_to_read)
   - Write the filled template to the swarm output directory path provided by the orchestrator
   - Every config JSON must use Orq.ai's exact field names (key, description, type, path, mcp.server_url, mcp.connection_type for MCP; function.name, function.description, function.parameters for function tools)

7. **Anti-patterns:**
   - DO NOT recommend MCP servers without web search verification
   - DO NOT give agents more than 5-6 tools -- if you identify more, prioritize by role relevance
   - DO NOT duplicate tool schemas that will appear in agent specs -- TOOLS.md provides the landscape; spec generator handles per-agent detail
   - DO NOT include difficulty ratings or estimated setup times
   - DO NOT use SSE transport -- Orq.ai uses HTTP-based MCP connections (`connection_type: "http"`)

8. **Example output snippet** -- Include a concise example showing one capability mapped through the full resolution chain (e.g., "Send Slack messages" -> check built-in (no) -> check catalog (Slack MCP found) -> verify via web search -> produce config JSON). Keep this under 20 lines.

Target prompt length: comparable to existing subagents (spec-generator.md is ~300 lines). Do not exceed 350 lines. The prompt must be specific enough that a different Claude instance could execute it without clarifying questions.
