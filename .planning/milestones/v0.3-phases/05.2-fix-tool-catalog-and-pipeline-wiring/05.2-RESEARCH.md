## Common Pitfalls

### Pitfall 1: Partial Identifier Fix

**What goes wrong:** Fixing tool-catalog.md but missing the same wrong identifiers in tool-resolver.md (or vice versa).
**Why it happens:** The audit only flagged tool-catalog.md. tool-resolver.md was identified during user discussion as also containing wrong identifiers.
**How to avoid:** Fix BOTH files in the same task. Verify with a search for the old identifiers (`write_to_memory_store`, `delete_from_memory_store`) across all files in the orq-agent directory after the fix.
**Warning signs:** Grep for old identifiers returns hits after the fix.

### Pitfall 2: Adding Research Brief to files_to_read as Static Path

**What goes wrong:** Adding a literal file path like `research-brief.md` to orchestration-generator.md's `<files_to_read>` block, which would fail because the path is dynamic (different per swarm run).
**Why it happens:** The user said "update orchestration-generator.md files_to_read to include the research brief path for explicitness."
**How to avoid:** The research brief is a runtime input, not a static reference file. Add a comment/note in the input contract section of orchestration-generator.md that the research brief is expected as a runtime input path. Do NOT add it to the literal `<files_to_read>` YAML block.
**Warning signs:** The orchestration generator would try to read a non-existent file at spawn time.

### Pitfall 3: KB Validation vs Pre-Output Validation Confusion

**What goes wrong:** Adding the KB Design data validation check to the existing Pre-Output Validation checklist instead of as a separate dedicated section.
**Why it happens:** The existing Pre-Output Validation checklist already has KB-related items, so it seems like a natural fit.
**How to avoid:** User explicitly said "a separate dedicated validation section" BEFORE the KB Design generation step. This is an INPUT validation (check research brief data exists), not an OUTPUT validation (check generated output is well-formed).
**Warning signs:** The validation runs too late (after generation) instead of before, or is lost among other checklist items.

### Pitfall 4: README Generator Wiring Ambiguity

**What goes wrong:** Unclear whether the research brief should be added to readme-generator.md's files_to_read or only to the orchestrator's spawn instructions.
**Why it happens:** User left this as Claude's discretion.
**How to avoid:** Follow the dataset generator pattern -- pass the research brief as a runtime input in the orchestrator's spawn instructions (orq-agent.md). The README generator does not need the research brief in its static files_to_read (it is a runtime artifact). Add the research brief path to the orchestrator's README generator input list alongside the existing 4 items.
**Warning signs:** README generator tries to load a non-existent static file.
