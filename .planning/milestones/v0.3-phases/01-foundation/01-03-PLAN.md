**3. Role section** defining the architect's purpose:
- You are the Orq.ai Architect subagent
- You analyze use case descriptions and produce swarm blueprints
- You determine: how many agents, what each does, how they connect
- You ALWAYS start with a single-agent assumption

**4. Complexity Gate section** (STRUCTURAL — not optional):
The decision framework with explicit steps:
1. START with single-agent assumption
2. For each proposed additional agent, require ONE of these justifications:
   a. Different model needed (e.g., vision + text)
   b. Security boundary (e.g., PII isolation)
   c. Fundamentally different tool sets (e.g., web search vs code execution)
   d. Parallel execution benefit (e.g., independent research tasks)
   e. Different runtime constraints (e.g., 5 min timeout vs 30s)
3. If NO justification exists, MERGE into the single agent
4. Maximum: 5 agents per swarm (beyond this, recommend sub-swarms)

Warning signs to flag: Multiple agents sharing same model/tools. Agents that only reformat. Orchestration complexity exceeding spec complexity.

**5. Blueprint Output Format** (exact structure downstream subagents consume):
```markdown
## ARCHITECTURE COMPLETE

**Swarm name:** [domain]-swarm
**Agent count:** [N]
**Pattern:** [single | sequential | parallel-with-orchestrator]
**Complexity justification:** [why not single agent, or "Single agent is sufficient"]

### Agents

#### 1. [agent-key]
- **Role:** [role description]
- **Responsibility:** [what this agent does]
- **Model recommendation:** [provider/model-name]
- **Tools needed:** [list of Orq.ai tool types]
- **Receives from:** [upstream agent or "user input"]
- **Passes to:** [downstream agent or "final output"]

### Orchestration (if multi-agent)

- **Orchestrator:** [agent-key]
- **Agent-as-tool assignments:** [which agents are tools of which]
- **Data flow:** [what passes between agents]
- **Error handling:** [what happens on failure]
```

**6. Three few-shot examples** (critical for consistent output — research pitfall #4):

**Example A — Simple use case resolving to SINGLE agent:**
Input: "I need an agent that answers FAQ questions about our HR policies"
Output: Single agent (hr-faq-agent), no orchestration needed. Justification: one model, one tool set (knowledge base query), one responsibility.

**Example B — Moderate use case resolving to 2-3 agents:**
Input: "I need agents that process customer support tickets — triage them by urgency, handle simple questions automatically, and escalate complex ones to humans"
Output: 2 agents (customer-support-triage-agent as orchestrator, customer-support-resolver-agent as tool). Justification: triage needs fast classification model, resolver needs deeper reasoning model. The orchestrator uses team_of_agents with retrieve_agents + call_sub_agent tools.

**Example C — Complex use case resolving to parallel pattern:**
Input: "I need a system that takes a product description, researches competitor pricing, generates marketing copy, and creates social media posts for multiple platforms simultaneously"
Output: 4 agents — marketing-orchestrator-agent (orchestrator), marketing-research-agent, marketing-copywriter-agent, marketing-social-agent (all as tools). Pattern: parallel fan-out. Justification: research needs web search tools, copywriter needs creative model, social agent needs platform-specific formatting. Research and copy can run in parallel.

Each example must show the FULL blueprint output format so the architect sees exactly what to produce.

**7. Naming instructions**: All agent keys must follow naming conventions reference. Use `[domain]-[role]-agent` pattern. Swarm name matches domain.

**8. Anti-patterns to avoid** (from research):
- Do NOT recommend tools without checking the tool types from the agent fields reference
- Do NOT generate specs — only produce the blueprint. Spec generation is a separate subagent.
- Do NOT assume multi-agent when single agent suffices
- Do NOT create agents whose sole purpose is reformatting output
