1. **The Rule**: Agent keys use `[domain]-[role]-agent` kebab-case convention
2. **Regex pattern**: `^[A-Za-z][A-Za-z0-9]*([._-][A-Za-z0-9]+)*$` with plain-English explanation
3. **10+ valid examples** with annotations: invoice-validator-agent, hr-onboarding-agent, customer-support-triage-agent, data-extraction-agent, etc.

5. **Swarm directory naming**: Directory matches domain portion of agent key (e.g., agents in `customer-support-*` go in `customer-support/` directory)
6. **Version tagging**: `@version-number` suffix support (e.g., `invoice-validator-agent@2`)

Keep both files focused on what subagents need to PRODUCE output, not everything in the API. No response schemas.

2. **Provider format table**: Provider name, format pattern, example. All providers from research: OpenAI (openai/), Anthropic (anthropic/), Google AI (google-ai/), AWS Bedrock (aws/), Azure (azure/), Groq (groq/), DeepSeek (deepseek/), Mistral (mistral/), Cohere (cohere/), Cerebras (cerebras/), Perplexity (perplexity/), Together AI (togetherai/), Alibaba (alibaba/), Minimax (minimax/).

   - Reasoning/complex tasks: anthropic/claude-sonnet-4-5, openai/o3, google-ai/gemini-2.5-pro
   - Classification/extraction: openai/gpt-4o-mini, anthropic/claude-3-5-haiku-20241022, groq/llama-3.3-70b-versatile
   - Generation/creative: anthropic/claude-sonnet-4-5, openai/gpt-4o
   - Fast/cheap: groq/llama-3.3-70b-versatile, cerebras/llama-3.3-70b, deepseek/deepseek-chat
   - Vision: openai/gpt-4o, google-ai/gemini-2.5-flash, anthropic/claude-sonnet-4-5
4. **How to choose section**: Criteria for model selection (task complexity, speed requirements, cost sensitivity, multimodal needs)
5. **Fallback model strategy**: How to pick ordered fallback_models (same tier from different provider)
6. **Last verified date**: 2026-02-24. Note that model landscape changes monthly.

Create `orq-agent/references/orchestration-patterns.md` (~800-1200 words) containing:

1. **Three patterns** with diagrams (ASCII art from research):
   - **Single Agent**: User Input -> [agent] -> Output. When: one model, one tool set, one responsibility. No orchestration config needed.
   - **Sequential Pipeline**: User Input -> [agent-a] -> [agent-b] -> [agent-c] -> Output. When: distinct processing phases, different models/tools per phase. Orq.ai mechanism: Task ID continuation.
   - **Parallel Fan-Out with Orchestrator**: User Input -> [orchestrator] -> sub-agents as tools -> assembled output. When: independent subtasks, different specializations. Orq.ai mechanism: team_of_agents + retrieve_agents + call_sub_agent.
2. **Pattern selection criteria table**: Input characteristics that suggest each pattern
3. **Orq.ai configuration for each pattern**: What fields to set (team_of_agents, tool types, task states)
4. **Complexity gate reference**: Default to single agent. Multi-agent requires justification per additional agent. Five valid justifications from research: different model needed, security boundary, fundamentally different tool sets, parallel execution benefit, different runtime constraints.
5. **Warning signs of over-engineering**: Multiple agents sharing same model/tools, agents that only reformat, orchestration docs longer than specs
6. **Max agent count**: Recommend max 5 per swarm; beyond that, decompose into sub-swarms
