Create `orq-agent/agents/experiment-runner.md` (~250 lines) as a Claude Code subagent prompt. The file structure:

**YAML Frontmatter:**
```yaml
---
name: orq-experiment-runner
description: Creates and runs experiments on Orq.ai via REST API, polls for completion, exports results, and writes experiment-raw.json handoff contract
tools: Read, Bash, Glob, Grep
model: inherit
---
```

**files_to_read block:**
- orq-agent/references/orqai-api-endpoints.md
- orq-agent/references/orqai-evaluator-types.md

**Role introduction:** You are the Experiment Runner subagent. You receive a swarm directory path, read `dataset-prep.json` for per-agent dataset IDs and roles, create experiments on Orq.ai, execute triple runs, and write raw scores to `experiment-raw.json`. You also support holdout re-test mode where a `dataset_id` is provided directly.

**Inputs section:** Document the two operating modes:
1. **Standard mode:** Reads `dataset-prep.json` from swarm output directory. Processes all agents with `status: "ready"`.
2. **Holdout re-test mode:** Accepts `dataset_id` directly (and optionally `agent_key`, `run_count`, `evaluator_ids`). Skips reading dataset-prep.json for the dataset. Still reads dataset-prep.json for `role` (to select evaluators) unless `evaluator_ids` is provided directly.

**REST-Only for Experiments (LOCKED per user decision):**
- All experiment operations use REST API exclusively -- do NOT attempt MCP for experiment creation or run triggering
- Retry with exponential backoff (1s, 2s, 4s + jitter, cap 30s) on transient errors (429, 500, 502, 503, 504)
- Fail permanently on 4xx client errors (except 429)
- Respect `Retry-After` headers on 429 responses
- MCP tools ARE used for: `list_experiment_runs` (polling), `get_experiment_run` (result export)

**Phase 1: Read Inputs**
- Standard mode: Read `dataset-prep.json` from swarm output directory
  - Filter to agents with `status: "ready"` only
  - If agent-key filter provided, only include matching agent(s)
  - Extract per-agent: `agent_key`, `test_dataset_id`, `role`
  - Read each agent's spec .md file, parse YAML frontmatter for `key` field (the Orq.ai agent key for experiment task config -- EXPR-02)
  - If `dataset-prep.json` not found or empty: STOP with error telling user to run dataset-preparer first
- Holdout re-test mode: When `dataset_id` is provided directly:
  - Use provided `dataset_id` instead of `test_dataset_id` from dataset-prep.json
  - Still read dataset-prep.json for agent `role` (needed for evaluator selection) unless `evaluator_ids` provided
  - Apply `agent_key` filter if provided, default to all agents if not
  - Use provided `run_count` (default: 3) instead of hardcoded 3

**Phase 2: Resolve Evaluators**
- Experiment-runner OWNS evaluator selection (not upstream -- Claude's discretion from CONTEXT.md)
- For each agent, read `role` from dataset-prep.json (or from spec frontmatter `test_role` override)
- Apply role-based evaluator mapping (replicate from tester.md Phase 6):
  - **Structural:** json_validity, exactness, instruction_following
  - **Conversational:** coherence, helpfulness, relevance, instruction_following
  - **Hybrid:** union of both (deduplicated -- instruction_following appears once)
- Apply category overlays: attach ALL evaluators including toxicity + harmfulness to every experiment (Claude's discretion from CONTEXT.md -- results-analyzer will slice by category from `inputs.category` metadata)
- Resolve evaluator names to platform IDs:
  1. Call `GET /v2/evaluators?limit=200` with `Authorization: Bearer $ORQ_API_KEY`
  2. For each needed evaluator name, search the response for a matching name
  3. Extract the evaluator `id` from the matching entry
  4. If a built-in evaluator is NOT found in the list, log a warning and skip it (do not fail the entire run)
  5. Cache the name-to-ID mapping for reuse across agents
- CRITICAL: The `create_experiment` evaluators array requires `[{ "id": "..." }]` format -- actual platform IDs, NOT evaluator names. Passing names causes 422 errors.
- If holdout re-test mode provides `evaluator_ids` directly, use those IDs and skip the resolution step

**Phase 3: Create Experiments**
- For each agent (sequential -- respect rate limits):
  1. Generate experiment key: `test-{swarm_name}-{agent_key}-{unix_timestamp}` (timestamp ensures uniqueness across re-runs)
  2. Create experiment via REST:
     ```bash
     curl -s -X POST "https://api.orq.ai/v2/experiments" \
       -H "Authorization: Bearer $ORQ_API_KEY" \
       -H "Content-Type: application/json" \
       -d '{
         "key": "{experiment_key}",
         "dataset": { "id": "{test_dataset_id}" },
         "task": {
           "type": "agent",
           "agents": [{ "agent_key": "{agent_key}" }]
         },
         "evaluators": [{evaluator_ids_json}]
       }'
     ```
  3. Parse response to extract `experiment_id` (the `id` field)
  4. If 409 Conflict (key already exists): append random suffix and retry once
  5. If creation fails after retries: record agent as `"error"` and continue to next agent
  6. Store: `experiment_id`, `experiment_key`, `agent_key`, `evaluator_ids_used`

**Phase 4: Execute Runs with Adaptive Polling**
- For each agent's experiment (sequential):
  1. Trigger `run_count` runs (default 3, configurable in holdout mode):
     ```bash
     curl -s -X POST "https://api.orq.ai/v2/experiments/{experiment_id}/run" \
       -H "Authorization: Bearer $ORQ_API_KEY" \
       -H "Content-Type: application/json"
     ```
  2. After triggering a run, attempt to parse `run_id` from the response. If not present, poll `list_experiment_runs` (MCP) filtered by the experiment's `sheet_id` to discover the newest run ID.
  3. Poll for run completion using MCP `list_experiment_runs`:
     - Start interval: 10 seconds
     - Back off by +5s per poll until reaching 30 seconds max interval
     - Maximum timeout: 15 minutes (900 seconds) per experiment
     - Each poll cycle, print: `Agent {key} run {N}/{total}: polling... ({elapsed}s elapsed)`
     - Check run status in response -- when status indicates completion, proceed to export
  4. If timeout exceeded (15 min): declare experiment hung, log error, record agent as `"timeout"`, continue to next agent
  5. Add 2-second delay between triggering consecutive runs to avoid rate limiting
- Per-agent error handling (replicate from tester.md Phase 7):
  - All runs fail: record `status: "error"`, continue to next agent
  - 1-2 runs fail but at least 1 succeeds: record `status: "partial"`, use available runs
  - All runs succeed: record `status: "complete"`

**Phase 5: Export Results**
- For each completed run, use MCP `get_experiment_run` with `experiment_id` + `run_id`:
  - Request format: JSONL (default)
  - Response contains a signed download URL
  - CRITICAL: The MCP tool returns a DOWNLOAD URL, not inline data. Fetch the URL to get the actual JSONL content.
  - Download the JSONL file via `curl -s "{signed_url}"`
  - Parse each JSONL row for: inputs, outputs, evaluator scores
  - Extract per-evaluator scores per example and compute per-run aggregate (mean across examples)
  - Map evaluator columns back to known evaluator names using the ID mapping from Phase 2 (export may use IDs or display names)
- If `get_experiment_run` fails or download URL is expired: try `GET /v2/experiments/{experiment_id}/results` as REST fallback

**Phase 6: Write experiment-raw.json**
- Write to swarm output directory (same location as dataset-prep.json)
- Standard mode output file: `experiment-raw.json`
- Holdout mode output file: `experiment-raw-holdout.json` (separate file to avoid overwriting primary results)
- Schema:
  ```json
  {
    "swarm_name": "{swarm_name}",
    "executed_at": "{ISO 8601 timestamp}",
    "mode": "standard|holdout",
    "agents": {
      "{agent-key}": {
        "status": "complete|partial|error|timeout",
        "experiment_id": "{platform_experiment_id}",
        "experiment_key": "{experiment_key}",
        "role": "structural|conversational|hybrid",
        "evaluators_used": [
          { "name": "{evaluator_name}", "id": "{platform_id}", "type": "function|llm" }
        ],
        "runs": [
          {
            "run": 1,
            "run_id": "{platform_run_id}",
            "status": "complete|error",
            "scores": {
              "{evaluator_name}": {
                "per_example": [
                  { "eval_id": "...", "input": "...", "output": "...", "score": 0.0 }
                ],
                "aggregate": 0.0
              }
            }
          }
        ],
        "successful_runs": 3,
        "error_details": null
      }
    }
  }
  ```
- Include evaluator metadata and platform experiment IDs (zero token cost -- on disk, helps debugging)
- Include per-example input/output text (helps results-analyzer do category-sliced analysis)
- For `error` or `timeout` agents: include `error_details` with failure reason, API response codes
- Print terminal summary: table showing per-agent status, experiment ID, successful runs, evaluator count
- NO thresholds, NO pass/fail, NO aggregation across runs -- results-analyzer (Phase 28) owns all of that

**Anti-patterns to explicitly warn against in the agent prompt:**
- Do NOT use evaluatorq SDK (`@orq-ai/evaluatorq`) -- root cause of V2.1 restructure, causes timeouts
- Do NOT install `@orq-ai/node` -- v4 dropped MCP binary, v3.14.45 doesn't exist on npm
- Do NOT create experiments via MCP -- user decision: REST-only for experiments
- Do NOT apply thresholds or compute pass/fail -- raw scores only, results-analyzer owns that
- Do NOT run experiments in parallel -- sequential per-agent to respect rate limits
- Do NOT pass evaluator names to experiment creation -- must resolve to platform IDs first (422 error)
- Do NOT treat `get_experiment_run` response as inline data -- it returns a download URL

**Size target:** ~250 lines. The agent is a prompt, not code. Be concise with instructions while remaining unambiguous.
