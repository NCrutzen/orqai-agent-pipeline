---
name: orq-dataset-preparer

**files_to_read block:**
- orq-agent/references/orqai-api-endpoints.md
- orq-agent/templates/dataset.md

**Role introduction:** You are the Dataset Preparer subagent. You receive a swarm directory path and an optional agent-key filter. You prepare and upload correctly formatted datasets to Orq.ai and produce a dataset-prep.json handoff contract.

**MCP-First / REST-Fallback Pattern section** (LOCKED -- inherited from deployer):
- Copy the retry/backoff pattern from deployer.md exactly (per-operation, 3 retries, exponential backoff 1s/2s/4s + jitter, cap 30s, fail on 4xx except 429)
- Key difference from tester.md: dataset creation uses MCP `create_dataset` first with REST fallback. Row upload uses REST `POST /v2/datasets/{id}/rows` PREFERRED (because MCP `create_datapoints` schema lacks the `messages` top-level field)
- Do NOT record which channel (MCP vs REST) was used -- just make it work

**Phase 1: Pre-check Deployment**
- Read ORCHESTRATION.md to discover agents
- If agent-key filter provided, only include matching agent(s)
- Read each agent spec .md file, parse YAML frontmatter between `---` delimiters
- Verify each agent has `orqai_id` field -- if ANY agent lacks it, STOP with clear error message telling user to run /orq-agent:deploy first
- Collect: agent_key, orqai_id, key (from frontmatter), spec_content, dataset_paths

**Phase 2: Parse V1.0 Datasets**
- For each agent, find `{agent-key}-dataset.md` and `{agent-key}-edge-dataset.md` in `datasets/` directory
- If no dataset files found for an agent, record status as `skipped` with reason "No dataset files found"
- Parse markdown tables: find "## Eval Pairs" heading (clean) or "## Adversarial Test Cases" (edge), read lines until next `##` or EOF
- Split each row by `|` delimiter, trim whitespace, skip header and separator lines (containing `---`)
- Extract: ID, Input, Expected Output, Pass Criteria (clean) or ID, Input, Attack Vector, Expected Behavior (edge)
- Tag each example with category: clean dataset rows get `happy-path`, edge dataset rows get categories from their Attack Vector or default to `adversarial`
- Tag each example with source: `original`

**Phase 3: Augment to 30+ Examples**
- If an agent already has 30+ examples after parsing, skip augmentation
- Apply exactly 4 augmentation techniques (replicate from tester.md):
  1. **Parameter swaps** — Change specific values (names, numbers, dates) while preserving structure
  2. **Complexity variations** — Add/remove optional fields, change nesting depth
  3. **Format variations** — Reword the same request differently (formal/informal, verbose/terse)
  4. **Rephrasings** — Same meaning expressed with different vocabulary and sentence structure
- CRITICAL: Each augmented example MUST have an adapted expected output that reflects the specific input changes. Do NOT copy expected outputs verbatim from the source example.
- Tag augmented examples with source: `augmented` and category matching the source example's category
- Continue augmenting until total reaches at least 30 examples per agent

**Phase 4: Merge and Split 60/20/20 Stratified**
- Merge clean + edge + augmented examples into a single pool per agent
- Group examples by category (happy-path, variation, boundary, adversarial, edge-case, stress)
- For each category: shuffle randomly, assign 60% train / 20% test / 20% holdout
- Round up for test/holdout on odd counts (prefer more test/holdout over train)
- Validate: each split has proportional category distribution

**Phase 5: Smoke Test**
- Before uploading full datasets, verify the row format works:
  1. Create a smoke-test dataset via MCP `create_dataset` (display_name: `smoke-{swarm_name}-{agent_key}`, path: `/`)
  2. Upload 1 representative row via REST `POST /v2/datasets/{dataset_id}/rows` with the full field format:
     ```json
     {
       "inputs": { "text": "...", "category": "...", "source": "original", "eval_id": "..." },
       "messages": [{ "role": "user", "content": "..." }],
       "expected_output": "..."
     }
     ```
  3. Create experiment via MCP `create_experiment`:
     - key: `smoke-{swarm_name}-{agent_key}`
     - dataset.id: smoke_dataset_id
     - task.type: `agent`
     - agents: [{ agent_key: first_agent_key }]
     - auto_run: true
  4. Poll for results with adaptive interval (start 10s, back off to 30s)
  5. Verify evaluator scores are non-null
  6. Delete smoke dataset via MCP `delete_dataset`
  7. If scores are null → ABORT with clear error about row format: "Smoke test failed: evaluator scores are null. The row format may not be compatible with the experiment engine."
- Use the first agent that passes pre-check (has orqai_id). If agent-key filter is active, use that specific agent.

**Phase 6: Upload Datasets**
- For each agent (sequential -- respect rate limits):
  1. Create 3 datasets via MCP `create_dataset` with REST fallback:
     - `test-{SWARM_NAME}-{AGENT_KEY}-train`
     - `test-{SWARM_NAME}-{AGENT_KEY}-test`
     - `test-{SWARM_NAME}-{AGENT_KEY}-holdout`
  2. Upload rows to each dataset via REST `POST /v2/datasets/{id}/rows` (preferred) with MCP `create_datapoints` fallback
     - Each row MUST include: `messages: [{role: "user", content: input_text}]` as top-level field
     - Each row MUST include: `expected_output` as top-level field
     - Each row MUST include: `inputs` with `text`, `category`, `source`, `eval_id` fields
     - If REST endpoint accepts batch (array), use it. If 422, fall back to individual row upload.
     - MCP `create_datapoints` maxItems is 100 per call -- batch accordingly
  3. Record dataset IDs (train_dataset_id, test_dataset_id, holdout_dataset_id)

**Phase 7: Infer Agent Roles**
- For each agent, analyze spec_content for role signals:
  - **Structural signals:** JSON output, schema, structured data, extraction, formatting, parsing, json_schema tool type
  - **Conversational signals:** conversation, chat, support, explanation, advice, natural language
  - **Hybrid:** both structural AND conversational signals present
  - **Override:** `test_role` field in YAML frontmatter takes precedence over heuristic
  - **Default:** `hybrid` (safest -- gets union of evaluators)

**Phase 8: Write dataset-prep.json**
- Write to swarm output directory (alongside where test-results.json would go)
- Schema:
  ```json
  {
    "swarm_name": "{swarm_name}",
    "prepared_at": "{ISO 8601 timestamp}",
    "agents": {
      "{agent-key}": {
        "status": "ready|skipped|error",
        "role": "structural|conversational|hybrid",
        "test_dataset_id": "{ulid}",
        "train_dataset_id": "{ulid}",
        "holdout_dataset_id": "{ulid}",
        "example_counts": {
          "original": N,
          "augmented": N,
          "total": N,
          "per_split": { "train": N, "test": N, "holdout": N }
        }
      }
    }
  }
  ```
- For `skipped` agents: include `"reason": "No dataset files found"`, role is null, no dataset IDs
- For `error` agents: include `"reason": "{error details}"`, role is null, no dataset IDs
- Print terminal summary: table showing per-agent status, role, example counts, dataset IDs

**Anti-patterns to explicitly warn against in the agent prompt:**
- Do NOT install `@orq-ai/node@latest` (v4 dropped MCP binary) or pin `^3.14.45` (doesn't exist on npm)
- Do NOT upload datasets in parallel (sequential to respect rate limits)
- Do NOT put `messages` inside `inputs` -- experiment engine reads it as a top-level field
- Do NOT skip the smoke test -- null evaluator scores from missing `messages` are silent failures
- Do NOT copy expected outputs verbatim for augmented examples -- each needs adapted output

**Size target:** ~250 lines. The agent is a prompt, not code. Be concise with instructions while remaining unambiguous.
