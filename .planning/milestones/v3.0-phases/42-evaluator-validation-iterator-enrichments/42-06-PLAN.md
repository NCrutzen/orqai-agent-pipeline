`# Orq.ai Evaluator Validator`

Opening paragraph stating role, inputs (evaluator spec, agent swarm dir, criterion name), outputs (`{swarm_dir}/evaluator-validations/{name}.json` + Annotation Queue on Orq.ai).

### 9 SKST sections (in this order before/around phases):

1. **Constraints** (NEVER/ALWAYS + "Why these constraints:"):
   - NEVER default new LLM-as-judge evaluators to continuous scales — default is binary Pass/Fail; continuous requires explicit justification (EVLD-01).
   - NEVER bundle multiple failure modes into one evaluator — one evaluator per failure mode, split bundled criteria automatically (EVLD-02).
   - NEVER leak dev/test labels into few-shot examples in the judge prompt (EVLD-05).
   - ALWAYS collect 100+ human labels per criterion, balanced ~50 Pass / ~50 Fail (EVLD-04).
   - ALWAYS measure TPR and TNR on held-out test set with ≥30 Pass / ≥30 Fail before marking evaluator validated (EVLD-06).
   - ALWAYS compute inter-annotator agreement when ≥2 humans label same item; flag IAA < 85% for re-calibration (EVLD-10).

2. **When to use** — trigger phrases: "validate my evaluator", "/orq-agent:iterate flags evaluator-quality issue", failure-diagnoser emits evaluator-quality action plan.

3. **When NOT to use** — anti-triggers: pre-built guardrails (`orq_*`); judge prompt not yet drafted → use prompt-optimization; <100 labels available → return to Annotation Queue first.

4. **Companion Skills** — directional handoffs:
   - ← `failure-diagnoser` — receives evaluator-quality issue from iteration-proposals.json
   - ← `/orq-agent:iterate` — invoked for evaluator-version A/B
   - → `hardener` — emits evaluator-validations/{name}.json consumed by Phase 2.0 TPR/TNR gate
   - ↔ `results-analyzer` — consumes IAA when ≥2 humans labeled same datapoint

5. **Done When** — falsifiable checklist:
   - [ ] Annotation Queue / Human Review entity created on Orq.ai (EVLD-09)
   - [ ] 100+ balanced labels collected (≥50 Pass, ≥50 Fail) (EVLD-04)
   - [ ] Train/dev/test split disjoint, 10-20% / 40-45% / 40-45% ratio (EVLD-05)
   - [ ] TPR and TNR measured on held-out test (≥30 Pass / ≥30 Fail) (EVLD-06)
   - [ ] Inter-annotator agreement computed + flagged if < 85% (EVLD-10)
   - [ ] Judge prompt uses 4-component template (role, task, criterion + pass/fail defs, examples, CoT-before-answer JSON) (EVLD-03)
   - [ ] One evaluator per failure mode — bundled criteria split (EVLD-02)
   - [ ] Default scale is binary Pass/Fail unless explicit justification provided (EVLD-01)
   - [ ] evaluator-validations/{name}.json written to swarm dir

6. **Destructive Actions** — AskUserQuestion confirm required before:
   - Creating Annotation Queue on Orq.ai (modifies workspace)
   - Marking evaluator as validated (affects downstream hardener promotion gate)

7. **Anti-Patterns** table — ≥5 rows including:
   - Using continuous scale without justification → default binary Pass/Fail
   - Bundling multiple criteria into one judge → split one-per-failure-mode
   - Reusing test labels as few-shot → enforce disjoint split
   - Skipping IAA when multiple humans labeled → always compute when N≥2
   - Marking validated at 89% TPR → hard gate at 90% for both TPR and TNR

8. **Open in orq.ai** — deep links:
   - **Annotation Queues:** https://my.orq.ai/annotation-queues
   - **Evaluators:** https://my.orq.ai/evaluators
   - **Human Reviews:** https://my.orq.ai/human-reviews

9. **Documentation & Resolution** — standard 4-tier trust order (same as existing agents).

### 7 Phases:

- **Phase 1: Intake** — read evaluator criterion + target agent context; detect bundled criteria and split into N one-per-failure-mode evaluators (EVLD-02).
- **Phase 2: Judge Prompt Generation** — emit 4-component template with `<role>`, `<task>`, `<criterion>` (with explicit Pass/Fail definitions), `<examples>` (from train split only, NEVER dev/test), `<output>` JSON schema `{"reasoning": "...", "verdict": "pass"|"fail"}` — CoT before answer (EVLD-03). Default scale binary (EVLD-01); continuous requires user justification via AskUserQuestion.
- **Phase 3: Create Annotation Queue via MCP/REST (EVLD-09)** — MCP-first (`annotation-queues-create`), REST fallback (`POST /v2/annotation-queues`). Fields: name, description, categorical Pass/Fail + optional sentiment/numeric/free-text. Emit queue URL to user.
- **Phase 4: Guide 100+ Label Collection (EVLD-04)** — show user the queue URL, target counts (≥50 Pass, ≥50 Fail), poll `annotations-list --queue {id}` until balanced threshold met. Emit progress updates.
- **Phase 5: Split Labels (EVLD-05)** — disjoint partition into train (10-20%) / dev (40-45%) / test (40-45%). Shuffle with fixed seed for reproducibility. Enforce no overlap.
- **Phase 6: Measure TPR/TNR (EVLD-06)** — run judge prompt on test split. TPR = TP/(TP+FN), TNR = TN/(TN+FP). Require ≥30 Pass + ≥30 Fail in test set. Reference `resources/tpr-tnr-methodology.md`.
- **Phase 7: Compute IAA + Emit Validation JSON (EVLD-10)** — if multiple annotators labeled same item, compute Cohen's kappa or simple % agreement. Flag < 85% for re-calibration. Write `{swarm_dir}/evaluator-validations/{name}.json` with schema documented in plan context.
