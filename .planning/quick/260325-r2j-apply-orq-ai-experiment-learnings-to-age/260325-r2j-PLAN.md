The 6 experiment learnings to apply:

1. **Model Response Format** -- Always use `json_schema` with `strict: true`, never `json_object`. The json_object format causes hallucinated field names and missing required fields. json_schema with strict mode enforces exact schema compliance.

2. **A/B Testing** -- Use the `@orq-ai/evaluatorq` framework for A/B experiments, NOT the MCP experiment API. The MCP experiment tools have unreliable schemas and cause silent failures. NOTE: experiment-runner.md already moved to REST API (even further from evaluatorq) -- verify tester.md references are consistent and add learnings to anti-patterns where needed.

3. **Two-Evaluator Pattern (Mandatory)** -- Every experiment MUST use BOTH a code-evaluator (function type: json_validity, exactness, regex_match, etc.) AND an LLM-evaluator (coherence, relevance, helpfulness, etc.) together. Using only one type gives incomplete signal -- code catches structural issues, LLM catches semantic quality.

4. **LLMs Can't Do Math** -- Use the portionOptimizer pattern: LLM for selection/reasoning, code tool for calculations. Never rely on LLM to compute percentages, totals, or numeric transformations. Always pair with a code tool or function tool for math.

5. **KB Tools on Agents Are Unreliable** -- When an agent has a detailed, comprehensive system prompt, remove KB tools (retrieve_knowledge_bases, query_knowledge_base) and bake the knowledge directly into the system prompt. KB retrieval adds latency, costs tokens, and returns inconsistent chunks. Only use KB tools when the knowledge corpus is too large to fit in the prompt (>5000 words of reference material) or changes frequently.

6. **API Endpoints** -- Correct endpoint usage documentation. Specifically:
   - Agent execution: `POST /v2/agents/{agent_id}/execute` (not `/invoke`)
   - Experiment creation requires `evaluators: [{ "id": "..." }]` format (platform IDs, not names)
   - Dataset row upload requires `messages` as top-level field (not nested in `inputs`)
