Variant A — Modal-driven

Click Replay → modal opens with intent dropdown + optional note + Cancel/Confirm. Reclassify-as-noise opens a similar modal with the 4-noise-key dropdown. Close opens a minimal confirm modal. Standard pattern.

Clear separation of "view row" vs "act on row" — modal is unambiguous about destination state. Each modal can carry contextual notes / form fields without crowding the main pane. Familiar pattern to operators (consistent with most SaaS dashboards).
Heavyweight for what's mostly a one-tap operation. Operator who wants same-intent replay (no edit) still pays the modal-open + confirm tax. ≥3 clicks to handle one row vs the inline patterns. Backdrop blocks reading the email body or list while deciding — bad if operator needs to re-read the body to choose the intent. Modal stacking gets weird for keyboard navigation across rows.

★ Variant B — Inline-expand within pane

Click Replay → the action button area expands into an intent editor inline (replaces just the Replay button; other actions dim). One Enter to confirm, Esc to cancel. Reclassify-as-noise expands the same way with the 4-noise-key dropdown. Close fires immediately (operator can manually close without confirm — Cmd-Z within 5s reverses if needed; out of scope for v1). No modals. Same-intent Replay is effectively two clicks (Replay → ⏎). Different-intent Replay is three (Replay → pick → ⏎).

Email body + ranked output stay visible while operator picks the intent — no blocked context. Two-click happy path for same-intent replay. Inline Replay editor matches the Stage 1 widget pattern from Bulk Review (CONTEXT.md §Specific Ideas reuses that component shape). Pane never loses scroll position. Keyboard-driven: ⏎ confirms, Esc reverts. Reclassify uses identical UX with a different field — operator learns one pattern.
Pane height grows when the editor expands — small viewports might need to scroll. Optional "note" field for audit context has nowhere obvious to go (could expand below the dropdown — slight chrome cost). The dimmed action buttons during editing are slightly awkward visually; could be hidden entirely instead. Close-without-confirm is a minor risk if operator misclicks (mitigation: brief undo toast).

Variant C — Dropdown-on-button (split-button)

The Replay button is a "split-button": clicking the main label confirms with the current Stage 3 pick (one tap → done). Clicking the chevron (or the dropdown menu shown here) reveals the intent picker for editing. Same shape for Reclassify (split-button with default noise key visible inline). Close fires immediately.

Most compact happy path: same-intent replay = ONE tap (no expansion, no modal). Different-intent replay = open dropdown + pick (still two interactions but feels integrated, not modal-flow). The current intent is visible on the button itself, so operator confirms by reading the button label.
Split-button discoverability — operators who don't know about the chevron will never find the edit path. Note field has nowhere to go (would need a secondary expand). On smaller pane widths the button label gets crowded. Reclassify shows a noise key on the main button which might confuse operators ("did I already pick auto_reply?"). The pattern works well for power users but adds a learning curve.
